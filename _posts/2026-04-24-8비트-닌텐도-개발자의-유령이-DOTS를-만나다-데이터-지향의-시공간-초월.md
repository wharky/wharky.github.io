### **문제 제기: 자원의 한계, 그리고 풀리지 않는 숙제**

게임을 개발한다는 것은 끊임없이 자원(CPU, Memory, GPU, 개발 시간)의 한계와 싸우는 일입니다. 레트로 닌텐도 게임 개발자들은 킬로바이트 단위의 RAM과 MHz 단위의 CPU라는, 지금 상상조차 하기 힘든 극악의 제약 속에서 경이로운 결과물을 만들어냈습니다. 그들은 어떻게 그 모든 것을 해냈을까요?

반면, 현대 게임 개발은 어떨까요? 우리는 테라바이트급 스토리지, 기가바이트급 RAM, GHz 단위의 멀티코어 CPU, 그리고 괴물 같은 GPU를 가지고 있습니다. 하지만 여전히 "게임이 버벅인다", "프레임 드롭이 심하다", "복잡한 시뮬레이션이 어렵다" 같은 문제에 직면합니다. 하드웨어 스펙은 무한히 늘어났는데, 왜 우리는 여전히 자원 한계에 부딪히는 걸까요?

저는 이 두 시대의 개발자들이 공통적으로 씨름했던 '문제의 본질'이 있다고 봅니다. 그리고 그 해답은 의외로 '데이터'에 있었습니다.

### **1부: 8비트 시대의 강제된 데이터 지향 - 본능적인 최적화**

저는 감히 말합니다. 레트로 게임 개발자들은 비록 'Data-Oriented Design(DOD)'이라는 용어를 몰랐을지언정, 그들 스스로가 *가장 순도 높은 데이터 지향적 개발자*였습니다. 그들은 선택의 여지 없이, 컴퓨터 아키텍처의 근본 원리(예: 캐시 효율성, 데이터 인접성)를 본능적으로 파고들 수밖에 없었죠.

**어떤 문제가 있었고, 어떻게 해결했을까요?**

*   **문제 1: 극악의 메모리 제약 (수 KB 단위!)**
    *   **해결책: 타일 기반 렌더링 및 데이터 구조.**
        *   레트로 게임의 배경은 대부분 '타일'로 구성됩니다. 화면 전체를 비트맵으로 그리는 대신, 작고 재활용 가능한 타일 이미지들의 인덱스(데이터)로 맵을 구성했습니다. `맵[y][x]`에는 '어떤 타일을 그릴지'에 대한 *데이터*만 저장하고, 실제 타일 이미지는 ROM의 특정 영역에 미리 로드해두는 식이죠. 심지어 충돌 판정 정보까지 이 타일 데이터에 포함시켜, 메모리 중복을 최소화했습니다. 이는 현대 DOTS에서 'Shared Component'나 'Singleton Component'로 참조하는 '순수 데이터 블록'과 개념적으로 매우 흡사합니다.

        ```
        // 의사 코드: 레트로 게임의 맵 데이터 구조
        struct Tile {
            byte TileGraphicIndex;  // 어떤 그래픽 타일을 그릴지
            bool IsSolid;           // 충돌 가능한 타일인가?
            byte ItemOnTileID;      // 타일 위에 아이템이 있다면?
        };

        Tile TileMap[MAP_HEIGHT][MAP_WIDTH]; // 실제 게임 맵은 이 Tile 객체 배열
        ```

*   **문제 2: 느린 CPU (수 MHz 단위!)**
    *   **해결책: 오브젝트 재활용(Object Pooling) 및 단순한 상태 머신.**
        *   화면에 동시에 존재할 수 있는 스프라이트(움직이는 객체)의 수는 극히 제한적이었습니다. 개발자들은 적이나 총알 같은 오브젝트를 미리 배열로 만들어 놓고, 필요할 때 활성화하고 비활성화하는 '오브젝트 풀링' 기법을 본능적으로 사용했습니다. 새로운 객체를 생성하고 소멸하는 오버헤드를 줄이는 것이죠.
        *   또한, 적 캐릭터의 AI나 게임 오브젝트의 동작은 대부분 매우 단순한 '상태 머신(State Machine)'으로 구현되었습니다. `enum`으로 상태를 정의하고 `switch` 문으로 데이터를 조작하는 식이죠. 이 모든 것은 `캐릭터_데이터_배열[i].x`, `캐릭터_데이터_배열[i].y`, `캐릭터_데이터_배열[i].state` 와 같이, 데이터 덩어리를 효율적으로 순회하며 처리하는 방식이었습니다.

        ```
        // 의사 코드: 레트로 게임의 적 캐릭터 데이터와 업데이트 로직
        struct Enemy {
            short x, y;
            byte state; // 0: Idle, 1: Patrol, 2: Attack, 3: Dying
            byte type;  // Enemy Type ID
            byte hp;
            // ... 기타 데이터
        };

        Enemy Enemies[MAX_ENEMIES]; // 적 캐릭터 데이터 배열

        void UpdateEnemies() {
            for (int i = 0; i < MAX_ENEMIES; i++) {
                if (Enemies[i].hp > 0) { // 활성화된 적만 처리
                    switch (Enemies[i].state) {
                        case 0: // Idle
                            // ...
                            break;
                        case 1: // Patrol
                            Enemies[i].x += EnemyData[Enemies[i].type].Speed;
                            break;
                        // ...
                    }
                    // 충돌 검사 등 다른 로직
                }
            }
        }
        ```
        보세요. 객체지향의 상속이나 가상 함수 호출 같은 복잡한 오버헤드 없이, 순수 데이터 덩어리를 반복문으로 처리하는 방식. 이 얼마나 DOTS스러운가요!

### **2부: 현대의 근육, DOTS - 명시적인 효율성의 재림**

이제 시공간을 넘어 현대로 와봅시다. Unity의 DOTS는 바로 이러한 '데이터 지향'의 철학을 현대적인 아키텍처와 도구로 명시적으로 구현한 것입니다. Unity는 과거 OOP(Object-Oriented Programming) 기반의 GameObject 중심 설계에서 발생하던 성능 병목 현상, 특히 CPU와 메모리 캐시 비효율성 문제를 해결하기 위해 DOTS를 내놓았습니다.

**어떤 문제가 있었고, DOTS로 어떻게 해결했을까요?**

*   **문제 1: OOP의 한계와 CPU 캐시 미스.**
    *   OOP는 추상화와 재사용성에는 강하지만, 메모리상에서 데이터가 파편화되기 쉽습니다. `GameObject`와 그에 붙는 `MonoBehaviour` 컴포넌트들은 힙(Heap) 메모리의 여기저기에 흩어져 있어, CPU가 데이터를 처리할 때마다 캐시에 올리고 내리는 비효율이 발생합니다 (캐시 미스). 이는 멀티코어 CPU 시대에 더욱 치명적입니다.
    *   **해결책: Entity Component System (ECS)과 데이터 인접성.**
        *   DOTS의 핵심인 ECS는 Entities(개념적인 ID), Components(순수 데이터), Systems(데이터 처리 로직)로 구성됩니다. Component는 객체의 "속성 데이터" 그 자체이며, 메모리상에 연속적으로(contiguous) 배치됩니다. 특정 시스템이 `Translation`과 `Velocity` Component를 가진 모든 Entity를 처리할 때, 이 데이터들은 메모리에서 붙어있기 때문에 CPU 캐시 효율이 극대화됩니다. 이는 8비트 시절 개발자들이 배열에 모든 데이터를 때려 박아 효율을 높였던 것과 정확히 같은 원리입니다.

*   **문제 2: 멀티코어 CPU 활용의 어려움.**
    *   단일 스레드 기반의 `MonoBehaviour` 업데이트 루틴은 멀티코어 CPU의 잠재력을 완전히 활용하기 어렵습니다. 복잡한 로직을 병렬화하는 것은 매우 까다로운 작업입니다.
    *   **해결책: C# Job System과 Burst Compiler.**
        *   C# Job System은 스레드 세이프한 방식으로 작업을 작은 단위로 쪼개어 여러 코어에서 병렬로 실행할 수 있게 합니다. 여기에 Burst Compiler는 C# 코드를 네이티브에 가까운 고성능 기계어로 컴파일하여, 극한의 성능을 뽑아냅니다. 레트로 개발자들이 어셈블리어로 한땀 한땀 최적화했던 작업을, DOTS는 현대적인 도구와 프레임워크로 가능하게 하는 셈입니다. 마치 8비트 콘솔의 한정된 병렬 처리 장치(스프라이트 컨트롤러 등)를 최대한 활용했던 것과 비슷한 맥락이죠.

### **3부: 퓨전! 레트로 최적화 기법을 DOTS로 재해석하기**

이제 이 두 개념을 실제 예시로 엮어봅시다. 레트로 게임에서 수많은 총알이나 적 캐릭터를 효과적으로 관리하는 기법을 DOTS로 어떻게 재해석할 수 있을까요?

**상황**: 화면에 수백 발의 총알이 날아다니고, 각 총알은 매 프레임 위치를 업데이트하고 화면 밖으로 나가면 비활성화되어야 합니다.

**레트로 방식 (개념적)**:
`Bullet`이라는 `struct` 배열을 만들어 놓고, `IsActive` 플래그로 활성화/비활성화를 제어합니다. `UpdateBullets` 함수에서 이 배열을 순회하며 `x, y`를 업데이트하고, `IsActive` 상태를 변경합니다.

```csharp
// 레트로적 사고를 C#으로 표현 (DOTS 이전)
public struct BulletData
{
    public float X, Y;
    public float VelocityX, VelocityY;
    public bool IsActive;
}

public class BulletManager : MonoBehaviour
{
    private BulletData[] bullets;
    private const int MAX_BULLETS = 500;

    void Start() {
        bullets = new BulletData[MAX_BULLETS];
        // ... 초기화
    }

    void Update() {
        for (int i = 0; i < MAX_BULLETS; i++) {
            if (bullets[i].IsActive) {
                bullets[i].X += bullets[i].VelocityX * Time.deltaTime;
                bullets[i].Y += bullets[i].VelocityY * Time.deltaTime;

                if (bullets[i].X < 0 || bullets[i].X > Screen.width || ...) {
                    bullets[i].IsActive = false; // 화면 밖으로 나가면 비활성화
                }
            }
        }
    }
}
```
이 코드는 사실 이미 꽤나 데이터 지향적입니다. `BulletData`는 순수한 데이터 덩어리이고, `BulletManager`는 이 데이터 배열을 순회하며 로직을 처리합니다. 문제는 `Time.deltaTime` 같은 `MonoBehaviour` 의존성과, 단일 스레드에서 `MAX_BULLETS`가 커질수록 발생하는 병목이죠.

**DOTS 방식 (실무적)**:
이제 이 레트로적 사고를 DOTS 프레임워크에 넣어봅시다. `BulletData`는 `IComponentData`로, `BulletManager`의 로직은 `SystemBase`나 `IJobEntityBatch` 기반의 `System`으로 바뀝니다.

1.  **Component 정의**: 총알의 데이터만 담는 `struct`를 정의합니다.

    ```csharp
    // Bullet.cs
    using Unity.Entities;
    using Unity.Mathematics; // float2 사용

    public struct BulletTranslation : IComponentData
    {
        public float2 Value;
    }

    public struct BulletVelocity : IComponentData
    {
        public float2 Value;
    }

    public struct BulletLifetime : IComponentData
    {
        public float Value; // 남은 수명 (초)
    }

    // IsActive 대신 Entity 자체의 존재 유무로 관리
    ```

2.  **System 정의**: 총알 데이터를 처리하는 시스템을 만듭니다. `IJobEntityBatch`를 사용하여 병렬 처리를 극대화합니다.

    ```csharp
    // BulletMovementSystem.cs
    using Unity.Burst;
    using Unity.Collections;
    using Unity.Entities;
    using Unity.Jobs;
    using Unity.Mathematics;

    [BurstCompile] // 성능 최적화
    public partial struct BulletMovementSystem : ISystem
    {
        public void OnCreate(ref SystemState state) { }
        public void OnDestroy(ref SystemState state) { }

        [BurstCompile]
        public void OnUpdate(ref SystemState state)
        {
            float deltaTime = SystemAPI.Time.DeltaTime; // DeltaTime 가져오기

            // 모든 BulletTranslation과 BulletVelocity 컴포넌트를 가진 엔티티에 대해 Job 실행
            new MoveBulletsJob
            {
                DeltaTime = deltaTime
            }.ScheduleParallel(); // 병렬로 Job 스케줄링
        }
    }

    [BurstCompile]
    public partial struct MoveBulletsJob : IJobEntity
    {
        public float DeltaTime;

        // IJobEntity는 특정 ComponentData를 가진 모든 Entity를 반복합니다.
        // [Ref]는 쓰기 가능한 Component, [ReadOnly]는 읽기 전용입니다.
        void Execute(ref BulletTranslation translation, in BulletVelocity velocity, ref BulletLifetime lifetime)
        {
            translation.Value += velocity.Value * DeltaTime;
            lifetime.Value -= DeltaTime; // 수명 감소

            // 화면 밖으로 나가는 로직이나 수명 초과 로직은
            // 다른 시스템이나 여기서 EntityCommandBuffer를 사용하여 Entity를 제거
            // 예: if (lifetime.Value <= 0) ecb.DestroyEntity(entity);
        }
    }
    ```
    (실제 DOTS에서는 `EntityCommandBuffer`를 사용하여 엔티티 생성/삭제를 시스템의 메인 스레드 밖에서 처리하거나, 화면 밖 검출 로직은 다른 `IJobEntity`를 활용하는 것이 일반적입니다.)

**이것이 왜 레트로 정신의 현대적 계승일까요?**

*   **순수 데이터:** `BulletTranslation`, `BulletVelocity` 등은 오직 데이터만을 담고 있습니다. `MonoBehaviour`의 비대하고 추상화된 구조와는 거리가 멀죠. 8비트 시절의 `struct Bullet { ... }`와 본질적으로 동일합니다.
*   **배열 순회 최적화:** `MoveBulletsJob`은 `BulletTranslation`, `BulletVelocity` 컴포넌트들을 메모리상에서 연속적으로 가져와 일괄 처리합니다. 이는 CPU 캐시 친화적이며, 마치 레트로 게임이 `for` 루프를 돌며 `BulletData` 배열을 훑는 것과 똑같은 효과를 냅니다.
*   **병렬 처리:** `ScheduleParallel()`을 통해 수백, 수천 개의 총알 업데이트가 멀티코어 CPU에서 동시에 처리됩니다. 8비트 시절에는 꿈도 못 꿨던 '병렬성'을, 데이터 지향 설계를 통해 완벽하게 활용하는 것이죠.

### **저의 주관적인 견해: DOTS는 강제로 우리를 천재로 만든다**

어떤 이들은 DOTS가 너무 복잡하고, 러닝 커브가 높다고 말합니다. 하지만 저는 다르게 생각합니다. DOTS는 우리에게 '강제된 최적화'의 환경을 제공합니다. 마치 8비트 시절 개발자들이 하드웨어의 제약 때문에 '천재적인 데이터 지향 설계'를 본능적으로 익혔던 것처럼, DOTS는 현대 개발자들에게 '명시적인 데이터 지향 설계'를 요구하며 우리를 더 나은, 더 근본적인 문제 해결자로 만듭니다.
