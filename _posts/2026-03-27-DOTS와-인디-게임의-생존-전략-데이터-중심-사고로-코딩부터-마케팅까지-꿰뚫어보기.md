### 1. 문제 제기: 인디 개발의 만성적인 자원 부족과 DOTS의 탄생

인디 게임 개발은 언제나 자원과의 싸움입니다. 제한된 인력, 한정된 시간, 부족한 예산. 이 모든 제약 속에서 우리는 어떻게든 기발하고 재미있는 게임을 만들어내야 합니다. 특히 Unity를 주로 사용하는 인디 개발자들에게 가장 큰 골칫거리 중 하나는 바로 '성능'과 '유지보수성'입니다.

전통적인 Unity의 MonoBehaviour 기반 개발은 편리하고 직관적이지만, 특정 규모 이상으로 엔티티(GameObject)가 많아지거나 복잡한 로직이 얽히기 시작하면 곧바로 한계에 부딪힙니다. `Update()` 메서드 안에서 모든 것을 처리하려다 보면, 캐시 미스(cache miss)가 빈번해지고, GC(Garbage Collection) 스파이크가 발생하며, 코드는 스파게티처럼 엉켜 유지보수가 지옥이 됩니다. "아, 이 정도면 괜찮겠지"라고 생각했던 코드가 런타임에 발목을 잡고, 최적화에 발을 동동 구르다 보면 출시일은 저 멀리 사라지는 악몽. 인디 팀에게는 치명적인 시간 낭비이자 기회비용입니다.

이런 고질적인 문제에 대한 Unity의 근본적인 해답이 바로 **Data-Oriented Technology Stack (DOTS)**입니다. DOTS는 단순히 "빠른" 기술이 아닙니다. 그것은 '데이터'를 최우선으로 생각하고, 코드를 '시스템'적으로 바라보는 완전히 새로운 패러다임입니다. 마치 고대 스파게티 코드를 깔끔한 공장식 생산 라인으로 재편하는 것과 같습니다.

### 2. DOTS의 핵심: 데이터 중심 설계로 문제 해결

DOTS의 핵심은 크게 세 가지 축으로 이루어져 있습니다:

*   **ECS (Entity Component System)**: 게임 오브젝트를 '데이터 묶음(Component)'으로 이루어진 '개념적 존재(Entity)'로 보고, 이 데이터를 처리하는 '로직(System)'을 분리합니다.
*   **Job System**: 멀티코어 프로세서의 잠재력을 최대한 활용할 수 있도록 병렬 처리를 안전하고 효율적으로 지원합니다.
*   **Burst Compiler**: 고성능 C# 코드를 네이티브 코드로 변환하여 미친 듯한 속도 향상을 가져옵니다.

어떻게 이 세 가지가 앞서 언급한 문제들을 해결할까요?
가장 큰 변화는 '데이터'를 중심으로 사고하게 된다는 점입니다. 모든 것은 '데이터'이고, 시스템은 그 데이터를 변환하는 '함수'입니다.

#### **2.1. 문제 해결 예시: 수천 개의 오브젝트를 효율적으로 처리하기**

수많은 유닛, 발사체, 충돌체, 혹은 환경 오브젝트가 동시에 움직이는 게임을 상상해봅시다. 전통적인 `MonoBehaviour` 방식은 각 오브젝트마다 클래스 인스턴스가 존재하고, 각각의 `Update()`가 호출되며, 이는 캐시 미스와 CPU 파이프라인의 낭비로 이어집니다.

**DOTS 방식의 해결책:**

1.  **데이터 분리 (Components):**
    *   각 오브젝트가 가져야 할 순수한 데이터만 `IComponentData`로 정의합니다.
        ```csharp
        // 위치 데이터
        public struct Position : IComponentData
        {
            public float3 Value; // Unity.Mathematics의 float3 사용
        }

        // 속도 데이터
        public struct Velocity : IComponentData
        {
            public float3 Value;
        }

        // 수명 데이터 (예: 발사체)
        public struct Lifetime : IComponentData
        {
            public float Value;
        }
        ```
    *   이 컴포넌트들은 클래스가 아닌 '구조체(struct)'이므로 메모리에 연속적으로 배치될 가능성이 높아 데이터 접근 효율(캐시 효율)이 극대화됩니다.

2.  **로직 분리 (Systems):**
    *   이 데이터들을 처리하는 로직은 `SystemBase`를 상속받는 시스템에서 정의합니다. 시스템은 특정 컴포넌트 조합을 가진 엔티티들을 쿼리하여 일괄적으로 데이터를 처리합니다.
    *   **의사 코드 (MovementSystem):**
        ```csharp
        // MovementSystem (SystemBase를 상속받아 OnUpdate에서 로직 처리)
        public partial class MovementSystem : SystemBase
        {
            protected override void OnUpdate()
            {
                float deltaTime = SystemAPI.Time.DeltaTime; // DOTS 1.0+ 스타일

                // Job을 활용하여 병렬 처리 (아래는 Simplified Entities.ForEach 예시)
                Entities.ForEach((ref Position position, in Velocity velocity) => {
                    position.Value += velocity.Value * deltaTime;
                }).ScheduleParallel(); // ScheduleParallel로 Job System에 스케줄링
            }
        }
        ```
    *   **의사 코드 (LifetimeSystem):**
        ```csharp
        // LifetimeSystem (수명이 다한 엔티티 제거)
        public partial class LifetimeSystem : SystemBase
        {
            protected override void OnUpdate()
            {
                float deltaTime = SystemAPI.Time.DeltaTime;
                var ecb = SystemAPI.GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>().CreateCommandBuffer(World.Unmanaged); // 엔티티 생성/삭제를 위한 ECB

                Entities.ForEach((Entity entity, ref Lifetime lifetime) => {
                    lifetime.Value -= deltaTime;
                    if (lifetime.Value <= 0f)
                    {
                        ecb.DestroyEntity(entity); // 수명이 다하면 엔티티 제거
                    }
                }).ScheduleParallel();
            }
        }
        ```
    *   `ScheduleParallel()`은 Unity Job System을 활용하여 수천, 수만 개의 엔티티에 대한 처리를 여러 CPU 코어에 분산시킵니다. 여기에 Burst Compiler가 이 C# Job 코드를 초고속 네이티브 코드로 컴파일해주니, 성능은 압도적으로 향상됩니다.

#### **2.2. 아키텍처 다이어그램 (개념도)**

```

+----------------+       +----------------+       +----------------+
      |                          |                          |
      v                          v                          v

+------------------+     +------------------+     +------------------+
      |                          |                          |
      v                          v                          v
```

이 다이어그램은 엔티티가 컴포넌트들의 묶음일 뿐이며, 시스템이 독립적으로 해당 컴포넌트 데이터를 처리하는 방식을 보여줍니다. 데이터와 로직의 완벽한 분리죠.

### 3. DOTS의 '데이터 중심' 사고, 인디 게임 마케팅과 포스트모템으로 확장하기

자, 이제 이 두 키워드를 억지스럽지 않게 연결해볼 시간입니다. 저는 DOTS를 통해 얻은 '데이터 중심' 사고방식이 코드 레벨을 넘어, 인디 게임 개발의 모든 과정, 심지어 **마케팅과 포스트모템**에까지 적용될 수 있다고 굳게 믿습니다.

#### **3.1. 마케팅: 직감 대신 데이터로 '실험'하라**

인디 게임 마케팅은 종종 '감'과 '운'에 의존하곤 합니다. 멋져 보이는 트레일러를 만들고, 밈 같은 문구를 쓰고, 커뮤니티에 열심히 홍보하는 것. 이 모든 노력이 헛되지 않기를 바라지만, 실제로 어떤 활동이 효과적이었는지 객관적으로 분석하기는 어렵습니다.

**문제**: "내가 만든 게임은 최고야! 모두가 좋아할 거야!"라는 막연한 기대감으로 마케팅을 시작합니다. 하지만 데이터가 없으면 어떤 부분이 성공했고 어떤 부분이 실패했는지 알 수 없습니다. 한정된 마케팅 예산은 순식간에 사라지고, 회수율은 바닥을 칩니다.

**DOTS의 비유적 적용**: DOTS가 게임 로직을 '데이터(컴포넌트)'와 '처리 시스템'으로 분리하듯이, 마케팅도 '마케팅 데이터'와 '마케팅 시스템'으로 분리해야 합니다.

*   **마케팅 컴포넌트 (데이터)**:
    *   스팀 위시리스트 전환율
    *   트레일러 조회수, 이탈률
    *   광고 캠페인 클릭률 (CTR), 전환율 (CVR)
    *   커뮤니티(Discord, Twitter) 참여율, 긍정/부정 감성 분석
    *   스토어 페이지 A/B 테스트 결과 (설명, 스크린샷, 태그)
    *   인플루언서 마케팅 ROI (투자 대비 수익)
*   **마케팅 시스템 (로직)**:
    *   A/B 테스트 시스템 (두 가지 버전의 스토어 페이지를 동시에 운영)
    *   광고 캠페인 최적화 시스템 (데이터 기반으로 타겟층, 예산, 크리에이티브 조정)
    *   콘텐츠 바이럴 확산 시스템 (커뮤니티와 소셜 미디어 분석 후 유효한 콘텐츠 재생산)
    *   보도자료 배포 및 추적 시스템

제가 직접 겪었던 일입니다. 한때 저희 팀은 '아름다운 그래픽'에만 몰두한 트레일러를 만들고는 "이거면 충분할 거야!"라고 생각했습니다. 하지만 초기 위시리스트 전환율은 처참했습니다. DOTS를 개발하며 체득한 데이터 중심 사고로, 우리는 곧바로 A/B 테스트에 돌입했습니다. 게임의 핵심 '게임플레이'가 돋보이는 트레일러, 직관적인 게임 로고, 그리고 핵심 기능을 명확히 설명하는 스토어 페이지 설명을 여러 버전으로 만들고, 데이터를 추적했습니다. 결과는 놀라웠습니다. 데이터는 '아름다움'보다는 '재미'와 '명확성'이 구매 결정에 더 큰 영향을 미친다는 것을 여실히 보여주었고, 우리는 곧바로 마케팅 전략을 수정할 수 있었습니다. 마치 복잡한 게임 시스템에서 병목 지점을 찾고 최적화하듯, 마케팅에서도 데이터를 통해 병목을 찾고 개선한 것입니다.

**저의 의견**: "만약 당신이 코드에서 데이터 중심 설계를 외친다면, 마케팅에서도 직감이 아닌 데이터로 '실험'하고 '개선'하는 시스템을 구축해야 합니다. 코드만큼이나 마케팅도 데이터 주도적이어야 합니다."

#### **3.2. 포스트모템: 감정 없이 객관적인 데이터로 '배움'의 시스템을 만들다**

게임 출시 후 진행되는 포스트모템은 매우 중요하지만, 때로는 감정적으로 흐르거나 책임 전가로 변질되기도 쉽습니다. "누구 잘못이다", "이게 문제였다" 등의 주관적인 의견이 난무하고, 결국 다음 프로젝트에 실질적인 교훈으로 이어지지 못하는 경우가 많습니다.

**문제**: 실패의 원인 분석이 주관적인 경험과 감정에 치우쳐 실제 문제를 해결할 수 있는 객관적인 인사이트를 얻지 못합니다. 성공의 요인 분석 또한 모호하여 재현하기 어렵습니다.

**DOTS의 비유적 적용**: DOTS가 코드의 결함을 '데이터'와 '시스템'의 관점에서 분석하듯이, 포스트모템 역시 감정을 배제하고 객관적인 '데이터'를 수집하고, 이를 분석하는 '시스템'을 구축해야 합니다.

*   **포스트모템 컴포넌트 (데이터)**:
    *   개발 일정 대비 실제 진행률 데이터 (각 모듈별 개발 시간, 버그 발생률)
    *   성능 측정 데이터 (FPS, 메모리 사용량, 로딩 시간)
    *   플레이어 피드백 데이터 (버그 리포트, 건의 사항, 만족도 조사)
    *   수익 데이터 (판매량, 평균 판매 가격, 지역별 수익)
    *   마케팅 비용 대비 수익률 (ROI)
    *   팀원 만족도 데이터 (익명 설문조사)
*   **포스트모템 시스템 (로직)**:
    *   데이터 수집 및 시각화 시스템 (각 데이터 포인트를 차트, 그래프로 정리)
    *   핵심 문제점 도출 시스템 (데이터를 기반으로 가장 큰 병목, 실패 요인 식별)
    *   액션 플랜 수립 시스템 (도출된 문제점에 대한 구체적인 개선 방안 및 책임자 지정)
    *   지식 베이스 구축 시스템 (포스트모템 결과를 문서화하여 다음 프로젝트에 활용)

제가 참여했던 한 프로젝트에서는 개발 막바지에 예상치 못한 성능 저하 문제로 곤욕을 치렀습니다. DOTS를 적용하지 않았던 구형 프로젝트였죠. 포스트모템에서 우리는 "누가 이 코드를 짰냐"며 서로를 비난하기 바빴습니다. 그러나 다음 프로젝트에서는 DOTS의 '데이터 중심' 사고를 포스트모템에 적용했습니다. 개발 과정에서 주기적으로 성능 프로파일링 데이터를 기록했고, 각 기능별 개발 시간과 버그 리포트 수를 추적했습니다. 출시 후에는 판매 데이터, 유저 이탈률, 심지어는 부정적인 리뷰의 키워드까지 '데이터'로 수집하고 분석했습니다. 그 결과, 우리는 "OOO 모듈의 초기 설계 결함이 성능 저하의 주된 원인이었다"는 객관적인 결론을 도출할 수 있었고, 다음 프로젝트에서는 해당 설계 패턴을 피하는 명확한 '시스템'을 구축할 수 있었습니다. 감정적 비난 대신, 데이터가 문제를 명확히 보여준 것입니다.

**저의 의견**: "진정한 DOTS 마인드의 포스트모템은 단순히 회고가 아닙니다. 그것은 객관적인 데이터에 기반하여 지속적인 개선을 위한 '학습 시스템'을 구축하는 것입니다. 실패도, 성공도 모두 데이터가 됩니다."

### 4. 결론: 데이터 중심 사고로 무장한 인디 개발자의 힘

DOTS는 단순한 기술 스택을 넘어, '데이터를 어떻게 다루고 활용할 것인가'에 대한 근본적인 철학을 제시합니다. 이 철학은 고성능 게임 개발뿐만 아니라, 자원과 경험이 부족한 인디 개발자들이 마케팅과 포스트모템이라는 어려운 과제를 현명하게 해결할 수 있는 강력한 도구가 될 수 있습니다.

코드에서 `Entities.ForEach`로 수만 개의 데이터를 병렬 처리하며 성능을 극한으로 끌어올리듯, 마케팅에서도 수많은 고객 데이터를 A/B 테스트로 처리하며 최적의 전략을 찾아내야 합니다. 포스트모템에서는 감정적 편향 없이, 명확한 데이터를 기반으로 실패 원인을 진단하고 학습하는 시스템을 구축해야 합니다.

DOTS가 여러분의 게임을 더 빠르고 효율적으로 만들듯이, DOTS에서 얻은 데이터 중심 사고는 여러분의 인디 게임 개발 여정 전체를 더 현명하고 성공적으로 이끌어 줄 것입니다. 이제, 여러분의 코드뿐만 아니라 마케팅과 포스트모템에도 '데이터 중심'이라는 강력한 렌즈를 씌워보지 않으시겠습니까?
