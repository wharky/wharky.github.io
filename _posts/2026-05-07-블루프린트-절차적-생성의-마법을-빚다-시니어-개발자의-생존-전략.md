### 챕터 1: 절차적 생성, 블루프린트의 덫에 빠지다 (Problem)

저의 개발 경력 초창기, 저는 블루프린트가 만능이라고 믿었습니다. 특히 '절차적 생성'은 디자이너와 협업하기 좋고, 빠르게 결과를 볼 수 있다는 장점 때문에 블루프린트로 구현하기 시작했죠. 단순한 맵 타일 생성부터 시작해서, 던전 레이아웃, 심지어 아이템 속성 조합까지… 모든 것을 블루프린트로 밀어붙였습니다.

**문제는 금방 드러났습니다.**
*   **읽기 힘든 그래프:** 던전 방 하나를 생성하는 로직이 스크린 3개 크기에 달하는 거대한 그래프로 펼쳐졌습니다. 디버깅은 고사하고, 제가 어제 짠 코드도 오늘 다시 보면 남의 코드 같았죠.
*   **성능 저하:** 복잡한 반복문과 오브젝트 생성 로직이 메인 스레드를 잡아먹어 게임이 뚝뚝 끊겼습니다. 특히 초기 로딩 시 심각했죠.
*   **유지보수의 악몽:** 기능 하나를 추가하거나 수정하면, 예상치 못한 곳에서 버그가 터져 나왔습니다. 의존성이 거미줄처럼 얽혀 있어 건드리기조차 두려웠습니다.
*   **확장성 부재:** 새로운 유형의 던전이나 아이템을 추가하려면, 기존 로직을 거의 새로 짜야 하는 상황이 발생했습니다.

이쯤 되면 "아니, 그냥 C++로 짜면 되는 거 아니냐?"라고 생각하실 수 있습니다. 맞습니다. 무거운 계산 로직은 C++이 훨씬 유리합니다. 하지만 모든 팀이 C++ 개발에 충분한 리소스를 할당할 수 있는 것도 아니고, 빠른 이터레이션이 중요한 경우 블루프린트의 유연성은 포기하기 어렵습니다. 결국 문제는 '블루프린트를 어떻게 하면 잘 쓸 수 있는가'로 귀결되었습니다.

### 챕터 2: 시니어 개발자의 생존 전략: 블루프린트 모범 사례로 절차적 생성 길들이기 (Solution)

절차적 생성은 본질적으로 복잡한 알고리즘과 데이터 처리를 요구합니다. 이를 블루프린트의 직관적인 비주얼 스크립팅으로 효율적으로 구현하려면, 몇 가지 '황금률'이 필요하다는 것을 깨달았습니다. 마치 고대 마법의 복잡한 주문을 단순화하고, 재사용 가능한 룬으로 나누는 것과 비슷하다고 할까요?

#### 2.1. 모듈화의 힘: 함수, 매크로, 그리고 컴포넌트

가장 먼저 적용한 것은 **'모듈화'**였습니다. 복잡한 로직을 작은 단위로 쪼개고, 각각의 기능을 명확히 정의하는 것이죠.

*   **Pure Functions (순수 함수):** 입력을 받아서 오직 출력만 반환하고, 외부 상태를 변경하지 않는 함수는 재사용성과 테스트 용이성이 극대화됩니다. 특히 절차적 생성에서 '위치 계산', '충돌 여부 판단' 같은 순수한 계산 로직에 필수적입니다.
*   **Functions (함수):** 특정 액터의 상태를 변경하거나 월드에 영향을 미치는 로직은 함수로 캡슐화합니다. 예를 들어, `SpawnRoomActor`, `PlaceDoorAtLocation` 같은 함수들입니다.
*   **Macros (매크로):** 특정 노드 조합을 묶어 시각적인 깔끔함을 유지하면서, 인라인으로 동작하게 합니다. 순수 함수로 만들기 어려운 복잡한 흐름 제어(ex: for-each loop with break)나 자주 사용되는 패턴에 유용합니다.
*   **Actor Components (액터 컴포넌트):** 여러 액터에서 공통적으로 사용되는 기능(예: `ProceduralMeshGeneratorComponent`, `RandomizerComponent`)은 컴포넌트로 분리하여 재사용성을 높입니다. 이는 객체 지향 프로그래밍의 '구성(Composition)' 개념과 일맥상통합니다.

**Pseudo-code & Logic Example (모듈화 적용 전/후)**

**[문제 상황: 거대한 이벤트 그래프]**
```
// Event BeginPlay on DungeonGenerator_BP
BEGIN_EVENT_GRAPH
    GENERATE_MAP_DATA_ARRAY
    LOOP N TIMES FOR ROOMS:
        CALCULATE_RANDOM_SIZE
        CALCULATE_RANDOM_POSITION
        CHECK_OVERLAP_WITH_EXISTING_ROOMS (Complex Branching)
        IF NO_OVERLAP:
            SPAWN_ROOM_ACTOR (Setting various properties)
            ADD_ROOM_TO_LIST
            GENERATE_DOORS_FOR_ROOM (More complex logic)
            // ... Many more nodes ...
        ELSE:
            RETRY_POSITION_CALCULATION
    CONNECT_ROOMS_WITH_CORRIDORS
    SPAWN_PLAYER
END_EVENT_GRAPH
```

**[해결 방안: 모듈화된 함수/컴포넌트]**
```
// DungeonGenerator_BP Event Graph
Event BeginPlay:
    Call InitializeDungeonSettings (Data-driven: uses Data Table)
    Call GenerateRoomPlacementLogic (Function)
    Call ConnectRoomsWithCorridors (Function)
    Call FinalizeDungeon (Spawns player, loot, etc.)

// Function: GenerateRoomPlacementLogic
BEGIN_FUNCTION
    Set RoomConfig = GetRoomConfigByType (Pure Function: from Data Table)
    LOOP N TIMES FOR ROOMS:
        RandomSize = GetRandomRoomSize (Pure Function: uses RoomConfig)
        RandomPos = GetRandomMapPosition (Pure Function)
        ProposedRoomBounds = CalculateRoomBounds (Pure Function: takes Pos, Size)

        IF CanPlaceRoom(ProposedRoomBounds, GeneratedRoomsList) (Pure Function):
            NewRoomActor = SpawnActor_BP_Room(RandomPos, RandomSize) (Function)
            NewRoomActor.SetupDoors(RoomConfig.DoorCountRange) (Function Call on RoomActor)
            NewRoomActor.PopulateProps(RoomConfig.PropDensity) (Function Call on RoomActor)
            Add RoomActor to GeneratedRoomsList
        ELSE:
            // Handle retry or skip
END_FUNCTION

// Blueprint Pure Function: CanPlaceRoom (Often in a utility BP Library or specific Component)
bool CanPlaceRoom(ProposedBounds, ExistingRooms):
    FOR EACH ExistingRoom IN ExistingRooms:
        IF UKismetMathLibrary.OverlapAABB(ProposedBounds, ExistingRoom.Bounds):
            RETURN false
    RETURN true
```
이렇게 로직을 분리하니, `GenerateRoomPlacementLogic` 함수 안에서 무슨 일이 일어나는지 한눈에 파악할 수 있고, `CanPlaceRoom` 로직에 문제가 생기면 그 함수만 집중해서 디버깅할 수 있게 됩니다.

#### 2.2. 데이터 중심 설계 (Data-Driven Design)

절차적 생성은 수많은 파라미터(방의 크기 범위, 문 개수, 재료 타입 등)에 의해 결과가 달라집니다. 이 값들을 블루프린트 그래프 안에 하드코딩하면 유지보수가 지옥이 됩니다. 해결책은 **'데이터 테이블(Data Table)'**과 **'구조체(Struct)'**를 활용하는 것입니다.

*   **구조체(Struct):** 관련된 데이터를 하나의 묶음으로 정의하여 가독성과 재사용성을 높입니다. 예: `F_DungeonRoomConfig` (MinSize, MaxSize, DoorCountRange, PropDensity 등).
*   **데이터 테이블(Data Table):** 구조체를 기반으로 수많은 데이터를 Excel이나 CSV 형태로 관리합니다. 게임 밸런스, 레벨 디자인 파라미터 등을 이곳에 넣어두고, 블루프린트에서는 그 데이터를 읽어와서 사용합니다.

```
// Blueprint Logic using Data Table
Event BeginPlay:
    // DungeonType.BasicRoom 같은 Enum 값을 키로 사용
    RoomConfigRow = GetDataTableRow(DungeonRoomDataTable, DungeonType.BasicRoom.ToString())
    IF RoomConfigRow is Valid:
        MinRoomSize = RoomConfigRow.MinSize
        MaxRoomSize = RoomConfigRow.MaxSize
        // ...
```
데이터와 로직을 분리하면, 개발자가 그래프를 수정하지 않고도 디자이너가 직접 파라미터를 조절하여 다양한 절차적 생성 결과를 실험할 수 있게 됩니다. 이것이야말로 블루프린트가 추구하는 진정한 가치라고 생각합니다.

#### 2.3. 성능 최적화: 블루프린트의 한계 인식과 C++의 현명한 활용

아무리 모듈화를 잘 하고 데이터를 분리해도, 복잡한 수학 계산이나 대량의 데이터 처리는 블루프린트의 한계를 뛰어넘기 어렵습니다. 이때는 **'C++의 현명한 활용'**이 필요합니다.

*   **무거운 계산은 C++로:** Perlin Noise 계산, 복잡한 행렬 연산, 수만 개의 메시 버텍스 처리 등은 C++로 구현하고, 블루프린트에서는 이를 호출하는 형태로 만듭니다. `UFUNCTION(BlueprintCallable)`을 사용하면 C++ 함수를 블루프린트에서 쉽게 호출할 수 있습니다.
*   **비동기 처리(Async Tasks):** 던전 생성과 같이 시간이 오래 걸리는 작업은 게임의 메인 스레드를 막지 않도록 비동기로 처리합니다. 언리얼 엔진의 Async Task 노드를 활용하거나, C++에서 별도의 스레드를 생성하여 처리 후 메인 스레드로 결과를 전달합니다. `Blueprint Latent Action`을 커스텀 구현하는 것도 좋은 방법입니다.
*   **오브젝트 풀링(Object Pooling):** 절차적으로 생성되는 오브젝트(예: 던전 타일, 풀잎, 파편)가 많다면, 생성/파괴 비용이 큽니다. 오브젝트 풀을 만들어서 재활용하면 성능을 크게 향상시킬 수 있습니다.

**Opinionated Storytelling:** 제가 만든 초기 던전 생성기는 `Event BeginPlay`에 모든 로직을 때려 박아 시작하자마자 게임이 5초 동안 멈추는 기적을 보여줬습니다. 당시 사수에게 등짝 스매시를 맞고(?) 비동기 처리의 중요성을 깨달았죠. 별도의 `UProceduralGenerationManager` C++ 클래스를 만들고, 던전 생성 로직을 비동기로 돌린 결과, 로딩 시간이 1초 이내로 줄어들었습니다. 마법 같았죠!

#### 2.4. 가독성 & 유지보수: 깔끔한 그래프는 곧 행복

마지막으로, 블루프린트 그래프 자체의 **'가독성'과 '유지보수성'**은 아무리 강조해도 지나치지 않습니다.

*   **명확한 명명 규칙:** 변수, 함수, 이벤트 등 모든 것에 통일된 명명 규칙을 적용합니다 (예: 변수 `V_PlayerHealth`, 함수 `F_CalculateDamage`).
*   **주석(Comments)과 Reroute Nodes:** 중요한 로직 블록에는 주석을 달아 설명하고, 선이 꼬이는 것을 방지하기 위해 `Reroute Node`를 적극적으로 활용합니다.
*   **Collapsed Graph:** 복잡한 부분을 `Collapsed Graph`로 묶어서 전체적인 시야를 확보합니다. (너무 자주 쓰면 디버깅이 어려워질 수 있으니 적절히 사용)
*   **인터페이스(Blueprint Interfaces):** 서로 다른 액터 간의 통신은 블루프린트 인터페이스를 통해 명확하게 정의합니다. 이는 결합도를 낮추고 유연성을 높여줍니다.

### 마무리: 블루프린트는 마법 지팡이, 마법사가 되세요!

언리얼 엔진 블루프린트는 분명 강력한 도구입니다. 하지만 마법 지팡이를 들었다고 해서 바로 마법사가 되는 것은 아니죠. 절차적 생성처럼 복잡하고 방대한 시스템을 블루프린트로 구현할 때는, 오늘 제가 이야기한 모듈화, 데이터 중심 설계, 성능 최적화, 그리고 가독성이라는 '마법사의 비법'을 반드시 숙지해야 합니다.

처음에는 번거롭게 느껴질 수 있지만, 이 습관들이 쌓이면 여러분의 블루프린트는 더 이상 스파게티가 아닌, 정교하게 설계된 예술 작품이자 강력한 생산 도구가 될 것입니다. 엉킨 선들을 풀고, 깔끔하게 정리된 그래프를 보면서 디버깅의 고통 없이 새로운 기능을 추가할 때의 쾌감은 이루 말할 수 없습니다.

절차적 생성은 끊임없이 새로운 도전 과제를 던져줍니다. 하지만 제대로 된 블루프린트 베스트 프랙티스와 함께라면, 여러분은 그 도전을 즐거운 창작의 과정으로 만들 수 있을 겁니다. 다음 포스팅에서는 좀 더 구체적인 절차적 생성 알고리즘 구현 예시를 다뤄볼 기회가 있기를 바라며, 오늘 이야기는 여기서 마치겠습니다. Happy Blueprinting!
