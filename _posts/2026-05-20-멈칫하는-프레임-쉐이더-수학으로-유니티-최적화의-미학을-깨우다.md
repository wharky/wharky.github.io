### ✨ 해결: 쉐이더 수학으로 직접 최적화의 '마법'을 부리다

저는 한때 모바일 게임 개발 프로젝트에서 꽤 고생했던 기억이 있습니다. 특정 이펙트가 화면에 나타나는 순간, 프레임이 뚝뚝 떨어지는 현상이었죠. 초기에는 복잡한 파티클 시스템이나 여러 레이어의 텍스처 블렌딩으로 구현했는데, 이게 모바일 GPU에는 치명타였습니다. 에셋 스토어에서 여러 쉐이더를 가져다 써봐도 비슷했죠.

그때 저는 깨달았습니다. "누군가가 만들어 놓은 것을 쓰는 것을 넘어, 내가 직접 픽셀을 그려야 하는구나!" 그렇게 **GLSL** (유니티에서는 HLSL/CG가 주로 쓰이지만, GLSL의 사고방식과 수학적 원리는 모든 쉐이더 언어에 통용됩니다)의 세계로 깊숙이 파고들었고, 그 결과는 놀라웠습니다.

쉐이더 수학을 깊이 이해하고 직접 코드를 깎아 만든 쉐이더는, 기존의 해결책보다 훨씬 가볍고 빨랐습니다. 왜냐고요? 불필요한 연산을 제거하고, GPU의 병렬 처리 능력을 최대한 활용하며, 메모리 접근을 최소화했기 때문입니다. 이는 단순한 '최적화 트릭'이 아니라, **렌더링의 본질을 이해하고 직접 설계하는 과정**이었습니다.

#### 🕹️ Deep Dive: '절대 지연 없는' 프로시저럴 노이즈 & 그라디언트

가장 흔하게 접하면서도 최적화의 묘미를 느낄 수 있는 예시는 바로 '프로시저럴(Procedural) 노이즈와 그라디언트'입니다. UI 애니메이션 배경, 혹은 게임 월드의 미묘한 안개나 빛 효과를 표현할 때 자주 사용되죠.

**기존 방식의 문제:**

*   **텍스처 사용:** 고해상도 노이즈/그라디언트 텍스처를 사용하면 메모리를 잡아먹고, 확대 시 픽셀레이션 문제가 발생할 수 있습니다. 동적인 효과를 위해서는 여러 텍스처를 블렌딩해야 할 수도 있어 오버헤드가 큽니다.
*   **CPU 연산:** 노이즈 함수를 CPU에서 계산하여 텍스처로 업로드하는 방식은 CPU에 부담을 주고, GPU로의 데이터 전송 비용이 발생합니다.

**쉐이더 수학을 이용한 해결책:**

우리는 Fragment Shader 안에서 순수하게 수학적 연산만으로 동적인 노이즈와 그라디언트를 생성할 수 있습니다. 이렇게 하면 텍스처 메모리를 절약하고, CPU 오버헤드를 없애며, GPU의 엄청난 병렬 처리 능력을 활용하여 각 픽셀마다 독립적으로 빠르게 연산할 수 있습니다. 이것이 바로 **유니티 엔진 최적화의 핵심적인 '트릭' 중 하나**입니다.

다음은 매우 간소화된 GLSL(또는 HLSL/CG에 적용 가능한) 의사코드입니다. 핵심 로직과 수학적 접근 방식을 살펴보세요.

```glsl
// Fragment Shader Logic (Pseudo-code, inspired by GLSL/HLSL)

struct Varyings
{
    float4 pos       : SV_POSITION;
    float2 uv        : TEXCOORD0; // 텍스처 좌표 (0.0 ~ 1.0)
    // ... 기타 필요한 데이터
};

// 메인 함수
float4 MyFragmentShader(Varyings i) : SV_TARGET
{
    float2 uv = i.uv;
    float time = _Time.y; // Unity 엔진에서 전달받는 현재 시간

    // 1. 역동적인 그라디언트 생성 (Shader Math: Lerp, Sin, Time)
    // 두 색상 사이를 부드럽게 보간합니다.
    // Sin 함수를 이용하여 시간에 따라 그라디언트 방향과 강도를 변화시킵니다.
    vec3 colorA = vec3(0.1, 0.2, 0.4); // 시작 색상 (짙은 파랑)
    vec3 colorB = vec3(0.8, 0.6, 0.3); // 끝 색상 (따뜻한 주황)

    // UV의 X좌표와 시간에 따라 그라디언트 강도를 계산합니다.
    // 3.0은 그라디언트의 반복 횟수, 0.5+0.5는 Sin 함수 결과(-1~1)를 (0~1)로 매핑.
    float gradientFactor = sin(uv.x * 3.0 + time * 0.5) * 0.5 + 0.5;

    // mix (또는 lerp) 함수로 두 색상을 보간합니다.
    vec3 finalColor = mix(colorA, colorB, gradientFactor);

    // 2. 간단한 프로시저럴 노이즈 추가 (Shader Math: Dot, Sin, Frac)
    // 텍스처 샘플링 없이 UV 좌표와 몇 가지 상수를 이용해 유사 난수를 생성합니다.
    // 이는 Simplex/Perlin 노이즈만큼 정교하진 않지만, 미묘한 디테일 추가에 효과적입니다.
    float noise = frac(sin(dot(uv * 123.456, vec2(78.9, 12.3))) * 456.789);

    // 생성된 노이즈를 최종 색상에 미미하게 더합니다. (노이즈의 강도는 0.1)
    finalColor += noise * 0.1;

    // 최종 색상을 반환합니다. (알파 값은 1.0으로 불투명하게 설정)
    return vec4(finalColor, 1.0);
}
```

**로직 설명:**

1.  **`uv` (텍스처 좌표):** 화면의 각 픽셀 위치를 0.0에서 1.0 사이의 값으로 정규화한 좌표입니다. 이는 쉐이더 수학에서 공간적 위치를 표현하는 가장 기본적인 입력입니다.
2.  **`_Time.y` (시간):** 유니티에서 제공하는 현재 시간 값입니다. 이 값을 활용하여 이펙트를 동적으로 만들 수 있습니다.
3.  **`sin(x)`:** 파형을 만들어내는 삼각 함수입니다. `uv.x`에 `sin`을 적용하고 시간을 더해 움직이는 파동 형태의 그라디언트를 만듭니다. `0.5 + 0.5`를 하는 이유는 `sin` 함수의 결과가 `-1`에서 `1` 사이이므로, 이를 `0`에서 `1` 사이로 정규화하여 색상 보간에 사용하기 위함입니다.
4.  **`mix(a, b, t)` (또는 `lerp`):** 두 값 `a`와 `b`를 `t` 비율로 선형 보간하는 함수입니다. `t`가 `0`이면 `a`, `1`이면 `b`를 반환합니다.
5.  **`dot(a, b)` (내적):** 두 벡터의 내적을 계산합니다. 노이즈 함수에서 UV 좌표를 섞어 특정 패턴을 만드는 데 자주 사용됩니다. 벡터의 방향 유사도를 측정하는 강력한 도구죠.
6.  **`frac(x)`:** `x`의 소수 부분만을 취하는 함수입니다. `sin` 함수와 `dot` 연산을 조합하여 복잡한 유사 난수를 만들고, `frac`으로 그 값을 `0`에서 `1` 사이로 유지시켜 무작위성을 부여합니다.

이 간단한 코드는 텍스처 한 장 없이, CPU 개입 없이, 오직 GPU에서 픽셀 단위로 동적이고 복잡한 시각 효과를 만들어냅니다. 이것이 바로 **GLSL과 쉐이더 수학이 유니티 최적화에 기여하는 방식**의 정수입니다.

#### 🚀 Unity Engine Optimization Tricks: 실질적인 이점

위와 같은 쉐이더 수학 기반의 접근 방식은 유니티 프로젝트에 다음과 같은 실질적인 최적화 이점을 가져다줍니다.

*   **메모리 절약:** 텍스처를 사용하지 않으므로 VRAM 사용량을 크게 줄입니다. 이는 특히 모바일 환경에서 매우 중요합니다.
*   **드로우콜 감소 & 배치 최적화:** 같은 재질을 사용하는 여러 오브젝트는 드로우콜을 합쳐 배치될 가능성이 높아집니다. 쉐이더를 직접 만들면 재질 속성을 최소화하여 이점을 얻기 쉽습니다.
*   **CPU 오버헤드 감소:** 텍스처 데이터 업로드, CPU 기반의 연산 등이 사라지므로 CPU 부담이 줄어듭니다.
*   **GPU 병렬 처리 극대화:** 픽셀 쉐이더는 각 픽셀을 독립적으로 병렬 처리하므로, 복잡한 연산이라도 효율적으로 처리될 수 있습니다.
*   **높은 유연성과 동적 제어:** 게임 플레이 상황이나 사용자 입력에 따라 시각 효과를 훨씬 더 세밀하고 역동적으로 제어할 수 있습니다.

### 💡 제 견해: 수학은 쉐이더 개발자의 '언어'이자 '무기'

"수학이라니, 너무 어려워!"라고 생각하실 수도 있습니다. 하지만 저는 감히 단언컨대, **쉐이더 개발에 있어 수학은 단순한 도구가 아니라, 당신의 의도를 GPU에게 전달하는 '언어'이자 문제를 해결하는 '강력한 무기'**입니다.

복잡한 물리 기반 렌더링(PBR)의 조명 모델부터, 볼류메트릭 렌더링, 사인드 디스턴스 필드(SDF)를 이용한 프로시저럴 지오메트리까지, 모든 고급 렌더링 기법의 밑바닥에는 수학적 원리가 깔려 있습니다. 이 원리를 이해하지 못하면, 우리는 쉐이더가 어떻게 작동하는지 '어렴풋이' 알 뿐, 진정으로 제어하고 최적화할 수는 없습니다.

**제가 드리고 싶은 조언은 다음과 같습니다:**

1.  **두려워 말고 시작하세요:** 벡터, 행렬, 삼각 함수 등 기본적인 것부터 차근차근 익혀나가세요. 온라인에 훌륭한 자료들이 많습니다.
2.  **작게 시작하고 반복하세요:** 복잡한 쉐이더를 처음부터 만들려고 하지 마세요. 간단한 색상 그라디언트, UV 조작부터 시작하여 조금씩 복잡도를 늘려가세요.
3.  **프로파일링은 필수:** 아무리 수학적으로 완벽해 보여도, 실제 성능은 다를 수 있습니다. 유니티 프로파일러와 GPU 디버거를 적극 활용하여 병목 지점을 찾고 개선하세요.
4.  **오픈 소스와 에셋을 '해부'하세요:** 다른 사람들이 만든 쉐이더 코드를 읽고 분석하는 것은 훌륭한 학습 방법입니다. 어떻게 수학적 개념이 실제 코드에 적용되었는지 직접 확인하세요.
