### **문제 1: "PPU가 비명을 질러요!" - 과도한 Draw Call과 Overdraw 해결하기**

요즘 유니티 프로젝트에서 가장 흔하게 마주하는 성능 문제 중 하나는 바로 과도한 Draw Call과 Overdraw입니다. 복잡한 씬, 수많은 오브젝트, 다양한 재질들이 GPU를 끊임없이 괴롭히죠. 프로파일러를 열면 `Camera.Render`가 심장병 환자의 그래프처럼 널뛰는 것을 볼 수 있습니다.

**레트로 콘솔 개발자의 시선:**
NES의 PPU(Picture Processing Unit)는 한 스캔라인에 최대 8개의 스프라이트만 그릴 수 있었습니다. 배경은 고정된 타일셋으로 이루어져 있었고, 팔레트도 극히 제한적이었죠. SNES 역시 스프라이트와 배경 레이어 관리의 제약이 심했습니다. 당시 개발자들은 이 제한 속에서 어떻게 수많은 캐릭터와 화려한 배경을 표현했을까요? 그들은 '재활용', '뭉쳐 그리기', '안 보이는 건 안 그리기'의 장인이었습니다.

**해결책 (Unity Practical): 레트로 정신으로 Batching과 Culling을 마스터하자!**

### **문제 2: "GC가 터졌어요!" - 예기치 않은 프레임 드랍, GC Spike 해결하기**

C#을 사용하는 유니티 개발자라면 누구나 한 번쯤 겪어봤을 골치 아픈 문제입니다. 게임 도중 갑자기 화면이 멈칫하거나 프레임이 뚝 떨어지는 현상. 프로파일러를 돌려보면 `GC.Collect`가 빨간불을 깜빡이며 경고하고 있습니다. 바로 가비지 컬렉터(Garbage Collector)가 쓰레기를 치우느라 잠시 게임 스레드를 멈추는 바람에 발생한 일이죠.

**레트로 콘솔 개발자의 시선:**
NES의 RAM은 고작 2KB였고, SNES도 128KB에 불과했습니다. 이 좁디좁은 메모리에서 `new GameObject()` 같은 건 상상조차 할 수 없었습니다. 그들에게는 "동적 할당"이라는 개념 자체가 사치였죠. 모든 것은 미리 정해진 공간에, 정해진 형태로 존재하거나, 재활용되었습니다. 그들에게 GC Spike는 존재할 수 없는 문제였습니다. 왜냐고요? 애초에 버릴 쓰레기조차 만들지 않았으니까요!

**해결책 (Unity Practical): 짠돌이 메모리 관리, 오브젝트 풀링으로 GC를 길들이자!**

1.  **오브젝트 풀링 (Object Pooling):**
    새로운 오브젝트(총알, 폭발 이펙트, 적 등)가 필요할 때마다 `Instantiate()` 하는 대신, 미리 일정한 개수를 만들어두고 필요할 때마다 비활성화된 것 중 하나를 꺼내 쓰는 방식입니다. 사용이 끝나면 파괴하는 대신 다시 비활성화하여 풀로 돌려보냅니다. 레트로 콘솔 게임의 스프라이트 재활용(스프라이트 수를 넘지 않기 위해)과 동일한 원리입니다.
    ```csharp
    // 간단한 오브젝트 풀링 의사코드
    public class SimpleObjectPool : MonoBehaviour
    {
        public GameObject prefab;
        public int initialPoolSize = 10;
        private List<GameObject> pooledObjects = new List<GameObject>();

        void Awake()
        {
            for (int i = 0; i < initialPoolSize; i++)
            {
                GameObject obj = Instantiate(prefab);
                obj.SetActive(false);
                pooledObjects.Add(obj);
            }
        }

        public GameObject GetPooledObject()
        {
            foreach (GameObject obj in pooledObjects)
            {
                if (!obj.activeInHierarchy) // 비활성화된 오브젝트 찾기
                {
                    obj.SetActive(true);
                    return obj;
                }
            }
            // 풀이 부족하면 새로 생성 (최악의 경우, GC 발생)
            GameObject newObj = Instantiate(prefab);
            pooledObjects.Add(newObj);
            return newObj;
        }

        public void ReturnPooledObject(GameObject obj)
        {
            obj.SetActive(false);
        }
    }
    ```
    *   **내 생각:** 오브젝트 풀링은 현대 개발자가 레트로 개발자에게 바치는 오마주입니다. `new`나 `Instantiate`는 마치 용돈을 물 쓰듯이 쓰는 철없는 자식 같은 거죠. 미리 계획하고 아껴 써야 합니다.

2.  **핫 패스 (Hot Path)에서의 `new` 피하기:**
    `Update()`, `FixedUpdate()`, `LateUpdate()` 또는 자주 호출되는 함수 내에서 `new` 키워드를 사용해 새로운 객체를 생성하는 것을 극도로 지양해야 합니다. 매 프레임마다, 혹은 매 호출마다 새 객체가 생성되면 쓰레기가 산더미처럼 쌓여 GC를 유발합니다.
    *   **예시:** `Vector3 temp = new Vector3(x, y, z);` 대신 `Vector3 temp; temp.x = x; temp.y = y; temp.z = z;` (물론 `Vector3`는 `struct`라 괜찮지만, `class`의 경우를 상상해 보세요)
    *   **팁:** 빈 리스트를 매번 `new List<T>()`로 생성하는 대신, 한 번만 생성하고 `list.Clear()`를 사용하는 것이 훨씬 효율적입니다.

3.  **String Concatenation (문자열 결합) 주의:**
    C#에서 `+` 연산자를 이용한 문자열 결합은 매번 새로운 문자열 객체를 생성합니다. 특히 반복문 안에서 자주 발생하면 심각한 GC를 유발할 수 있습니다.
    *   `string log = "Hello " + name + " and " + message;` (GC 유발 가능)
    *   대신 `StringBuilder`를 사용하세요:
        ```csharp
        // C# 문자열 빌더 사용
        using System.Text;
        private StringBuilder sb = new StringBuilder(); // 한번만 생성

        void LogMessage(string name, string message)
        {
            sb.Clear(); // 재활용
            sb.Append("Hello ");
            sb.Append(name);
            sb.Append(" and ");
            sb.Append(message);
            Debug.Log(sb.ToString()); // 최종적으로 한 번만 새 문자열 생성
        }
        ```
    *   **내 생각:** 문자열은 눈에 보이지 않는 쓰레기를 가장 많이 만드는 주범 중 하나입니다. 옛날에는 문자 하나하나를 비트맵으로 찍어가며 아껴 썼다는 것을 상기합시다.

### 마무하며: 레트로 정신, 현대 개발의 나침반이 되다

오늘 우리는 유니티 최적화라는 현대적 과제를 해결하기 위해, NES와 SNES 개발자들의 '짠내 나는' 지혜를 빌려와 보았습니다. 보잘것없는 하드웨어 속에서 최상의 경험을 뽑아내기 위해 씨름했던 그들의 정신은, 비록 하드웨어 스펙이 넘쳐나는 시대일지라도, 여전히 우리에게 유효한 교훈을 줍니다.

*   **제약 조건 안에서 창의성을 발휘하라.**
*   **낭비하지 말고, 재활용하라.**
*   **보이지 않는 것은 과감히 잘라내라.**
*   **미리 계획하고, 효율적으로 관리하라.**

이 모든 것은 비단 레트로 콘솔만의 이야기가 아닙니다. 모바일 게임, VR/AR, 혹은 AAA급 게임에서 최고 성능을 뽑아내려는 모든 현대 개발자들이 새겨야 할 금언입니다. 다음번에 유니티 프로파일러를 보며 한숨 쉬는 일이 생긴다면, 잠시 눈을 감고 NES PPU의 비명 소리, 16KB RAM의 압박을 상상해 보세요. 그리고 그 속에서 현명한 해결책을 찾아냈던 과거의 개발자들에게서 영감을 얻어봅시다.

결국, 최적화는 단순히 버그를 고치거나 숫자를 줄이는 기술적인 작업이 아닙니다. 그것은 제한된 자원 안에서 최고의 사용자 경험을 제공하고자 하는, 개발자의 '실력'이자 '철학'입니다. 저와 함께 이 레트로 정신을 다시 불태워, 우리의 게임을 더욱 빛나게 만들어 나갑시다!
