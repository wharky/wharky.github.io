### 1. Old School Cool: 예측 불가능은 용납되지 않던 시대의 미학

저는 가끔 요즘 개발자들이 수 기가바이트의 메모리, 수십 코어 CPU를 가지고도 "최적화가 어렵다"고 하는 말을 들으면 피식 웃음이 나곤 합니다. 농담이 아니에요! NES 시절의 CPU는 MHz 단위였고, RAM은 고작 수 킬로바이트에 불과했습니다. SNES나 PS1에 와서 좀 나아졌지만, 여전히 요즘 PC와 비교하면 장난감 수준이었죠.

**[문제]:** 이 극악의 자원 제약 속에서 어떻게 버그 없이 복잡한 게임을 매끄럽게 구동했을까요? 어떻게 화면에 수많은 스프라이트를 그려내고, 물리 연산을 수행하며, 심지어 3D 공간까지 구현했을까요?
**[해결책]:** 당시 개발자들은 말 그대로 '하드웨어의 신'이었습니다. 그들은 모든 CPU 사이클, 모든 메모리 바이트를 꿰뚫고 있었죠. 예측 불가능한 것은 곧 버그였고, 예측 불가능한 시스템은 게임을 망치는 지름길이었습니다.

*   **NES의 PPU 타이밍:** 8비트 기기에서 수많은 스프라이트를 화면에 구현하기 위해 개발자들은 PPU(Picture Processing Unit)의 VBlank 타이밍을 초 단위로 계산해 스프라이트의 위치를 업데이트하고 심지어 더 많은 스프라이트를 깜빡거리게 만들어 한계를 우회했습니다. 모든 것이 정교하게 '결정론적(Deterministic)'으로 작동해야만 했습니다.
*   **SNES의 Mode 7:** 고작 수십 KB의 RAM으로 환상적인 원근감과 확대/축소를 구현한 Mode 7은 고정소수점(Fixed-point) 연산의 꽃이었습니다. 부동소수점(Floating-point) 연산이 하드웨어 지원도 미비했고 느렸기에, 개발자들은 모든 좌표와 각도를 정수 형태로 관리하며 오차 없이 '정확히' 계산했습니다.
*   **PS1의 폴리곤 연산:** Z-버퍼(Depth Buffer)조차 없던 시절, PS1 개발자들은 오직 폴리곤의 렌더링 순서와 고정소수점 연산을 통해 3D 공간을 재구성했습니다. 이는 곧 게임 세계의 모든 계산이 플랫폼과 무관하게 *항상 동일한 결과*를 보장해야 한다는 것을 의미했습니다.

이때의 개발은 단순한 코딩이 아니라, 시스템의 본질을 꿰뚫는 예술에 가까웠습니다. 모든 것이 예측 가능해야만 했고, 다음 프레임에 무슨 일이 일어날지 개발자 스스로가 완벽히 이해하고 제어해야 했습니다.

### 2. 시간을 거스르는 자들: 현대 네트워크 동기화의 숙명과 롤백 넷코드

이제 2024년으로 돌아와 봅시다. 우리는 엄청난 컴퓨팅 파워를 가지고 있지만, 여전히 극복하기 어려운 한계에 부딪힙니다. 바로 **네트워크 지연 시간(Latency)**이죠. 아무리 빠른 인터넷이라도 빛의 속도를 넘어설 수는 없고, 서버와 클라이언트 간의 물리적인 거리는 항상 존재합니다.

**[문제]:** 온라인 멀티플레이어 게임에서 이 지연 시간은 어떻게 플레이어 경험을 망칠까요? 나의 공격이 상대방에게 닿았는데 서버에서는 아니라고 한다거나, 나는 분명히 피했는데 맞았다고 판정되는 불쾌한 상황.
**[해결책]:** 이 문제를 해결하기 위한 여러 기법 중, 가장 고도화되고 효과적이라고 평가받는 것이 바로 **Rollback Netcode(롤백 넷코드)**입니다.

*   **Dead Reckoning (추측 항법):** 클라이언트가 과거 데이터를 기반으로 상대방의 다음 움직임을 예측하고, 서버에서 올바른 데이터가 오면 살짝 보정하는 방식입니다. 구현은 비교적 쉽지만, 예측 오차가 커지면 캐릭터가 뚝뚝 끊기는 '스냅(Snap)' 현상이 발생해 몰입감을 해칩니다.
*   **Rollback Netcode (롤백 넷코드):** 클라이언트는 서버의 응답을 기다리지 않고 자신의 입력과 예측되는 상대방의 입력으로 게임을 진행합니다. 만약 서버에서 더 정확한 입력 정보가 도착했는데 클라이언트의 예측이 틀렸다면? 클라이언트는 잠시 '시간을 되돌려(Rollback)' 과거의 특정 시점(보통 서버에서 권한을 가진 프레임)으로 게임 상태를 되돌린 후, 새로 받은 정확한 입력으로 *다시 시뮬레이션*하여 현재 프레임까지 빠르게 도달합니다. 이 과정이 충분히 빠르다면 플레이어는 지연 시간을 거의 느끼지 못합니다.

롤백 넷코드는 그야말로 시간을 거스르는 마법과도 같습니다. 하지만 이 마법의 가장 중요한 전제 조건은, 게임의 모든 로직과 상태 변화가 **완벽하게 '결정론적(Deterministic)'**이어야 한다는 것입니다. 즉, 어떤 환경에서든 (클라이언트 A, 클라이언트 B, 서버 등) *동일한 시작 상태와 동일한 입력이 주어지면 항상 동일한 다음 상태에 도달해야* 합니다.

### 3. 과거와 현재의 교차점: 결정론적 아키텍처의 재발견

여기서 바로 Retro Console Architecture와 Rollback Netcode의 놀라운 연결고리가 드러납니다.
NES, SNES, PS1 개발자들이 제한된 자원 속에서 완벽한 예측 가능성을 추구했던 그 '결정론적 사고'가, 현대의 롤백 넷코드 구현에 있어서는 절대적인 필수 조건이 되는 것이죠.

**[핵심 연결고리]:**
Rollback Netcode에서 '시간을 되돌려 다시 시뮬레이션'하는 과정이 성공하려면, 되돌린 시점부터 현재까지의 모든 게임 로직이 클라이언트와 서버에서 100% 동일하게 실행되어야 합니다. 단 한 비트라도 차이가 나면 게임 상태가 어긋나고, 이는 곧 치명적인 동기화 오류로 이어집니다.

**[실무적 예시]: 부동소수점 연산과 고정소수점 연산**

대부분의 현대 게임 엔진은 물리 연산이나 위치 계산에 부동소수점(float/double)을 사용합니다. 하지만 부동소수점 연산은 CPU 아키텍처, 컴파일러, 최적화 레벨 등 미묘한 환경 차이에 따라 결과값이 미세하게 달라질 수 있습니다. 이 작은 오차가 롤백 넷코드에서는 치명적인 비결정성으로 작용합니다.

**[문제]:** 클라이언트와 서버에서 같은 물리 연산을 수행했는데, 부동소수점 오차 때문에 위치값이 0.00001 차이가 난다면? 롤백 시뮬레이션 결과가 달라지고, 캐릭터가 엉뚱한 곳으로 순간이동하는 '오류'가 발생합니다.

**[해결책]: 옛것에서 배우다 - 고정소수점(Fixed-point) 연산의 부활**

```pseudo
// 부동소수점 연산 (Rollback 환경에서 비결정적일 가능성 농후)
struct Vec2 { float x, y; };

void UpdatePhysics(Vec2& pos, const Vec2& vel, float deltaTime) {
    pos.x += vel.x * deltaTime;
    pos.y += vel.y * deltaTime;
}

// 고정소수점 연산 (Retro 콘솔 개발의 유산이자 Rollback의 심장)
// 예: 16.16 고정소수점 (정수부 16비트, 소수부 16비트 -> 스케일링 팩터 = 2^16 = 65536)
typedef long long FixedPoint;
const FixedPoint FP_ONE = 65536; // 1.0을 나타내는 고정소수점 값

struct FpVec2 { FixedPoint x, y; };

// float -> FixedPoint 변환
FixedPoint FloatToFixed(float val) {
    return (FixedPoint)(val * FP_ONE);
}

// FixedPoint -> float 변환
float FixedToFloat(FixedPoint val) {
    return (float)val / FP_ONE;
}

// 고정소수점 곱셈 (오버플로우 방지 고려)
FixedPoint FixedMultiply(FixedPoint a, FixedPoint b) {
    return (a * b) / FP_ONE; // 간단한 예시, 실제 구현은 더 복잡할 수 있음
}

void UpdatePhysicsFixed(FpVec2& pos, const FpVec2& vel, FixedPoint fixedDeltaTime) {
    pos.x += FixedMultiply(vel.x, fixedDeltaTime);
    pos.y += FixedMultiply(vel.y, fixedDeltaTime);
}

// 사용 예시
// FpVec2 playerPos = { FloatToFixed(10.5f), FloatToFixed(20.0f) };
// FpVec2 playerVel = { FloatToFixed(1.2f), FloatToFixed(0.5f) };
// FixedPoint dt = FloatToFixed(1.0f / 60.0f); // 60fps
// UpdatePhysicsFixed(playerPos, playerVel, dt);
```
NES 개발자들이 느린 CPU에서 정밀한 연산을 위해 고정소수점을 사용했던 그 방식이, 현대 롤백 넷코드에서는 플랫폼 독립적인 '결정론'을 보장하는 핵심 도구가 됩니다. 심지어 랜덤 시드(Random Seed) 관리, 비결정적인 외부 라이브러리 함수 호출 제한, 운영체제나 하드웨어에 의존적인 시간 함수 사용 금지 등, 모든 면에서 '예측 가능성'을 최우선으로 두는 아키텍처를 설계해야 합니다. 마치 retro 콘솔 개발자들이 모든 것을 수동으로 제어했던 것처럼요.

### 4. 그래서, 우리에게는 무엇이 필요한가?

제가 말하고 싶은 핵심은 이겁니다. 현대 게임 개발은 점점 더 많은 추상화와 편의성을 제공합니다. 엔진은 복잡한 물리나 렌더링을 알아서 처리해주고, 라이브러리는 네트워크 통신을 쉽게 만들어주죠. 하지만 이런 '추상화의 달콤함' 뒤에는 때로 독이 숨어있을 수 있습니다.

**[Opinionated Summary]:** Retro 콘솔 개발자들이 가졌던 '자원 제약 속의 장인정신'과 '완벽한 예측 가능성에 대한 집착'은 단지 과거의 유물이 아닙니다. 이는 오늘날 롤백 넷코드를 성공시키는 가장 중요한 사고방식이며, 현대 개발자들이 다시금 되새겨야 할 핵심 역량입니다.

**[Practical Advice]:**
*   **저수준(Low-level) 디테일 이해:** 여러분이 사용하는 엔진이나 라이브러리가 '내부적으로 어떻게 작동하는지'에 대한 호기심을 잃지 마세요. 특히 물리 연산이나 무작위성(randomness) 처리 같은 핵심 로직에서는 더욱 그렇습니다.
*   **결정론적 설계:** 멀티플레이어 게임을 개발한다면, 처음부터 '이 시스템은 결정론적이어야 한다'는 마인드셋으로 설계하세요. 모든 입력과 상태 변화가 예측 가능하도록, side-effect 없는 순수 함수(pure function)와 데이터 중심(data-oriented) 설계를 지향하는 것이 좋습니다.
*   **고정소수점 연산의 재고:** 특정 상황(특히 롤백 넷코드가 필요한 경쟁 게임)에서는 부동소수점 대신 고정소수점 연산을 적극적으로 고려해 보세요. 이는 성능 향상뿐 아니라 동기화 안정성에도 큰 도움이 될 수 있습니다.

제가 처음 롤백 넷코드를 본격적으로 공부하고 구현할 때, 과거 NES 개발 문서들을 뒤적이며 '아, 결국 모든 건 여기서 시작되었구나! 예측 가능성이 모든 것의 열쇠였어!' 깨달았던 순간이 있습니다. 마치 시대를 거슬러 올라가 개발자 선배들의 지혜를 훔쳐오는 듯한 기분이었죠.

결론적으로, 'Retro Console Architecture'는 단순히 과거의 향수를 자극하는 기술 트렌드가 아닙니다. 그것은 극심한 제약 속에서 탄생한 '결정론적 시스템 설계'의 정수이며, 이 원칙은 'Server-Side Network Sync'의 최전선, 특히 Rollback Netcode와 같은 고성능 동기화 아키텍처를 구축하는 데 있어 변함없이 강력한 지침이 됩니다. 과거는 현재의 스승이며, 미래의 길잡이가 될 수 있습니다. 우리에게 필요한 것은 그 지혜를 발굴하고, 현대의 문제에 적용하는 통찰력 아닐까요?
