### 1. 문제: "영화를 게임처럼, 게임을 영화처럼" 그 경계의 무거움

인터랙티브 무비 게임은 시청자와 플레이어 사이의 경계를 허물고, 이야기의 흐름을 플레이어의 선택에 맡기는 매력적인 장르입니다. 하지만 개발자 관점에서 보면, 이 '매력' 뒤에는 엄청난 기술적 도전이 숨어있습니다.

**핵심 문제점:**
1.  **방대한 영상 데이터:** 고해상도 비디오 클립들이 시퀀스 단위로 연결되는데, 이 파일들은 크기가 어마어마합니다. 잘못 관리하면 로딩 지연, 메모리 과부하, 심지어 게임 크기 자체가 감당할 수 없게 커지는 문제에 직면합니다.
2.  **복잡한 분기 로직:** 플레이어의 선택에 따라 이야기가 수십, 수백 갈래로 나뉘는데, 이 모든 경로와 상태 변화를 효율적으로 관리해야 합니다. 잘못된 설계는 디버깅 지옥으로 이어지죠.
3.  **매끄러운 전환:** 비디오 클립 간의 전환, 혹은 비디오에서 인게임 플레이(QTE 등)로의 전환은 영화처럼 부드러워야 합니다. 미세한 끊김이나 프레임 드랍은 몰입을 순식간에 깨뜨립니다.
4.  **입력 반응성:** 플레이어의 선택 입력이 즉각적으로 반영되어야 합니다. 로딩 때문에 입력이 씹히거나 늦게 반응하면 '내가 뭘 선택했지?'하는 불쾌감을 줍니다.

저도 과거에 이런 유형의 게임을 개발하면서 "어떻게 하면 이 무거운 비디오들을 끊김 없이 보여줄 수 있을까?"라는 질문에 밤잠을 설치곤 했습니다. 특히 모바일 환경에서는 그 압박이 더욱 심했죠.

### 2. 해결책: 인터랙티브 무비 시스템 아키텍처와 유니티 최적화의 만남

이러한 문제들을 해결하기 위해 우리는 몇 가지 핵심적인 아키텍처 설계와 유니티 최적화 기법을 결합해야 합니다.

#### 2.1. 방대한 영상 데이터 관리: Addressables와 비디오 스트리밍 최적화
**문제:** 모든 비디오 클립을 한 번에 로드하는 것은 비현실적입니다. 필요한 비디오만 효율적으로 로드하고, 더 이상 필요 없는 비디오는 빠르게 언로드해야 합니다.

**해결책:**
*   **Unity Addressables System 활용:** 비디오 클립들을 `Addressables`로 관리하여, 런타임에 필요한 비디오만 비동기적으로 로드합니다. 이는 초기 빌드 사이즈를 줄이고, 메모리 사용량을 최적화하는 데 필수적입니다.
*   **예측 로딩(Predictive Loading):** 플레이어의 현재 선택 지점 이후로 발생할 수 있는 가장 유력한 다음 비디오 클립들을 미리 로딩 버퍼에 넣어둡니다. 이는 갑작스러운 로딩 지연을 최소화합니다.
*   **비디오 코덱 및 해상도 최적화:** 타겟 플랫폼(PC, 모바일, 콘솔)에 맞춰 최적의 비디오 코덱(예: H.264, VP8)과 해상도를 선택하고, 비트레이트를 조절하여 파일 크기를 줄이면서도 시각적 품질을 유지합니다.
*   **VideoPlayer Pooling:** `VideoPlayer` 컴포넌트도 일종의 오브젝트로 보고, 필요할 때마다 생성/삭제하기보다는 미리 풀링하여 재활용하는 전략을 사용합니다.

**핵심 로직 (Pseudo-code):**
```csharp
// 1. Addressables를 이용한 비동기 비디오 로딩
public class VideoLoader : MonoBehaviour
{
    private Dictionary<string, AsyncOperationHandle<VideoClip>> loadingHandles = new Dictionary<string, AsyncOperationHandle<VideoClip>>();
    private Dictionary<string, VideoClip> loadedClips = new Dictionary<string, VideoClip>();

    public async Task<VideoClip> LoadVideoAsync(string addressablePath)
    {
        if (loadedClips.TryGetValue(addressablePath, out VideoClip clip))
        {
            return clip; // 이미 로드됨
        }

        if (loadingHandles.ContainsKey(addressablePath))
        {
            // 이미 로딩 중인 경우, 완료될 때까지 기다림
            await loadingHandles[addressablePath].Task;
            return loadedClips[addressablePath];
        }

        Debug.Log($"[VideoLoader] Loading {addressablePath} asynchronously...");
        var handle = Addressables.LoadAssetAsync<VideoClip>(addressablePath);
        loadingHandles[addressablePath] = handle;
        clip = await handle.Task;

        if (handle.Status == AsyncOperationStatus.Succeeded)
        {
            loadedClips[addressablePath] = clip;
            Debug.Log($"[VideoLoader] Loaded {addressablePath} successfully.");
            return clip;
        }
        else
        {
            Debug.LogError($"[VideoLoader] Failed to load {addressablePath}: {handle.OperationException}");
            return null;
        }
    }

    public void ReleaseVideo(string addressablePath)
    {
        if (loadedClips.ContainsKey(addressablePath))
        {
            Addressables.Release(loadingHandles[addressablePath]);
            loadedClips.Remove(addressablePath);
            loadingHandles.Remove(addressablePath);
            Debug.Log($"[VideoLoader] Released {addressablePath}.");
        }
    }

    // 다음 시퀀스 예측 로딩
    public void PreloadNextVideos(List<string> potentialNextPaths)
    {
        foreach (var path in potentialNextPaths)
        {
            if (!loadedClips.ContainsKey(path) && !loadingHandles.ContainsKey(path))
            {
                // 백그라운드에서 로드 시작 (await 하지 않음)
                LoadVideoAsync(path).Forget(); // 'Forget'은 C# 9.0 이상에서 Task를 기다리지 않고 실행하기 위한 확장 메서드
            }
        }
    }
}

// 2. VideoPlayer 관리 (풀링)
public class VideoPlayerPool : MonoBehaviour
{
    public GameObject videoPlayerPrefab; // VideoPlayer 컴포넌트가 붙은 프리팹
    private Queue<VideoPlayer> availablePlayers = new Queue<VideoPlayer>();
    private List<VideoPlayer> activePlayers = new List<VideoPlayer>();
    public int poolSize = 3;

    void Awake()
    {
        for (int i = 0; i < poolSize; i++)
        {
            GameObject obj = Instantiate(videoPlayerPrefab, transform);
            VideoPlayer player = obj.GetComponent<VideoPlayer>();
            player.gameObject.SetActive(false);
            availablePlayers.Enqueue(player);
        }
    }

    public VideoPlayer GetPlayer()
    {
        if (availablePlayers.Count > 0)
        {
            VideoPlayer player = availablePlayers.Dequeue();
            player.gameObject.SetActive(true);
            activePlayers.Add(player);
            return player;
        }
        else
        {
            // 풀이 부족하면 새로 생성 (최적화 관점에서 추천하지 않음, 풀 크기 조절 필요)
            Debug.LogWarning("VideoPlayer pool exhausted, creating new player.");
            GameObject obj = Instantiate(videoPlayerPrefab, transform);
            VideoPlayer player = obj.GetComponent<VideoPlayer>();
            player.gameObject.SetActive(true);
            activePlayers.Add(player);
            return player;
        }
    }

    public void ReturnPlayer(VideoPlayer player)
    {
        player.Stop();
        player.clip = null; // 클립 해제
        player.gameObject.SetActive(false);
        activePlayers.Remove(player);
        availablePlayers.Enqueue(player);
    }
}
```

#### 2.2. 복잡한 분기 로직 및 상태 관리: 내러티브 그래프와 스크립터블 오브젝트
**문제:** 수많은 선택지, 이에 따른 이야기 분기, 그리고 플레이어의 행동에 따라 변하는 게임 상태를 어떻게 일관성 있게 관리할까요? 이 부분이 꼬이면 디버깅은 물론, 새로운 콘텐츠 추가도 힘들어집니다.

**해결책:**
*   **내러티브 그래프 시스템:** 모든 비디오 시퀀스, 선택지, 그리고 다음 시퀀스로의 연결을 노드 기반의 그래프 형태로 설계합니다. 이 그래프는 `ScriptableObject`로 구성하여 에디터에서 쉽게 편집하고 관리할 수 있도록 합니다.
*   **Game State Manager (Singleton):** 플레이어의 선택, 획득한 아이템, 캐릭터와의 관계 등 모든 게임 상태를 하나의 중앙 관리 시스템에서 제어합니다. 이 상태는 다음 비디오 시퀀스를 결정하는 데 중요한 역할을 합니다.
*   **Decision Tree/FSM 활용:** 이야기의 흐름을 유한 상태 기계(FSM)나 의사 결정 트리(Decision Tree)로 모델링하여, 현재 상태와 플레이어의 선택에 따라 어떤 다음 노드로 이동할지 명확하게 정의합니다.

**아키텍처 다이어그램 (개념적):**

```mermaid
graph TD
    A[Game Manager] --> B[Narrative Flow Manager]
    B --> C{Current Narrative Node (ScriptableObject)}
    C --> D[Video Player Manager]

    C --> F[Game State Manager (Singleton)]
    F --> B

    G -- "Preload/Load Video Clips" --> D

    F -- "State Data" --> C

```

#### 2.3. 매끄러운 전환과 입력 반응성: 비동기 로딩과 렌더링 최적화
**문제:** 비디오 전환 시 검은 화면, 로딩 스피너, 입력 씹힘은 플레이어 몰입을 심각하게 해칩니다. 영화 같은 경험을 위해서는 티 안 나는 전환이 필수입니다.

**해결책:**
*   **`SceneManager.LoadSceneAsync` 및 `AllowSceneActivation`:** 만약 각 시퀀스를 별도의 씬으로 구성했다면, `LoadSceneAsync`를 사용하고 `allowSceneActivation = false`로 설정하여 백그라운드에서 다음 씬을 로드한 후, 모든 준비가 완료되었을 때 `allowSceneActivation = true`로 전환하여 매끄럽게 장면을 바꿉니다.
*   **커스텀 비디오 전환:** `VideoPlayer`의 `renderMode`를 `RenderTexture`로 설정한 뒤, 두 개의 `RenderTexture`를 사용하여 현재 비디오와 다음 비디오를 동시에 준비합니다. 그 후 커스텀 쉐이더를 이용하여 페이드 인/아웃, 와이프 등 영화적인 전환 효과를 구현합니다. 이때 GPU 친화적인 쉐이더를 사용하는 것이 중요합니다.
*   **입력 버퍼링:** 플레이어의 입력은 실시간으로 받아서 버퍼에 저장해두고, 시스템이 다음 선택 지점에 도달했을 때 버퍼의 입력을 처리하는 방식으로 입력 손실을 방지합니다.

**렌더링 전환 로직 (개념적):**
```csharp
public class VideoTransitionManager : MonoBehaviour
{
    public RawImage currentVideoDisplay; // 현재 비디오 렌더텍스처를 표시할 UI
    public RawImage nextVideoDisplay;    // 다음 비디오 렌더텍스처를 표시할 UI
    public Material transitionMaterial;  // 전환 효과를 위한 쉐이더 (예: dissolve)

    private RenderTexture rtCurrent, rtNext;
    private VideoPlayer vpCurrent, vpNext;

    void Awake()
    {
        // 렌더텍스처 초기화 (풀링 고려)
        rtCurrent = new RenderTexture(1920, 1080, 24);
        rtNext = new RenderTexture(1920, 1080, 24);
        currentVideoDisplay.texture = rtCurrent;
        nextVideoDisplay.texture = rtNext;

        // VideoPlayer 셋업 (VideoPlayerPool에서 가져온다고 가정)
        vpCurrent = VideoPlayerPool.Instance.GetPlayer();
        vpNext = VideoPlayerPool.Instance.GetPlayer();
        vpCurrent.targetTexture = rtCurrent;
        vpNext.targetTexture = rtNext;
        nextVideoDisplay.gameObject.SetActive(false); // 다음 비디오는 숨겨둠
    }

    public async Task PerformTransition(VideoClip newClip)
    {
        // 다음 비디오 플레이어에 클립 설정 및 버퍼링 시작
        vpNext.clip = newClip;
        vpNext.Prepare();
        while (!vpNext.isPrepared)
        {
            await Task.Yield(); // 준비될 때까지 기다림
        }

        // 전환 시작
        nextVideoDisplay.gameObject.SetActive(true);
        vpNext.Play();

        float transitionDuration = 1.0f; // 전환 시간
        float timer = 0f;
        while (timer < transitionDuration)
        {
            timer += Time.deltaTime;
            float t = timer / transitionDuration;
            transitionMaterial.SetFloat("_TransitionProgress", t); // 쉐이더 파라미터 업데이트
            await Task.Yield(); // 한 프레임 대기
        }

        // 전환 완료 후, current/next 역할 교체
        VideoPlayerPool.Instance.ReturnPlayer(vpCurrent);
        vpCurrent = vpNext;
        rtCurrent = rtNext; // 렌더텍스처도 교체
        currentVideoDisplay.texture = rtCurrent;
        nextVideoDisplay.gameObject.SetActive(false); // 이전 비디오 숨김

        // 다음 렌더텍스처를 새로 할당하거나 풀에서 가져옴
        rtNext = new RenderTexture(1920, 1080, 24);
        vpNext = VideoPlayerPool.Instance.GetPlayer();
        vpNext.targetTexture = rtNext;
    }
}
```

### 3. 시니어 개발자의 쓴소리 & 단맛: "프로파일러는 친구다, 그러나 가끔은 적이 되기도 한다"

제가 이 분야에서 얻은 가장 큰 교훈은 **"최적화는 선택이 아니라 생존이다"** 라는 것입니다. 특히 인터랙티브 무비처럼 자원을 많이 쓰는 장르에서는요. 처음부터 완벽한 최적화를 할 수는 없지만, 개발 초기부터 성능에 대한 의식을 가지고 설계하는 것이 중요합니다.

*   **"프로파일러는 우리의 가장 친한 친구입니다."** 유니티 프로파일러는 GC.Alloc, CPU 스파이크, 메모리 누수 등을 잡아내는 데 핵심적인 도구입니다. 비디오 로딩, 재생, 전환 각 단계마다 프로파일링을 해보세요. 예상치 못한 병목 지점을 발견하고 "아차!" 싶을 때가 많을 겁니다. 특히 대규모 `Addressables` 로딩 시 발생하는 GC 스파이크는 늘 경계해야 합니다.
*   **"메모리 관리: 보물을 캐는 광부의 마음으로."** 비디오는 말 그대로 '덩어리'입니다. 필요 없는 비디오 클립을 제때 `Addressables.Release` 하지 않으면 메모리가 감당할 수 없게 불어납니다. LRU(Least Recently Used) 캐시 정책 등을 도입하여 사용 빈도가 낮은 자원을 자동으로 언로드하는 시스템을 구축하는 것도 좋은 방법입니다.
*   **"비동기의 미학: 메인 스레드를 자유롭게."** `async/await` 패턴은 유니티에서 비동기 작업을 처리하는 강력한 도구입니다. 무거운 파일 로딩, 네트워크 요청 등을 메인 스레드에서 분리하여 UI가 멈추거나 프레임 드랍이 발생하는 것을 막을 수 있습니다. 다만, `Task` 사용 시 `UnityEngine.Object`에 접근할 때는 메인 스레드로 돌아와야 한다는 점을 잊지 마세요. `UniTask` 같은 라이브러리는 유니티 환경에서 `async/await`를 더 유연하게 사용할 수 있도록 돕습니다.
