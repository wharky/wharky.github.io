### 1. 문제: "잦은 객체 생성/삭제가 프레임을 잡아먹는다!" (GC 압박)
우리 모두 경험해보지 않았나요? 게임 중에 총알이 나가거나 이펙트가 터질 때마다 `Instantiate`와 `Destroy`를 반복하다가, 갑자기 게임이 멈칫하는 현상. 바로 가비지 컬렉터(GC)가 부과하는 과세입니다. 짧은 순간에도 수많은 객체가 생성되고 소멸되는 환경에서 GC는 성능의 가장 큰 적이 됩니다. 마치 언리얼 블루프린트에서 매 프레임마다 불필요한 노드 생성과 삭제를 반복하는 것과 같죠. 블루프린트는 한 번 만들어진 노드 그래프가 계속 실행되는 것을 전제로 합니다.

*   **해결책**: **오브젝트 풀링(Object Pooling)**
    사용할 객체들을 미리 생성해두고 필요할 때 꺼내 쓰고, 다 쓴 후에는 파괴하는 대신 다시 풀(Pool)에 반환하여 재활용하는 기법입니다.

*   **핵심 로직 (Pseudo-code)**:
    ```csharp
    public class ObjectPool<T> where T : MonoBehaviour
    {
        private readonly List<T> _pool = new List<T>();
        private readonly T _prefab;
        private readonly Transform _parent;
        private int _initialSize;

        public ObjectPool(T prefab, int initialSize, Transform parent = null)
        {
            _prefab = prefab;
            _initialSize = initialSize;
            _parent = parent;
            InitializePool();
        }

        private void InitializePool()
        {
            for (int i = 0; i < _initialSize; i++)
            {
                CreateNewObject();
            }
        }

        private T CreateNewObject()
        {
            T obj = GameObject.Instantiate(_prefab, _parent);
            obj.gameObject.SetActive(false); // 비활성화 상태로 풀에 보관
            _pool.Add(obj);
            return obj;
        }

        public T Get()
        {
            foreach (T obj in _pool)
            {
                if (!obj.gameObject.activeInHierarchy) // 비활성화된 객체 찾기
                {
                    obj.gameObject.SetActive(true);
                    return obj;
                }
            }
            // 풀에 여유가 없으면 새로 생성 (옵션)
            T newObj = CreateNewObject();
            newObj.gameObject.SetActive(true);
            return newObj;
        }

        public void Return(T obj)
        {
            obj.gameObject.SetActive(false);
            // 필요하다면 객체 상태 초기화 로직 추가
        }
    }

    // 사용 예시:
    // ObjectPool<Bullet> bulletPool = new ObjectPool<Bullet>(bulletPrefab, 100);
    // Bullet myBullet = bulletPool.Get();
    // myBullet.Shoot();
    // ...
    // bulletPool.Return(myBullet);
    ```

*   **나의 견해**: 오브젝트 풀링은 게임 개발의 ABC 중 하나입니다. 언리얼 블루프린트에서 잘 만든 함수나 매크로를 재활용하듯, 유니티에서는 객체를 재활용하는 거죠. 초기에 조금 번거롭더라도, 이 패턴을 도입하면 GC 스파이크는 거의 사라지고 훨씬 부드러운 플레이를 경험할 수 있습니다. 특히 모바일 게임에서는 생명줄이나 다름없죠. "미리 준비하고, 아껴 쓰고, 나눠 쓰는" 자세가 중요합니다.

### 2. 문제: "매번 `GetComponent`를 호출해서 컴포넌트를 찾는다?" (불필요한 탐색 비용)
`Update`나 `LateUpdate` 같은 매 프레임 호출되는 함수 안에서 `GetComponent<T>()`를 사용하는 건 마치 언리얼 블루프린트에서 매번 `Get Player Character` 노드를 호출하여 특정 컴포넌트를 캐스팅하려 하는 것과 같습니다. 이 함수는 생각보다 비싼 연산입니다. 게임 오브젝트 계층 구조를 탐색해야 하거든요.

*   **해결책**: **컴포넌트 참조 캐싱(Caching Component References)**
    `Awake()`나 `Start()`에서 필요한 컴포넌트의 참조를 한 번만 가져와 변수에 저장해두고, 이후에는 이 저장된 참조를 사용하는 것입니다.

*   **핵심 로직 (Pseudo-code)**:
    ```csharp
    public class MyPlayerController : MonoBehaviour
    {
        private Rigidbody _rb; // private 필드로 선언하여 외부 접근 제한

        void Awake()
        {
            // 컴포넌트 참조를 한 번만 가져와 캐싱
            _rb = GetComponent<Rigidbody>(); 
            if (_rb == null)
            {
                Debug.LogError("Rigidbody 컴포넌트를 찾을 수 없습니다!");
            }
        }

        void FixedUpdate()
        {
            // 캐싱된 참조를 사용하여 물리 연산 수행
            if (_rb != null)
            {
                _rb.AddForce(transform.forward * 10f);
            }
        }
    }
    ```

*   **나의 견해**: 이건 너무나도 기본 중의 기본이라 말하기도 민망할 정도입니다. 하지만 놀랍게도 초보 개발자뿐만 아니라 가끔은 숙련된 개발자들도 놓치는 부분입니다. 언리얼 블루프린트에서는 변수를 직접 연결하는 방식이 기본이죠? 특정 액터의 컴포넌트를 참조할 때도 `Get Component` 노드를 거친 후 `Set` 노드로 변수에 저장해두고 사용하는 것이 모범 사례입니다. 유니티에서도 마찬가지입니다. 한 번의 명확한 연결로 이후의 모든 불필요한 탐색을 없애는 것, 이것이 바로 효율적인 설계의 시작입니다.

### 3. 문제: "UI가 느리다고? 왜?" (UI Canvas Rebuild 비용)
유니티의 UI 시스템은 강력하지만, 잘못 사용하면 엄청난 성능 저하를 일으킬 수 있습니다. 특히 동적인 UI 요소가 많거나, 자주 텍스트가 바뀌는 경우, UI 요소의 속성이 변경될 때마다 전체 캔버스(또는 관련된 부분)가 다시 그려지는 '메시 리빌드(Mesh Rebuild)'가 발생하여 CPU에 큰 부담을 줍니다. 마치 블루프린트에서 `Tick` 이벤트에 복잡한 UI 업데이트 로직을 매번 때려 박는 것과 비슷하죠.

*   **해결책**: **캔버스 분할 및 UI 요소 최소화, 배치(Batching) 최적화**
    1.  **캔버스 분할**: 자주 변경되는 UI와 정적인 UI를 다른 캔버스에 분리합니다. 작은 변경이 전체 캔버스를 다시 그리게 만드는 것을 방지합니다.
    2.  **`Layout Group` & `Content Size Fitter` 주의**: 이 컴포넌트들은 유연하지만, 변경 시 UI 재계산을 유발하기 쉬우므로 필요한 경우에만 사용하고, 가능하다면 수동으로 배치하는 것을 고려합니다.
    3.  **텍스트 업데이트 최소화**: 매 프레임 텍스트를 업데이트하는 대신, 꼭 필요할 때만 업데이트하도록 로직을 설계합니다. 예를 들어, FPS 카운터는 0.1초마다 업데이트하는 것으로 충분할 수 있습니다.
    4.  **`Graphic Raycaster` 최적화**: 화면 전체에 퍼져있는 여러 개의 `Graphic Raycaster`는 불필요한 Raycast 연산을 유발합니다. 필요한 UI 그룹에만 Raycaster를 두거나, 레이어 마스크를 잘 활용하여 체크 범위를 제한합니다.

*   **핵심 로직 (아키텍처 설명)**:
    *   **Layered Canvas Structure**:
        ```
        Root Canvas (Screen Space Overlay)
        ├── Static UI Canvas (e.g., Background, Borders)
        │   └── Image (Logo), Text (Game Title)
        ├── Dynamic UI Canvas (e.g., HP Bar, Score, Inventory)
        │   ├── PlayerHUD Canvas
        │   │   ├── Text (Score), Slider (HP)
        │   │   └── Image (Ability Icon, frequently changed)
        │   └── PopUp Canvas (for temporary dialogs)
        │       └── Panel (Confirmation Dialog)
        └── World Space Canvas (if any, for floating names)
        ```
    *   **Text Update Optimization**:
        ```csharp
        public class OptimizedTextUpdater : MonoBehaviour
        {
            public TextMeshProUGUI targetText; // 또는 UnityEngine.UI.Text
            private string _currentValue = "";

            public void SetText(string newValue)
            {
                if (_currentValue != newValue) // 값이 변경되었을 때만 업데이트
                {
                    _currentValue = newValue;
                    targetText.text = _currentValue;
                }
            }
        }
        ```

*   **나의 견해**: "UI는 괜찮겠지" 하는 안일한 생각은 개발 막바지에 지옥을 선물합니다. 언리얼 블루프린트의 위젯(UMG) 또한 복잡해지면 성능 문제가 발생하기 쉽습니다. 핵심은 "불필요한 작업을 하지 않는 것"입니다. 유니티의 캔버스는 일종의 렌더링 단위인데, 이 단위를 잘게 쪼개서 변경의 파급 효과를 최소화해야 합니다. 정적인 부분과 동적인 부분을 명확히 분리하고, 데이터 바인딩 시에도 'dirty check' 로직을 넣어 변경된 부분만 업데이트하는 습관을 들이세요.

### 4. 문제: "물리가 너무 무겁다!" (Collision Matrix & Raycasting)
특히 복잡한 환경에서 수많은 콜라이더와 리지드바디가 상호작용할 때 물리 연산은 엄청난 부하를 유발합니다. 매 프레임마다 모든 가능한 충돌 조합을 검사하는 것은 비효율적이죠. 블루프린트에서 불필요한 콜리전 프리셋이나 복잡한 라인 트레이스 로직을 남발하는 것과 유사합니다.

*   **해결책**: **레이어 기반 충돌 매트릭스, `NonAlloc` 메서드 활용**
    1.  **`Physics.queriesHitBackfaces` & `Physics.queriesHitTriggers`**: 불필요한 연산을 줄이기 위해 이 값들을 필요한 경우에만 `true`로 설정합니다.
    2.  **레이어 기반 충돌 매트릭스**: `Edit -> Project Settings -> Physics` 또는 `Physics 2D`에서 레이어 간의 충돌 검사 여부를 설정합니다. 특정 레이어 오브젝트끼리는 아예 충돌 검사를 하지 않도록 설정하여 물리 연산을 크게 줄일 수 있습니다.
    3.  **`NonAlloc` 메서드**: `Physics.OverlapSphereNonAlloc`, `Physics.RaycastNonAlloc` 등 `NonAlloc` 접미사가 붙은 메서드들은 결과값을 미리 할당된 배열에 저장하여 가비지 생성을 줄입니다. 대규모 물리 쿼리를 자주 수행할 때 필수적입니다.
    4.  **`Rigidbody` 슬립 임계값 조정**: 움직임이 거의 없는 `Rigidbody`는 '슬립' 상태로 전환되어 연산 부하를 줄입니다. 이 임계값을 조정하여 더 빨리 슬립 상태로 진입하게 할 수 있습니다.

*   **핵심 로직 (설명)**:
    *   **Collision Matrix Configuration (Example)**:
        |           | Player | Enemy | Environment | Projectile | Trigger |
        | :-------- | :----- | :---- | :---------- | :--------- | :------ |
        | **Player**  | ✔️     | ✔️    | ✔️          | ✔️         | ✔️      |
        | **Enemy**   | ✔️     | ✔️    | ✔️          | ✔️         | ✔️      |
        | **Environment**| ✔️     | ✔️    | ✔️          | ✔️         |         |
        | **Projectile**| ✔️     | ✔️    |             |            |         |
        | **Trigger** | ✔️     | ✔️    |             |            |         |
        이런 식으로 `Projectile`은 `Environment`와 `Trigger`와는 충돌 검사를 하지 않도록 설정하는 등, 불필요한 충돌을 제거합니다.

    *   **`OverlapSphereNonAlloc` Usage**:
        ```csharp
        public class AOEDamageDealer : MonoBehaviour
        {
            public float radius = 5f;
            public int maxColliders = 10;
            private Collider[] _overlapResults;

            void Awake()
            {
                _overlapResults = new Collider[maxColliders]; // 미리 배열 할당
            }

            public void DealDamage()
            {
                int numColliders = Physics.OverlapSphereNonAlloc(transform.position, radius, _overlapResults);

                for (int i = 0; i < numColliders; i++)
                {
                    // _overlapResults[i]에 있는 Collider들에 피해를 주거나 로직 수행
                    Debug.Log($"Hit: {_overlapResults[i].name}");
                }
            }
        }
        ```

*   **나의 견해**: "물리 연산은 공짜가 아니다!"를 항상 명심해야 합니다. 특히 모바일이나 저사양 타겟에서는 더욱 그렇죠. 블루프린트에서도 `Collision Presets`을 통해 어떤 액터와 충돌할지 세밀하게 설정하듯, 유니티에서는 레이어 매트릭스를 적극적으로 활용하여 쓸데없는 계산을 없애야 합니다. 그리고 Raycast/Overlap 같은 쿼리 함수는 무조건 `NonAlloc`을 기본으로 사용하세요. 이게 진정한 프로의 습관입니다.

### 5. 문제: "내 `Update` 루프가 너무 복잡해!" (Monobehaviour Bloat & Execution Order)
수십, 수백 개의 `MonoBehaviour` 스크립트가 각자의 `Update`, `LateUpdate`를 호출하면서 생기는 오버헤드는 간과하기 쉽습니다. 게다가 이들의 실행 순서는 종종 예측하기 어렵습니다. 언리얼 블루프린트에서는 이벤트 그래프에서 노드 연결로 명확한 실행 흐름을 제어하죠. 하지만 유니티는 기본적으로 각 스크립트가 독립적으로 작동합니다.

*   **해결책**: **커스텀 업데이트 매니저, 그리고 `Job System` & Burst Compiler로의 확장**
    1.  **중앙 집중식 업데이트 매니저**: 모든 업데이트 로직을 하나의 `MonoBehaviour` (혹은 `ScriptableObject`)에서 관리하는 매니저를 만들고, 다른 객체들은 이 매니저에 자신을 등록하여 업데이트를 받는 방식입니다. 이렇게 하면 `MonoBehaviour`의 `Update` 호출 오버헤드를 줄이고, 실행 순서를 명확하게 제어할 수 있습니다.
    2.  **`Job System` & Burst Compiler**: 고성능이 필요한 병렬 연산의 경우, 유니티의 `Job System`과 `Burst Compiler`를 사용하여 C# 코드를 네이티브 코어에 최적화된 고성능 코드로 변환할 수 있습니다. 이는 데이터 지향 기술 스택(DOTS)의 핵심 요소이기도 합니다.

*   **핵심 로직 (Pseudo-code & Concept)**:
    *   **Custom Update Manager**:
        ```csharp
        public interface IUpdatable
        {
            void OnUpdate(float deltaTime);
        }

        public class GameManager : MonoBehaviour
        {
            private List<IUpdatable> _updatables = new List<IUpdatable>();

            public static GameManager Instance { get; private set; } // Singleton

            void Awake() { Instance = this; }

            public void RegisterUpdatable(IUpdatable updatable)
            {
                _updatables.Add(updatable);
            }

            public void UnregisterUpdatable(IUpdatable updatable)
            {
                _updatables.Remove(updatable);
            }

            void Update()
            {
                float dt = Time.deltaTime;
                for (int i = 0; i < _updatables.Count; i++)
                {
                    _updatables[i].OnUpdate(dt);
                }
            }
        }

        public class MyEnemyLogic : MonoBehaviour, IUpdatable
        {
            void OnEnable()
            {
                GameManager.Instance.RegisterUpdatable(this); // 활성화 시 매니저에 등록
            }

            void OnDisable()
            {
                GameManager.Instance.UnregisterUpdatable(this); // 비활성화 시 등록 해제
            }

            public void OnUpdate(float deltaTime)
            {
                // 적 로직 처리
                transform.position += Vector3.forward * deltaTime;
            }
        }
        ```

    *   **Job System & Burst Compiler (Concept)**:
        복잡한 계산(예: 수천 개의 유닛 AI 경로 계산, 파티클 시뮬레이션)을 메인 스레드에서 분리하여 병렬로 실행하고, `Burst Compiler`가 이를 초고속으로 처리하도록 합니다. 이는 언리얼의 멀티 스레딩 아키텍처나 C++에서의 `Task Graph` 시스템과 유사한 접근 방식입니다.

*   **나의 견해**: `Update` 함수는 우리의 친구이면서도 가장 교활한 적이 될 수 있습니다. 저는 언리얼 블루프린트의 `Tick` 이벤트를 남발하는 것과 같은 맥락이라고 봅니다. 모든 것을 `Update`에 때려 박는 것은 게으른 개발의 증거입니다. 커스텀 업데이트 매니저는 조금 더 구조화된 방식이며, 나아가 `Job System`과 `Burst Compiler`는 유니티에서 극한의 성능을 뽑아내고 싶을 때 반드시 고려해야 할 '게임 체인저'입니다. 이건 단순한 최적화 트릭을 넘어선 '아키텍처 설계'의 영역입니다. 처음에는 어렵게 느껴질 수 있지만, 미래를 위한 투자라고 생각하세요.
