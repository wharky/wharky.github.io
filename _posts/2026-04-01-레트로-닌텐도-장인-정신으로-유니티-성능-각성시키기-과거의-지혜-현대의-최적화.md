### 1. 문제: 흩어진 데이터, 지쳐가는 CPU — 닌텐도의 '밀도 높은 배치'에서 배우다

**문제 인식**: 현대 객체 지향 프로그래밍(OOP)은 코드의 가독성과 유지보수성을 높여주지만, 데이터를 메모리에 흩뿌려 놓는 경향이 있습니다. 각 오브젝트가 자체 데이터를 여기저기 들고 있으면, CPU 캐시 미스(Cache Miss)가 빈번하게 발생하고, 이는 곧 성능 저하로 이어지죠. 특히 수많은 게임 오브젝트가 동시에 업데이트되는 상황에서는 이 문제가 극대화됩니다.

**레트로 닌텐도의 지혜**: NES나 SNES 시절, 개발자들은 기가바이트 단위의 램은커녕, 메가바이트도 아닌 *킬로바이트* 단위의 램을 가지고 씨름했습니다. 데이터의 배치는 생존의 문제였죠. 그들은 관련 데이터를 최대한 가깝게, 그리고 연속적으로 배치하여 CPU가 효율적으로 접근할 수 있도록 설계했습니다. 예를 들어, 수백 개의 스프라이트가 움직여야 한다면, 각 스프라이트의 위치, 속도, 애니메이션 프레임 등 핵심 정보를 `struct`의 배열 형태로 꽉꽉 채워 넣었을 겁니다. OOP의 '데이터와 행위의 캡슐화'보다는 '데이터의 효율적인 접근'이 우선이었죠.

**유니티 솔루션: 데이터 지향 설계(Data-Oriented Design)의 부활**

유니티도 이 문제의식에 공감하여 ECS(Entity Component System), 즉 DOTS(Data-Oriented Technology Stack)를 도입했습니다. ECS는 데이터를 중심으로 설계하여 CPU 캐시 효율을 극대화하는 방식입니다. 하지만 당장 DOTS로 전환하기 어렵다면, 기존 Monobehavior 환경에서도 이 원칙을 적용할 수 있습니다.

**핵심 로직**:
*   **객체 대신 구조체 배열 활용**: `Transform`이나 `Rigidbody` 같은 컴포넌트 데이터에 자주 접근해야 한다면, 각 컴포넌트에서 필요한 핵심 데이터만 `struct`로 묶어 별도의 `NativeArray`나 일반 `List<MyComponentData>`에 저장하고, 업데이트 로직을 Job System이나 일반 `for` 루프에서 순회하게 만드세요.
*   **데이터의 압축 및 재활용**: 예를 들어, 적 캐릭터들의 `Health`, `Position`, `Velocity`가 각각 다른 스크립트에 흩어져 있다면, 이를 하나의 `struct`로 묶고 이 `struct`의 배열을 한 곳에서 관리하는 식입니다.

```csharp
// 문제 상황 (OOP 스타일, 데이터 분산)
public class Enemy : MonoBehaviour
{
    public float health;
    public Vector3 position;
    public Vector3 velocity;
    // ... 기타 데이터
}

// 해결책 (DOD 스타일, 데이터 집약)
public struct EnemyData
{
    public float health;
    public Vector3 position;
    public Vector3 velocity;
    public int enemyId; // 참조를 위한 ID
    // ... 기타 필요한 데이터
}

public class EnemyManager : MonoBehaviour
{
    private List<EnemyData> _enemyDataList = new List<EnemyData>();
    private List<EnemyRenderer> _enemyRenderers = new List<EnemyRenderer>(); // 시각적 표현은 별도 관리

    // Update 루프 (의사 코드)
    void Update()
    {
        for (int i = 0; i < _enemyDataList.Count; i++)
        {
            EnemyData currentData = _enemyDataList[i];
            currentData.position += currentData.velocity * Time.deltaTime;
            _enemyDataList[i] = currentData; // struct는 값 타입이므로 다시 할당
            _enemyRenderers[i].SetPosition(currentData.position); // 시각적 업데이트
        }
    }

    // 새 적 생성 시
    public void AddEnemy(Vector3 startPos, Vector3 initialVel)
    {
        EnemyData newData = new EnemyData
        {
            health = 100f,
            position = startPos,
            velocity = initialVel,
            enemyId = _enemyDataList.Count
        };
        _enemyDataList.Add(newData);

        // 시각적 표현은 인스턴싱 등으로 별도 처리
        GameObject enemyGO = Instantiate(enemyPrefab);
        enemyGO.transform.position = startPos;
        _enemyRenderers.Add(enemyGO.GetComponent<EnemyRenderer>());
    }
}
```
이런 접근은 `Enemy` 오브젝트 하나하나에 대한 `GetComponent` 호출이나 `Transform` 접근을 줄이고, 데이터를 메모리상에서 연속적으로 처리하게 하여 훨씬 효율적인 CPU 사용을 가능하게 합니다.

### 2. 문제: 무한정 늘어나는 드로우 콜과 리소스 — 닌텐도의 '제한적이지만 똑똑한 활용'

**문제 인식**: 유니티에서 게임을 만들다 보면 씬에 오브젝트를 마구 던져 넣게 됩니다. 각 오브젝트마다 재질(Material)이 다르고, 텍스처가 다르고, 심지어 그림자 설정까지 다르다면? CPU는 GPU에게 너무 많은 '드로우 콜'을 날리느라 바쁘고, 이는 곧 병목 현상으로 이어집니다. 또한, 방대한 그래픽 리소스는 빌드 용량을 늘리고 로딩 시간을 지연시키는 주범이 되죠.

**레트로 닌텐도의 지혜**: 레트로 게임들은 드로우 콜이라는 개념은 없었지만, 한 프레임에 화면에 뿌릴 수 있는 스프라이트나 타일의 개수가 하드웨어적으로 엄격하게 제한되어 있었습니다. 컬러 팔레트도 몇십 개 수준이었고요. 개발자들은 이 제약을 극복하기 위해 눈물겨운 노력을 했습니다.
*   **팔레트 스왑**: 마리오와 루이지가 같은 스프라이트 데이터에 다른 팔레트 인덱스만 적용하여 색만 바꾸는 방식.
*   **스프라이트 재활용**: 적 캐릭터가 화면에서 사라지면 그 스프라이트 슬롯을 즉시 다른 적에게 재할당.
*   **타일맵 기반**: 복잡한 배경도 작은 타일 조각들을 조합하여 메모리와 렌더링 오버헤드를 최소화.
*   **애셋 버지팅**: 롬 용량이 곧 게임의 크기였기에, 모든 텍스처와 사운드, 애니메이션 프레임을 철저히 예산화하고 압축했습니다.

**유니티 솔루션: GPU 인스턴싱과 애셋 관리의 마스터가 되라!**

유니티에서도 닌텐도 개발자들의 정신을 이어받아 드로우 콜을 줄이고 리소스를 효율적으로 관리할 수 있는 다양한 방법들이 있습니다.

**핵심 로직**:
*   **드로우 콜 최적화**:
    *   **GPU 인스턴싱**: 동일한 메시와 재질을 사용하는 오브젝트가 수백 개 있다면, GPU 인스턴싱을 활성화하세요. 단 한 번의 드로우 콜로 수많은 오브젝트를 렌더링할 수 있습니다.
    *   **스프라이트 아틀라스**: 2D 게임이라면, 여러 스프라이트를 하나의 큰 텍스처(아틀라스)에 모아 사용하세요. 텍스처 스왑 횟수를 줄여 드로우 콜을 줄일 수 있습니다.
    *   **Static/Dynamic Batching**: 정적 배치(Static Batching)는 움직이지 않는 오브젝트들을 미리 합쳐 하나의 메쉬로 만들고, 동적 배치(Dynamic Batching)는 특정 조건(적은 정점 수, 같은 재질)을 만족하는 작은 메쉬들을 런타임에 합쳐줍니다.
    *   **SRP Batcher**: URP나 HDRP를 사용한다면 SRP Batcher를 활용하여 재질의 프로퍼티 블록을 효율적으로 관리할 수 있습니다.
*   **리소스 관리 및 애셋 버지팅**:
    *   **오브젝트 풀링(Object Pooling)**: 총알이나 적 유닛처럼 자주 생성되고 파괴되는 오브젝트들은 미리 만들어두고 재활용하는 오브젝트 풀링을 사용하세요. `Instantiate`/`Destroy`의 오버헤드를 크게 줄일 수 있습니다.
    *   **Addressables**: 유니티 Addressables를 사용하면 애셋을 빌드 시점에 패키징하고 런타임에 필요할 때만 로드/언로드할 수 있습니다. 롬 카트리지의 용량 제약을 극복하려던 닌텐도 개발자들의 노력이 현대에는 Addressables로 발현된 셈이죠.
    *   **텍스처 압축 및 미드맵**: 모든 텍스처에 적절한 압축 설정(ETC2, ASTC 등)을 적용하고, 미드맵(Mipmap)을 사용해 거리에 따라 해상도를 낮춰 GPU 메모리 사용량을 줄이세요.

```csharp
// 오브젝트 풀링 의사 코드
public class BulletPoolManager : MonoBehaviour
{
    public GameObject bulletPrefab;
    private Queue<GameObject> _bulletPool = new Queue<GameObject>();
    public int poolSize = 10;

    void Start()
    {
        for (int i = 0; i < poolSize; i++)
        {
            GameObject bullet = Instantiate(bulletPrefab);
            bullet.SetActive(false);
            _bulletPool.Enqueue(bullet);
        }
    }

    public GameObject GetBullet()
    {
        if (_bulletPool.Count > 0)
        {
            GameObject bullet = _bulletPool.Dequeue();
            bullet.SetActive(true);
            return bullet;
        }
        else
        {
            // 풀이 부족할 경우 추가 생성 (선택 사항)
            GameObject newBullet = Instantiate(bulletPrefab);
            return newBullet;
        }
    }

    public void ReleaseBullet(GameObject bullet)
    {
        bullet.SetActive(false);
        _bulletPool.Enqueue(bullet);
    }
}
```
이러한 기법들은 단순한 트릭이 아니라, 제한된 자원 속에서 최적의 퍼포먼스를 뽑아내기 위한 근본적인 사고방식의 전환입니다. 닌텐도 개발자들이 그랬던 것처럼, 우리도 매 순간 '이 리소스는 정말 필요한가?', '더 효율적인 방법은 없을까?'를 고민해야 합니다.

### 3. 문제: 복잡한 알고리즘의 오버헤드 — 닌텐도의 '우아한 단순성'

**문제 인식**: 현대 게임 개발에서는 복잡한 물리 시뮬레이션, 정교한 AI, 실시간 길 찾기 등 고도화된 알고리즘을 쉽게 적용하곤 합니다. 하지만 이 모든 것이 CPU/GPU에 상당한 부담을 줍니다. 때로는 과도하게 정교한 알고리즘이 플레이 경험에 큰 영향을 주지 않으면서 성능만 잡아먹는 경우가 발생합니다.

**레트로 닌텐도의 지혜**: 슈퍼 마리오의 점프 물리학은 완벽한 현실 고증을 거친 물리 엔진의 결과물이 아닙니다. 제한된 연산 자원 안에서 플레이어에게 '납득 가능한' 재미있는 점프 경험을 제공하기 위해, 닌텐도 개발자들은 몇 개의 간단한 수학 공식과 상수 값을 조합하여 점프의 궤적을 제어했습니다. `속도 = 초기속도 + 중력 * 시간` 이라는 간단한 공식과 몇 가지 예외 처리만으로도 충분히 몰입감 있는 경험을 만들어낼 수 있었죠. 적 AI 역시 복잡한 행동 트리나 머신러닝 대신, '플레이어가 가까이 오면 돌진', '벽에 부딪히면 방향 전환' 같은 단순한 규칙 기반의 로직으로 충분했습니다.

**유니티 솔루션: 프로파일링 기반의 '적절한' 알고리즘 선택**

"과유불급"이라는 말이 있듯이, 유니티 최적화에서도 '적절한' 알고리즘을 선택하는 것이 중요합니다. 모든 문제에 가장 복잡하고 최신 기술을 적용할 필요는 없습니다.

**핵심 로직**:
*   **프로파일링이 핵심**: 어떤 알고리즘이 병목 현상을 일으키는지, 또는 특정 기능이 예상보다 많은 리소스를 소모하는지 정확히 파악해야 합니다. 유니티 프로파일러는 우리의 가장 친한 친구입니다.
*   **간단한 대안 고려**: A* 같은 복잡한 길 찾기 알고리즘이 필요 없는 상황이라면, 단순히 그리드 기반의 BFS/DFS나 심지어는 미리 계산된 경로(Precomputed Path)를 사용하는 것이 훨씬 효율적일 수 있습니다.
*   **경량 물리 시뮬레이션**: 특정 오브젝트(예: 잔해 파편, 나뭇잎)에 대해서는 유니티 기본 물리 엔진 대신, 직접 `Transform.position`을 제어하거나, 간단한 충돌 판정만을 수행하는 경량 물리 시뮬레이션을 구현하는 것을 고려해볼 수 있습니다.

```csharp
// 레트로 스타일의 간단한 캐릭터 컨트롤러 (물리 엔진 없이)
public class SimplePlayerController : MonoBehaviour
{
    public float moveSpeed = 5f;
    public float jumpForce = 10f;
    public float gravity = -20f; // 마리오식 중력
    private Vector3 _velocity;
    private bool _isGrounded;

    void Update()
    {
        // 횡 이동
        float horizontalInput = Input.GetAxis("Horizontal");
        _velocity.x = horizontalInput * moveSpeed;

        // 중력 적용
        _velocity.y += gravity * Time.deltaTime;

        // 점프 (땅에 있을 때만)
        if (Input.GetButtonDown("Jump") && _isGrounded)
        {
            _velocity.y = jumpForce;
            _isGrounded = false;
        }

        // 실제 이동
        transform.position += _velocity * Time.deltaTime;

        // 간단한 바닥 충돌 처리 (의사 코드)
        if (transform.position.y < 0) // 예시: y=0이 바닥이라고 가정
        {
            transform.position = new Vector3(transform.position.x, 0, transform.position.z);
            _velocity.y = 0;
            _isGrounded = true;
        }
    }
}
```
이런 접근 방식은 유니티의 강력한 물리 엔진을 사용하지 않아도, 플레이어가 원하는 핵심적인 '조작감'과 '반응성'을 충분히 제공할 수 있음을 보여줍니다. '정밀함'보다 '납득 가능한 경험'에 집중하는 것이죠.

### 시니어 개발자의 썰 (feat. 개인적인 견해)

저는 가끔 후배 개발자들에게 이런 농담을 합니다. "요즘 친구들은 100만 원짜리 컴퓨터로 코드를 돌리면서, 왜 10만 원짜리 컴퓨터를 기준으로 생각해야 하는지 이해를 못 해." 농담이지만, 여기에 핵심이 담겨 있습니다. 우리는 항상 최악의 상황, 즉 최소 사양의 기기에서도 게임이 잘 돌아갈 수 있도록 최적화를 고려해야 합니다.

예전에 한 프로젝트에서, 복잡한 UI 애니메이션 때문에 저사양 기기에서 프레임 드롭이 심각했던 적이 있습니다. 온갖 최신 UI 최적화 기법을 동원했지만 개선이 미미했죠. 결국 해결책은 UI 애니메이션의 '갯수'와 '복잡성' 자체를 줄이는 것이었습니다. 닌텐도 게임에서 과도하게 화려한 UI 애니메이션을 본 적이 있나요? 거의 없을 겁니다. 그들은 최소한의 움직임으로 최대한의 정보를 전달했죠. 그 순간, 저는 '최적화는 기술 이전에 마인드셋의 문제'라는 것을 다시 한번 깨달았습니다.

과거 닌텐도 개발자들은 마치 조각가가 돌덩이에서 불필요한 부분을 깎아내듯, 제한된 자원 속에서 '무엇을 남길 것인가'에 집중했습니다. 그들의 코드는 예술적인 압축과 효율의 결정체였죠. 유니티 최적화 역시 마찬가지입니다. 우리에게 필요한 것은 단순히 새로운 기술을 익히는 것을 넘어, 이러한 '제한적 사고'와 '문제 해결을 위한 근본적인 접근'을 내면화하는 것입니다.
