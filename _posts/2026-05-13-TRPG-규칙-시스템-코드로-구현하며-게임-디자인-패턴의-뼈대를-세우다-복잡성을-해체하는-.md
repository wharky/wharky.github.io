### 📜 문제: 규칙은 살아 숨 쉬는데, 코드는 왜 항상 뻣뻣할까?

### ✨ 해결: TRPG 규칙, 설계 패턴의 렌즈로 재탄생시키다

이런 문제를 해결하기 위해 저는 TRPG 규칙들을 코드로 옮길 때, 단순히 '구현'하는 것을 넘어 '설계'의 관점에서 접근하기 시작했습니다. 그리고 그 중심에는 언제나 **게임 디자인 패턴**들이 있었습니다. TRPG 규칙의 복잡성을 해체하고, 재사용 가능하며 유지보수가 쉬운 코드를 만드는 데 필요한 지혜를 얻을 수 있었죠.

자, 이제 몇 가지 핵심적인 TRPG 규칙 시스템을 코드로 어떻게 구현하고, 그 과정에서 어떤 디자인 패턴이 빛을 발하는지 심층적으로 파고들어 볼까요?

#### 🛠️ Case 1: 캐릭터 스탯 및 속성 시스템 – 속성(Attribute)과 수정자(Modifier)로 유연성을 확보하다

TRPG에서 캐릭터 스탯(힘, 민첩, 지능 등)은 기본값에 장비, 버프, 디버프, 종족 특성 등 수많은 요소에 의해 실시간으로 변동됩니다. 이걸 단순히 `character.strength = baseStrength + itemBonus + buffBonus;` 이런 식으로 계산하면, 계산 순서나 복잡한 조건(예: 특정 버프는 다른 버프에 영향을 받지 않음)을 처리하기가 불가능해집니다.

**문제**: 스탯 변경 로직이 분산되어 있고, 계산 순서가 복잡하며, 새로운 수정자(Modifier) 타입 추가 시 모든 스탯 계산 로직을 변경해야 합니다.

**해결**: **컴포넌트 패턴(Component Pattern)**과 **전략 패턴(Strategy Pattern)**을 결합하여 `Attribute` 클래스와 `AttributeModifier` 시스템을 설계합니다.

**핵심 로직**:

1.  **Attribute 클래스**: 각 스탯(힘, 민첩)을 표현하는 클래스로, 기본값을 가지고 여러 `AttributeModifier` 객체를 리스트 형태로 관리합니다.
2.  **AttributeModifier 클래스**: 스탯에 가해지는 모든 변경(아이템 보너스, 버프, 디버프)을 추상화합니다. `ModifierType` (예: `Flat`, `PercentageAdditive`, `PercentageMultiplicative`)을 정의하여 다양한 계산 전략을 적용할 수 있도록 합니다.
3.  **계산 순서 관리**: `AttributeModifier`에 `Order` 값을 부여하여, 수정자들이 적용되는 순서를 명시적으로 제어합니다.

```csharp
// C# 유사 코드
public enum ModifierType
{
    Flat,                   // 고정값 추가/제거
    PercentageAdditive,     // % 가산 (ex: +10% 힘)
    PercentageMultiplicative // % 곱산 (ex: 최종 힘에 1.2배)
    // 기타 필요한 Modifier 타입 추가 가능
}

public class AttributeModifier
{
    public float Value { get; private set; }
    public ModifierType Type { get; private set; }
    public int Order { get; private set; } // 적용 우선순위 (낮을수록 먼저 적용)
    public object Source { get; private set; } // 누가 이 수정자를 부여했는지 (아이템, 버프 등)

    public AttributeModifier(float value, ModifierType type, int order, object source = null)
    {
        Value = value;
        Type = type;
        Order = order;
        Source = source;
    }
}

public class GameAttribute
{
    private float _baseValue;
    private List<AttributeModifier> _modifiers = new List<AttributeModifier>();

    public string Name { get; private set; }

    public GameAttribute(string name, float baseValue)
    {
        Name = name;
        _baseValue = baseValue;
    }

    public void AddModifier(AttributeModifier modifier)
    {
        _modifiers.Add(modifier);
        _modifiers.Sort((a, b) => a.Order.CompareTo(b.Order)); // 우선순위에 따라 정렬
    }

    public void RemoveModifier(AttributeModifier modifier)
    {
        _modifiers.Remove(modifier);
    }

    public float GetFinalValue()
    {
        float finalValue = _baseValue;
        float percentageAdditiveSum = 0f;
        float percentageMultiplicativeProduct = 1f;

        // Order 1: Flat Modifiers
        foreach (var mod in _modifiers.Where(m => m.Type == ModifierType.Flat))
        {
            finalValue += mod.Value;
        }

        // Order 2: Percentage Additive Modifiers (먼저 합산)
        foreach (var mod in _modifiers.Where(m => m.Type == ModifierType.PercentageAdditive))
        {
            percentageAdditiveSum += mod.Value;
        }
        finalValue *= (1 + percentageAdditiveSum); // 한 번에 적용

        // Order 3: Percentage Multiplicative Modifiers (곱하기)
        foreach (var mod in _modifiers.Where(m => m.Type == ModifierType.PercentageMultiplicative))
        {
            finalValue *= (1 + mod.Value);
        }

        // TODO: Min/Max clamp, etc.

        return finalValue;
    }
}

// 사용 예시
// var characterStrength = new GameAttribute("Strength", 10);
// characterStrength.AddModifier(new AttributeModifier(5, ModifierType.Flat, 1, "Basic Sword")); // 검 장착
// characterStrength.AddModifier(new AttributeModifier(0.2f, ModifierType.PercentageAdditive, 2, "Heroic Buff")); // 버프
// Console.WriteLine($"Current Strength: {characterStrength.GetFinalValue()}"); // 10 + 5 = 15. 15 * (1 + 0.2) = 18
```

**제 생각**: 이 방식은 스탯 계산 로직을 `GameAttribute` 클래스 안에 캡슐화하고, 모든 변경 사항을 `AttributeModifier` 객체로 추상화하여 관리합니다. 새로운 버프나 장비가 추가되어도, 새로운 `AttributeModifier` 객체를 생성해서 리스트에 추가하기만 하면 됩니다. 스파게티 코드 대신 깔끔하고 확장 가능한 구조가 만들어지는 거죠. 마치 TRPG에서 GM이 플레이어 시트에 새로운 버프를 쓱쓱 적어 넣듯이 말입니다!

#### 🛠️ Case 2: 스킬 및 주문 시스템 – 명령(Command)과 전략(Strategy)으로 무한한 스킬을 창조하다

TRPG에서 스킬과 주문은 그 종류와 효과가 무궁무진합니다. '화염구'는 피해를 주고, '치유'는 체력을 회복시키고, '도발'은 적의 어그로를 끕니다. 이 모든 스킬을 각각 독립적인 함수로 구현하면 코드 중복과 유지보수 지옥이 펼쳐집니다.

**문제**: 스킬의 종류가 늘어날수록 코드량이 폭증하고, 스킬 발동 조건, 효과 계산, 타겟팅 로직 등이 뒤죽박죽 섞여 관리가 어렵습니다.

**해결**: **커맨드 패턴(Command Pattern)**을 활용하여 스킬 발동을 추상화하고, **전략 패턴(Strategy Pattern)**으로 스킬의 실제 효과를 캡슐화합니다.

**핵심 로직**:

1.  **SkillCommand 인터페이스/추상 클래스**: 모든 스킬이 공통으로 구현해야 할 `Execute()`, `CanExecute()` 등을 정의합니다. 이는 스킬 발동의 '명령' 그 자체를 나타냅니다.
2.  **구체적인 SkillCommand 클래스**: `FireballCommand`, `HealCommand` 등 각 스킬에 대한 구체적인 명령을 구현합니다. 이 클래스들은 스킬 발동에 필요한 파라미터(시전자, 대상)를 가지고 있습니다.
3.  **SkillEffectStrategy 인터페이스**: 스킬의 실제 효과(피해 계산, 치유량 계산, 버프 적용 등)를 추상화합니다. `CalculateDamageStrategy`, `ApplyBuffStrategy` 등이 될 수 있습니다.
4.  **스킬 데이터 주도 설계(Data-Driven Design)**: 실제 스킬 데이터(JSON, XML 등)에서 어떤 `SkillEffectStrategy`를 사용할지, 어떤 파라미터로 사용할지 정의합니다.

```csharp
// C# 유사 코드
// 1. 커맨드 패턴의 추상화: 모든 스킬의 기본 골격
public abstract class SkillCommand
{
    protected Character _caster;
    protected GameObject _target; // 타겟은 캐릭터, 지점, 오브젝트 등 다양할 수 있음

    public SkillCommand(Character caster, GameObject target)
    {
        _caster = caster;
        _target = target;
    }

    public abstract bool CanExecute(); // 마나, 쿨다운, 범위 등 검사
    public abstract void Execute();    // 스킬 발동 로직
}

// 2. 전략 패턴의 추상화: 스킬 효과의 종류
public interface ISkillEffectStrategy
{
    void ApplyEffect(Character caster, GameObject target, float[] parameters);
}

// 구체적인 전략: 피해 계산
public class DamageEffectStrategy : ISkillEffectStrategy
{
    public void ApplyEffect(Character caster, GameObject target, float[] parameters)
    {
        float baseDamage = parameters[0]; // 스킬 데이터에서 넘어온 기본 피해량
        // 추가 파라미터: 피해 유형, 치명타 확률 등

        float finalDamage = baseDamage + caster.GetAttribute("Strength").GetFinalValue() * 0.5f; // 예시
        target.GetComponent<HealthComponent>()?.TakeDamage(finalDamage);
        Console.WriteLine($"{caster.Name} deals {finalDamage} damage to {target.Name}!");
    }
}

// 구체적인 전략: 치유
public class HealEffectStrategy : ISkillEffectStrategy
{
    public void ApplyEffect(Character caster, GameObject target, float[] parameters)
    {
        float baseHeal = parameters[0];
        float finalHeal = baseHeal + caster.GetAttribute("Intelligence").GetFinalValue() * 0.8f;
        target.GetComponent<HealthComponent>()?.Heal(finalHeal);
        Console.WriteLine($"{caster.Name} heals {target.Name} for {finalHeal} health!");
    }
}

// 3. 구체적인 스킬 커맨드 (여기서 전략을 사용)
public class FireballSkillCommand : SkillCommand
{
    private ISkillEffectStrategy _damageStrategy;
    private float[] _damageParameters;

    public FireballSkillCommand(Character caster, GameObject target, ISkillEffectStrategy damageStrategy, float[] damageParameters)
        : base(caster, target)
    {
        _damageStrategy = damageStrategy;
        _damageParameters = damageParameters;
    }

    public override bool CanExecute()
    {
        // 마나 체크, 쿨다운 체크, 시야/거리 체크 등
        return _caster.HasEnoughMana(50) && !_caster.IsOnCooldown("Fireball");
    }

    public override void Execute()
    {
        if (CanExecute())
        {
            _caster.ConsumeMana(50);
            _caster.SetCooldown("Fireball", 3f);
            Console.WriteLine($"{_caster.Name} casts Fireball at {_target.Name}!");
            _damageStrategy.ApplyEffect(_caster, _target, _damageParameters);
            // 시각 효과, 사운드 등
        } else {
            Console.WriteLine("Cannot cast Fireball!");
        }
    }
}

// var fireballDamageStrategy = new DamageEffectStrategy();
// var fireballCommand = new FireballSkillCommand(player, enemy, fireballDamageStrategy, new float[] { 30f }); // 기본 피해량 30
```

**제 생각**: 커맨드 패턴은 '어떤 스킬을 사용한다'는 의도를 캡슐화하고, 전략 패턴은 '그 스킬이 정확히 어떤 효과를 내는가'를 정의합니다. 이를 통해 새로운 스킬을 추가하는 것은 `SkillCommand`를 상속받고 적절한 `ISkillEffectStrategy`를 조합하는 것으로 끝납니다. 스킬 데이터만 변경하여 새로운 스킬을 '생성'할 수도 있죠. TRPG 룰북에 새로운 주문을 추가하듯이 유연하게 확장할 수 있습니다.

#### 🛠️ Case 3: 이벤트 및 조건 시스템 – 관찰자(Observer) 패턴으로 시스템 간의 비동기 소통을 돕다

TRPG에서는 "몬스터가 죽으면 경험치를 얻고, 아이템이 드롭되며, 퀘스트 카운트가 증가한다"와 같은 수많은 이벤트와 그에 따른 반응들이 존재합니다. 이 모든 것을 직접적으로 호출하는 방식으로 구현하면, 시스템 간의 결합도가 높아져 나중에 변경하기 매우 어려워집니다.

**문제**: 특정 이벤트 발생 시 여러 시스템이 반응해야 하는데, 직접 호출하면 시스템 간의 의존성이 강해집니다. `HealthComponent`가 `QuestManager`, `InventoryManager`, `UIManager`를 모두 알고 있어야 하는 불필요한 결합이 발생합니다.

**해결**: **관찰자 패턴(Observer Pattern)**, 혹은 좀 더 발전된 형태인 **이벤트 버스(Event Bus) 시스템**을 사용하여 시스템 간의 느슨한 결합을 유도합니다.

**핵심 로직**:

1.  **EventManager**: 모든 이벤트를 중계하는 중앙 허브 역할을 합니다.
2.  **Subscribe/Unsubscribe**: 각 시스템은 자신이 관심 있는 이벤트에 대해 `EventManager`에 구독합니다.
3.  **Publish**: 이벤트가 발생하면, 해당 이벤트를 발생시킨 주체는 `EventManager`에 이벤트를 발행(Publish)합니다.
4.  **EventHandler**: 구독한 모든 시스템은 발행된 이벤트를 받아 각자의 로직을 수행합니다.

```csharp
// C# 유사 코드
// 이벤트 클래스 (데이터만 포함)
public abstract class GameEvent { }

public class CharacterDeathEvent : GameEvent
{
    public Character DeadCharacter { get; private set; }
    public Character Killer { get; private set; }
    public CharacterDeathEvent(Character dead, Character killer) { DeadCharacter = dead; Killer = killer; }
}

public class ItemPickedUpEvent : GameEvent
{
    public Item PickedItem { get; private set; }
    public Character Picker { get; private set; }
    public ItemPickedUpEvent(Item item, Character picker) { PickedItem = item; Picker = picker; }
}

// 2. EventManager (싱글톤 또는 정적 클래스)
public static class EventManager
{
    // Dictionary<이벤트 타입, 해당 이벤트를 처리할 액션 리스트>
    private static Dictionary<Type, List<Action<GameEvent>>> _subscribers = new Dictionary<Type, List<Action<GameEvent>>>();

    public static void Subscribe<T>(Action<T> handler) where T : GameEvent
    {
        Type eventType = typeof(T);
        if (!_subscribers.ContainsKey(eventType))
        {
            _subscribers[eventType] = new List<Action<GameEvent>>();
        }
        // 제네릭 핸들러를 일반 GameEvent 핸들러로 래핑
        _subscribers[eventType].Add(e => handler((T)e));
    }

    public static void Unsubscribe<T>(Action<T> handler) where T : GameEvent
    {
        // 실제 구현에서는 Delegate.Remove를 사용해야 함.
        // 여기서는 예시를 위해 단순화
        Type eventType = typeof(T);
        if (_subscribers.ContainsKey(eventType))
        {
            // _subscribers[eventType].RemoveAll(action => action.Target == handler.Target && action.Method == handler.Method);
            // 위 코드는 래핑된 Action에 대해 직접 작동하지 않을 수 있으므로, 실제 구현에서는 좀 더 복잡한 로직이 필요
            // 혹은, 구독 시 ID 부여 등 다른 관리 방식 고려
        }
    }

    public static void Publish<T>(T eventData) where T : GameEvent
    {
        Type eventType = typeof(T);
        if (_subscribers.ContainsKey(eventType))
        {
            // 리스트 복사 후 순회하여, 이벤트 도중 구독/취소로 인한 컬렉션 변경 오류 방지
            foreach (var handler in _subscribers[eventType].ToList())
            {
                handler(eventData);
            }
        }
    }
}
```

### 💡 TRPG와 디자인 패턴, 그 이상의 시너지

지금까지 살펴본 TRPG 규칙 시스템의 구현 예시들은 단순히 '어떻게 구현할까?'를 넘어 '어떻게하면 더 유연하고 확장성 있게 설계할까?'에 대한 답을 제시합니다. TRPG 규칙들은 본질적으로 모듈화되어 있고, 데이터 주도적이며, 강력한 상호작용성을 가지고 있습니다. 이는 현대 게임 개발에서 추구하는 견고하고 유연한 아키텍처와 놀랍도록 일치하죠.

**저의 경험을 한 토막 풀어보자면...**

예전에 한 MMORPG 프로젝트에서 스킬 시스템을 처음부터 구현할 때, 신입 개발자들과 함께 아무 패턴 없이 '각 스킬마다 별도 함수 만들고, 스킬 데이터는 Enum으로 관리!'라는 패기 넘치는 계획을 세운 적이 있습니다. 10개 스킬까지는 괜찮았죠. 50개가 넘어가면서부터는 버그 리포트가 산더미처럼 쌓이고, 신규 스킬 추가는 며칠 밤샘 작업이 필요했습니다. 결국 엎고 위에 설명드린 커맨드 & 전략 패턴으로 재구현했습니다. 그때의 '아, 진작 이렇게 할 걸!' 하는 후회와 함께 얻은 깨달음은 지금도 저의 개발 철학의 큰 부분을 차지하고 있습니다.
