### 1. 문제 제기: '수만 개의 객체'라는 현실과 환상의 간극

게임을 개발하다 보면 항상 마주치는 벽이 있습니다. 바로 '성능'과 '비주얼'의 조화죠. 특히, 수만 개의 나뭇잎이 바람에 흔들리고, 수백 대의 유닛이 동시에 전투를 벌이며, 수천 개의 파티클이 폭발 효과를 만들어내는 등, **수많은 객체가 동시에 저마다의 개성을 뽐내며 움직여야 하는 상황**에 직면하게 됩니다.

초창기에는 이런 문제를 CPU에서 각 객체의 상태를 일일이 계산하고, 매번 별도의 Draw Call을 날리는 방식으로 해결하려 했습니다. 결과는 뻔했죠. CPU는 비명을 지르고, 프레임 드랍은 기본이며, 개발자는 밤샘 야근으로 다크서클이 턱까지 내려오는 지옥을 경험하게 됩니다. "어떻게 하면 CPU를 덜 괴롭히면서 이 수많은 객체들을 아름답고 부드럽게 그려낼 수 있을까?" 이 질문이 저를 셰이더의 세계로 이끌었습니다.

### 2. 솔루션의 시작: GPU의 심장, Shader Math & GLSL

이 문제의 핵심은 **CPU의 병목 현상**입니다. CPU는 범용적인 연산에 강하지만, 수천만 개의 픽셀이나 버텍스를 동시에 처리하는 데는 한계가 명확합니다. 이때 GPU가 구원투수로 등장합니다. GPU는 수천 개의 코어를 이용해 단순하고 반복적인 연산을 미친 듯이 병렬 처리하는 데 특화되어 있습니다. 그리고 이 GPU를 움직이는 언어가 바로 **GLSL (OpenGL Shading Language)**입니다.

GLSL과 셰이더 매스(Shader Math)를 활용하면, 우리는 단순히 텍스처를 붙이는 것을 넘어, 버텍스 하나하나, 픽셀 하나하나의 색상, 위치, 노멀 등을 수학적으로 계산하여 동적으로 만들어낼 수 있습니다. 예를 들어, 움직이는 파도를 텍스처 애니메이션으로 처리하려면 엄청난 메모리와 데이터가 필요하지만, 셰이더에서는 사인(sin) 함수 몇 번과 노이즈 함수 조합만으로 실시간으로 변형되는 파도를 구현할 수 있습니다. 이것이 바로 '프로시저럴(Procedural)'의 마법이죠.

#### 실전 예시: 수만 개의 나뭇잎이 바람에 흔들리는 장면

수만 개의 나뭇잎을 각각 CPU에서 연산하여 움직이게 하면? 상상만 해도 끔찍합니다. 하지만 셰이더를 활용하면, 우리는 이 문제를 GPU로 넘길 수 있습니다.

1.  **버텍스 셰이더 (Vertex Shader)**: 각 나뭇잎 메시의 버텍스들을 GPU에서 계산하여 실시간으로 움직입니다.
    *   **문제**: 모든 나뭇잎이 똑같이 움직이면 부자연스럽습니다.
    *   **해결**: 각 나뭇잎의 고유 ID나 위치를 기반으로 `per-instance` 데이터를 활용, 바람의 강도나 방향을 미묘하게 다르게 적용합니다. `sin` 함수와 노이즈 함수를 조합하여 각 버텍스의 위치를 시간에 따라 변형시킵니다.

```glsl
// --- Vertex Shader Pseudo-code ---
#version 460 core
layout (location = 0) in vec3 aPos;         // 모델 로컬 좌표
layout (location = 1) in vec3 aNormal;       // 모델 로컬 노멀
layout (location = 2) in vec2 aTexCoords;    // 텍스처 좌표

// 인스턴스별 데이터 (CPU에서 GPU로 전달)
layout (location = 3) in mat4 instanceTransform;    // 각 인스턴스의 World Transform 행렬
layout (location = 7) in float instanceWindOffset; // 각 인스턴스별 바람 흔들림 오프셋

out vec2 TexCoords;

uniform mat4 projection; // 투영 행렬
uniform mat4 view;       // 뷰 행렬
uniform float time;      // 게임 시간
uniform vec3 windDirection; // 바람 방향 (월드 공간)
uniform float windStrength; // 바람 강도

void main()
{
    TexCoords = aTexCoords;

    // 1. 모델 로컬 좌표를 월드 공간으로 변환
    vec4 worldPos = instanceTransform * vec4(aPos, 1.0);

    // 2. 바람에 의한 버텍스 변형 계산 (Shader Math의 핵심!)
    // instanceWindOffset으로 각 나뭇잎 인스턴스별 움직임 위상(phase)을 다르게 함
    float windPhase = (time * 2.0 + instanceWindOffset) * windStrength * 0.5;
    float swayAmount = sin(windPhase) * 0.1; // 바람에 흔들리는 최대 정도

    // 나뭇잎 아래쪽 버텍스일수록 덜 흔들리게 하여 자연스러움 증가
    // (1.0 - aPos.y)는 나뭇잎의 밑동이 0, 끝이 1이라고 가정했을 때 적절한 가중치 제공
    vec3 offset = windDirection * swayAmount * (1.0 - aPos.y);

    worldPos.xyz += offset; // 월드 위치에 변형 적용

    // 3. 최종 클립 공간 위치 계산
    gl_Position = projection * view * worldPos;
}
```

이 셰이더는 GPU에서 수만 번 병렬로 실행되며, 각각의 나뭇잎 버텍스를 독립적으로 변형시킵니다. CPU는 단지 `time`, `windDirection`, `windStrength` 같은 몇 가지 `uniform` 값과 각 인스턴스의 `transform` 및 `windOffset` 데이터만 넘겨주면 됩니다.

### 3. 데이터의 오케스트라: DOTS와 셰이더의 시너지

여기서 '수만 개의 인스턴스 데이터(`instanceTransform`, `instanceWindOffset`)'를 어떻게 효율적으로 GPU에 전달할지가 관건이 됩니다. 전통적인 MonoBehaviour 방식은 GameObject 하나하나에 스크립트를 붙여 `Update()`에서 데이터를 준비하면 CPU 오버헤드가 극심합니다. 바로 이때, **Data-Oriented Technology Stack (DOTS)**의 진정한 가치가 빛을 발합니다.

DOTS는 데이터를 중심으로 사고하는 패러다임을 제공합니다. Entity Component System (ECS)으로 데이터와 로직을 분리하고, Job System과 Burst Compiler를 통해 CPU 연산을 극한으로 끌어올립니다.

**어떤 문제가 있었고, DOTS로 어떻게 해결하는가?**

*   **문제**: 수만 개의 나뭇잎 인스턴스 데이터를 CPU에서 효율적으로 생성, 관리, 그리고 GPU가 읽을 수 있는 형태로 준비하는 것. 예를 들어, `Matrix4x4` 수만 개와 `float` 수만 개를 매 프레임 업데이트하고 `MaterialPropertyBlock`을 통해 전달하는 것은 엄청난 부담입니다.
*   **해결**: DOTS는 이 문제를 다음과 같이 해결합니다.
    1.  **ECS로 데이터 관리**: 각 나뭇잎을 `Entity`로 만들고, `LocalToWorld` (Matrix) 컴포넌트와 `WindData` (float offset) 컴포넌트 등을 붙입니다. `LocalToWorld`는 오브젝트의 최종 월드 변환을 나타내며, 셰이더의 `instanceTransform`에 직접 매핑됩니다. `WindData` 컴포넌트는 각 나뭇잎의 고유 `windOffset` 값을 가질 수 있습니다.
    2.  **Job System으로 데이터 준비**: `IJobParallelFor`를 사용하여 수만 개의 나뭇잎 엔티티의 `WindData`를 계산하거나 업데이트합니다. 예를 들어, 특정 조건(지형 높이, 근접성)에 따라 `windOffset`을 다르게 부여하는 로직을 Burst 컴파일된 Job으로 돌려 병렬 처리합니다. 이 과정에서 필요한 `instanceTransform` 값들도 `LocalToWorld` 컴포넌트로부터 효율적으로 수집할 수 있습니다.
    3.  **ComputeBuffer로 GPU 전송**: Job System에서 준비된 `LocalToWorld` 매트릭스 배열과 `WindData` 배열을 `ComputeBuffer`에 담아 GPU에 직접 전달합니다. Unity의 `Graphics.DrawMeshInstancedIndirect` 같은 API를 활용하면 단일 Draw Call로 수만 개의 인스턴스를 그릴 수 있습니다. 이 과정에서 DOTS의 `NativeArray`와 `ComputeBuffer` 간의 효율적인 데이터 전송 메커니즘이 큰 이점을 제공합니다.

#### 아키텍처 다이어그램 (개념)

```
+-------------------+       +--------------------+       +-------------------+
|      CPU (DOTS)   |       |   Compute Buffer   |       |       GPU (GLSL)    |
|-------------------|       | (Instanced Data)   |       |-------------------|
| ECS (Entities)    |       |--------------------|       | Vertex Shader     |
| - LocalToWorld    |<----- | Mat4[N], Float[N]  |----->| (instanceTransform)|
| - WindData        |       +--------------------+       | (instanceWindOffset)|
| - Other Params    |               ^                    |  -> Shader Math   |
|                   |               |                    |  -> Final Pos     |
| Job System        |               |                    +-------------------+
| (Burst Compiled)  |               |                            |
| - Collect / Update|---------------|                            V
|   Instance Data   |                                    +-------------------+
| (e.g., Wind Offset|                                    | Fragment Shader   |
|  based on noise)  |                                    | (PBR Lighting, etc.)|
+-------------------+                                    +-------------------+
```

이 흐름은 제가 여러 프로젝트에서 '수만 개의 전투 유닛'이나 '광활한 필드의 풀떼기'를 처리하며 얻은 핵심 노하우입니다. CPU는 데이터 준비에 집중하고 (그것도 병렬적으로!), GPU는 렌더링 연산에 집중하는, 진정한 의미의 책임 분담이죠. DOTS가 없었다면 CPU가 이 모든 데이터를 준비하는 과정에서 또 다른 병목에 시달렸을 겁니다.

### 4. 시니어의 견해: 데이터 중심 사고방식의 중요성

솔직히 말씀드리면, 셰이더 프로그래밍은 처음에는 어렵게 느껴질 수 있습니다. '수학'이라는 장벽 때문이죠. 하지만 한번 이 '데이터 중심', '병렬 처리'라는 마인드셋을 갖게 되면, 게임 개발의 많은 난제들이 풀리기 시작합니다.

DOTS 역시 마찬가지입니다. 초기에 개념을 잡기가 쉽지 않지만, 본질은 '데이터를 어떻게 효율적으로 다루고, 예측 가능하며, 병렬적으로 처리할 것인가'에 있습니다. GLSL의 셰이더 매스 역시 결국 "입력 데이터를 어떻게 출력 데이터로 효율적으로 변환할 것인가"의 문제입니다.

제가 늘 강조하는 것이지만, 좋은 개발자는 단순히 기술 스택을 많이 아는 것을 넘어, **문제를 정의하고, 그 문제에 가장 적합한 '사고방식'과 '기술'을 연결할 줄 아는 사람**입니다. GLSL과 DOTS는 서로 다른 영역의 기술 같지만, 뿌리에는 '데이터 중심의 고성능 병렬 처리'라는 공통된 철학이 흐르고 있습니다.

이 두 기술을 깊이 이해하고 결합한다면, 여러분은 단순한 게임 개발자를 넘어, 성능과 비주얼이라는 두 마리 토끼를 동시에 잡는 진정한 '기술 아티스트'가 될 수 있을 것이라 확신합니다.
