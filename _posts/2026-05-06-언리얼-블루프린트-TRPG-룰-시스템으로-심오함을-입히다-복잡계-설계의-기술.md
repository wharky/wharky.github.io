### 문제: 블루프린트 스파게티와 휘발성 룰셋

언리얼 엔진의 블루프린트는 정말 멋진 도구입니다. 코딩 지식 없이도 아이디어를 빠르게 시각화하고, 복잡한 로직을 구현할 수 있게 해주죠. 하지만, 이 강력함 뒤에는 그림자도 있습니다. 제대로 된 설계 원칙 없이 사용하면, 프로젝트가 커질수록 블루프린트는 걷잡을 수 없는 '스파게티 코드'가 되기 십상입니다. 변수들이 어디서 어떻게 바뀌는지, 이 노드가 왜 여기에 붙어있는지, 심지어 내가 짠 코드인데도 며칠 뒤면 "이게 뭐였더라?" 하게 되죠.

여기에 TRPG 룰 시스템을 코드로 구현하는 과제를 얹어봅시다. TRPG는 캐릭터의 능력치, 스킬, 전투 공식, 상태 이상, 아이템 효과 등 방대하고 유기적으로 연결된 룰로 구성됩니다. 게다가, 이 룰들은 게임 디자이너의 피와 땀으로 끊임없이 수정되고, 밸런스가 조절되며, 새로운 요소가 추가됩니다. 만약 이런 복잡한 룰셋을 블루프린트에 하드코딩하거나, 일관성 없는 방식으로 구현한다면 어떻게 될까요? 디자이너가 스킬 대미지 1만 바꿔도 개발자는 온갖 블루프린트를 뒤져야 하고, 새로운 상태 이상 하나 추가하려면 기존 로직을 통째로 뜯어고쳐야 하는 재앙이 펼쳐질 겁니다.

마치 던전 마스터(DM)가 플레이어들의 모든 행동을 실시간으로 코딩하려는 것과 같아요. 불가능하겠죠? DM은 '룰북'이라는 명확한 데이터와 '룰'이라는 정의된 로직을 기반으로 상황을 조율합니다. 우리의 게임 시스템도 이와 같아야 합니다.

### 해결책: TRPG 룰 시스템의 설계 철학을 블루프린트에 녹여내다

그렇다면, 이 '블루프린트 스파게티'를 피하고, TRPG처럼 유연하고 확장 가능한 룰 시스템을 만들려면 어떻게 해야 할까요? 저는 TRPG의 핵심 설계 철학을 블루프린트 베스트 프랙티스와 결합하는 것이 해답이라고 봅니다. 바로 '모듈성(Modularity)', '데이터 중심 설계(Data-Driven Design)', 그리고 '명확한 인터페이스(Clear Interfaces)'입니다.

#### 1. 데이터 중심 설계: 룰북은 코드 속에 존재해야 한다! (Data-Driven Design)

TRPG 룰북은 단순한 텍스트 묶음이 아닙니다. 캐릭터 생성 규칙, 스킬 목록, 몬스터 정보, 마법 공식 등 모든 게임 플레이의 근간이 되는 '데이터베이스'죠. 우리의 게임도 마찬가지여야 합니다.

*   **문제**: 스킬 대미지, 아이템 속성, 캐릭터 기본 능력치 등을 블루프린트 내부에 숫자 리터럴로 박아 넣는 행위. 이건 '룰북'을 DM의 머릿속에만 넣어두는 것과 같습니다. 디자이너가 접근하기 어렵고, 수정할 때마다 개발자가 블루프린트를 열어봐야 합니다.
*   **해결책**: **Data Assets (데이터 애셋)과 Data Tables (데이터 테이블)**을 적극 활용하세요.
    *   `UDataAsset`은 특정 형태의 데이터를 저장하는 데 이상적입니다. `FCharacterStats`, `FSkillDefinition`, `FStatusEffectData`와 같은 구조체(Struct)를 정의하고, 각 캐릭터 클래스, 스킬, 상태 이상별로 데이터 애셋을 생성하여 값을 할당합니다.
    *   `UDataTable`은 대량의 반복적인 데이터를 스프레드시트처럼 관리하는 데 유용합니다. 모든 아이템 정보, 몬스터 스탯, 지역별 드롭 테이블 등을 여기에 넣어두세요.

```cpp
// (Pseudo-code) SkillDefinition.h
USTRUCT(BlueprintType)
struct FSkillDefinition
{
    GENERATED_BODY()

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FName SkillID; // 고유 ID
    
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FText SkillName;
    
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    int32 BaseDamage; // 기본 피해량
    
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    float CooldownTime; // 쿨다운
    
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    TArray<TEnumAsByte<EStatusEffectType>> AppliedStatusEffects; // 부여하는 상태 이상
};

// UDataAsset에서 FSkillDefinition을 사용
// 블루프린트에서 "새로운 데이터 애셋 생성" -> "SkillDataAsset" 선택 후 각 스킬 데이터 생성.
```
**Opinionated**: "초반에는 귀찮아 보여도, 이 습관 하나로 나중에 수많은 백트래킹과 야근을 줄일 수 있습니다. '나중에 데이터화해야지'라는 생각은 죄악입니다. 지금 당장 데이터화하세요. 디자이너가 직접 값만 바꿔 게임 밸런스를 조절할 수 있다면, 개발자는 훨씬 더 중요한 시스템 개발에 집중할 수 있습니다."

#### 2. 인터페이스와 컴포넌트: 역할 분담과 유연한 상호작용 (Interfaces & Components)

TRPG에서 캐릭터는 '전사', '마법사' 같은 클래스 외에도 '도적'의 특기를 가진 '멀티 클래스'일 수도 있고, 특정 장비를 착용해 새로운 능력을 얻기도 합니다. 이처럼 유연하게 능력을 조합하고 상호작용하는 것은 '역할 분담'이 명확하기 때문입니다.

*   **문제**: 모든 기능을 하나의 거대한 캐릭터 블루프린트에 때려 넣는 것. `BP_PlayerCharacter` 하나에 인벤토리 로직, 스킬 사용 로직, 전투 로직, 상태 이상 로직이 전부 들어가 있으면? 신이시여. 게다가 다른 캐릭터가 이 기능을 재활용하려면 `Cast To BP_PlayerCharacter`의 늪에 빠질 겁니다.
*   **해결책**: **Blueprint Interfaces (블루프린트 인터페이스)와 Actor Components (액터 컴포넌트)**를 활용하여 역할과 책임을 명확히 분리하세요.
    *   **컴포넌트**: 특정 기능을 캡슐화하는 데 사용합니다. 예를 들어, `UInventoryComponent`는 아이템 관리 로직만, `UAbilitySystemComponent`는 스킬 사용 로직만, `UStatusEffectComponent`는 상태 이상 적용/해제 로직만 담당하게 만듭니다. 이 컴포넌트들을 필요한 액터에 추가하기만 하면 됩니다.
    *   **인터페이스**: 컴포넌트나 액터 간의 약한 결합(Loose Coupling)을 위해 필수적입니다. "나는 대미지를 줄 수 있는 대상과 소통하고 싶어. 그게 뭔지는 몰라도 돼." 이때 `ICombatant` 인터페이스를 만들고 `ApplyDamage`, `Heal` 함수를 정의합니다. 대미지를 주는 블루프린트는 특정 클래스를 아는 대신, `ICombatant`를 구현하는 모든 액터에게 대미지를 줄 수 있게 됩니다.

```
// (Conceptual Blueprint Interface Diagram)

                   [ICombatant Interface]
                   - ApplyDamage(Amount, DamageType)
                   - ApplyStatusEffect(EffectData)
                           |
                           | implements
                           V
    [UAbilitySystemComponent] -- (owns) --> [BP_PlayerCharacter]
                                   ^
                                   | (delegates to)
    [UStatusEffectComponent] -------
    [UInventoryComponent] ----------
```

**Opinionated**: "블루프린트 인터페이스는 '계약'입니다. '네가 이 계약을 지킨다면, 나는 너와 이런 방식으로 소통할게.' 이 계약 덕분에 상호작용이 훨씬 유연해지고, 미래에 어떤 새로운 '전투원'이 추가되더라도 기존 시스템을 수정할 필요가 없죠. 컴포넌트는 레고 블록과 같습니다. 재사용 가능한 기능 블록을 만들어 끼워 맞추세요. 이 두 가지를 잘 활용하면 복잡한 TRPG 룰셋도 훨씬 체계적으로 관리할 수 있습니다."

#### 3. FSM (State Machine) 활용: 전투 및 상태 관리 (Finite State Machine)

TRPG 전투는 '턴 시작', '공격', '방어', '상태 이상 적용', '턴 종료' 등 명확한 단계와 상태 전이로 이루어집니다. 이를 효과적으로 구현하려면 상태 머신이 필수적입니다.

*   **문제**: 복잡한 `Branch` 노드의 연속으로 캐릭터의 행동 상태를 관리하는 것. `If (IsAttacking) Then ... Else If (IsDefending) Then ...` 이런 식으로 상태를 관리하면, 새로운 상태가 추가될 때마다 전체 로직을 건드려야 합니다. 버그 유발의 지름길이죠.
*   **해결책**: **Finite State Machine (FSM)**을 직접 구현하거나, 언리얼의 Behavior Tree (비헤이비어 트리) 같은 내장 기능을 활용하여 상태 관리를 체계화합니다.
    *   예를 들어, `UCombatStateComponent`를 만들고, 이 컴포넌트 내에서 `Idle`, `Attacking`, `Defending`, `Stunned`, `Dead` 등의 전투 상태를 관리합니다. 각 상태에 진입할 때(OnEnter), 유지될 때(OnTick), 나갈 때(OnExit)의 로직을 명확히 정의합니다.
    *   상태 전이(State Transition)는 특정 이벤트(예: `TakeDamage`, `AttackButton_Pressed`)가 발생했을 때만 이루어지도록 합니다.

```
// (Conceptual Blueprint FSM Flow)

[Event: OnTakeDamage]
    |
    V
[GetCurrentState] --> [Switch on ECombatState]
    |
    |-- Case: Idle --> [SetState(Stunned)]
    |-- Case: Attacking --> [SetState(Defending)] // Counter Attack!
    |-- Case: Stunned --> [PlayStunAnimation]
    V
[Event: OnStateExited(PrevState)]
[Event: OnStateEntered(NewState)]
```
**Opinionated**: "FSM은 복잡한 동작을 예측 가능하고 디버깅하기 쉽게 만듭니다. '이 캐릭터는 지금 어떤 상태인가?'라는 질문에 명확하게 답할 수 있게 되죠. 특히 TRPG의 상태 이상(독, 마비, 기절 등)을 구현할 때 FSM만큼 명쾌한 답은 없습니다. 각 상태 이상을 하나의 '상태'로 보고, 해당 상태에서만 발동하는 효과와 해제 조건을 명확히 하면 됩니다."

#### 4. 명확한 함수와 매크로: 룰의 명확한 정의 (Pure Functions & Macros)

TRPG 룰북은 복잡한 계산식이나 효과를 정의할 때, 최대한 명확하고 재사용 가능한 형태로 설명합니다. "주사위 굴림(d20)에 숙련 보너스와 능력치 수정치를 더한다"와 같이 말이죠.

*   **문제**: 동일한 계산 로직이 여러 블루프린트에 흩어져 있거나, 로직이 너무 길고 복잡하여 한눈에 파악하기 어려운 경우.
*   **해결책**: **Pure Functions (순수 함수)와 Macros (매크로)**를 활용하여 재사용 가능한 로직 블록을 만드세요.
    *   **순수 함수**: 입력값을 받아 출력값을 반환하며, 외부 상태를 변경하지 않는 함수입니다. `CalculateDamage(BaseDamage, Resistance, CriticalChance)` 같은 계산 로직에 최적입니다.
    *   **매크로**: 여러 노드를 그룹화하여 복잡한 로직을 시각적으로 단순화하는 데 좋습니다. `ApplyStatusEffectMacro` 같은 것을 만들어, 필요한 곳에 끌어다 놓기만 하면 됩니다.
    *   **Enums (열거형)과 Structs (구조체)**: 마법 숫자(Magic Number)나 모호한 문자열 대신, `EAbilityType` (공격, 방어, 유틸리티), `EDamageType` (물리, 마법, 화염) 같은 열거형과 `FDamageInstance` 같은 구조체를 사용해 데이터와 로직을 명확히 합니다.

**Opinionated**: "순수 함수는 마치 TRPG의 '공식'과 같습니다. 이 공식을 알면 어떤 값을 넣었을 때 어떤 결과가 나올지 정확히 예측할 수 있죠. 매크로는 특정 행동을 묶어주는 '루틴'입니다. 반복되는 로직은 무조건 함수나 매크로로 분리하세요. 그리고 제발, 제발 `Boolean` 변수 5개를 엮어 만드는 조건문은 피하세요. 그게 무슨 의미인지 3개월 뒤의 당신은 모를 겁니다."

### 결론: 스파게티를 넘어, 아키텍처의 미학으로

오늘 우리는 언리얼 블루프린트의 베스트 프랙티스를 TRPG 룰 시스템 구현이라는 흥미로운 렌즈로 살펴보았습니다. TRPG 룰셋의 복잡성과 유연함은, 데이터 중심 설계, 인터페이스/컴포넌트 기반 모듈화, 상태 머신 활용, 그리고 명확한 함수 정의가 왜 게임 개발에서 중요한지 아주 훌륭하게 보여주는 예시입니다.
