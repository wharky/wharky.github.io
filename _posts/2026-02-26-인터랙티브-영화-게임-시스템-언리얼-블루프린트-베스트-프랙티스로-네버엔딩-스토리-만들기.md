### 문제: 인터랙티브 스토리, 그 끝없는 가지치기의 늪

인터랙티브 무비 게임, 혹은 내러티브 중심의 어드벤처 게임은 플레이어의 선택이 이야기의 흐름을 바꾸는 매력적인 장르입니다. 하지만 이 매력 뒤에는 개발자의 고통이 숨어 있습니다. 수십 개의 분기점, 각 분기마다 달라지는 대사와 영상 클립, 그리고 플레이어의 선택이 게임 세계와 캐릭터 관계에 미치는 영향까지. 이 모든 것을 직관적인 블루프린트로 구현하다 보면, 금세 '스파게티 코드'의 악몽에 빠지기 십상입니다.

처음에는 쉬워 보입니다. `Sequence` 노드 몇 개와 `Branch` 노드로 선택지를 구현하죠. 하지만 챕터 1이 끝나기도 전에 블루프린트 그래프는 거대한 괴물처럼 변해버립니다. 특정 선택지가 다른 선택지에 영향을 미치거나, 게임 진행에 따라 이전 선택지가 잠기는 등 복잡한 로직이 추가될 때마다, 그래프는 한없이 복잡해지고, 수정은 두려워지며, 디버깅은 좌절의 연속이 됩니다. 결국, 이 복잡성 때문에 야심 찬 스토리는 미완의 꿈으로 남기 쉽습니다.

### 해결책: 블루프린트 베스트 프랙티스로 구축하는 우아한 시스템

이러한 문제를 해결하기 위해 제가 제안하는 것은 바로 **'모듈화된, 데이터 중심의, 이벤트 주도형 아키텍처'**입니다. 언리얼 엔진의 블루프린트가 가진 강력한 기능을 베스트 프랙티스와 결합하여, 유연하고 확장 가능한 인터랙티브 무비 게임 시스템을 구축하는 것이 핵심입니다.

#### 1. 데이터 중심의 내러티브 설계 (Data-Driven Narrative)

**문제**: 모든 대사, 선택지, 영상 경로, 그리고 그 결과 로직을 블루프린트에 직접 구현하면 수정하기 어렵고, 콘텐츠 추가가 비효율적입니다. 스토리 작가나 기획자가 직접 내용을 수정하기는 거의 불가능해집니다.

**해결책**: 모든 내러티브 콘텐츠와 흐름을 데이터 테이블(Data Table) 또는 커스텀 `UDataAsset`을 통해 관리합니다.

**로직/의사코드**:
각 인터랙티브 '노드'(스토리의 한 단위)를 구조체로 정의하고, 이를 데이터 테이블의 한 행으로 관리합니다.

```cpp
// FInteractiveMovieNode.h (C++ Struct, 블루프린트에서 사용 가능)
USTRUCT(BlueprintType)
struct FChoiceData
{
    GENERATED_BODY()

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    FText ChoiceText;             // 선택지 텍스트
    
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    FName NextNodeID;             // 이 선택 후 이동할 노드 ID
    
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    TArray<FName> ConsequenceFlags; // 이 선택으로 인해 설정될 게임 상태 플래그
};

USTRUCT(BlueprintType)
struct FInteractiveMovieNode : public FTableRowBase // 데이터 테이블용
{
    GENERATED_BODY()

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    FName NodeID;                 // 고유 노드 ID (참조용)

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    TSoftObjectPtr<UFileMediaSource> VideoClip; // 재생할 영상 클립 (Soft Reference로 메모리 최적화)

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    FText DialogueText;           // 노드에 표시될 대사

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    TArray<FChoiceData> Choices;  // 플레이어가 선택할 수 있는 선택지 목록

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    FName AutoNextNodeID;         // 선택지 없이 자동으로 다음 노드로 넘어갈 경우
    
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    TArray<FName> RequiredFlags;  // 이 노드에 도달하기 위한 필요 조건 플래그
    // ... 기타 상태 변경, 이벤트 트리거 정보 등
};
```

중앙 `BP_InteractiveMovieManager`는 이 데이터 테이블을 로드하여 현재 `NodeID`에 해당하는 정보를 가져와 게임을 진행합니다. 이 방식은 기획자가 직접 엑셀 시트나 데이터 에셋 편집기에서 스토리 흐름을 변경할 수 있게 하여, 개발자와의 의존성을 크게 줄여줍니다.

#### 2. 상태 머신(State Machine) 기반의 흐름 제어

**문제**: '영상 재생 중', '선택지 대기 중', '선택 처리 중' 등 노드 내부의 복잡한 상태를 블루프린트의 실행 핀(Execution Pin)만으로 관리하면 오류 발생 가능성이 높고, 가독성이 떨어집니다.

**해결책**: 간단한 Enum 기반의 상태 머신이나 언리얼 엔진의 내장된 State Machine (State and Conduits) 기능을 활용하여, 각 노드의 내부 흐름을 명확하게 제어합니다.

**로직/의사코드**:
`BP_InteractiveMovieManager` 내에서 사용할 `EInteractiveMovieState` Enum을 정의합니다.

```
UENUM(BlueprintType)
enum class EInteractiveMovieState : uint8
{
    IMS_Idle UMETA(DisplayName="Idle"),
    IMS_LoadingNode UMETA(DisplayName="Loading Node Data"),
    IMS_PlayingVideo UMETA(DisplayName="Playing Video"),
    IMS_PresentingDialogue UMETA(DisplayName="Presenting Dialogue"),
    IMS_PresentingChoices UMETA(DisplayName="Presenting Choices"),
    IMS_ProcessingChoice UMETA(DisplayName="Processing Choice"),
    IMS_Transitioning UMETA(DisplayName="Transitioning to Next Node")
};
```

각 상태는 특정 로직만을 수행하고, 다음 상태로의 전환 조건을 명확히 합니다.

```blueprint
// BP_InteractiveMovieManager (부분 의사 코드)
// 현재 상태 변수: CurrentMovieState (EInteractiveMovieState)

Function ProcessNode(FName InNodeID):
    CurrentNodeData = GetNodeDataFromDataTable(InNodeID)
    SetCurrentMovieState(EInteractiveMovieState::LoadingNode)
    // ... 비동기 데이터 로딩 로직 ...
    
    OnDataLoaded -> SetCurrentMovieState(EInteractiveMovieState::PlayingVideo)
    Call BPI_VideoPlayer.PlayVideo(CurrentNodeData.VideoClip) // 인터페이스 호출

Event OnVideoPlaybackFinished:
    SetCurrentMovieState(EInteractiveMovieState::PresentingDialogue)
    Call BPI_DialoguePresenter.ShowDialogue(CurrentNodeData.DialogueText)

Event OnDialogueFinished:
    If CurrentNodeData.Choices.IsEmpty():
        SetCurrentMovieState(EInteractiveMovieState::Transitioning)
        ProcessNode(CurrentNodeData.AutoNextNodeID)
    Else:
        SetCurrentMovieState(EInteractiveMovieState::PresentingChoices)
        Call BPI_ChoicePresenter.ShowChoices(CurrentNodeData.Choices)

Event OnChoiceMade(FChoiceData ChosenChoice):
    SetCurrentMovieState(EInteractiveMovieState::ProcessingChoice)
    UpdateGameState(ChosenChoice.ConsequenceFlags) // BPI_GameStateManager 사용
    ProcessNode(ChosenChoice.NextNodeID)
```
이처럼 상태를 명확히 나누면, 현재 게임이 어떤 상황인지 한눈에 파악할 수 있고, 예상치 못한 버그를 줄일 수 있습니다.

#### 3. 인터페이스(Interfaces)와 이벤트 디스패처(Event Dispatchers)를 통한 느슨한 결합

**문제**: 비디오 플레이어, UI 위젯, 게임 상태 관리자 등 다양한 컴포넌트 간의 직접적인 참조는 시스템을 경직되게 만들고 재사용성을 저해합니다.

**해결책**: 블루프린트 인터페이스와 이벤트 디스패처를 적극 활용하여, 컴포넌트 간의 결합도를 낮춥니다. `BP_InteractiveMovieManager`는 특정 구현체를 직접 호출하는 대신, 인터페이스 메시지를 보냅니다.

**아키텍처 설명**:

*   **`BPI_VideoPlayer`**: `PlayVideo(MediaSource)`, `StopVideo()`, `OnVideoPlaybackFinished` (이벤트 디스패처) 등의 함수를 정의합니다. 실제 영상 재생은 `BP_MediaPlayerActor` (또는 UMG 위젯 내 미디어 플레이어)가 이 인터페이스를 구현하여 처리합니다.
*   **`BPI_ChoicePresenter`**: `ShowChoices(Array<FChoiceData>)`, `HideChoices()`, `OnChoiceMade(FChoiceData)` (이벤트 디스패처) 등을 정의합니다. 실제 선택지 UI는 `BP_ChoiceWidget`이 이 인터페이스를 구현합니다.
*   **`BPI_GameStateManager`**: `UpdateFlags(Array<FName>)`, `CheckFlag(FName)` 등을 정의합니다. 플레이어의 선택이 게임 상태(예: 호감도, 아이템 소유)에 미치는 영향을 관리하는 `BP_GameStateManagerActor`가 이를 구현합니다.

`BP_InteractiveMovieManager`는 시작 시 `BPI_VideoPlayer`, `BPI_ChoicePresenter`, `BPI_GameStateManager` 인터페이스를 구현하는 액터/위젯을 찾아 참조하고, 필요한 기능을 인터페이스 메시지로 호출합니다. 이렇게 하면 영상 플레이어나 UI 위젯의 구현이 변경되더라도 `BP_InteractiveMovieManager`는 거의 수정할 필요가 없습니다.

#### 4. 부모-자식 블루프린트와 컴포넌트를 통한 재사용성

**문제**: 여러 종류의 인터랙티브 요소(예: 대화 노드, QTE 노드, 탐색 노드)마다 비슷한 기본 로직을 반복해서 작성하게 됩니다.

**해결책**: 공통 로직을 담은 부모 블루프린트를 만들고, 이를 상속받아 특화된 자식 블루프린트를 만듭니다. 또는 재사용 가능한 컴포넌트로 분리하여 필요할 때마다 붙여 사용합니다.

*   **예시**: `BP_BaseInteractiveNodeActor`를 만들고, 이 액터에 모든 인터랙티브 노드가 공통적으로 가질 법한 로직(예: 노드 ID 설정, 다음 노드 전환 로직)을 넣습니다.
    *   `BP_DialogueNode` (부모로부터 상속): 대화와 선택지 표시에 특화된 로직 추가.
    *   `BP_QTE_Node` (부모로부터 상속): QTE 미니게임 로직 추가.
    *   `BP_ExploreNode` (부모로부터 상속): 플레이어 탐색 및 특정 오브젝트 상호작용 로직 추가.

컴포넌트의 경우, `BP_NarrativeTriggerComponent`를 만들어 플레이어가 특정 영역에 진입하거나 오브젝트를 사용했을 때 `BP_InteractiveMovieManager`에 '특정 노드를 시작하라'는 이벤트를 보낼 수 있습니다.

### 나의 경험과 실용적인 조언: "삽질하지 마세요!"

저는 처음 인터랙티브 무비 게임을 만들었을 때, 모든 것을 블루프린트 그래프 안에 구겨 넣다가 밤샘 디버깅으로 피폐해진 경험이 있습니다. 그 경험을 통해 얻은 교훈은 바로 **'계획'**과 **'분리'**의 중요성입니다.

*   **C++ vs. 블루프린트**: 핵심 데이터 구조(`FInteractiveMovieNode`, `FChoiceData` 등)는 C++로 정의하는 것이 좋습니다. 안정성과 성능 면에서 유리하고, 블루프린트에서도 유연하게 사용할 수 있습니다. 실제 게임 로직의 대부분은 블루프린트로 구현하되, 복잡하거나 성능이 중요한 부분(예: 대규모 AI 의사결정)은 C++로 옮기는 것을 고려하세요.
*   **명확한 명명 규칙**: 팀원들과의 협업을 위해 `BP_`, `BPI_`, `DT_` 등 명확한 접두사와 일관된 명명 규칙을 따르세요. '이게 뭔지 나중에라도 알 수 있게'가 아니라 '누가 봐도 바로 알 수 있게'라는 마음으로 작성해야 합니다.
*   **주석과 정리**: 블루프린트 그래프는 보기 좋게 정리하고, 중요한 부분에는 반드시 주석을 달아두세요. 나중에 여러분 자신이나 다른 개발자가 유지보수할 때 엄청난 시간을 절약해 줄 것입니다. 함수와 매크로로 자주 사용되는 로직을 캡슐화하는 것도 필수입니다.
*   **테스트와 디버깅**: 복잡한 분기점을 가진 시스템은 테스트하기 어렵습니다. 특정 노드로 바로 이동할 수 있는 디버그 기능이나, 특정 조건 플래그를 강제로 설정하는 툴을 만들어두면 개발 속도가 비약적으로 빨라집니다.
*   **성능 최적화**: 비디오 재생은 생각보다 리소스를 많이 잡아먹습니다. 언리얼 엔진의 미디어 프레임워크를 올바르게 사용하고, 필요 없는 비디오는 미리 언로드(Unload)하여 메모리 부하를 줄여야 합니다. UMG 위젯의 복잡성도 프레임 드랍의 주범이 될 수 있으니 주의 깊게 프로파일링하세요.

### 결론: 스토리가 살아 숨 쉬는 게임을 향해
