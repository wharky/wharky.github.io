### 1. 문제 제기: 영화와 게임 사이의 '버퍼링'

저희가 마주한 가장 큰 문제는 다음과 같았습니다.
*   **방대한 데이터 관리:** 수십, 수백 개의 영상 클립을 어떻게 효율적으로 로드하고 언로드할 것인가? 빌드 용량은 물론, 런타임 메모리 사용량도 폭발적입니다.
*   **끊김 없는 전환:** 플레이어의 선택에 따라 다음 영상 클립으로 넘어갈 때, 로딩 스크린이나 버벅임 없이 영화처럼 자연스럽게 이어지게 하려면?
*   **복잡한 브랜칭 로직:** 플레이어의 선택과 내부 변수(선택 누적, 관계도 등)에 따라 수많은 분기가 실시간으로 결정되어야 하는데, 이걸 코드에 하드코딩하면 유지보수가 지옥이 됩니다.
*   **UI 오버레이 성능:** 영상 위에 선택지 UI, 자막, 퀵타임 이벤트(QTE) UI 등이 실시간으로 나타나고 사라질 때 발생하는 GC(Garbage Collection) 스파이크와 프레임 드랍.

이런 문제들을 해결하지 못하면, 플레이어는 몰입감을 잃고 "아, 또 로딩이네", "화면이 왜 이래?"라고 불평하며 게임을 꺼버릴 것입니다. 저희의 목표는 "이게 영화인지 게임인지 구분이 안 갈 정도로!" 매끄러운 경험을 제공하는 것이었습니다.

### 2. 해결책: '예측 로딩' 기반의 적응형 내러티브 스트리밍 시스템

저희는 이 문제에 대한 해결책으로 **'예측 로딩(Predictive Loading)'**과 **'비동기 에셋 스트리밍(Asynchronous Asset Streaming)'**에 기반한 커스텀 **'적응형 내러티브 스트리밍 시스템(Adaptive Narrative Streaming System)'**을 구축했습니다. 핵심은 플레이어가 선택을 하기 전, 또는 특정 분기점에 도달하기 전에 미리 다음 영상 및 관련 데이터를 백그라운드에서 로드하여 '버퍼링 없는' 전환을 가능하게 하는 것이었죠.

#### 2.1. Deep Dive: 내러티브 구조화 & 데이터 모델링

가장 먼저, 복잡한 내러티브 브랜칭을 효과적으로 관리할 방법이 필요했습니다. 코드로 모든 분기를 관리하는 것은 불가능에 가깝다는 것을 지난 프로젝트에서 뼈저리게 느꼈죠.

*   **문제:** 거미줄처럼 얽힌 스토리 분기를 코드로 관리하는 건 악몽입니다. 디자이너가 직접 스토리를 시각적으로 설계하고 테스트할 방법이 없었어요.
*   **해결:** 저희는 유니티의 **ScriptableObject**를 활용하여 내러티브 그래프 시스템을 만들었습니다. 각 `NarrativeNode`는 비디오 재생, 선택지 제시, 특정 이벤트 트리거 등 다양한 역할을 수행하며, 다음 노드(들)로의 연결 포인트를 가집니다.

```csharp
// 의사 코드: NarrativeNode.cs (ScriptableObject)
public abstract class NarrativeNode : ScriptableObject
{
    public string NodeID; // 고유 ID
    public List<NarrativeNode> NextNodes; // 다음으로 이어질 노드들 (브랜치)

    public abstract void EnterNode(NarrativeDirector director);
    public abstract void ExitNode(NarrativeDirector director);
}

// 비디오 재생 노드
[CreateAssetMenu(fileName = "VideoNode", menuName = "Narrative/Video Node")]
public class VideoNode : NarrativeNode
{
    public AssetReference VideoClipRef; // Addressables를 활용한 비디오 클립 참조
    public float Duration; // 재생 시간 (선택지가 없을 경우)

    public override void EnterNode(NarrativeDirector director)
    {
        Debug.Log($"[VideoNode] Playing video: {VideoClipRef.AssetGUID}");
        // Director가 VideoPlayer를 통해 VideoClipRef를 재생하도록 지시
        // 그리고 다음 노드를 예측 로딩 시작
        director.StartPredictiveLoading(NextNodes);
    }
    // ...
}

// 선택지 제시 노드
[CreateAssetMenu(fileName = "ChoiceNode", menuName = "Narrative/Choice Node")]
public class ChoiceNode : NarrativeNode
{
    public List<ChoiceData> Choices; // { Text, TargetNodeID, Condition }
    public AssetReference BackgroundVideoRef; // 선택지가 표시될 동안 재생될 배경 영상

    public override void EnterNode(NarrativeDirector director)
    {
        Debug.Log($"[ChoiceNode] Presenting choices.");
        // Director가 UIManager를 통해 Choices를 표시하도록 지시
        // 각 선택지의 TargetNode를 미리 로딩 시작
        foreach (var choice in Choices)
        {
            NarrativeNode targetNode = director.GetNodeByID(choice.TargetNodeID);
            director.StartPredictiveLoading(new List<NarrativeNode> { targetNode });
        }
    }
    // ...
}

// NarrativeDirector.cs (메인 컨트롤러)
public class NarrativeDirector : MonoBehaviour
{
    public NarrativeNode StartNode;
    private NarrativeNode _currentNode;
    private Dictionary<string, NarrativeNode> _nodeMap; // ID로 노드를 빠르게 찾기 위함

    private void Awake()
    {
        // 모든 NarrativeNode ScriptableObject를 로드하여 _nodeMap 초기화
        // Addressables로 관리하면 더 효율적입니다.
    }

    public void StartGame()
    {
        _currentNode = StartNode;
        _currentNode.EnterNode(this);
    }

    public void MoveToNode(NarrativeNode nextNode)
    {
        _currentNode.ExitNode(this);
        _currentNode = nextNode;
        _currentNode.EnterNode(this);
    }

    // 예측 로딩 로직 (자세한 내용은 다음 섹션)
    public void StartPredictiveLoading(List<NarrativeNode> potentialNextNodes)
    {
        // ... (Async/Await 또는 Coroutine으로 비동기 로딩 처리)
    }

    public NarrativeNode GetNodeByID(string id) => _nodeMap.TryGetValue(id, out var node) ? node : null;
}
```

이 구조 덕분에 디자이너는 유니티 에디터에서 `NarrativeNode`들을 생성하고, 드래그앤드롭으로 연결하며 스토리를 시각적으로 구성할 수 있게 되었습니다. 에디터 툴을 좀 더 만들면, 노드 간 연결을 그래프 뷰로 볼 수 있어 더욱 직관적이겠죠.

#### 2.2. Deep Dive: 유니티 최적화 트릭의 진수 - 예측 로딩과 비동기 스트리밍

이제 이 내러티브 시스템이 '버벅임 없는 흐름'을 만들어내기 위한 유니티 최적화 기술들을 살펴볼 차례입니다.

*   **문제:** 거대한 영상 파일이 게임 빌드 용량을 부풀리고, 로드 시 메모리를 한 번에 점유하며, 네트워크 스트리밍도 불안정합니다.
*   **해결:** **Unity Addressables**와 **비동기 로딩(Async Loading)**을 조합하고, **예측 로딩 캐시(Predictive Loading Cache)**를 구현했습니다.

1.  **Addressables로 에셋 모듈화:**
    각 영상 클립, 자막 데이터, UI 프리팹 등을 Addressables 그룹으로 묶어 관리합니다. 이렇게 하면 필요한 에셋만 그때그때 로드하고, 더 이상 필요 없는 에셋은 언로드하여 메모리를 효율적으로 사용할 수 있습니다. 또한, 원격 서버에 영상을 배치하여 온디맨드(On-demand) 방식으로 다운로드하거나, 패치 용량을 줄이는 데도 유리하죠.

2.  **비동기 예측 로딩 구현:**
    `NarrativeDirector`는 현재 노드에서 다음으로 이어질 수 있는 모든 잠재적인 노드들을 분석합니다. 예를 들어, `VideoNode`가 재생 중일 때, 다음에 `ChoiceNode`가 나올 것이 예상되면, 그 `ChoiceNode`가 참조하는 모든 `VideoClipRef` (다음 영상들)과 UI 프리팹들을 백그라운드에서 미리 로드하기 시작합니다.

```csharp
// 의사 코드: Predictive Video Streamer & Asset Loader
public class PredictiveStreamer : MonoBehaviour
{
    private Dictionary<string, AsyncOperationHandle<GameObject>> _loadedUIPrefabs = new Dictionary<string, AsyncOperationHandle<GameObject>>();
    private Dictionary<string, AsyncOperationHandle<VideoClip>> _loadedVideoClips = new Dictionary<string, AsyncOperationHandle<VideoClip>>();

    // 예측 로딩 시작 (다음 노드의 모든 잠재적 에셋들을 로드)
    public async Task PreloadAssetsAsync(List<NarrativeNode> nodesToPreload)
    {
        List<Task> loadTasks = new List<Task>();
        foreach (var node in nodesToPreload)
        {
            if (node is VideoNode videoNode)
            {
                if (!_loadedVideoClips.ContainsKey(videoNode.VideoClipRef.AssetGUID))
                {
                    loadTasks.Add(LoadVideoClipAsync(videoNode.VideoClipRef));
                }
            }
            else if (node is ChoiceNode choiceNode)
            {
                // 선택지 UI 프리팹 로드 (예시)
                if (!_loadedUIPrefabs.ContainsKey("ChoiceUIPrefab")) // 선택지 UI는 공통 프리팹으로 가정
                {
                    loadTasks.Add(LoadUIPrefabAsync("ChoiceUIPrefabReference"));
                }
                // 배경 영상이 있다면 로드
                if (choiceNode.BackgroundVideoRef != null && !_loadedVideoClips.ContainsKey(choiceNode.BackgroundVideoRef.AssetGUID))
                {
                     loadTasks.Add(LoadVideoClipAsync(choiceNode.BackgroundVideoRef));
                }
            }
            // ... 다른 노드 타입 처리
        }
        await Task.WhenAll(loadTasks); // 모든 로딩이 완료될 때까지 대기
        Debug.Log("Predictive loading complete!");
    }

    private async Task LoadVideoClipAsync(AssetReference videoRef)
    {
        var handle = videoRef.LoadAssetAsync<VideoClip>();
        _loadedVideoClips[videoRef.AssetGUID] = handle;
        await handle.Task;
        Debug.Log($"Video clip loaded: {videoRef.AssetGUID}");
    }

    private async Task LoadUIPrefabAsync(string referenceId)
    {
        var handle = Addressables.LoadAssetAsync<GameObject>(referenceId);
        _loadedUIPrefabs[referenceId] = handle;
        await handle.Task;
        Debug.Log($"UI prefab loaded: {referenceId}");
    }

    // 더 이상 필요 없는 에셋 언로드
    public void UnloadAsset(string assetGUID)
    {
        if (_loadedVideoClips.TryGetValue(assetGUID, out var handle))
        {
            Addressables.Release(handle);
            _loadedVideoClips.Remove(assetGUID);
            Debug.Log($"Video clip unloaded: {assetGUID}");
        }
        // ... UI 프리팹 등 다른 에셋도 처리
    }
}
```

이 `PredictiveStreamer`는 `NarrativeDirector`에 의해 제어되며, `VideoPlayer` 컴포넌트와 연동됩니다. `VideoPlayer`는 이미 로드된 `VideoClip`을 즉시 받아 재생하고, 필요하다면 `VideoPlayer.Prepare()`를 호출하여 미리 준비 상태로 만들 수도 있습니다.

3.  **UI & 선택지 오버레이 최적화:**
    *   **문제:** 선택지가 나타날 때마다 UI 요소를 `Instantiate()`하면 GC 스파이크가 발생합니다.
    *   **해결:** **Object Pooling**을 적극적으로 활용했습니다. 선택지 버튼이나 자막 UI 패널 등은 미리 풀링해두고, 필요할 때만 활성화하여 재사용합니다. 이렇게 하면 런타임에 새로운 객체를 할당하는 비용을 최소화하여 프레임 드랍을 방지할 수 있습니다.
    *   **UGUI 배치 최적화:** `Canvas`를 적절히 분리하고, `Graphic Raycaster` 설정을 최소화하며, 복잡한 UI는 Static Batching을 고려하는 등 UGUI의 렌더링 오버헤드를 줄이는 일반적인 최적화 기법도 중요합니다.

### 3. 아키텍처 다이어그램 (텍스트 버전)

이 흐름 속에서 `NarrativeDirector`는 스토리의 중심을 잡고, `PredictiveStreamer`는 미래를 대비하며 에셋을 조달하고, `UIManager`는 플레이어와의 상호작용을 매끄럽게 중개합니다. 이 세 요소가 유기적으로 움직여 하나의 '끊김 없는 영화' 경험을 만들어내는 것이죠.

### 4. 시니어의 팁: "대충은 없다, 프로파일링만이 살 길!"

제가 이 시스템을 만들면서 가장 중요하게 생각했던 것은 **"대충 될 거야"라는 안일한 생각을 버리고, 항상 프로파일러를 켜고 개발하는 습관**이었습니다. 특히 인터랙티브 무비 게임처럼 영상 스트리밍과 실시간 상호작용이 복합적으로 일어나는 시스템에서는 작은 최적화 요소들이 쌓여 큰 성능 저하를 일으킬 수 있거든요.

기억나는 에피소드가 하나 있습니다. 한창 개발 중인데, 특정 선택지를 누를 때마다 미세하게 0.1초 정도의 멈춤이 발생하는 겁니다. 육안으로는 잘 안 느껴지지만, 저는 이 '찰나의 끊김'이 플레이어의 몰입을 방해할 것이라고 확신했어요. 프로파일러를 돌려보니, 작은 UI 아이콘 몇 개가 선택지가 나타날 때마다 `Instantiate`되고 `Destroy`되면서 미미한 GC 스파이크를 일으키고 있던 겁니다. 겨우 아이콘 몇 개인데? 놀랍죠. 풀링으로 바꾸자마자 그 찰나의 멈춤은 사라졌고, 저는 "역시 프로파일링만이 살 길!"이라고 외치며 팀원들에게 또 잔소리를 늘어놓았죠. (다들 저를 '프로파일러 꼰대'라고 부르지만, 이게 다 게임을 위함이라는 걸 언젠가 알아줄 겁니다.)

### 맺음말

'System of Interactive Movie Games'는 단순한 기술 구현을 넘어, 플레이어에게 깊은 몰입감과 새로운 경험을 선사하는 예술의 영역이라고 생각합니다. 그리고 그 예술을 매끄럽게 전달하기 위한 핵심은 바로 **'Unity Engine Optimization Tricks'**입니다. 메모리 관리, 비동기 로딩, 오브젝트 풀링, 그리고 정교한 내러티브 구조화까지. 이 모든 것이 유기적으로 결합될 때 비로소 우리는 기술적인 한계를 넘어선, 진정한 '인터랙티브 무비'를 만들어낼 수 있습니다.

물론, 제가 오늘 공유한 내용은 저희 팀의 접근 방식 중 일부일 뿐입니다. 각자의 프로젝트와 목표에 따라 더 많은, 더 기발한 최적화 기법들이 존재할 것입니다. 하지만 중요한 것은 **문제의 본질을 이해하고, 그것을 해결하기 위해 어떤 유니티 기능과 기법들을 활용할 수 있을지 깊이 고민하는 자세**라고 생각합니다.

여러분의 프로젝트에도 오늘 제가 드린 이야기가 작게나마 영감을 줄 수 있다면 더할 나위 없이 기쁠 것 같습니다. 다음번에는 더 흥미로운 기술 이야기로 찾아오겠습니다.
