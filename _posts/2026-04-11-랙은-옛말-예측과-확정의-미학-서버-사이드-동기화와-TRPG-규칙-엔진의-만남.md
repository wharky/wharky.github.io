우리가 만드는 게임은 대부분 '실시간'이라는 환상을 플레이어에게 제공해야 합니다. 하지만 현실은 녹록지 않죠. 유저와 서버 사이의 물리적인 거리가 만들어내는 '지연(Latency)'은 이 환상을 깨뜨리는 주범입니다. 특히 빠른 반응 속도가 중요한 액션 게임이나 격투 게임에서는 100ms의 핑 차이만으로도 플레이어 경험은 나락으로 떨어집니다. "내가 분명히 눌렀는데!"라는 절규는 개발자에게 가장 듣기 싫은 피드백 중 하나일 겁니다.

이 문제를 해결하기 위해 우리는 클라이언트와 서버가 게임 상태를 동기화하는 다양한 기법들을 발전시켜 왔습니다. 그중 대표적인 두 가지가 바로 Dead Reckoning과 Rollback Netcode입니다.

### 1. Dead Reckoning: "일단 너 믿고 간다, 틀리면 고쳐줄게!"

Dead Reckoning은 이름 그대로 '사전 예측'에 기반한 동기화 방식입니다. 클라이언트는 자신의 입력과 현재 상태를 기반으로 다음 게임 상태를 예측하고, 그 예측된 상태를 화면에 보여줍니다. 서버는 클라이언트로부터 받은 입력을 처리하고, 실제 게임 상태를 계산한 뒤 주기적으로 클라이언트에게 '정답'을 보내줍니다. 클라이언트는 이 정답과 자신의 예측을 비교하여 오차가 발생하면 부드럽게 보정합니다.

*   **언제 유용한가?**: 캐릭터 이동, 투사체 움직임 등 연속적이고 비교적 오차 허용 범위가 넓은 상황에서 빛을 발합니다. MMORPG 같은 게임에서 플레이어 캐릭터나 NPC의 움직임이 다소 늦게 보정되더라도 큰 문제가 없는 경우에 주로 사용됩니다.
*   **동작 방식 (Pseudo-code)**:

    ```
    // 클라이언트 측
    func Update(deltaTime):
        // 1. 플레이어 입력 처리 (예: 좌측 키 눌림)
        input = GetLocalPlayerInput()

        // 2. 현재 상태 기반으로 다음 상태 예측
        predictedPosition = currentPosition + input.direction * moveSpeed * deltaTime

        // 3. 예측된 상태 렌더링
        Render(predictedPosition)

        // 4. 서버로 입력 전송
        SendInputToServer(input)

        // 5. 서버로부터 보정 메시지 수신 시 처리
        if (receivedServerState):
            // 내 예측과 서버의 정답 비교
            if (abs(predictedPosition - receivedServerState.position) > threshold):
                // 오차가 크면 서버 상태로 부드럽게 보간 (Lerp)
                currentPosition = Lerp(currentPosition, receivedServerState.position, correctionRate * deltaTime)
            else:
                currentPosition = predictedPosition // 오차가 적으면 예측 상태 유지
    ```

*   **단점**: 예측이 크게 빗나가면 캐릭터가 순간이동(teleport)하거나 '뚝뚝 끊기는' 현상이 발생할 수 있습니다. 특히 입력에 대한 즉각적인 피드백과 정확성이 중요한 복잡한 상호작용에서는 한계가 명확합니다.

### 2. Rollback Netcode: "없던 일로 하자, 시간을 되돌릴게!"

여기서 게임 개발의 진정한 마법이 시작됩니다. Rollback Netcode는 Dead Reckoning이 가진 예측의 한계를 정면으로 돌파합니다. 클라이언트는 상대방의 입력이 아직 도착하지 않았더라도, 마치 입력이 도착한 것처럼 '예측'하여 게임을 시뮬레이션합니다. 그리고 나중에 진짜 입력이 도착하면, **과거로 돌아가(Rollback)** 해당 시점의 게임 상태를 복원하고, 올바른 입력으로 게임을 다시 시뮬레이션하여 현재까지의 상태를 **재계산(Resimulation)**합니다. 이 과정이 굉장히 빠르게 이루어지기 때문에 플레이어는 랙을 거의 느끼지 못하고 즉각적인 반응성을 경험하게 됩니다.

*   **언제 유용한가?**: 격투 게임, 리듬 게임, 또는 핵앤슬래시처럼 '프레임 단위'의 정확한 입력과 즉각적인 반응이 필수적인 장르에서 압도적인 성능을 자랑합니다.
*   **핵심 요구사항: 완벽한 결정론(Perfect Determinism)**
    Rollback이 작동하려면 게임의 시뮬레이션 로직이 **완벽하게 결정론적**이어야 합니다. 즉, 동일한 초기 상태와 동일한 입력이 주어지면, 언제나 정확히 동일한 결과 상태를 만들어내야 합니다. 단 한 비트라도 다르면 안 됩니다. 부동 소수점 연산 오차, 시스템 시간 의존성, 순서가 보장되지 않는 해시 테이블 등 결정론을 깨뜨리는 모든 요소를 철저히 배제해야 합니다.
*   **동작 방식 (핵심 로직)**:

    ```
    // 클라이언트 측 (개념적)
    struct GameState { /* 캐릭터 위치, 체력, 아이템 등 모든 게임 상태 */ }
    var historyStates = new Dictionary<int, GameState>() // 프레임별 게임 상태 저장
    var inputBuffer = new Dictionary<int, PlayerInput[]>() // 프레임별 플레이어 입력 저장
    var currentFrame = 0

    func AdvanceFrame(inputs):
        // 현재 프레임 상태 저장 (롤백 시 필요)
        historyStates[currentFrame] = Clone(currentState)
        inputBuffer[currentFrame] = inputs

        // 모든 입력으로 게임 상태 업데이트 (여기서 게임 로직이 실행됨)
        UpdateGameState(currentState, inputs)
        currentFrame++

    func OnReceiveRemoteInput(frame, remotePlayerInput):
        // 1. 해당 프레임의 과거 상태 로드 (Rollback)
        var restoredState = historyStates[frame]
        var restoredInputBuffer = inputBuffer[frame]
        restoredInputBuffer[remotePlayerID] = remotePlayerInput // 올바른 입력으로 교체

        // 2. 해당 프레임부터 현재 프레임까지 다시 시뮬레이션 (Resimulation)
        for f = frame to currentFrame:
            var inputsForFrame = inputBuffer[f]
            UpdateGameState(restoredState, inputsForFrame) // 게임 로직 실행
            // f 프레임의 상태 저장 (재시뮬레이션 결과로 업데이트)
            historyStates[f] = Clone(restoredState)

        // 현재 상태를 재시뮬레이션 결과로 업데이트
        currentState = restoredState
    ```

    보시면 아시겠지만, Rollback은 단순히 예측을 넘어 시간을 되돌리는 마법에 가깝습니다. 하지만 그 마법의 전제 조건은 바로 '모든 것을 예측 가능하고, 일관되게 재현할 수 있는' 시스템, 즉 완벽한 결정론입니다.

### TRPG 규칙 시스템, 결정론의 정수(Essence of Determinism)

여기서 TRPG 규칙 시스템 이야기가 나옵니다. "아니, 턴제 게임에 웬 Rollback?"이라고 생각하실 수도 있습니다. 물론 TRPG는 Rollback Netcode가 필요한 초고속 액션 게임과는 거리가 있습니다. 하지만 TRPG 규칙을 코드로 구현하는 과정은 Rollback Netcode를 위한 게임 코드를 작성하는 것과 놀라울 정도로 유사한 사고방식을 요구합니다.

TRPG 규칙을 코드로 구현한다는 것은, 주사위 굴림, 캐릭터 스탯, 스킬 사용, 마법 효과 등 복잡한 상호작용의 결과를 '애매함 없이' 확정적인 로직으로 만들어내는 것을 의미합니다.

*   **주사위 굴림**: `RollDice(numFaces, count)` 함수는 특정 시드(seed)와 함께 호출될 때 항상 동일한 결과 시퀀스를 반환해야 합니다. (`Random.Next()` 같은 내장 함수는 피해야 할 1순위입니다. 고유한 결정론적 난수 생성기(Deterministic RNG)를 구현해야 합니다.)
*   **스킬 / 마법 계산**: 캐릭터의 힘, 대상의 방어력, 시전자의 레벨, 스킬의 특성치 등 모든 변수를 인자로 받으면, 데미지 계산이든 성공 확률 계산이든 언제나 정확히 같은 결과가 나와야 합니다.
*   **상태 변화**: 아이템 사용, 버프/디버프 적용, HP 감소 등 모든 상태 변화는 외부 요인 없이 오직 현재 상태와 입력된 액션에 의해서만 결정되어야 합니다.

### 왜 이 두 키워드를 함께 봐야 할까요?

제 경험상, Rollback Netcode를 성공적으로 구현하려 노력하는 개발팀은 필연적으로 자신들의 핵심 게임 로직(시뮬레이션 레이어)을 완벽한 결정론 구조로 재편하게 됩니다. 이는 마치 TRPG 마스터가 복잡한 규칙들을 '만국 공통의 언어'로 명확하게 정의하듯, 게임의 모든 규칙과 상호작용을 수학적이고 논리적인 상태 전이 함수로 만들어내는 과정과 같습니다.

저는 종종 TRPG 규칙을 코드로 구현할 때, "이 규칙은 네트워크 게임에서 롤백 가능한가?"라는 질문을 던져봅니다. 비록 실제로 롤백이 필요한 게임이 아니더라도, 이 질문은 제 코드를 훨씬 더 견고하고, 테스트 가능하며, 예측 가능한 상태로 만듭니다.

*   **순수 함수(Pure Function)의 중요성**: 외부 상태에 의존하지 않고, 오직 주어진 입력에 의해서만 결과를 내는 함수들은 결정론적 시스템의 핵심입니다. `CalculateDamage(attackerStats, defenderStats, abilityData)` 와 같은 함수는 언제나 동일한 입력을 받으면 동일한 데미지를 반환해야 합니다.
*   **Side Effect 배제**: 함수 내부에서 전역 변수를 수정하거나, 입출력 작업을 수행하는 등의 '부수 효과(Side Effect)'는 결정론을 해칩니다. 모든 상태 변화는 명시적인 반환 값이나 인자 전달을 통해 이루어져야 합니다.
*   **명확한 상태 정의**: 게임의 모든 '상태' (캐릭터 위치, 인벤토리, 버프 목록, 심지어 주사위 시퀀스까지)는 명확하게 정의되고 직렬화(serialize) 가능해야 합니다. 이는 Rollback에서 과거 상태를 저장하고 복원하는 데 필수적입니다.

### 깊이 있는 실무 팁: TRPG 규칙 엔진을 Rollback 친화적으로 만들기

TRPG 규칙을 코드로 구현하면서 동시에 Rollback Netcode의 철학을 녹여내는 몇 가지 팁을 드립니다.

1.  **시드 기반 결정론적 난수 생성기(Seed-based Deterministic RNG) 사용**: 절대 `System.Random` 이나 `Math.random()` 같은 비결정론적 난수 생성기를 쓰지 마세요. 게임의 프레임 또는 특정 이벤트 시퀀스에 고정된 시드를 주입하여, 모든 클라이언트가 동일한 주사위 굴림 결과를 얻도록 해야 합니다.

    ```
    // Pseudocode for Deterministic RNG
    class DeterministicRNG:
        var seed: int

        constructor(initialSeed: int):
            seed = initialSeed

        func NextInt(min: int, max: int): int:
            // LCG (Linear Congruential Generator) 예시
            // 상수들은 실제 구현에서 잘 정의된 값을 사용 (e.g., C++11 std::minstd_rand)
            seed = (seed * 1103515245 + 12345) % 2^31
            return min + (seed % (max - min + 1))
    ```

    게임 루프에서 매 프레임마다, 혹은 특정 액션마다 이 RNG의 `NextInt()`를 호출하여 필요한 난수를 얻습니다. 이 RNG 객체 자체도 게임 상태의 일부로 직렬화되어야 합니다.

2.  **모든 게임 로직을 '순수 함수'로**: TRPG의 모든 규칙 (데미지 계산, 스킬 발동, 상태 이상 적용 등)은 현재 게임 상태와 입력된 액션을 인자로 받아 새로운 게임 상태를 반환하는 순수 함수로 구현하세요.

    ```
    // Pseudocode for Pure Function
    func ApplyAttack(state: GameState, attackerId: int, targetId: int, skillId: int): GameState:
        // 새로운 GameState 객체를 생성하여 반환 (기존 state를 직접 수정 금지)
        var newState = Clone(state)
        var attacker = newState.GetCharacter(attackerId)
        var target = newState.GetCharacter(targetId)
        var skill = GetSkillData(skillId)

        // 1. 공격력/방어력 계산 (Deterministic RNG로 주사위 굴림)
        // newState의 rng 객체를 사용하여 난수를 생성해야 합니다!
        var damageRoll = newState.rng.NextInt(1, 6) * 2 // 예시: 2d6 주사위
        var finalDamage = CalculateFinalDamage(attacker.stats, target.stats, skill.properties, damageRoll)

        // 2. 대상의 체력 감소
        target.hp -= finalDamage
        if (target.hp <= 0):
            target.isDead = true

        // 기타 상태 변화 (버프, 디버프 등)

        return newState // 변경된 새 상태 반환
    ```

3.  **Simulation Layer와 Presentation Layer 분리**: 게임 로직(Simulation)은 네트워크 동기화가 가능하도록 완벽하게 결정론적으로 만들고, 그래픽이나 사운드 같은 시각적/청각적 요소(Presentation)는 이 시뮬레이션 결과를 단순히 렌더링하도록 만드세요. Rollback이 일어날 때, 시뮬레이션은 과거로 돌아가지만 Presentation은 현재 예측된 상태를 유지하다가, 재시뮬레이션이 끝나면 새로운 '정답' 상태로 부드럽게 전환될 수 있습니다.

### 마치며

'Server-Side Network Sync'와 'TRPG Rule Systems'는 표면적으로 달라 보이지만, 결국 '게임의 규칙을 코드로 어떻게 견고하고 예측 가능하게 만들 것인가?'라는 본질적인 질문으로 수렴합니다. Rollback Netcode의 결정론적 요구사항은 여러분의 TRPG 규칙 엔진을 더욱더 강력하고, 오류에 강하며, 확장 가능한 구조로 만들어줄 겁니다.

이러한 사고방식은 단순히 네트워크 게임에만 국한되지 않습니다. 저는 복잡한 비즈니스 로직이나 시뮬레이션 시스템을 설계할 때도, '이 시스템이 롤백 가능하다면 어떤 모습일까?'라는 질문을 던져보곤 합니다. 여러분의 코드가 시간 여행을 할 수 있다고 상상해보세요. 그러면 자연스럽게 순수 함수, 불변 객체, 명확한 상태 전이 같은 좋은 디자인 원칙을 따르게 될 겁니다.

기술은 결국 우리가 상상하는 세계를 현실로 만들어주는 도구입니다. 복잡한 TRPG 세계의 규칙들을 코드로 아름답게 직조하고, 동시에 네트워크 너머의 플레이어들에게 '랙 없는' 경험을 선사하는 것. 이 두 가지를 모두 잡는 것은 개발자로서 정말 도전적이고, 또 짜릿한 경험이 될 것입니다.
