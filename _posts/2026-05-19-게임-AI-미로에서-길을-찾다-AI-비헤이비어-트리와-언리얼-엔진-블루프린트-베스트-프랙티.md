### 챕터 1: 혼돈 속의 질서 – 왜 우리는 비헤이비어 트리를 쓰는가? (Problem & Solution)

저는 경력 초반에 상태 머신(State Machine)으로 AI를 구현하며 꽤나 고생했습니다. NPC의 행동이 조금만 복잡해져도 상태 전이(State Transition) 조건이 거미줄처럼 엉켜 디버깅은커녕 이해조차 힘들어지는 '스파게티 코드'의 늪에 빠지기 일쑤였죠. "이 NPC는 왜 여기서 갑자기 이걸 하지?"라는 질문에 답을 찾기 위해 몇 시간씩 그래프를 쫓아다니던 기억이 생생합니다.

**문제:** 전통적인 상태 머신은 유한한 상태와 명확한 전이를 가진 AI에는 적합하지만, 복잡하고 유연하며 계층적인 행동 결정이 필요한 AI에서는 관리하기 어렵고 확장성이 떨어집니다. 특히 다수의 조건과 우선순위를 고려해야 할 때, 상태 폭발(State Explosion) 현상에 직면하게 됩니다.

**해결책:** 바로 여기에 **AI 비헤이비어 트리(Behavior Tree, BT)**가 등장합니다. 비헤이비어 트리는 행동을 계층적인 트리 구조로 모델링하여, 복잡한 AI 로직을 보다 직관적이고 모듈화된 방식으로 관리할 수 있게 해줍니다. 마치 의사결정 과정을 시각화한 것과 같죠. `Selector`, `Sequence`, `Parallel`, `Decorator`, `Task` 같은 노드들이 결합하여 NPC의 행동 우선순위와 조건을 명확하게 정의합니다. "적이 보이면 추격하고 공격해라, 아니면 경계를 서라, 그것도 아니면 순찰해라" 같은 복잡한 판단 흐름을 한눈에 파악할 수 있게 되는 거죠.

언리얼 엔진은 이 비헤이비어 트리를 기본으로 제공하며, 직관적인 에디터를 통해 시각적으로 AI 로직을 구성할 수 있도록 지원합니다. 하지만 '시각적'이라는 단어가 항상 만능은 아닙니다. 제대로 된 설계 철학 없이는 블루프린트도 스파게티가 될 수 있듯이, 비헤이비어 트리도 자칫하면 또 다른 형태의 혼돈을 낳을 수 있습니다.

### 챕터 2: 블루프린트, 칼날 위의 춤 – 베스트 프랙티스 (Practical & Opinionated)

언리얼 엔진에서 비헤이비어 트리는 대부분 블루프린트(Blueprint)를 통해 구현됩니다. 강력한 시각 스크립팅 도구인 블루프린트는 빠른 프로토타이핑과 이터레이션을 가능하게 하지만, 동시에 잘못 사용하면 성능 저하와 유지보수 지옥을 초래할 수 있습니다. 그래서 **Unreal Engine Blueprints Best Practices**를 비헤이비어 트리 개발에 적용하는 것이 중요합니다.

**1. 모듈화와 재사용성: 작은 단위로 쪼개고, 또 쪼개라!**
가장 중요한 원칙입니다. 거대한 하나의 비헤이비어 트리보다는, 특정 행동 덩어리(예: 전투 행동, 순찰 행동, 대피 행동)를 모듈화된 서브 트리(Subtree)로 구성하는 것을 강력히 추천합니다.

*   **Custom BT Tasks (C++ & Blueprint):**
    성능이 중요하거나 복잡한 로직은 C++로 `UBehaviorTreeTask_BlueprintBase`를 상속받아 구현하고, 이를 블루프린트에서 사용하도록 노출하세요. 간단한 로직이나 데이터 기반의 Task는 블루프린트 자체 Task로 구현해도 좋습니다.
    예를 들어, "적이 시야에 있는지 확인" 같은 Perception 관련 로직은 C++에서 빠르게 처리하고, "랜덤하게 이동할 지점 찾기" 같은 간단한 로직은 블루프린트 Task로 구현하는 식이죠.

    ```cpp
    // Pseudo-code for a C++ Custom BT Task
    // UBTTask_CheckLineOfSight.h
    UCLASS()
    class MYGAME_API UBTTask_CheckLineOfSight : public UBTTask_BlueprintBase
    {
        GENERATED_BODY()
    public:
        UBTTask_CheckLineOfSight();
        virtual EBTNodeResult::Type ExecuteTask(UBehaviorTreeComponent& OwnerComp, uint8* NodeMemory) override;

    protected:
        // C++에서만 사용하거나, BlueprintEditable로 노출하여 블루프린트에서 설정 가능
        UPROPERTY(EditAnywhere, Category = "Target")
        FBlackboardKeySelector TargetActorKey;
    };

    // UBTTask_CheckLineOfSight.cpp
    EBTNodeResult::Type UBTTask_CheckLineOfSight::ExecuteTask(UBehaviorTreeComponent& OwnerComp, uint8* NodeMemory)
    {
        // 빠르게 시야 검사 로직 수행 (C++ 강점 활용)
        // 결과에 따라 Succeeded 또는 Failed 반환
        return EBTNodeResult::Succeeded; // 혹은 Failed
    }
    ```

*   **Blueprint Function Libraries & Interfaces:**
    AI 전반에서 공통으로 사용되는 유틸리티 함수(예: 가장 가까운 적 찾기, 안전 지점 계산)는 Blueprint Function Library로 만들어 재사용성을 높이세요. AI 캐릭터 간의 소통이나, AI와 환경 간의 상호작용은 Blueprint Interface를 활용하여 느슨한 결합(Loose Coupling)을 유지하는 것이 좋습니다.

**2. Blackboard의 현명한 활용: AI의 뇌를 구조화하라!**
비헤이비어 트리의 핵심 파트너는 바로 **블랙보드(Blackboard)**입니다. AI의 현재 상태, 목표, 인지 정보 등 모든 데이터를 블랙보드 키-값 쌍으로 관리해야 합니다. 이는 AI의 '단기 기억'과 같죠.

*   **핵심 원칙:** 비헤이비어 트리 노드 자체에서 직접 데이터를 저장하거나 복잡한 계산을 하는 대신, 블랙보드에서 데이터를 읽고 쓰는 데 집중하세요. 이렇게 하면 각 노드가 더욱 단순해지고, AI의 상태를 블랙보드만으로도 쉽게 파악하고 디버깅할 수 있습니다.
*   **블랙보드 키 네이밍:** 명확하고 일관된 네이밍 컨벤션(예: `TargetActor`, `DestinationLocation`, `IsEnemyInSight`)은 필수입니다. 저는 보통 **`BB_`** 접두사를 사용해 블랙보드 키임을 명시합니다.

**3. 가독성과 유지보수: 미래의 나를 위해 투자하라!**
아무리 잘 설계된 로직이라도 알아보지 못하면 무용지물입니다.

*   **명확한 네이밍 컨벤션:** 모든 비헤이비어 트리 노드, 블루프린트 그래프, 함수, 변수에 의미 있는 이름을 부여하세요. `BT_Sequence_AttackPatrol`, `BT_Task_ChaseTarget`, `Decorator_IsInMeleeRange` 등 역할과 타입을 명확히 드러내는 이름이 좋습니다.
*   **주석과 설명:** 복잡한 로직 앞에는 반드시 주석을 달고, 각 노드의 Description 패널을 활용하여 의도를 명확히 밝히세요. "이 시퀀스는 적을 발견했을 때 전투 준비를 하는 로직입니다" 와 같이요.
*   **그래프 정렬:** 언리얼 엔진의 자동 정렬 기능을 활용하거나, 수동으로 노드들을 깔끔하게 배치하여 시각적인 가독성을 극대화해야 합니다.

**4. 성능 최적화: 블루프린트의 늪에 빠지지 마라!**
블루프린트는 C++보다 느릴 수 있습니다. 특히 틱(Tick)마다 호출되거나 반복적으로 실행되는 고성능이 요구되는 로직은 주의해야 합니다.

*   **C++ vs. Blueprint Balance:** '데이터는 블루프린트, 로직은 C++'이라는 만트라를 기억하세요. 또는 '빠른 이터레이션은 블루프린트, 성능 최적화는 C++'도 좋은 기준입니다. 퍼셉션(Perception) 컴포넌트나 경로 탐색(Pathfinding) 같은 엔진 단의 로직은 C++로, 비헤이비어 트리와 상호작용하는 부분은 블루프린트로 구현하는 식의 균형이 중요합니다.
*   **비헤이비어 트리의 Tick 최적화:** `Observer Aborts`와 `Conditional Flow`를 적절히 활용하여 불필요한 노드 평가를 최소화하세요. Decorator 노드의 조건이 충족되지 않으면 해당 서브 트리는 평가되지 않으므로, 이를 통해 성능을 최적화할 수 있습니다.

### 챕터 3: Deep Dive - 전투 AI 행동 트리의 아키텍처 (Code & Logic)

실제 전투 AI의 간략한 비헤이비어 트리 구조를 통해 위에서 언급한 베스트 프랙티스를 어떻게 적용하는지 살펴보겠습니다.

**[그림 1: 전투 AI 비헤이비어 트리 간략화 아키텍처 다이어그램]**
(텍스트 기반 다이어그램으로 표현)

```
Root
 └ Selector (전체 행동 우선순위)
    ├ Decorator: CanSeeEnemy? (적이 보이는가?)
    │  └ Sequence: Combat (전투 행동)
    │     ├ Task: StopMovement (이동 멈추기)
    │     ├ Selector: AttackDecision (공격 의사결정)
    │     │  ├ Decorator: IsTargetInMeleeRange? (근접 공격 범위인가?)
    │     │  │  └ Task: MeleeAttack (근접 공격)
    │     │  └ Decorator: IsTargetInRangedRange? (원거리 공격 범위인가?)
    │     │     └ Task: RangedAttack (원거리 공격)
    │     └ Task: AdjustPosition (위치 조정 - 엄폐, 회피 등)
    ├ Sequence: Patrol (순찰 행동)
    │  ├ Decorator: HasPatrolPath? (순찰 경로가 있는가?)
    │  │  └ Task: FollowPatrolPath (경로 따라 순찰)
    │  └ Task: WanderRandomly (랜덤하게 배회)
    └ Task: Idle (대기)
```

**설명:**

1.  **Root Selector:** AI의 최상위 의사결정 노드입니다. 우선순위는 위에서 아래로 흐릅니다.
2.  **CanSeeEnemy? Decorator:** AI가 적을 감지했는지 확인합니다. 이 조건이 참일 경우에만 'Combat' 시퀀스로 진입합니다. 거짓이라면 다음 우선순위인 'Patrol' 시퀀스를 시도합니다. 이 Decorator는 `Observer Aborts Self`로 설정하여, 전투 중 적이 시야에서 사라지면 즉시 전투 행동을 중단하고 다음 행동을 모색하게 할 수 있습니다.
3.  **Combat Sequence:** 적을 발견했을 때 실행되는 일련의 행동입니다.
    *   `Task: StopMovement`: 현재 이동을 멈춥니다.
    *   `AttackDecision Selector`: 근접 공격이 가능한지, 원거리 공격이 가능한지에 따라 우선순위를 결정합니다.
        *   `IsTargetInMeleeRange?` 데코레이터가 참이면 `MeleeAttack` Task를 실행합니다.
        *   그렇지 않고 `IsTargetInRangedRange?` 데코레이터가 참이면 `RangedAttack` Task를 실행합니다.
    *   `Task: AdjustPosition`: 공격 후 혹은 공격 전에 엄폐, 회피, 재장전 등을 위해 위치를 조정하는 Task입니다. 이는 `BTTask_FindCover`와 같은 C++ 기반 Task로 구현될 수 있습니다.
        *   **블랙보드 활용 예시:** `Task: AdjustPosition`은 `Blackboard`에서 `TargetLocation` 키를 읽어와 새로운 이동 목표 지점을 설정할 수 있습니다. `BTTask_FindCover`가 성공하면 `Blackboard`의 `DestinationLocation` 키를 업데이트하는 식이죠.
4.  **Patrol Sequence:** 적이 없을 때 실행되는 순찰 행동입니다. `HasPatrolPath?` 데코레이터로 순찰 경로 유무를 확인하고, 없으면 `WanderRandomly` Task를 실행합니다.
5.  **Idle Task:** 모든 조건이 충족되지 않을 때 실행되는 기본 대기 행동입니다.

이러한 구조는 각 행동의 책임과 우선순위를 명확히 하고, 새로운 행동을 추가하거나 기존 행동을 수정할 때 다른 부분에 미치는 영향을 최소화합니다.

### 챕터 4: 나의 개발 이야기 – 실수와 깨달음 (Storytelling & Opinionated)

한때 저는 무지성(?)으로 모든 AI 로직을 단일 거대한 블루프린트 비헤이비어 트리에 밀어 넣으려 했습니다. 당연히 프로젝트 후반부로 갈수록 그 거대한 BT는 '나만 아는 미로'가 되었고, 사소한 버그 하나를 잡기 위해 며칠 밤낮을 새워야 했습니다. "이 작은 변화가 다른 AI 행동에 왜 이런 부작용을 일으키는 거지?"라며 머리를 쥐어뜯던 경험은 지금도 잊을 수 없습니다.

그때 깨달은 것이 바로 '분리(Separation)'와 '모듈화(Modularity)'의 중요성입니다. 비헤이비어 트리는 AI의 의사결정을 담당하지만, AI Controller는 퍼셉션과 Blackboard 관리를, Pawn/Character는 애니메이션과 실제 행동 실행을 담당하는 식으로 명확하게 역할을 분리해야 한다는 것이죠. 그리고 각각의 역할은 Blueprint Interface나 C++ 컴포넌트를 통해 느슨하게 연결되어야 합니다.

결국 AI 개발은 단순히 코드를 짜는 것을 넘어, 'AI의 인지-사고-행동' 흐름을 어떻게 효율적이고 재사용 가능하게 구조화할 것인가에 대한 아키텍처 설계와 같습니다. 블루프린트가 제공하는 놀라운 유연함에 취해 '쉽고 빠르게'만 추구하다가는, 결국 '복잡하고 느리게' 개발하는 자신을 발견하게 될 겁니다.
