언뜻 보면, 네트워크 동기화는 저수준의 구현 기술이고 디자인 패턴은 고수준의 아키텍처 원칙이라 괴리감이 느껴질 수도 있습니다. 하지만 저는 단언컨대, 이 둘은 떼려야 뗄 수 없는 관계라고 말씀드리고 싶습니다. 견고하고 버그 없는 네트워크 코드를 만들려면, 디자인 패턴이 제시하는 구조와 원칙이 마치 뼈대와 같습니다. 복잡한 네트워크 로직 속에서 헤매지 않고, 미래의 확장성과 유지보수성을 확보하려면 반드시 디자인 패턴의 도움을 받아야 합니다.

### 🎮 문제: 랙과 지연 속에서 '진실'을 찾아라!

온라인 멀티플레이어 게임을 개발하는 모든 개발자의 공통된 악몽은 무엇일까요? 바로 '랙(Lag)'과 '데싱크(Desync)'입니다. 플레이어는 분명 적을 쏘았는데 맞지 않았다거나, 나는 분명 피했는데 맞았다고 서버가 주장하는 상황. 우리는 이것을 "서버가 구라를 친다"고 표현하기도 하죠. 이런 불쾌한 경험은 플레이어 이탈의 가장 큰 원인 중 하나입니다.

네트워크 지연은 피할 수 없는 현실입니다. 아무리 네트워크 환경이 좋아져도 빛의 속도를 뛰어넘을 수는 없습니다. 문제는 이 지연 속에서 어떻게 모든 클라이언트가 일관된 게임 상태를 공유하고, 동시에 플레이어에게 즉각적인 반응성을 제공할 것인가입니다.

이 문제를 해결하기 위한 두 가지 강력한 무기가 바로 **Dead Reckoning (클라이언트 예측과 서버 보정)** 과 **Rollback (입력 지연 및 상태 재시뮬레이션)** 입니다.

### 🚀 해결책 1: Dead Reckoning – '내가 예측하고, 서버가 보정한다'

**Dead Reckoning**은 클라이언트가 자신의 입력과 다른 개체들의 예측된 상태를 기반으로 게임 세계를 미리 시뮬레이션하고 렌더링하여 플레이어에게 즉각적인 반응성을 제공하는 기법입니다. 그러다 서버로부터 '정확한' 상태 업데이트가 오면, 클라이언트는 자신의 예측과 서버의 진실을 비교하여 차이가 있을 경우 보정합니다.

**문제점:** 보정 과정에서 화면이 뚝뚝 끊기거나(Snapping), 개체가 갑자기 텔레포트하는(Teleporting) 현상이 발생할 수 있습니다. 이를 완화하기 위해 부드러운 보간(Interpolation) 기법을 사용하기도 합니다.

**🌟 Game Design Patterns의 적용:**

Dead Reckoning의 핵심은 클라이언트가 게임 상태를 예측하고, 서버가 보정하는 과정입니다. 이때 **`State Pattern`**과 **`Command Pattern`**이 빛을 발합니다.

1.  **`State Pattern` (상태 패턴):**
    *   각 게임 개체(플레이어, 몬스터, 발사체 등)는 '예측된 상태(Predicted State)'와 '확정된 상태(Authoritative State)'를 가질 수 있습니다.
    *   클라이언트에서는 `PlayerPredictedState`, `MonsterPredictedState`와 같은 상태 객체를 두어, 입력이 들어오면 해당 상태 객체가 움직임을 처리하도록 합니다. 이는 개체의 행동 로직을 깔끔하게 분리하고, 예측 로직을 모듈화하는 데 큰 도움을 줍니다.
    *   서버에서 보정 데이터가 오면, 클라이언트는 예측된 상태를 서버의 확정된 상태로 전환하거나, 그 사이를 부드럽게 보간하는 `Transition State`를 가동시킬 수 있습니다.

    ```pseudo
    // PlayerState.cs (C# 예시)
    interface IPlayerState {
        void HandleInput(Player player, InputData input);
        void Update(Player player, float deltaTime);
        void OnEnter(Player player);
        void OnExit(Player player);
    }

    class PredictedMovingState : IPlayerState {
        public void HandleInput(Player player, InputData input) { /* 예측 이동 로직 */ }
        public void Update(Player player, float deltaTime) { player.Position += player.PredictedVelocity * deltaTime; }
        // ...
    }

    class CorrectingState : IPlayerState { // 서버 보정 시 진입하는 상태
        Vector3 targetPosition;
        float startTime;
        public CorrectingState(Vector3 targetPos) { targetPosition = targetPos; }
        public void Update(Player player, float deltaTime) {
            // targetPosition까지 부드럽게 보간
            player.Position = Vector3.Lerp(player.Position, targetPosition, (Time.time - startTime) / CorrectionDuration);
            if (Vector3.Distance(player.Position, targetPosition) < Epsilon) player.ChangeState(new PredictedMovingState());
        }
        // ...
    }

    // Player.cs
    class Player {
        IPlayerState currentState;
        public Vector3 Position;
        public Vector3 PredictedVelocity;

        public void ChangeState(IPlayerState newState) {
            currentState?.OnExit(this);
            currentState = newState;
            currentState.OnEnter(this);
        }

        public void OnServerCorrection(Vector3 serverPos) {
            if (Vector3.Distance(Position, serverPos) > CorrectionThreshold) {
                ChangeState(new CorrectingState(serverPos));
            }
        }
    }
    ```

2.  **`Command Pattern` (커맨드 패턴):**
    *   클라이언트가 입력(이동, 공격 등)을 받으면, 이를 즉시 `MoveCommand`, `AttackCommand`와 같은 커맨드 객체로 캡슐화하여 자신의 게임 상태를 예측 시뮬레이션하는 데 사용합니다. 동시에 이 커맨드 객체를 서버로 전송합니다.
    *   서버는 이 커맨드를 받아 유효성을 검사하고, 자신의 Authoritative한 게임 상태에 적용한 후, 그 결과를 다시 클라이언트에게 보냅니다.
    *   이렇게 하면 클라이언트와 서버 간의 입력 처리 로직이 명확히 분리되고, 예측과 보정 로직을 각 커맨드 내에 담아 관리할 수 있습니다.

### 🚀 해결책 2: Rollback – '시간을 되돌려, 진실을 재현한다'

**Rollback Netcode**는 Dead Reckoning보다 훨씬 복잡하지만, 특히 격투 게임이나 매우 빠른 반응 속도를 요구하는 장르에서 플레이어 경험을 극대화하는 궁극적인 해결책으로 여겨집니다.

**원리:**
1.  클라이언트는 상대방의 입력을 기다리지 않고, 자신의 입력과 상대방의 **예측된 입력**을 기반으로 즉시 게임을 시뮬레이션하고 렌더링합니다. (이 부분은 Dead Reckoning과 유사합니다.)
2.  네트워크 지연으로 인해 뒤늦게 상대방의 **실제 입력**이 도착하면, 게임은 해당 입력이 발생해야 했던 과거 시점(프레임)으로 **롤백(Rollback)** 합니다.
3.  과거 시점의 저장된 게임 상태를 불러와, 이제 막 도착한 상대방의 실제 입력을 포함하여 그 시점부터 현재 프레임까지 **재시뮬레이션(Re-simulate)** 합니다.
4.  재시뮬레이션 결과와 현재 화면의 차이가 있다면, 시각적인 아티팩트(Artifact)를 최소화하면서 화면을 업데이트합니다.

**문제점:** 과거 상태를 저장하고 불러오며, 게임 전체를 재시뮬레이션하는 과정은 엄청난 자원과 정교한 설계가 필요합니다. 잘못 구현하면 버그의 온상이 되기 쉽습니다.

**🌟 Game Design Patterns의 적용:**

Rollback은 "시간을 되돌린다"는 개념 때문에 **`Command Pattern`**, **`Memento Pattern`**, 그리고 견고한 게임 상태 관리(`ECS` 또는 `State Pattern`의 확장)가 필수적입니다.

1.  **`Command Pattern` (커맨드 패턴):**
    *   Rollback의 핵심은 '입력 이력'과 '상태 재시뮬레이션'입니다. 모든 플레이어의 입력을 `InputCommand` 형태로 캡슐화하여 매 프레임 저장합니다.
    *   클라이언트가 롤백할 때, 특정 프레임의 `InputCommand`들을 순차적으로 다시 `Execute`하여 게임 상태를 재현합니다. 서버 역시 모든 클라이언트의 입력 커맨드를 받아 Authority 한 시뮬레이션을 수행합니다.
    *   이 패턴을 통해 입력 처리 로직이 명확해지고, 롤백 시점에 어떤 입력을 어떤 순서로 적용해야 하는지가 명확해집니다.

2.  **`Memento Pattern` (메멘토 패턴):**
    *   Rollback은 과거의 게임 상태로 돌아가야 합니다. 이를 위해 매 프레임마다 게임의 중요한 상태(모든 엔티티의 위치, 속도, 체력, 애니메이션 상태 등)를 스냅샷으로 저장해 두어야 합니다. 이것이 바로 **`Memento Pattern`**의 완벽한 사용처입니다.
    *   `Originator`는 게임 상태(Game World)이며, `Memento`는 특정 프레임의 게임 상태 스냅샷입니다. `Caretaker`는 이 `Memento`들을 저장하고 관리하는 `GameStateHistory` 같은 객체가 됩니다.

    ```pseudo
    // Memento.cs
    class GameStateMemento {
        public int FrameNumber;
        public Dictionary<int, EntityState> EntityStates; // Key: Entity ID, Value: Entity's serializable state
        // ... 기타 전역 게임 상태 (예: 시간, 점수, 환경 요소)
    }

    // GameWorld.cs (Originator)
    class GameWorld {
        private List<Entity> entities;
        // ...

        public GameStateMemento SaveState(int frame) {
            var memento = new GameStateMemento { FrameNumber = frame, EntityStates = new Dictionary<int, EntityState>() };
            foreach (var entity in entities) {
                memento.EntityStates[entity.ID] = entity.GetState(); // 각 엔티티의 상태를 직렬화
            }
            return memento;
        }

        public void RestoreState(GameStateMemento memento) {
            FrameNumber = memento.FrameNumber;
            foreach (var kvp in memento.EntityStates) {
                if (entities.Any(e => e.ID == kvp.Key)) {
                    entities.First(e => e.ID == kvp.Key).SetState(kvp.Value);
                }
            }
        }
    }

    // GameStateManager.cs (Caretaker)
    class GameStateManager {
        private Dictionary<int, GameStateMemento> history = new Dictionary<int, GameStateMemento>();
        private GameWorld gameWorld;
        private const int MaxHistoryFrames = 120; // 약 2초치 (60fps 기준)

        public GameStateManager(GameWorld world) { gameWorld = world; }

        public void SaveCurrentState(int frame) {
            if (history.Count >= MaxHistoryFrames) {
                history.Remove(history.Keys.Min()); // 가장 오래된 상태 제거
            }
            history[frame] = gameWorld.SaveState(frame);
        }

        public void RestoreToState(int frame) {
            if (history.TryGetValue(frame, out var memento)) {
                gameWorld.RestoreState(memento);
            } else {
                // 에러 처리: 해당 프레임의 상태가 없음
            }
        }
    }
    ```

3.  **`Component-Entity-System (ECS)` 또는 `State Pattern`의 확장:**
    *   Rollback을 성공적으로 구현하려면, 게임 상태가 매우 잘 정의되고 쉽게 저장/복원될 수 있어야 합니다. 엔티티 간의 의존성이 낮고, 순수 데이터 위주로 상태가 관리될수록 유리합니다.
    *   `ECS`는 이런 요구사항에 매우 적합합니다. 각 `Component`는 데이터만 가지며, `System`은 이 데이터를 기반으로 동작을 처리합니다. `Memento`를 만들 때는 각 `Component`의 데이터를 직렬화하고, 복원할 때는 다시 역직렬화하여 `Component`에 할당하면 됩니다. 이는 게임 상태를 모듈화하고 롤백 가능하게 만드는 데 탁월합니다.
    *   복잡한 `GameObject` 계층 구조나 싱글톤에 강하게 의존하는 구조에서는 Rollback 구현이 거의 불가능에 가깝습니다. 각 엔티티의 `State Pattern`이 더 세분화되어, 각 상태마다 필요한 데이터를 명확히 정의하는 것도 방법입니다.

    ```pseudo
    // Rollback Game Loop (Client-Side Simplified)
    function GameLoop():
        current_frame++
        gameStateManager.SaveCurrentState(current_frame); // Memento 패턴으로 현재 상태 저장
        inputHistory.AddInput(current_frame, GetLocalInput()); // Command 패턴으로 로컬 입력 저장

        // 예측 입력 처리 (상대방 입력이 아직 안왔을 경우)
        // SimulateCurrentFrame(inputHistory.GetInputs(current_frame)); 

        if (Network.HasNewServerData()) {
            serverData = Network.GetServerData();
            if (serverData.ConfirmedFrame > lastConfirmedFrame) {
                // 1. Rollback: 확정된 프레임 이전 상태로 되돌린다.
                gameStateManager.RestoreToState(serverData.ConfirmedFrame - 1);

                // 2. Re-simulate: 확정된 프레임부터 현재 프레임까지 다시 시뮬레이션한다.
                for (frame = serverData.ConfirmedFrame; frame <= current_frame; frame++) {
                    inputs = serverData.GetAuthoritativeInputs(frame); // 서버가 확정한 입력
                    if (inputs == null) { // 아직 서버가 확정하지 않은 프레임이면
                        inputs = inputHistory.GetInputs(frame); // 클라이언트 예측 입력 사용
                    }
                    SimulateSingleFrame(inputs); // Command 패턴으로 입력 실행
                    
                    // (옵션) 비주얼 보정: 롤백으로 인한 시각적 끊김을 부드럽게 처리
                    VisualCorrector.SmoothChanges(); 
                }
                lastConfirmedFrame = serverData.ConfirmedFrame;
            }
        }
        RenderCurrentState();
    ```

### 💡 저의 경험담: "아키텍처가 없으면 지옥을 맛본다"

제가 과거에 참여했던 한 FPS 프로젝트에서 데드 레코닝을 구현할 때의 일입니다. 처음에는 단순히 클라이언트에서 예측하고, 서버에서 좌표만 보내주면 되겠지 하는 안일한 생각으로 시작했습니다. 하지만 엔티티의 상태가 복잡해지고, 물리 엔진과의 상호작용, 애니메이션, 그리고 여러 특수 스킬들이 추가되면서 예측과 보정 로직이 난잡하게 꼬이기 시작했습니다.

어떤 버그는 특정 플레이어만 벽을 뚫고 지나가는 현상이었고, 또 다른 버그는 캐릭터가 순간이동하는 현상이었습니다. 디버깅이 정말 지옥 같았죠. `State Pattern`으로 각 엔티티의 상태(Idle, Moving, Attacking, Stunned 등)와 이에 따른 예측 로직을 분리하고, `Command Pattern`으로 모든 입력을 캡슐화하기 전까지는 매 업데이트마다 새로운 버그가 튀어나왔습니다. 나중에는 `ECS` 구조로 갈아엎으면서 상태 저장이 훨씬 깔끔해졌고, 비로소 안정적인 데드 레코닝을 얻을 수 있었습니다.

이 경험을 통해 저는 강력히 깨달았습니다. 복잡한 네트워크 동기화 기술, 특히 Rollback과 같이 시간을 다루는 기술은 단순히 "어떻게 작동하는가"를 아는 것만으로는 부족합니다. "어떻게 하면 이 복잡성을 관리하고, 확장하며, 디버깅 가능한 형태로 만들 것인가"라는 질문에 대한 답이 바로 **게임 디자인 패턴**에 있다는 것을요.

### 🌟 결론: 복잡성에 질서를 부여하는 지혜

오늘 우리는 **Server-Side Network Sync (Rollback/Dead Reckoning)** 라는 고난도 기술과 **Game Design Patterns**가 어떻게 시너지를 내는지 깊이 파고들어 보았습니다. Dead Reckoning의 예측과 보정 로직에는 `State Pattern`과 `Command Pattern`이, Rollback의 시공간을 넘나드는 기술에는 `Memento Pattern`과 `ECS` 같은 구조적 패턴이 필수적이라는 것을 알 수 있었습니다.

네트워크 게임 개발은 결코 쉽지 않습니다. 하지만 올바른 도구와 철학을 가지고 접근한다면, 우리는 플레이어에게 최고의 경험을 선사할 수 있습니다. 디자인 패턴은 단순히 학술적인 개념이 아닙니다. 그것은 현업 개발자들이 수많은 시행착오 끝에 얻어낸 지혜의 결정체이며, 복잡한 문제에 질서를 부여하고 예측 불가능한 미래에 대비하게 해주는 강력한 무기입니다.
