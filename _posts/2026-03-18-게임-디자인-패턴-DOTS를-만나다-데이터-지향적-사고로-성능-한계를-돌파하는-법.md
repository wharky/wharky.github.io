### 문제 제기: '좋은 코드'가 '느린 코드'가 될 때

게임 개발을 하다 보면 누구나 '좋은 코드'를 작성하고 싶어 합니다. 재사용 가능하고, 확장성 좋고, 유지보수하기 쉬운 코드 말이죠. 그래서 우리는 GoF(Gang of Four) 디자인 패턴을 필두로 하는 다양한 'Game Design Patterns'을 배우고 적용합니다. 상속, 다형성, 캡슐화와 같은 객체 지향(OOP) 원칙에 기반한 이 패턴들은 코드의 구조를 아름답게 만들고 복잡성을 관리하는 데 큰 도움을 주었습니다.

하지만 솔직히 고백하자면, 저도 수많은 프로젝트에서 '좋은 코드'가 '느린 코드'로 이어지는 아이러니를 경험했습니다. 특히 다수의 개체가 동시에 상호작용하고, 실시간으로 엄청난 양의 데이터를 처리해야 하는 게임 환경에서는 더욱 심했죠. 예를 들어볼까요? 수백, 수천 개의 적 캐릭터가 각자의 상태(Idle, Patrol, Attack, Flee 등)를 가지고 복잡한 AI 로직을 수행하는 상황을 가정해봅시다.

전통적인 OOP 방식으로 이 **State Pattern**을 구현하면, 각 캐릭터 객체는 `IState` 인터페이스를 상속받은 여러 상태 객체를 가지고, `Update()` 메서드 내에서 현재 상태의 `Update()`를 호출하는 방식이 일반적입니다.

```csharp
// 전통적인 OOP State Pattern (개념 코드)
interface ICharacterState
{
    void Enter(Character character);
    void Update(Character character, float deltaTime);
    void Exit(Character character);
}

class IdleState : ICharacterState { /* ... */ }
class AttackState : ICharacterState { /* ... */ }

class Character // 수천 개의 Character 객체
{
    private ICharacterState _currentState;
    public void SetState(ICharacterState newState) { /* ... */ }
    public void Update(float deltaTime)
    {
        _currentState.Update(this, deltaTime); // 가상 함수 호출!
    }
}
```

이 코드는 분명 깔끔하고 확장성이 좋습니다. 하지만 `_currentState.Update()`를 호출할 때마다 발생하는 가상 함수 호출(Virtual Call)은 CPU의 예측 분기(Branch Prediction)를 방해하여 파이프라인 스톨(Stall)을 일으킬 수 있습니다. 게다가 각 캐릭터 객체와 그들의 상태 객체들이 메모리 여기저기에 흩어져 있으면, CPU 캐시 미스(Cache Miss)가 빈번하게 발생하여 데이터를 가져오는 데 엄청난 시간을 낭비하게 됩니다. 게임은 결국 데이터를 매우 빠르게 처리해야 하는 작업인데, 데이터가 불규칙하게 배치되어 있으면 속도를 낼 수가 없게 되는 거죠.

저는 이런 문제를 겪을 때마다 "디자인 패턴이 과연 만능인가?"라는 근본적인 질문을 던지게 되었습니다. 분명 패턴 자체는 좋지만, 현대 하드웨어의 특성(멀티코어, 캐시 계층)을 고려하지 않은 구현은 오히려 독이 될 수 있다는 것을요.

### DOTS의 등장: 데이터 지향적 사고의 전환

이런 고민 속에서 Unity의 **DOTS(Data-Oriented Technology Stack)**는 저에게 새로운 관점을 제시해주었습니다. DOTS는 ECS(Entity-Component-System) 아키텍처, Burst 컴파일러, Job System을 통해 데이터를 중심으로 사고하고, CPU의 멀티코어와 캐시 효율성을 극대화하는 것을 목표로 합니다.

여기서 핵심은 **"무엇이 무엇을 *가지고* 있는가"**가 아니라 **"어떤 *데이터*가 있고, 이 데이터를 *어떤 시스템*이 *어떻게 처리하는가*"**로 사고방식을 전환하는 것입니다. 객체가 행동을 정의하는 대신, 데이터(컴포넌트)가 상태를 정의하고, 시스템이 이 데이터를 읽고 써서 행동을 구현합니다.

### 딥 다이브: State Pattern을 DOTS 방식으로 재해석하기

그럼 앞에서 예로 들었던 State Pattern을 DOTS 방식으로 어떻게 재해석할 수 있을까요? 전통적인 OOP 방식의 문제점을 DOTS가 어떻게 해결하는지 살펴보겠습니다.

DOTS에서는 캐릭터 자체가 하나의 `Entity`가 됩니다. 그리고 각 상태는 별도의 `Component`로 표현됩니다. 중요한 점은 이 상태 컴포넌트들은 '행동'을 가지지 않고, 그저 '지금 이 엔티티가 어떤 상태인지'를 나타내는 **데이터 마커(Tag Component)**에 불과하다는 것입니다.

```csharp
// DOTS (ECS) 방식의 State Pattern (개념 코드)

// 1. 상태를 나타내는 순수 데이터 컴포넌트들
// C# record struct (또는 struct)로 정의, 멤버는 비어있거나 최소한의 상태 데이터만 포함
public record struct IsIdleTag : IComponentData { /* No members, just a tag */ }
public record struct IsWalkingTag : IComponentData { /* ... */ }
public record struct IsAttackingTag : IComponentData { /* ... */ }
public record struct IsDeadTag : IComponentData { /* ... */ }

// 캐릭터의 속성 데이터 컴포넌트
public record struct MovementData : IComponentData
{
    public float Speed;
    public float CurrentVelocity;
    public float3 Position;
}
public record struct HealthData : IComponentData
{
    public float CurrentHealth;
    public float MaxHealth;
}

// 2. 상태 전환 및 행동 처리를 담당하는 시스템들
// 이 시스템들은 특정 컴포넌트 조합을 가진 엔티티들을 쿼리하여 일괄 처리합니다.

// StateTransitionSystem: 상태 전환 로직 처리
public class StateTransitionSystem : SystemBase
{
    protected override void OnUpdate()
    {
        // 유니티 ECS API의 일부입니다 (Entities.ForEach)
        Entities
            .ForEach((Entity entity, ref HealthData health, in InputData input) =>
            {
                // 죽음 상태 체크 (가장 높은 우선순위)
                if (health.CurrentHealth <= 0 && !EntityManager.HasComponent<IsDeadTag>(entity))
                {
                    // 기존 상태 제거 및 Dead 상태 추가
                    EntityManager.RemoveComponent<IsIdleTag>(entity);
                    EntityManager.RemoveComponent<IsWalkingTag>(entity);
                    EntityManager.AddComponent<IsDeadTag>(entity);
                    Debug.Log($"Entity {entity} is now Dead!");
                }
                // 걷기 상태 전환
                else if (input.MoveInput != float2.zero && !EntityManager.HasComponent<IsWalkingTag>(entity))
                {
                    EntityManager.RemoveComponent<IsIdleTag>(entity);
                    EntityManager.AddComponent<IsWalkingTag>(entity);
                    Debug.Log($"Entity {entity} is now Walking.");
                }
                // 유휴 상태 전환
                else if (input.MoveInput == float2.zero && !EntityManager.HasComponent<IsIdleTag>(entity) &&
                         !EntityManager.HasComponent<IsDeadTag>(entity)) // 죽은 상태가 아닐 때만
                {
                    EntityManager.RemoveComponent<IsWalkingTag>(entity);
                    EntityManager.AddComponent<IsIdleTag>(entity);
                    Debug.Log($"Entity {entity} is now Idle.");
                }
            }).ScheduleParallel(); // Job System을 통해 병렬 처리
    }
}

// MovementSystem: 걷기 상태 엔티티의 이동 처리
public class MovementSystem : SystemBase
{
    protected override void OnUpdate()
    {
        float deltaTime = Time.DeltaTime;
        Entities
            .WithAll<IsWalkingTag>() // IsWalkingTag를 가진 엔티티만 필터링
            .ForEach((ref MovementData movementData) =>
            {
                // 모든 걷는 엔티티의 이동 데이터를 병렬로 업데이트
                movementData.Position += new float3(movementData.CurrentVelocity, 0, 0) * deltaTime;
                // 실제 이동 로직 (Transform 동기화 등)은 다른 시스템에서 처리 가능
            }).ScheduleParallel();
    }
}
```

이 DOTS 방식의 State Pattern은 다음과 같은 이점을 가집니다.

1.  **데이터 집중 및 캐시 효율성**: `MovementData`와 `IsWalkingTag`를 가진 모든 엔티티의 데이터는 메모리에 연속적으로 배치될 가능성이 높습니다. `MovementSystem`이 이 데이터를 처리할 때 CPU는 캐시에서 필요한 데이터를 한 번에 많이 가져올 수 있어 캐시 미스가 줄어듭니다.
2.  **가상 함수 호출 제거**: 더 이상 `currentState.Update()`와 같은 가상 함수 호출이 없습니다. 시스템은 직접적으로 필요한 데이터를 조회하고 처리합니다.
3.  **병렬 처리 용이성**: `Entities.ForEach(...).ScheduleParallel()`과 같은 구문은 Job System을 활용하여 수천 개의 엔티티를 여러 코어에서 동시에 처리할 수 있도록 해줍니다. 각 시스템은 독립적인 데이터 셋에 대해 작동하므로 병렬화가 자연스럽습니다.
4.  **명확한 관심사 분리**: 상태를 정의하는 데이터 컴포넌트와, 상태 전환을 처리하는 시스템, 그리고 각 상태에서의 행동을 처리하는 시스템이 명확하게 분리됩니다. 이는 유지보수성과 확장성을 높이는 데 기여합니다. 새로운 상태가 필요하면 새로운 컴포넌트를 추가하고, 그 상태를 처리하는 시스템을 만들거나 기존 시스템을 수정하면 됩니다.

### 패턴을 넘어선 사고방식의 전환

State Pattern의 예시에서 볼 수 있듯이, DOTS는 단순히 기존 패턴을 다른 문법으로 구현하는 것을 넘어, **패턴의 본질적인 '의도'를 데이터 지향적인 관점에서 재해석**하도록 요구합니다.

*   **Strategy Pattern**? 서로 다른 전략들은 각기 다른 컴포넌트를 가진 엔티티를 쿼리하는 별도의 시스템으로 구현될 수 있습니다.
*   **Object Pool Pattern**? DOTS의 엔티티 시스템 자체가 매우 효율적인 객체 풀링 메커니즘을 제공합니다. 엔티티를 비활성화하는 대신, 해당 엔티티에서 모든 컴포넌트를 제거하거나, 특정 `Disabled` 태그 컴포넌트를 추가하여 시스템이 처리하지 않도록 만들 수 있습니다. 데이터는 여전히 메모리에 남아있고, 필요할 때 컴포넌트만 추가하여 재활용하는 것이 훨씬 빠릅니다.

이것은 단순히 기술 스택의 변화를 넘어, 게임 개발자로서 문제에 접근하는 **사고방식의 근본적인 전환**을 의미합니다. "이 객체는 어떤 행동을 해야 할까?"에서 "이 데이터를 누가 어떻게 처리해야 가장 효율적일까?"로 질문이 바뀌는 것이죠. 처음에는 낯설고 어렵게 느껴질 수 있지만, 이 관점을 내재화하면 성능 병목 지점을 미리 예측하고 더욱 견고하며 확장성 있는 아키텍처를 설계할 수 있는 눈을 갖게 됩니다.

### 결론: 레거시를 넘어 미래로

오랜 시간 동안 게임 개발을 해오면서, 저는 언제나 '최적화'라는 숙제와 씨름해왔습니다. 코드의 우아함과 성능 사이에서 균형을 잡는 것은 쉽지 않은 일이었습니다. 하지만 DOTS를 통해 Game Design Patterns을 다시 들여다보면서, 우리는 이 두 마리 토끼를 모두 잡을 수 있는 새로운 길을 찾을 수 있다는 확신을 갖게 되었습니다.

물론 DOTS가 모든 문제의 해결책은 아닙니다. UI나 일부 비즈니스 로직처럼 OOP가 더 적합한 영역도 분명 존재합니다. 하지만 성능이 핵심적인 게임 로직, 대규모 시뮬레이션, 복잡한 물리 상호작용 등에서는 DOTS의 데이터 지향적 접근 방식이 강력한 해답을 제공할 것입니다.

선배 개발자로서 여러분께 권하는 바는 이렇습니다. 과거의 디자인 패턴 지식을 버리라는 것이 아닙니다. 오히려 그 패턴들의 본질적인 의도를 꿰뚫어 보고, 현대 하드웨어와 DOTS라는 새로운 도구 위에서 어떻게 더 강력하고 효율적으로 구현할 수 있을지 고민해보세요. 이 과정 자체가 여러분을 한 단계 더 성장시키는 계기가 될 것이라 확신합니다.
