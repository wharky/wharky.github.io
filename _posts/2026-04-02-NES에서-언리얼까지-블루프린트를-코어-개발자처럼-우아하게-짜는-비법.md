### 📜 문제: '너무 많은 자유'가 낳은 블루프린트 스파게티 코드

언리얼 엔진의 블루프린트는 정말 강력하고 접근성이 뛰어난 도구입니다. 코딩 없이도 복잡한 게임 로직을 빠르게 구현할 수 있죠. 하지만 이 '강력함'과 '자유로움'이 때로는 독이 되기도 합니다.

**여러분도 이런 경험 있지 않나요?**
*   프로젝트가 커질수록 블루프린트 그래프가 거대한 '스파게티'처럼 얽혀서 도저히 유지보수가 불가능해지는 현상.
*   `Event Tick`에 온갖 로직을 다 때려 박아서 불필요한 연산이 늘어나고 프레임 드랍이 발생하는 경우.
*   총알이나 이펙트처럼 자주 생성되고 파괴되는 액터들을 매번 `Spawn Actor` 하고 `Destroy Actor` 하다가 메모리 단편화나 성능 병목 현상을 겪는 일.
*   똑같은 로직을 여러 블루프린트에 복붙하다가 버그 수정할 때마다 노가다를 하는 상황.

우리는 흔히 "요즘 PC 성능이 워낙 좋으니 대충 짜도 돌아가겠지" 하는 안일함에 빠지곤 합니다. 하지만 이건 옛 선배들이 상상조차 할 수 없었던 '사치'였습니다.

### 🕹️ NES PPU: 제한 속의 예술, 효율의 미학

제가 처음 NES(패미컴)의 개발 문서를 들여다봤을 때 느꼈던 충격과 감탄을 잊을 수 없습니다. 특히 NES의 **PPU(Picture Processing Unit)**는 그 자체로 하나의 최적화 교과서입니다.

**NES PPU의 한계는 상상을 초월했습니다:**
*   **스프라이트 제한:** 화면에 동시에 띄울 수 있는 스프라이트는 단 64개. 심지어 한 줄(scanline)에는 8개만 표시 가능했습니다.
*   **메모리 제한:** 스프라이트의 위치, 인덱스, 팔레트 등의 정보를 담는 OAM(Object Attribute Memory)은 256바이트가 전부였습니다.
*   **컬러 제한:** 배경과 스프라이트에 쓸 수 있는 팔레트가 극히 제한적이었죠.

이런 극악무도한 환경에서 <슈퍼 마리오 브라더스>, <젤다의 전설> 같은 명작들이 탄생했다는 게 믿기시나요? 개발자들은 이 한계를 극복하기 위해 필사적이었습니다.

**그들은 어떤 해법을 찾았을까요?**
1.  **스프라이트 멀티플렉싱(Multiplexing):** 8개 스프라이트 제한을 우회하기 위해 한 줄에 8개 이상이 필요하면, 화면 밖으로 밀어내거나 (마리오 같은 경우), V-Blank 인터럽트 타이밍에 빠르게 OAM 정보를 교체하여 마치 더 많은 스프라이트가 있는 것처럼 눈속임했습니다. 이는 곧 '보여줘야 할 것만 효율적으로 관리하는' 지혜였습니다.
2.  **객체 풀링(Object Pooling)의 원조:** 적 캐릭터나 총알처럼 빈번하게 생성/소멸되는 객체들을 매번 메모리에 올리고 내리는 건 상상조차 할 수 없었습니다. 대신 미리 일정 수의 스프라이트 슬롯을 할당해두고, 필요할 때 '활성화'해서 쓰고, 역할을 다하면 '비활성화'해서 재활용하는 방식을 사용했습니다.
3.  **효율적인 상태 관리:** 마리오가 점프하거나, 걷거나, 공격할 때마다 어떤 스프라이트를 쓰고 어떤 충돌 영역을 가질지, 메모리에 최소한의 정보만 올려두고 상황에 맞춰 '상태'를 전환하며 관리했습니다. 모든 것이 정교한 **유한 상태 머신(FSM)**으로 엮여 있었죠.

결론적으로, NES 개발자들은 '모든 것을 효율적으로 관리해야 한다'는 강박 속에서 **데이터 흐름 최적화, 자원 재활용, 명확한 상태 관리**라는 보편적인 개발 원칙을 만들어냈습니다.

### 🚀 언리얼 블루프린트: NES의 지혜를 현대적으로 구현하기

이제 NES 개발자들이 보여준 지혜를 우리 블루프린트 개발에 어떻게 적용할지 이야기해볼 시간입니다. 사실 NES의 접근 방식은 현대 컴포넌트 기반 아키텍처와 데이터 중심 디자인(DOD)의 초기 형태라고 볼 수 있습니다.

#### 1. 문제 해결: 스파게티 코드를 끊어낼 '우아한 상태 머신(FSM)'과 '컴포넌트 분리'

NES에서 캐릭터의 모든 상태(대기, 걷기, 점프, 공격)를 정교하게 관리하여 스프라이트와 충돌 영역을 최적화했듯이, 블루프린트에서도 복잡한 로직을 명확한 상태로 분리하고, 기능을 컴포넌트로 쪼개서 재사용성을 높여야 합니다.

**[Blueprint FSM Pseudo-code Example]**

```
// 1. 상태를 정의하는 Enum 생성: EPlayerState
//    - Idle
//    - Run
//    - Jump
//    - Attack
//    - ...

// 2. 현재 상태를 저장하는 변수: CurrentState (Type: EPlayerState)

// 3. 상태 전환 로직 (예: 점프 입력 시)
// Event OnInput_Jump_Pressed:
//    IF CurrentState == EPlayerState::Idle OR CurrentState == EPlayerState::Run THEN
//        SET CurrentState = EPlayerState::Jump
//        CALL CustomEvent_StartJumpAnimationAndLogic // 상태 진입 시 실행될 이벤트
//    END IF

// 4. 상태별 로직 (예: 점프 시작 시)
// CustomEvent_StartJumpAnimationAndLogic:
//    AddImpulse(Z_Vector, ForceMode)
//    PlayAnimation(JumpMontage)
//    SET Timer by Event (CustomEvent_CheckGroundStatus, Time=0.1, Looping=true) // 착지 확인 타이머

// 5. 상태 종료 및 다음 상태 전환 (예: 착지 시)
// CustomEvent_CheckGroundStatus:
//    IF IsOnGround() THEN
//        CLEAR Timer by Event (CustomEvent_CheckGroundStatus)
//        SET CurrentState = EPlayerState::Idle
//        CALL CustomEvent_StartIdleAnimationAndLogic
//    END IF
```

이런 방식으로 로직을 쪼개고, 각 기능을 별도의 블루프린트 컴포넌트로 분리하면, 특정 액터의 로직이 아무리 복잡해져도 그래프는 깔끔하게 유지되고, 버그를 찾기도 훨씬 쉬워집니다.

#### 2. 문제 해결: 자원 낭비를 막는 '블루프린트 오브젝트 풀링'

NES가 스프라이트 시트를 재활용했듯이, 언리얼에서도 총알, 파티클 이펙트, 적 유닛처럼 자주 생성되고 파괴되는 액터들은 오브젝트 풀링(Object Pooling)을 통해 성능 부하를 크게 줄일 수 있습니다. `Spawn Actor`와 `Destroy Actor`는 생각보다 비싼 연산입니다.

**[Blueprint Object Pooling Pseudo-code Example]**

```
// 1. 오브젝트 풀을 위한 Array 변수: PooledProjectiles (Type: Actor Object Reference Array)
// 2. 풀링할 액터 클래스: ProjectileClass (Type: Actor Class Reference)
// 3. 초기 풀 크기 변수: InitialPoolSize (Type: Integer)

// Event BeginPlay:
//    FOR (Index from 0 to InitialPoolSize - 1)
//        SpawnedProjectile = Spawn Actor (Class=ProjectileClass, Transform=Vector_Zero)
//        Set Actor Hidden In Game (SpawnedProjectile, True) // 일단 숨기고
//        Set Actor Enable Collision (SpawnedProjectile, False) // 충돌도 끄고
//        Set Actor Tick Enabled (SpawnedProjectile, False) // Tick도 끈다
//        Add (SpawnedProjectile) to PooledProjectiles // 풀에 추가
//    END FOR

// Function GetPooledProjectile: (Return Value: Actor Object Reference)
//    IF PooledProjectiles Array IS NOT EMPTY THEN
//        ProjectileToUse = Get Last Item (PooledProjectiles)
//        Remove Last Item (PooledProjectiles) // 풀에서 제거
//        Set Actor Hidden In Game (ProjectileToUse, False) // 다시 보이게
//        Set Actor Enable Collision (ProjectileToUse, True) // 충돌 활성화
//        Set Actor Tick Enabled (ProjectileToUse, True) // Tick 활성화
//        RETURN ProjectileToUse
//    ELSE
//        // 선택 사항: 풀이 비었으면 새로 생성하여 반환 (성능 저하 가능성)
//        SpawnedProjectile = Spawn Actor (Class=ProjectileClass, Transform=Vector_Zero)
//        RETURN SpawnedProjectile
//    END IF

// Function ReturnPooledProjectile (Input: ProjectileActor_Ref):
//    Set Actor Hidden In Game (ProjectileActor_Ref, True)
//    Set Actor Enable Collision (ProjectileActor_Ref, False)
//    Set Actor Tick Enabled (ProjectileActor_Ref, False)
//    Set Actor Location (ProjectileActor_Ref, OffscreenLocation_Vector) // 화면 밖 안전한 곳으로 이동
//    Add (ProjectileActor_Ref) to PooledProjectiles // 풀에 다시 추가
```

이 방식을 사용하면, 런타임에 불필요한 메모리 할당과 해제를 최소화하여 훨씬 부드러운 게임 플레이를 만들 수 있습니다.

#### 3. 문제 해결: 불필요한 연산을 줄이는 '이벤트 기반 아키텍처'

NES는 V-Blank 인터럽트처럼 중요한 이벤트가 발생할 때만 CPU가 특정 작업을 수행하도록 했습니다. 블루프린트에서도 `Event Tick`에 너무 많은 것을 의존하지 말고, 필요한 시점에만 로직이 실행되도록 **커스텀 이벤트(Custom Events), 이벤트 디스패처(Event Dispatchers), 인터페이스(Interfaces)**를 적극 활용해야 합니다.

*   **Custom Events:** 특정 상황에서만 호출될 로직을 캡슐화합니다.
*   **Event Dispatchers:** 다른 블루프린트에서 특정 이벤트가 발생했음을 알리고, 관심 있는 블루프린트들이 구독하여 반응하도록 합니다. 이는 느슨한 결합(Loose Coupling)을 유도하여 유지보수를 쉽게 합니다.
*   **Interfaces:** 다양한 액터들이 공통된 기능을 구현할 수 있도록 '계약'을 정의합니다. `Cast To` 노드의 남발을 줄이고 확장성을 높입니다.

이러한 기법들은 `Event Tick`이 매 프레임마다 모든 액터에서 불필요한 검사를 하는 것을 방지하고, 필요한 로직만 효율적으로 실행되도록 돕습니다.

### 💡 나의 견해: 제약이 낳는 창의성과 효율성

제가 처음 게임 개발을 시작했을 때, 선배 개발자들은 항상 "메모리를 아껴라", "CPU 사이클을 아껴라"고 강조했습니다. 그때는 그저 당연한 조언이라고 생각했죠. 하지만 NES 같은 레트로 콘솔의 아키텍처를 깊이 파고들면서, 그 말들이 얼마나 깊은 지혜에서 나온 것인지 깨달았습니다. 그 시절 개발자들은 '제약'이라는 거대한 벽 앞에서 비로소 진짜 '창의성'을 발휘했습니다.

오늘날 우리는 훨씬 더 강력한 하드웨어와 엔진을 가지고 있지만, 오히려 이 '풍요로움' 때문에 기본적인 원칙들을 간과하고 비효율적인 코드를 짜기 쉽습니다. 블루프린트 개발은 때때로 '너무 많은 자유'가 독이 될 수 있다는 걸 여실히 보여줍니다.

레트로 콘솔 개발자들이 자원 제약 속에서 만들어낸 '우아한 해킹'들과 최적화 기법들은 단순히 과거의 기술이 아닙니다. 그것은 **데이터를 이해하고, 흐름을 제어하며, 자원을 현명하게 사용하는 방법에 대한 시대를 초월한 원칙**입니다. 이 원칙들을 블루프린트에 적용하면, 여러분은 단순히 작동하는 코드를 넘어, '아름답고 강력하며 유지보수하기 쉬운' 코드를 만들 수 있을 겁니다.

그러니 오늘부터 여러분의 블루프린트를 짤 때, 잠시 눈을 감고 NES 개발자들이 얼마나 치열하게 한 바이트, 한 사이클과 싸웠는지를 상상해보세요. 그 상상이 여러분의 블루프린트 로직을 더욱 견고하고 우아하게 만들 것입니다.
