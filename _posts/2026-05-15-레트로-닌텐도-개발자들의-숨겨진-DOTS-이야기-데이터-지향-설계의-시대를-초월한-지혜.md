### 1. 레트로 개발자들의 무심한 듯 시크한 '데이터 최적화'

어릴 적 닌텐도 게임을 하며 저는 늘 경외심을 느꼈습니다. '어떻게 저 작은 카트리지 안에 저렇게 멋진 세상이 들어있을까?' 단순히 콘텐츠의 양을 넘어, 제한된 리소스 안에서 생동감 넘치는 게임 플레이를 구현하는 방식은 항상 궁금증을 자아냈죠. 지금 돌이켜보면, 그 질문의 답은 바로 '데이터 최적화'에 있었습니다. 당시 개발자들은 '데이터 지향 설계'라는 용어조차 몰랐겠지만, 그들의 코드는 본질적으로 가장 효율적인 데이터 처리 방식에 맞춰져 있었습니다.

**[문제 상황]**
레트로 게임 개발 환경은 지금과는 상상할 수 없을 정도로 열악했습니다. MB 단위는커녕, KB 단위의 메모리(RAM)와 MHz 단위의 CPU 클럭이 전부였습니다. 오브젝트 지향 프로그래밍(OOP)은 개념조차 희미했고, 가상 함수 호출이나 복잡한 클래스 계층 구조는 사치에 가까웠죠. 수많은 적들, 발사체, 배경 요소들을 동시에 처리해야 하는데, 동적 할당은 꿈도 꿀 수 없었습니다.

**[레트로 개발자들의 해결책: 암묵적인 데이터 지향 설계]**
그들은 본능적으로 가장 효율적인 방법을 찾아냈습니다. 그것은 바로 데이터를 '팩(Pack)'하고, '배열(Array)'에 담아 순차적으로 처리하는 방식이었습니다.

1.  **"모든 것은 배열이다 (Everything is an Array)"**:
    수많은 적 캐릭터, 발사체, 파티클 효과들은 모두 미리 할당된 배열(Pre-allocated array)에 저장되었습니다. 예를 들어, `Enemy`라는 구조체(struct) 배열을 만들고, 각 엔트리에 적의 위치, 속도, 상태, 체력 등을 압축해서 저장하는 식이죠. 특정 적이 파괴되면, 단순히 `active` 플래그를 비활성화하거나 배열의 다른 인스턴스로 교체하는 방식으로 재활용했습니다.

2.  **데이터 압축 (Data Packing)**:
    메모리 1바이트조차 소중했기에, 개발자들은 데이터를 비트(bit) 단위로 압축했습니다. 한 바이트에 적의 타입, 방향, 애니메이션 프레임 정보를 몽땅 구겨 넣는 일도 다반사였죠.

3.  **순차적인 처리 (Sequential Processing)**:
    게임 루프에서는 모든 적이나 발사체 배열을 `for` 루프를 돌며 업데이트했습니다. 이는 메모리 캐시(Cache) 효율성을 극대화하는 결과를 낳았습니다. 같은 타입의 데이터가 메모리에 연속적으로 배치되니, CPU가 한 번에 많은 데이터를 불러와 빠르게 처리할 수 있었죠.

아래는 레트로 게임에서 수많은 발사체를 처리하는 방식에 대한 의사 코드입니다.

```cpp
// 레트로 스타일 발사체 구조체 (예: NES/SNES)
// 실제로는 비트 플래그 등으로 더 압축했을 겁니다.
struct Projectile {
    char active;    // 0: 비활성, 1: 활성
    short x, y;     // 위치 (고정 소수점)
    short vx, vy;   // 속도 (고정 소수점)
    char type;      // 발사체 종류 (예: 미사일, 레이저)
    char lifetime;  // 남은 수명
};

// 최대 발사체 수 미리 정의
#define MAX_PROJECTILES 128
Projectile projectiles[MAX_PROJECTILES];

// 게임 초기화 시 모든 발사체 비활성화
void InitProjectiles() {
    for (int i = 0; i < MAX_PROJECTILES; i++) {
        projectiles[i].active = 0;
    }
}

// 새로운 발사체 생성
int SpawnProjectile(short startX, short startY, short speedX, short speedY, char type) {
    for (int i = 0; i < MAX_PROJECTILES; i++) {
        if (projectiles[i].active == 0) {
            projectiles[i].active = 1;
            projectiles[i].x = startX;
            projectiles[i].y = startY;
            projectiles[i].vx = speedX;
            projectiles[i].vy = speedY;
            projectiles[i].type = type;
            projectiles[i].lifetime = 60; // 60프레임 수명
            return i;
        }
    }
    return -1; // 공간 없음
}

// 매 프레임 발사체 업데이트
void UpdateProjectiles() {
    for (int i = 0; i < MAX_PROJECTIELs; i++) {
        if (projectiles[i].active == 1) {
            projectiles[i].x += projectiles[i].vx;
            projectiles[i].y += projectiles[i].vy;

            // 수명 감소 및 소멸 처리
            projectiles[i].lifetime--;
            if (projectiles[i].lifetime <= 0) {
                projectiles[i].active = 0;
            }

            // 화면 밖 처리
            if (projectiles[i].x < 0 || projectiles[i].x > SCREEN_WIDTH ||
                projectiles[i].y < 0 || projectiles[i].y > SCREEN_HEIGHT) {
                projectiles[i].active = 0;
            }

            // 충돌 처리 (다른 오브젝트 배열과 루프 돌며 검사)
            // ...
        }
    }
}
```
보시다시피, 레트로 개발자들은 극한의 제약 속에서 자신도 모르게 '데이터 지향적 사고'를 실천하고 있었습니다. 그들의 코드는 성능 병목 현상을 최소화하고, CPU의 잠재력을 최대한 끌어내기 위한 순수한 형태의 데이터 최적화 그 자체였죠.

### 2. DOTS, 시대를 초월한 지혜를 현대적으로 재해석하다

현대 게임은 레트로 게임과는 비교할 수 없을 정도로 복잡합니다. 방대한 오픈월드, 정교한 물리 시뮬레이션, 수많은 AI 캐릭터 등 처리해야 할 데이터의 양과 종류가 폭발적으로 증가했습니다. 우리는 더 이상 KB 단위 메모리에 갇혀 있지 않지만, 여전히 성능 문제는 존재합니다. 특히 멀티코어 CPU 아키텍처가 보편화되면서, 기존 객체 지향 프로그래밍(OOP) 방식은 종종 CPU 캐시 미스(Cache Miss)와 동기화 문제로 성능 저하를 일으키는 주범이 되곤 합니다.

**[문제 상황]**
수많은 게임 오브젝트들이 각자의 상태를 가지고 독립적으로 동작하는 OOP 방식은 개발 편의성 측면에서는 유리하지만, CPU가 데이터를 효율적으로 접근하고 병렬 처리하는 데는 불리합니다. 메모리 여기저기에 흩어져 있는 데이터를 접근할 때마다 캐시 미스가 발생하고, 이는 파이프라인 스톨(Pipeline Stall)로 이어져 CPU 코어의 활용도를 떨어뜨립니다.

**[DOTS의 해결책: 명시적인 데이터 지향 설계]**
여기서 Unity의 DOTS(Data-Oriented Technology Stack)가 등장합니다. DOTS는 레트로 개발자들이 암묵적으로 추구했던 '데이터 지향'의 철학을 현대적인 멀티코어 환경에 맞춰 명시적으로 재해석하고, 이를 구현할 수 있는 강력한 도구들을 제공합니다.

1.  **ECS (Entities Component System)**:
    *   **Entity**: 단순히 ID 값으로, 데이터의 '주인'을 식별합니다. 레트로의 '배열 인덱스'와 유사하죠.
    *   **Component**: 순수한 데이터 덩어리입니다. `Translation`, `Velocity`, `Health` 등, 해당 Entity의 속성을 정의합니다. 이는 레트로의 `Projectile` 구조체 내부의 멤버 변수들이 독립적인 컴포넌트로 분리된 형태와 같습니다.
    *   **System**: 특정 컴포넌트를 가진 Entity들을 쿼리(Query)하여 로직을 처리합니다. 레트로의 `UpdateProjectiles()` 함수가 배열을 순회하며 데이터를 처리하는 방식과 정확히 일치합니다.
    이 구조는 같은 종류의 컴포넌트 데이터가 메모리에 연속적으로 배치되도록 유도하여 캐시 효율을 극대화합니다.

2.  **Jobs System**:
    게임 로직을 작은 작업 단위(Job)로 분리하고, 이를 멀티코어 CPU에서 병렬로 실행할 수 있도록 합니다. 이는 레트로 개발자들이 단일 CPU에서 순차적으로 처리해야 했던 로직을, 현대의 다중 코어에서 동시에 처리하여 성능을 비약적으로 향상시킵니다.

3.  **Burst Compiler**:
    C# 코드를 최적화된 머신 코드로 컴파일하여 C++에 버금가는 실행 속도를 제공합니다. 이는 Jobs System과 결합하여 데이터 처리 루프의 성능을 극한으로 끌어올리며, 레트로 개발자들이 어셈블리 언어 레벨에서 직접 최적화를 했던 노력과 일맥상통합니다.

아래는 DOTS에서 위 레트로 발사체 업데이트 로직을 구현하는 방식에 대한 개념적인 코드입니다.

```csharp
// DOTS 컴포넌트 정의
// 순수한 데이터만 담습니다. C# 구조체(struct)로 정의.
public struct ProjectileData : IComponentData
{
    public float Speed;
    public float MaxLifetime;
    public float CurrentLifetime;
    public bool IsActive; // ECS에서는 Active 상태를 컴포넌트 추가/제거로 관리할 수도 있습니다.
    public float Damage;
}

// 이동을 위한 기본 컴포넌트 (Unity.Transforms 네임스페이스)
// public struct Translation : IComponentData { public float3 Value; }
// public struct LocalToWorld : IComponentData { public float4x4 Value; }

// DOTS 시스템 정의
// Projectile들을 업데이트하는 Job을 스케줄링하는 시스템입니다.
[BurstCompile] // Job을 Burst 컴파일러로 최적화합니다.
public partial struct ProjectileMovementJob : IJobEntity
{
    public float DeltaTime; // 매 프레임의 시간 간격

    // IJobEntity는 특정 컴포넌트를 가진 각 Entity에 대해 Execute 메서드를 호출합니다.
    void Execute(ref Translation translation, ref ProjectileData projectile, in Velocity velocity)
    {
        if (projectile.IsActive) // 활성 상태인 Projectile만 처리
        {
            // 위치 업데이트
            translation.Value += velocity.Value * DeltaTime;

            // 수명 감소 및 비활성화 처리
            projectile.CurrentLifetime -= DeltaTime;
            if (projectile.CurrentLifetime <= 0f ||
                translation.Value.x < -100f || translation.Value.x > 100f || // 화면 밖
                translation.Value.y < -100f || translation.Value.y > 100f)
            {
                projectile.IsActive = false;
                // 실제 DOTS에서는 Entity를 Destroy하거나 Pooling하여 재활용합니다.
                // 여기서는 개념적인 설명을 위해 IsActive 플래그를 사용합니다.
            }

            // 충돌 처리 (별도의 System이나 Job에서 처리)
        }
    }
}

// Projectile을 관리하는 메인 시스템
public partial class ProjectileSystem : SystemBase
{
    protected override void OnUpdate()
    {
        // DeltaTime을 Job에 전달
        float deltaTime = SystemAPI.Time.DeltaTime;

        // Job 스케줄링. ScheduleParallel()을 통해 멀티코어에 분산 실행됩니다.
        new ProjectileMovementJob { DeltaTime = deltaTime }.ScheduleParallel();
        
        // 여기에 Projectile 생성, 충돌 감지, 소멸 등의 다른 Job들을 추가로 스케줄링할 수 있습니다.
    }
}
```

DOTS는 레트로 개발자들이 수동으로 구현했던 데이터 레이아웃 최적화와 반복 처리를, ECS를 통해 구조적으로 강제하고, Jobs System과 Burst Compiler를 통해 병렬 처리와 최적화된 성능으로 자동화합니다. 즉, 과거의 지혜를 현대의 기술 스택으로 완벽하게 재해석한 것이죠.

### 3. 실전 적용: 레트로 슈팅 게임의 탄막을 DOTS로 구현한다면?

그렇다면, 수천 개의 발사체가 화면을 뒤덮는 레트로 스타일의 '탄막 슈팅 게임'을 DOTS로 만든다면 어떨까요?

**[문제 상황]**
고전 명작 '그라디우스'나 '콘트라'를 생각해보세요. 화면을 꽉 채우는 적기와 탄환들. 현대 엔진으로 이걸 객체 지향 방식으로 구현하면, 각각의 탄환을 클래스 인스턴스로 만들고 `Update()` 메서드를 호출하는 순간 CPU는 비명을 지를 겁니다. 캐시 미스와 가상 함수 호출 오버헤드가 막대할 테니까요.

**[DOTS를 활용한 해결책]**
DOTS는 이런 상황에서 진가를 발휘합니다.

1.  **`Projectile` Entity 정의**:
    각 탄환을 하나의 `Entity`로 봅니다. 이 `Entity`에는 `Translation`(위치), `Velocity`(속도), `Rotation`(회전), `ProjectileData`(수명, 데미지 등 커스텀 데이터)와 같은 `Component`들을 붙여줍니다.

2.  **`ProjectileMovementSystem`**:
    이 시스템은 `Translation`, `Velocity`, `ProjectileData` 컴포넌트를 가진 모든 `Entity`를 쿼리하여 업데이트합니다. `IJobEntity`나 `IJobChunk` 인터페이스를 구현한 Job으로 만들면, 수천 개의 탄환 이동 로직이 멀티코어 CPU에서 병렬로, 그리고 Burst 컴파일러 덕분에 초고속으로 처리됩니다.

3.  **`CollisionDetectionSystem`**:
    탄환과 플레이어, 또는 탄환과 적과의 충돌 감지 또한 `IJobEntity` 또는 `IJobChunk`를 통해 병렬 처리됩니다. 공간 분할(Spatial Partitioning) 기술(예: Grid, Quadtree)과 함께 사용하면 훨씬 효율적인 충돌 처리가 가능합니다.

4.  **`SpawnProjectileSystem` 및 `CleanupProjectileSystem`**:
    탄환 생성 및 소멸(혹은 재활용을 위한 Pooling) 로직도 시스템으로 분리하여 관리합니다. `EntityManager`를 통해 `Entity`를 생성하고 파괴하는 작업은 매우 효율적입니다.

이러한 DOTS 기반 아키텍처는 레트로 게임 개발자들이 배열과 루프를 통해 손수 구현했던 '데이터 연속성'과 '배치 처리'의 장점을 현대적인 병렬 처리 프레임워크 위에서 제공합니다. 수천 개의 탄환이 움직여도 CPU에 부하를 거의 주지 않으며, 게임의 프레임 속도는 안정적으로 유지될 겁니다.

### 맺음말: 과거에서 미래를 보다

우리는 오늘 **'Development Techniques of Retro Nintendo Games'**가 보여주었던 근본적인 최적화 원리와 **'Data-Oriented Technology Stack (DOTS)'**가 지향하는 현대적 해결책이 사실은 '데이터 지향 설계'라는 하나의 철학으로 관통하고 있음을 보았습니다.

레트로 개발자들은 극한의 제약 속에서 '어떻게 하면 데이터를 가장 효율적으로 관리하고 처리할 수 있을까?'를 고민하며 본능적으로 데이터 지향적 사고를 했습니다. 그리고 오늘날 DOTS는 그 지혜를 잃지 않고, 현대의 복잡성과 멀티코어 환경에 맞춰 '어떻게 하면 CPU와 캐시의 잠재력을 최대한 활용할 수 있을까?'를 명시적으로 답하고 있습니다.

DOTS는 단순히 새로운 기술 스택이 아니라, 과거의 뛰어난 개발자들이 직면했던 문제와 그들이 찾아낸 본질적인 해결책을 현대적인 언어와 도구로 재구성한 결과라고 저는 생각합니다. 그러니 다음번 새로운 기술을 배울 때, 혹은 성능 문제에 부딪혔을 때, 잠시 옛 게임의 개발 방식을 상상해보는 건 어떨까요? 때로는 과거의 지혜 속에서 미래를 헤쳐나갈 영감을 얻을 수 있을 겁니다.

이 글이 여러분의 개발 여정에 작은 영감과 통찰을 제공했기를 바라며, 저는 다음 포스팅에서 또 다른 흥미로운 주제로 찾아오겠습니다!
