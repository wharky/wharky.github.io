### ☕️ 커피 한 잔 하며 나누는 옛날이야기: 대규모 렌더링의 딜레마

솔직히 고백하자면, 저도 한때는 대규모 오브젝트 렌더링 앞에서 좌절을 맛보던 시절이 있었습니다. 수천 개의 풀잎, 수백 마리의 적, 폭발 시 쏟아지는 엄청난 양의 파티클... Unity의 MonoBehavior나 Unreal의 Actor 방식으로는 꿈도 꾸기 힘들었죠. CPU는 Draw Call과 Component Update에서 비명을 질렀고, GPU는 매 프레임 새로운 메시 데이터를 기다리느라 지쳐 쓰러지기 일쑤였습니다.

**문제점의 핵심은 무엇이었을까요?**

1.  **CPU 바틀넥**:
    *   **오브젝트 지향 프로그래밍 (OOP)의 오버헤드**: 각 GameObject/Actor가 자체적인 로직과 데이터를 가지면서, 수많은 개별적인 업데이트와 가상 함수 호출이 발생합니다. CPU 캐시 효율은 바닥을 쳤죠.
    *   **Draw Call 폭증**: 각 오브젝트를 개별적으로 그리는 Draw Call은 CPU에게 엄청난 부담을 주었고, 이는 프레임 드랍으로 직결되었습니다.
2.  **GPU와의 비효율적인 통신**:
    *   CPU가 개별 오브젝트의 데이터를 갱신하고 GPU로 넘겨주는 과정 자체가 병목이었습니다. 아무리 멋진 쉐이더를 짜놔도, GPU에 데이터가 제때 도착하지 않으면 무용지물이 됩니다.

저는 늘 생각했습니다. "이 수많은 오브젝트들을 어떻게 하면 '하나의 거대한 데이터 덩어리'처럼 취급하고, 그걸 GPU에게 통째로 넘겨줄 수 없을까? 그리고 GPU는 그 거대한 덩어리 안에서 각 요소를 개별적으로, 하지만 효율적으로 렌더링할 수 없을까?" 이 질문에 대한 답을 찾기 시작하면서, 제 눈에 들어온 것이 바로 DOTS와 쉐이더의 강력한 시너지였습니다.

### 🚀 DOTS: 데이터가 흐르는 고속도로를 건설하다

DOTS는 저에게 '생각하는 방식' 자체를 바꾼 혁신이었습니다. 오브젝트 중심 사고에서 벗어나, **데이터 중심 (Data-Oriented)** 사고방식으로 전환하게 했죠. Entity-Component-System (ECS) 아키텍처는 게임 월드를 엔티티(고유 ID), 컴포넌트(데이터), 시스템(로직)으로 분리하여 데이터의 응집성을 높이고, C# Job System과 Burst Compiler는 이 데이터를 CPU 코어 전반에 걸쳐 효율적으로 병렬 처리할 수 있게 해주었습니다.

**DOTS가 문제를 어떻게 해결했는가?**

DOTS는 수많은 오브젝트의 데이터를 일관된 메모리 레이아웃으로 정리하여 CPU 캐시 효율을 극대화합니다. 그리고 이 데이터를 한 번에 대량으로 처리할 수 있는 Job 시스템을 제공하여, CPU 코어들을 쉴 틈 없이 일하게 만들죠.

예를 들어, 수많은 파티클이나 풀잎을 생각해봅시다. 이들은 각각 고유한 위치, 속도, 수명, 그리고 렌더링에 필요한 파라미터(색상, 크기 스케일, 애니메이션 위상 등)를 가질 것입니다. DOTS를 사용하면 이 모든 데이터를 구조체 기반의 컴포넌트로 정의하고, 시스템에서 이 컴포넌트들을 그룹으로 묶어 빠르게 갱신할 수 있습니다.

**[핵심 로직: DOTS 컴포넌트 & 시스템 설계 (Pseudo-code)]**

```csharp
// 1. 렌더링에 필요한 데이터(컴포넌트) 정의
// 각 인스턴스(풀잎, 파티클 등)가 가질 고유한 정보들입니다.
public struct RenderInstanceData : IComponentData
{
    public float3 Position; // 월드 위치
    public float4 Rotation; // 회전 (쿼터니언)
    public float3 Scale;    // 크기
    public float4 Color;    // 고유 색상 또는 베이스 색상
    public float AnimationPhase; // 쉐이더에서 사용할 애니메이션 위상
    public int MaterialIndex; // 특정 쉐이더 로직 선택용
}

// 2. 렌더링 데이터를 준비하는 시스템 (Job System 활용)
// 예를 들어, 매 프레임마다 AnimationPhase를 업데이트하거나,
// 특정 조건에 따라 Color를 변경하는 등의 로직을 처리합니다.
// 이 시스템은 수십만 개의 인스턴스 데이터를 병렬로 빠르게 갱신합니다.
[BurstCompile]
public partial struct PrepareRenderDataSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float deltaTime = SystemAPI.Time.DeltaTime;
        float elapsedTime = SystemAPI.Time.ElapsedTime;

        // Job을 통해 모든 RenderInstanceData 컴포넌트를 병렬로 업데이트합니다.
        // 이 예제에서는 AnimationPhase를 시간과 엔티티 ID에 따라 다르게 설정합니다.
        new UpdateRenderJob
        {
            DeltaTime = deltaTime,
            ElapsedTime = elapsedTime
        }.ScheduleParallel();
    }

    [BurstCompile]
    public partial struct UpdateRenderJob : IJobEntity
    {
        public float DeltaTime;
        public float ElapsedTime;

        void Execute(Entity entity, ref RenderInstanceData renderData)
        {
            // 각 엔티티에 고유한 애니메이션 위상을 적용
            renderData.AnimationPhase = (ElapsedTime + (float)entity.Index * 0.1f) * 0.5f;

            // 특정 조건에 따라 색상 변경 (예시)
            if (math.sin(renderData.AnimationPhase) > 0.8f)
            {
                renderData.Color = new float4(1.0f, 0.5f, 0.0f, 1.0f); // 주황색
            }
            else
            {
                renderData.Color = new float4(0.0f, 0.8f, 1.0f, 1.0f); // 파란색
            }

            // 위치, 회전 등 다른 데이터도 여기서 효율적으로 업데이트 가능
            // renderData.Position += someVelocity * DeltaTime;
        }
    }
}
```

이렇게 DOTS를 통해 효율적으로 가공된 데이터는 이제 GPU로 넘어가 멋진 그림을 그릴 준비를 마칩니다. 중요한 점은, DOTS는 이 데이터를 개별 오브젝트의 형태로 넘기는 대신, **하나의 큰 버퍼(Structured Buffer)** 형태로 GPU에게 전달할 수 있게 해준다는 것입니다. 이는 Draw Call을 획기적으로 줄이고, GPU가 인스턴싱(Instancing)이라는 기법을 활용할 수 있는 기반을 마련해 줍니다.

### ✨ Shader Math & GLSL: 데이터 고속도로 위에서 펼쳐지는 시각적 마법

DOTS가 CPU와 메모리 측면의 고속도로를 깔아주었다면, 이제 GPU가 이 도로 위에서 시각적 마법을 부릴 차례입니다. 앞서 DOTS로 준비된 대량의 인스턴스 데이터는 GPU의 Structured Buffer (또는 SSBO)로 전송됩니다. 이제 쉐이더는 이 버퍼에 접근하여 `gl_InstanceID`와 같은 내장 변수를 활용해 각 인스턴스에 고유한 특성을 부여할 수 있습니다.

**문제: 단순히 많이 그리는 것을 넘어, '다양하고 동적인' 표현은 어떻게?**

단순히 동일한 메시를 수천 번 그리는 것은 쉽습니다. 하지만 각 풀잎이 바람에 다르게 흔들리고, 각 파티클이 고유한 색상과 크기를 가지며 시간에 따라 변화하는 모습은 어떻게 표현할까요? 여기서 **Shader Math와 GLSL**이 빛을 발합니다.

DOTS가 넘겨준 `Position`, `Rotation`, `Scale`, `Color`, `AnimationPhase` 같은 인스턴스별 데이터를 활용하여, GLSL 쉐이더는 각 인스턴스에 고유한 월드 변환 행렬을 적용하고, 색상을 보정하며, 심지어는 복잡한 수학 함수를 통해 애니메이션 효과를 구현할 수 있습니다.

**[핵심 로직: GLSL 쉐이더 (Pseudo-code)]**

```glsl
// Vertex Shader (예: 풀잎 또는 파티클의 움직임)
#version 450 core

layout(location = 0) in vec3 aPos;         // 모델 로컬 공간 정점 위치
layout(location = 1) in vec3 aNormal;       // 모델 로컬 공간 정점 노멀
// ... 다른 정점 속성 (UV 등) ...

// DOTS에서 넘겨받은 인스턴스 데이터 구조
struct InstanceData {
    mat4 objectToWorld;     // 인스턴스별 월드 변환 행렬
    vec4 baseColorFactor;   // 인스턴스별 기본 색상
    float animationPhase;   // 인스턴스별 애니메이션 위상
    // ... 필요한 추가 데이터 ...
};

// SSBO (Shader Storage Buffer Object)를 통해 대량의 인스턴스 데이터 접근
// layout(std430, binding = 0)은 Unity의 GraphicsBuffer.SetData와 같은
// 구조화된 버퍼를 사용할 때 유용합니다.
layout(std430, binding = 0) readonly buffer InstanceBuffer {
    InstanceData instances[];
};

// Fragment Shader로 전달할 변수
out vec3 v_worldPos;
out vec3 v_normal;
out vec4 v_color;
out float v_animationPhase;

void main()
{
    // gl_InstanceID는 현재 렌더링되는 인스턴스의 고유 ID입니다.
    // DOTS에서 준비한 데이터 버퍼의 인덱스로 활용됩니다.
    InstanceData currentInstance = instances[gl_InstanceID];

    mat4 modelMatrix = currentInstance.objectToWorld;

    // AnimationPhase와 노이즈 함수를 사용하여 바람 효과 구현 (Shader Math!)
    // sin, cos, Perlin/Simplex 노이즈 등 다양한 수학 함수 활용
    float windStrength = 0.5f;
    float bendFactor = sin(currentInstance.animationPhase * 2.0f + aPos.x * 0.5f) * windStrength;
    
    // Y축이 높은 정점일수록 더 많이 꺾이도록 조정 (풀잎 위쪽이 휘도록)
    vec3 animatedPos = aPos;
    animatedPos.x += aPos.y * bendFactor; // 간단한 꺾임 효과

    v_worldPos = (modelMatrix * vec4(animatedPos, 1.0)).xyz;
    gl_Position = CameraProjectionMatrix * CameraViewMatrix * vec4(v_worldPos, 1.0);

    v_normal = mat3(modelMatrix) * aNormal;
    v_color = currentInstance.baseColorFactor;
    v_animationPhase = currentInstance.animationPhase; // 프래그먼트 쉐이더로 전달
}

// Fragment Shader (예: 풀잎 또는 파티클의 색상 및 디테일)
#version 450 core

in vec3 v_worldPos;
in vec3 v_normal;
in vec4 v_color;
in float v_animationPhase;

out vec4 FragColor;

void main()
{
    // 광원 계산 (생략)
    // vec3 lightDir = normalize(vec3(0.5, 1.0, 0.0));
    // float diff = max(dot(v_normal, lightDir), 0.0);
    // vec3 finalLight = vec3(0.3) + vec3(0.7) * diff;

    // 인스턴스별 색상에 시간과 노이즈를 더해 동적인 효과 연출 (Shader Math!)
    vec3 proceduralColor = v_color.rgb * (0.8 + 0.2 * sin(v_animationPhase * 3.0 + v_worldPos.x * 0.1));
    proceduralColor *= (0.7 + 0.3 * cos(v_worldPos.z * 0.2 + v_animationPhase)); // Z축에 따른 변화

    // 더 복잡한 노이즈 텍스처 룩업이나 프로시저럴 패턴 생성도 가능합니다.
    // 예를 들어, Perlin 노이즈 함수를 GLSL로 구현하여 풀잎의 불규칙한 색상 변이를 줄 수 있습니다.
    
    FragColor = vec4(proceduralColor, 1.0);
}
```

이 코드를 통해 우리는 DOTS가 마련해준 데이터 고속도로 위에서 GLSL이 어떻게 춤을 추는지 엿볼 수 있습니다. CPU는 데이터 준비에만 집중하고, GPU는 받은 데이터를 바탕으로 수천, 수만 개의 인스턴스를 각기 다른 모습으로, 고성능으로 렌더링하는 데 모든 역량을 쏟는 거죠. 더 이상 CPU가 "이거 다 계산하려면 너무 느려!"라고 소리칠 걱정 없이, GPU에게 "네가 멋진 그림을 그려봐!"라고 맡길 수 있게 된 것입니다.

### 💡 도전과 기회: 시니어 개발자의 시야

물론, DOTS와 Shader Math를 깊이 파고드는 것은 쉬운 일이 아닙니다.

*   **높은 학습 곡선**: 새로운 패러다임과 낯선 수학 개념들을 익혀야 합니다.
*   **디버깅의 복잡성**: CPU와 GPU 사이를 넘나들며 발생하는 문제를 디버깅하는 것은 인내를 요구합니다.
*   **툴링의 성숙도**: DOTS는 계속 발전 중이며, 쉐이더 디버깅 툴도 완벽하지 않을 때가 있습니다.

하지만 이 모든 어려움은 곧 엄청난 **기회**를 의미합니다. 저의 경험상, 이 두 영역을 깊이 이해하고 결합했을 때, 여러분은 기존의 개발 방식으로는 상상하기 어려웠던 규모와 디테일의 게임 월드를 구현할 수 있게 됩니다. 이는 곧 독창적이고 몰입감 넘치는 플레이 경험으로 이어질 것입니다.

### 🌟 결론: 미래 게임 개발의 핵심 시너지

Data-Oriented Technology Stack (DOTS)은 데이터를 효율적으로 관리하고 CPU의 한계를 뛰어넘게 해줍니다. 그리고 이 DOTS가 깔아준 강력한 기반 위에서 Shader Math와 GLSL은 수많은 인스턴스를 '그저 그려내는' 것을 넘어, 각 인스턴스에 생명력을 불어넣고 복잡하고 아름다운 시각적 효과를 창조할 수 있게 합니다.

이 둘의 시너지는 단순히 성능 최적화를 넘어, 여러분이 게임 디자인에서 더 과감하고 혁신적인 시도를 할 수 있는 자유를 선사합니다. 저는 이 기술들이 미래 게임 개발의 핵심 기둥이 될 것이라고 확신합니다.

동료 개발자 여러분, 이제 옛 방식을 고집하는 대신, 이 새로운 패러다임을 받아들이고 깊이 파고들어 보세요. 데이터가 흐르는 고속도로 위에서 쉐이더가 펼치는 마법 같은 이야기를 직접 만들어내는 경험은, 여러분의 개발 인생에 잊을 수 없는 전환점이 될 것입니다. 다음 포스팅에서 또 다른 깊은 기술 이야기로 찾아오겠습니다!
