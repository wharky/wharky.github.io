### 문제: 혼돈의 카오스, TRPG 규칙 시스템

제가 처음 게임 개발에 뛰어들었을 때, 가장 고통스러웠던 부분 중 하나가 바로 '규칙 시스템' 구현이었습니다. 특히 TRPG(Tabletop Role-Playing Game)에서 영감을 받은 RPG들은 캐릭터 스탯, 스킬, 버프/디버프, 장비 효과, 환경 요인 등 수많은 변수들이 복잡하게 얽혀 최종 결과를 도출합니다.

초보 때는 대략 이런 식이었죠:

```cpp
// 낡은 방식의 calculate_attack_damage() 함수 (상상만 해도 끔찍)
float calculate_attack_damage(Character attacker, Character defender, Weapon weapon, Skill skill, Environment env) {
    float damage = attacker.baseStrength;

    if (attacker.has_buff("Rage")) {
        damage *= 1.2f;
    }
    if (weapon.type == WeaponType.Sword) {
        damage += weapon.damage_modifier;
    }
    // ... 수십, 수백 개의 if-else if 블록 ...

    if (defender.has_debuff("Weakness")) {
        damage *= 0.8f;
    }
    // ... 방어자 관련 if-else if ...

    if (env.is_raining && skill.name == "LightningStrike") {
        damage *= 1.5f;
    }
    // ... 환경 관련 if-else if ...

    return damage - defender.armor; // 최종 계산
}
```

이런 코드는 금세 유지보수 지옥으로 변합니다. 새로운 스킬이나 아이템이 추가될 때마다 해당 함수를 열고 `if-else if` 블록을 추가해야 했고, 버그가 발생하면 어디서부터 잘못되었는지 추적하는 데만 하루가 꼬박 걸렸죠. 특정 조건에서 버프가 중첩되지 않거나, 디버프가 예상과 다르게 적용되는 문제는 거의 매 업데이트마다 저를 괴롭혔습니다. 시스템의 확장성? 예측 불가능성? '실력 있는' 개발자라고 부르기 민망할 정도였습니다.

그렇다면 이 복잡한 규칙들을 어떻게 하면 우아하고 확장성 있게 코드로 구현할 수 있을까요? 여기서 저는 **'Shader Math & GLSL'** 에서 얻은 영감을 공유하고 싶습니다.

### 해결책: 셰이더 파이프라인처럼, 규칙을 데이터와 함수로

셰이더는 수많은 픽셀 하나하나에 대해 병렬적으로, 그리고 결정론적으로 색상을 계산합니다. `main` 함수 안에서 수많은 수학적 연산과 조건 분기를 거쳐 최종 `gl_FragColor`를 결정하죠. 중요한 것은, 셰이더는 외부 상태에 크게 의존하지 않고, 주어진 입력(uniforms, varying)과 내부 로직만으로 출력을 만들어낸다는 점입니다. 그리고 각 연산 단계가 매우 모듈화되어 있어 재사용성이 높고, 이해하기 쉽습니다.

이러한 셰이더의 설계 철학을 TRPG 규칙 시스템에 적용해 보면 어떨까요?

#### 1. 입력(Input)과 출력(Output)의 명확한 정의

셰이더는 UV 좌표, 노멀 벡터, 텍스처 등 명확한 입력을 받아 픽셀 색상이라는 명확한 출력을 냅니다. 우리 TRPG 시스템도 마찬가지여야 합니다. "공격력 계산"이라는 작업에 필요한 모든 입력(공격자의 기본 능력치, 스킬, 활성화된 버프/디버프, 대상의 방어력 등)을 명확히 정의하고, 단 하나의 출력(최종 피해량)을 목표로 해야 합니다.

#### 2. 규칙의 모듈화 및 데이터 기반 설계

GLSL에서 복잡한 노이즈 함수나 라이팅 모델을 작은 함수들로 쪼개어 재사용하고 조합하듯이, TRPG 규칙도 작은 '수정자(Modifier)' 단위로 쪼개야 합니다.

**문제:** 캐릭터의 공격력은 '기본 힘', '무기 공격력', '버프', '디버프' 등 다양한 요소에 의해 결정됩니다. 이들을 어떻게 관리하고 적용할까요?

**해결:** 모든 수정자를 `IAttributeModifier` 인터페이스를 구현하는 객체로 만듭니다. 이 수정자들은 단순히 '값'과 '적용 방식(평면 증가, 퍼센트 증가, 곱셈 등)'을 가지고 있습니다. 캐릭터의 각 능력치(Attribute)는 이 수정자들의 리스트를 가집니다.

```cpp
// 의사 코드 (Pseudo-code)

// 수정자 인터페이스
interface IAttributeModifier {
    ModifierType GetType(); // FLAT, PERCENT_ADD, PERCENT_MULTIPLY 등
    float GetValue();
    int GetOrder(); // 적용 순서 (e.g., FLAT 먼저, PERCENT 나중)
    string GetSource(); // 어떤 스킬/아이템/버프에서 왔는지 (디버깅용)
}

// 스탯 (예: 힘, 지능, 체력)
class Attribute {
    string Name;
    float BaseValue;
    List<IAttributeModifier> Modifiers; // 이 스탯에 적용된 모든 수정자들

    // 최종 스탯 값 계산 (셰이더의 main() 함수처럼)
    float CalculateFinalValue() {
        // 1단계: 기본 값에 FLAT 수정자 적용
        float currentValue = BaseValue;
        foreach (modifier in Modifiers.Where(m => m.GetType() == ModifierType.FLAT).OrderBy(m => m.GetOrder())) {
            currentValue += modifier.GetValue();
        }

        // 2단계: 퍼센트 증가/감소 수정자 적용
        float percentModifier = 1.0f;
        foreach (modifier in Modifiers.Where(m => m.GetType() == ModifierType.PERCENT_ADD).OrderBy(m => m.GetOrder())) {
            percentModifier += modifier.GetValue(); // 예를 들어 0.1f는 10% 증가
        }
        foreach (modifier in Modifiers.Where(m => m.GetType() == ModifierType.PERCENT_MULTIPLY).OrderBy(m => m.GetOrder())) {
            currentValue *= modifier.GetValue(); // 예를 들어 1.2f는 20% 증가
        }

        return currentValue * percentModifier;
    }

    void AddModifier(IAttributeModifier modifier) { /* 중복 검사 등 */ Modifiers.Add(modifier); }
    void RemoveModifier(IAttributeModifier modifier) { /* 특정 source의 modifier 제거 */ Modifiers.Remove(modifier); }
}

// 캐릭터
class Character {
    Map<string, Attribute> Attributes; // "Strength", "Dexterity" 등
    List<ActiveEffect> ActiveEffects; // 현재 캐릭터에게 걸려있는 모든 버프/디버프

    void ApplyEffect(EffectData effectData) {
        // effectData에서 정의된 modifier들을 생성하여 Attributes에 AddModifier
        // 예: Strength.AddModifier(new FlatModifier(-5, "Poison"));
        // ActiveEffects에도 추가하여 나중에 제거할 때 사용
    }

    void RemoveEffect(EffectData effectData) {
        // ActiveEffects에서 찾아 제거하고, Attributes에서 해당 source의 modifier 제거
    }
}
```

이 방식은 마치 셰이더가 여러 텍스처와 노멀 맵, 라이팅 모델 함수들을 조합하여 최종 픽셀을 그려내듯이, 다양한 수정자들을 조합하여 최종 스탯 값을 계산합니다. 수정자의 `GetOrder()`와 `GetType()`은 셰이더의 연산 순서(예: 텍스처 샘플링 -> 라이팅 계산 -> 포스트 프로세싱)처럼 중요합니다. 순서가 바뀌면 결과가 완전히 달라질 수 있거든요!

더 나아가, 이 `IAttributeModifier` 구현체들은 코드에 하드코딩될 필요가 없습니다. 대부분의 정보(타입, 값, 적용 순서, 출처)는 JSON이나 XML 같은 데이터 파일로 정의하고, 런타임에 로드하여 객체로 인스턴스화할 수 있습니다. 마치 셰이더의 `uniform` 변수가 외부에서 데이터를 받아오듯이 말이죠. 새로운 버프나 장비 효과를 추가할 때, 이제는 프로그래머가 아닌 기획자가 데이터만 수정하면 됩니다. 이것이 진정한 '데이터 기반 설계'의 힘이죠.

#### 3. 결정론적 변환과 예측 가능한 결과

셰이더는 동일한 입력이 주어지면 *항상* 동일한 출력을 생성합니다. TRPG 시스템도 이러한 '결정론적 변환'을 지향해야 합니다. 물론 TRPG에는 주사위 굴림 같은 무작위 요소가 있지만, 이마저도 '시드(Seed)'라는 입력을 받아 결정론적으로 결과를 생성하는 함수로 볼 수 있습니다.

`CalculateFinalValue()` 함수는 `BaseValue`와 `Modifiers` 리스트라는 명확한 입력을 받아 항상 동일한 `finalValue`를 반환합니다. 이는 디버깅을 쉽게 하고, 예측 가능한 게임 플레이를 가능하게 합니다. "왜 이 스킬 데미지가 이렇게 나왔지?" 라는 질문에, `Modifiers` 리스트를 추적하면 명확한 답을 얻을 수 있습니다.

#### 4. 병렬적 사고의 확장

GLSL은 기본적으로 병렬 처리의 극치입니다. 수백만 개의 픽셀이 동시에 자신만의 계산을 수행하죠. TRPG 시스템 자체를 병렬 처리하라는 의미는 아니지만, `Attribute` 하나하나를 독립적인 계산 단위로 보고, 각 `Attribute`가 다른 `Attribute`에 최소한의 의존성을 가지도록 설계하는 것은 셰이더의 '병렬적 사고'에서 배울 점이 많습니다. 서로 간섭을 최소화하고 독립적으로 계산될 수 있도록 말이죠.

### 깊이 있는 통찰: 현업에서 얻은 교훈

제가 이 시스템으로 바꾸고 나서 가장 크게 느낀 점은 **'코드의 선형적인 흐름이 아니라, 데이터의 흐름과 변환에 집중하게 되었다'** 는 것입니다. 과거에는 `if-else` 문을 따라가며 코드를 읽었다면, 이제는 어떤 데이터(Attribute)가 있고, 어떤 수정자들(Modifiers)이 이 데이터를 어떤 순서로 변환(CalculateFinalValue)하는지를 파악하는 방식으로 생각하게 됩니다. 이는 셰이더 코드를 읽을 때 "이 UV 좌표가 어떤 함수들을 거쳐 최종 색상이 되었는가"를 분석하는 방식과 놀랍도록 유사합니다.

또한, 이 방식은 유닛 테스트 작성에 매우 유리합니다. 특정 `Attribute`에 특정 `Modifier`를 적용했을 때 `CalculateFinalValue()`가 예상대로 동작하는지 작은 단위로 검증할 수 있습니다. 셰이더의 작은 함수를 테스트하듯 말이죠.

오늘 우리는 **TRPG 규칙 시스템**의 복잡성을 해결하기 위해 **Shader Math와 GLSL**의 설계 원칙, 즉 '모듈화된 함수, 데이터 기반의 입력-출력 변환, 결정론적 처리'에서 영감을 얻어보았습니다. 겉보기에는 전혀 관련 없어 보이는 두 분야이지만, 본질적으로 복잡한 문제를 단순하고 우아하게 해결하려는 공통된 목표를 가지고 있습니다.

현업 개발자로서 제가 드리고 싶은 메시지는 이겁니다: 때로는 여러분의 주력 분야가 아닌 다른 기술 영역에서 놀라운 통찰을 얻을 수 있습니다. 그래픽스 셰이더의 수학적 우아함은 복잡한 게임 로직을 설계하는 데 영감을 줄 수 있고, 데이터베이스 최적화 기법은 게임 서버의 백엔드 성능을 비약적으로 끌어올릴 수도 있습니다. 시야를 넓히고, 다양한 분야의 설계 철학을 여러분의 문제에 대입해보세요.
