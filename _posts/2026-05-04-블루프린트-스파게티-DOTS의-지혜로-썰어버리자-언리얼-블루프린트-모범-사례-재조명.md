### 🩸 문제: 편리함 뒤에 숨은 블루프린트의 덫

처음 언리얼 엔진을 접했을 때, 블루프린트의 마법 같은 편리함에 다들 감탄하셨을 겁니다. 드래그 앤 드롭 몇 번으로 복잡한 로직을 구현하고, 코드를 직접 작성하지 않아도 게임이 움직이는 것을 보면 희열을 느끼죠. 하지만 프로젝트가 커지고 팀원들이 늘어나면서, 이 편리함은 종종 우리를 벼랑 끝으로 몰아세웁니다.

**우리가 흔히 겪는 문제들:**

1.  **스파게티 코드 (Spaghetti Graphs):** 노드와 와이어가 거미줄처럼 얽히고설켜, 한 번 로직을 수정하려면 온몸의 털이 곤두서는 경험.
2.  **성능 병목 현상 (Performance Bottlenecks):** 모든 Actor가 매 Tick마다 불필요한 연산을 수행하거나, 복잡한 레퍼런스 체인 때문에 프레임 드랍이 발생하는 경우.
3.  **높은 의존성 (Tight Coupling):** 특정 Actor나 Component에 대한 직접적인 참조가 너무 많아, 하나를 수정하면 다른 여러 곳에서 예상치 못한 버그가 터지는 상황.
4.  **확장성 및 유지보수의 어려움:** 새로운 기능을 추가하거나 기존 로직을 리팩토링하기가 거의 불가능에 가까워지는 악몽.

이런 문제들을 보면서 저는 종종 "과연 블루프린트가 잘못된 걸까?"라는 자문을 던지곤 했습니다. 아니오, 블루프린트는 잘못되지 않았습니다. 우리가 블루프린트를 다루는 '방식'이 문제였던 거죠. 그리고 저는 그 해답의 실마리를 DOTS의 데이터 지향적 사고방식에서 찾기 시작했습니다.

### 💡 해결책: DOTS 철학으로 블루프린트 아키텍처 재구성하기

DOTS의 핵심은 '데이터'와 '로직'을 명확히 분리하고, 데이터를 효율적으로 처리하며, 시스템들이 독립적으로 작동하도록 만드는 것입니다. 이를 통해 성능, 확장성, 병렬성을 극대화하죠. 우리는 언리얼 엔진의 블루프린트 환경에서 DOTS의 엔티티-컴포넌트-시스템(ECS) 패턴을 그대로 구현할 수는 없지만, 그 핵심 정신을 빌려 블루프린트의 고질적인 문제를 해결할 수 있습니다.

**핵심은 "데이터를 중심에 두고, 필요한 로직만 최소한으로 실행하며, 의존성을 끊어내자" 입니다.**

---

#### 1. 데이터와 로직 분리: Actor는 '컨테이너'이자 '식별자', Component가 '로직'이다

**문제:** 흔히 블루프린트를 만들 때, Actor 클래스 안에 캐릭터의 체력, 이동 속도, 공격력 같은 데이터는 물론, 이동 로직, 공격 로직, 피격 로직까지 모조리 때려 넣는 경우가 많습니다. 결과적으로 Actor 블루프린트는 거대한 '만능 주머니'가 되고, 특정 데이터를 수정하려 해도 온갖 로직이 얽혀 있어 변경이 두려워집니다.

**DOTS 관점에서의 해결책:** DOTS에서 엔티티(Entity)는 데이터의 단순한 식별자이고, 컴포넌트(Component)는 데이터 덩어리입니다. 로직은 시스템(System)에서 담당하죠. 블루프린트에서는 Actor를 '엔티티'처럼, Actor Component를 '컴포넌트'처럼 활용하여 데이터와 로직을 분리할 수 있습니다.

*   **Data-Only Blueprints & Data Assets:** 정적인 설정값이나 에셋 데이터는 `Data Asset`이나 `Data Table`을 활용하세요. 런타임에 변할 수 있는 데이터라면 `Struct`를 정의하고, 이 `Struct`를 단일 목적의 `Actor Component`에 담아 관리하는 것을 권장합니다.
*   **Component-Based Architecture 강화:**
    *   **Bad Example (Actor에 모든 것을 때려 넣기):**
        ```pseudo
        // Character_BP (Actor)
        Variables:
            Health: Float
            AttackPower: Float
            MovementSpeed: Float
        Functions:
            MoveForward() { ... }
            TakeDamage(DamageAmount) { Health -= DamageAmount; ... }
            Attack() { ... }
        Event Tick:
            UpdateMovement()
            UpdateAnimation()
        ```
    *   **Good Example (Component를 활용한 분리):**
        ```pseudo
        // S_CharacterStats (Struct)
        Variables:
            Health: Float
            AttackPower: Float
            MovementSpeed: Float

        // HealthComponent_BP (Actor Component)
        Variables:
            CharacterStats: S_CharacterStats // Struct 인스턴스
        Functions:
            InitializeHealth(InitialStats: S_CharacterStats)
            TakeDamage(DamageAmount: Float) { CharacterStats.Health -= DamageAmount; }
            OnDeath_EventDispatcher // 체력이 0이 되면 브로드캐스트

        // MovementComponent_BP (Actor Component)
        Variables:
            MovementSpeed: Float // 혹은 S_CharacterStats에서 가져옴
        Functions:
            SetTargetLocation(Location: Vector)
            MoveToTarget()
        Event Tick:
            // SetTargetLocation이 호출되었을 때만 이동 로직 수행
            // 아니면 특정 조건에서만 실행되도록 가드
        ```
        이처럼 Actor는 `HealthComponent`, `MovementComponent` 등을 가지는 컨테이너 역할을 하고, 각 Component는 자신의 역할과 관련된 데이터와 로직만 담당하게 됩니다. 이 방식은 재사용성을 높이고, 특정 기능을 수정할 때 다른 부분에 미치는 영향을 최소화합니다.

#### 2. 의존성 최소화: 직접 참조 대신 인터페이스와 이벤트를 활용하라

**문제:** 블루프린트에서 가장 흔하고 치명적인 실수는 바로 과도한 직접 참조(Direct Reference)와 `Cast To` 남용입니다. "Get All Actors Of Class"로 모든 액터를 가져와서 일일이 `Cast To`를 시도하거나, 특정 액터의 변수를 직접 가져와 조작하는 것은 극심한 성능 저하와 유지보수 지옥으로 가는 지름길입니다.

**DOTS 관점에서의 해결책:** DOTS에서는 시스템들이 엔티티의 특정 컴포넌트를 보고 반응합니다. 직접적으로 "이 엔티티가 저 엔티티의 특정 메소드를 호출하라"는 식의 명령이 아니라, "이 컴포넌트를 가진 모든 엔티티에 대해 이 로직을 실행하라"는 방식이죠. 블루프린트에서도 '메시지 기반' 또는 '데이터 기반' 소통 방식을 통해 직접 참조를 줄일 수 있습니다.

*   **Blueprint Interface:** Actor나 Component가 '무엇을 할 수 있는지'에 대한 계약을 정의합니다. 특정 함수를 호출하기 위해 해당 Actor의 클래스를 알 필요 없이, 단순히 인터페이스를 구현했는지만 확인하고 함수를 호출할 수 있습니다.
    *   **Bad Example:** `Cast To Enemy_BP` -> `Call TakeDamage()`
    *   **Good Example:** `Does Implement Interface (BPI_Damageable)` -> `Call TakeDamage (via interface message)`
    ```pseudo
    // BPI_Damageable (Blueprint Interface)
    Function: TakeDamage(DamageAmount: Float)

    // PlayerAttack_BP (Actor Component)
    OnOverlap (with other Actor):
        If OtherActor Implements BPI_Damageable:
            Call TakeDamage (via interface message) on OtherActor
        Else:
            // Do nothing or other logic
    ```
    인터페이스는 강력한 느슨한 결합(Loose Coupling) 메커니즘을 제공하여, 로직을 유연하게 확장하고 수정할 수 있게 합니다.

*   **Event Dispatcher:** 1대다(One-to-many) 통신에 이상적입니다. 특정 사건이 발생했음을 알리고, 관심 있는 다른 Actor나 Component들이 이를 구독하여 반응하게 합니다. 직접적인 참조 없이 이벤트를 통해 서로 소통하게 되죠.

*   **Gameplay Tags:** 특정 상태나 속성을 태그로 표현하고, 로직에서 이 태그를 활용해 대상을 필터링하거나 행동을 결정합니다. "이 액터가 '적' 태그를 가지고 있는가?"를 묻는 것이 "이 액터가 'EnemyCharacter_BP'인가?"를 묻는 것보다 훨씬 유연하고 데이터 지향적인 질문입니다.

#### 3. Tick 함수의 지옥에서 벗어나기: 필요한 순간에만 작동하는 효율적인 시스템

**문제:** 성능 저하의 주범 중 하나는 바로 무분별한 `Event Tick` 사용입니다. 매 프레임마다 불필요한 로직이 실행되거나, 데이터 변경 여부와 상관없이 계속해서 연산이 반복되면, 작은 기능이라도 수많은 Actor에 적용될 경우 치명적인 성능 저하로 이어집니다.

**DOTS 관점에서의 해결책:** DOTS에서는 시스템이 오직 '관심 있는 컴포넌트'의 '데이터 변경'에 대해서만 반응합니다. 블루프린트에서도 이러한 '이벤트 기반' 또는 '데이터 변화 감지' 방식으로 로직을 전환해야 합니다.

*   **Event-Driven Updates:** 대부분의 로직은 Tick 대신 특정 이벤트가 발생했을 때만 실행되도록 설계하세요.
    *   예시: `MovementComponent`는 `SetTargetLocation`이 호출되었을 때만 이동 로직을 실행하고, 목표에 도달하면 `StopMovement` 이벤트를 발생시킵니다.
    *   `HealthComponent`는 `TakeDamage`가 호출될 때만 체력 감소 로직을 처리하고, 체력이 0이 되면 `OnDeath_EventDispatcher`를 브로드캐스트합니다.
*   **Timers와 Gate:** 주기적인 업데이트가 필요하다면 `Set Timer by Event` 또는 `Gate` 노드를 활용하여 특정 조건에서만 로직이 실행되도록 제어할 수 있습니다.
*   **Blueprint Nativization:** 최종 단계에서 성능이 심각하게 문제가 되는 블루프린트는 Nativization을 고려해볼 수 있습니다. 하지만 이는 구조적인 개선이 선행된 후에 고려해야 할 사항입니다. 구조가 나쁘면 Nativization도 한계가 명확하거든요.
*   **프로파일링 필수:** 언리얼 엔진의 프로파일링 툴(CPU Profiler, GPU Profiler)을 적극적으로 활용하여, 실제로 어디에서 병목 현상이 발생하는지 정확히 파악해야 합니다. 막연한 추측은 개발 시간을 낭비하게 할 뿐입니다.

```pseudo
// MovementComponent_BP (Good Example - Event Driven)
Variables:
    bIsMoving: Boolean
    TargetLocation: Vector
    CurrentLocation: Vector

Event SetTargetLocation (NewTarget: Vector):
    TargetLocation = NewTarget
    bIsMoving = True
    // Start moving logic only now, e.g., set a timer for interpolated movement
    // Or, if small amount, just do it immediately.
    // Or, if using a character movement component, call its MoveTo function.

Event Tick: // Only if absolutely necessary for continuous, complex physics or AI
    If bIsMoving:
        // Interpolate movement towards TargetLocation
        CurrentLocation = FMath::VInterpTo(CurrentLocation, TargetLocation, GetWorldDeltaSeconds(), InterpSpeed)
        If Vector_Distance(CurrentLocation, TargetLocation) < Threshold:
            bIsMoving = False
            OnMovementFinished_EventDispatcher.Broadcast()
```
이 로직은 매 Tick마다 `MoveToTarget`을 호출하는 대신, `SetTargetLocation`이 호출될 때만 `bIsMoving` 플래그를 활성화하고, `Tick`에서는 이 플래그가 True일 때만 실제 이동 연산을 수행하도록 합니다. 목표에 도달하면 플래그를 다시 끄고, 필요 없는 Tick 연산을 멈추는 거죠.

### 👴🏼 나의 경험담: 블루프린트를 탓하기 전에 나 자신을 돌아보라

저도 한때 블루프린트의 매력에 푹 빠져 앞뒤 안 가리고 쓰다가, 프로젝트 막바지에 새벽까지 버그를 잡고 성능 최적화에 매달렸던 경험이 있습니다. "블루프린트라서 느린 거야!"라고 툴 탓을 하기도 했죠. 하지만 시간이 지나고 다양한 프로젝트를 경험하면서, 문제는 툴이 아니라 '툴을 다루는 나의 방식'에 있었다는 것을 깨달았습니다.

DOTS는 유니티에서 나온 기술이지만, 그 깊은 철학은 특정 엔진에 국한되지 않는다고 생각합니다. '데이터를 효율적으로 다루는 것'은 모든 프로그래밍 패러다임과 언어, 엔진에서 핵심적인 가치이기 때문입니다. 블루프린트 개발자로서 우리는 그 편리함에 취해 '데이터'라는 근본적인 개념을 소홀히 할 때가 많습니다. 하지만 DOTS의 관점에서 블루프린트를 다시 바라보면, Actor가 품고 있는 데이터의 의미, Component의 역할, 그리고 로직이 언제 어떻게 실행되어야 하는지에 대한 통찰을 얻을 수 있습니다.

블루프린트는 단순한 '드래그 앤 드롭' 도구가 아닙니다. 그것은 견고한 게임 로직을 설계하는 '시각적 프로그래밍 언어'입니다. 이 언어를 마스터하기 위해서는 단순히 노드를 연결하는 것을 넘어, 데이터 흐름, 의존성 관리, 그리고 효율적인 실행 방식에 대한 깊은 이해가 필요합니다.

### 🚀 결론: 블루프린트를 '시니어'답게 다루는 법

블루프린트는 언리얼 엔진의 강력한 강점이며, 빠른 프로토타이핑과 이터레이션을 가능하게 하는 보석 같은 존재입니다. 하지만 그 힘은 신중하게 다루지 않으면 오히려 독이 될 수 있습니다. DOTS에서 영감을 받은 데이터 지향적 사고방식을 블루프린트 개발에 적용함으로써, 우리는 다음과 같은 이점을 얻을 수 있습니다.

*   **명확한 구조:** 데이터와 로직이 분리되어 코드가 훨씬 이해하기 쉬워집니다.

*   **향상된 성능:** 불필요한 연산을 줄이고 필요한 로직만 실행하여 프레임 드랍을 최소화합니다.
