### **문제 1: 끝없는 드로우 콜 지옥과 쉐이더 오버헤드**
**"우리 게임은 왜 항상 CPU 바운드인가요?"**

어느 날 저에게 찾아온 주니어 개발자의 한탄이었습니다. 프로파일러를 돌려보니, 수천, 수만 개의 드로우 콜이 CPU를 갉아먹고 있었죠. 유니티의 Batching 시스템이 분명 있는데, 왜 이럴까요?

#### **레트로 콘솔의 지혜: 스프라이트 배치(Sprite Batching)와 텍스처 아틀라스(Texture Atlas)**
NES나 SNES 시절, VRAM은 극히 제한적이었고, CPU는 오늘날의 계산기보다도 느렸습니다. 당시 개발자들은 화면에 수많은 스프라이트를 뿌리기 위해 '드로우 콜'이라는 개념조차 없었지만, 본질적으로는 CPU가 그래픽 컨트롤러에 명령을 내리는 횟수를 최소화해야 했습니다. 그들은 어떻게 했을까요?

1.  **스프라이트 아틀라스:** 여러 작은 이미지를 하나의 큰 텍스처에 모아두고, 필요한 부분만 잘라 사용했습니다. 이는 메모리 효율성뿐만 아니라, 나중에 등장할 '드로우 콜' 감소의 초석이 됩니다.
2.  **수동 배치:** 텍스처와 쉐이더(개념은 없지만)가 같은 오브젝트들을 모아서 한 번에 그리도록 데이터를 구성했습니다. 이른바 수동 "배치" 작업이었죠. PS1 시절에는 T-Buffer에 필요한 텍셀을 직접 전송하는 등, GPU에 데이터를 넘기는 과정을 극도로 제어했습니다.

#### **유니티 적용: Batching, Instancing, 그리고 Combine Meshes의 재발견**
유니티는 우리를 위해 Static Batching, Dynamic Batching, GPU Instancing 같은 훌륭한 시스템을 제공합니다. 하지만 이것만으로는 부족할 때가 많습니다.

*   **Static Batching:** 정적 오브젝트는 Mesh를 합쳐버려 드로우 콜을 크게 줄여줍니다. `Mesh.CombineMeshes()` API를 직접 활용하여 특정 조건에서만 동작하는 커스텀 Static Batching을 구현할 수도 있습니다.
*   **GPU Instancing:** 동일한 Mesh와 Material을 사용하는 수많은 오브젝트를 단 한 번의 드로우 콜로 렌더링할 수 있게 해주는 마법 같은 기술입니다. (Material에서 Enable GPU Instancing을 켜는 것을 잊지 마세요!)
*   **Custom Batching/Combine:** 레트로 개발자들이 그랬듯, 수동으로 배치 그룹을 만들고 Mesh를 합치는 전략은 여전히 유효합니다. 예를 들어, 특정 지형 오브젝트나 수많은 작은 프랍들은 런타임에 하나의 Mesh로 합쳐 버리면 획기적인 성능 향상을 볼 수 있습니다.

```csharp
// 의사코드: Custom Mesh Combine for Static Props
public class PropMeshCombiner : MonoBehaviour
{
    public float combineDistanceThreshold = 100f; // 일정 거리 내의 프랍만 합치기
    private List<MeshFilter> nearbyProps = new List<MeshFilter>();
    private MeshFilter combinedMeshFilter;
    private MeshRenderer combinedMeshRenderer;

    void Start()
    {
        combinedMeshFilter = gameObject.AddComponent<MeshFilter>();
        combinedMeshRenderer = gameObject.AddComponent<MeshRenderer>();
        // 모든 Prop들을 찾아서 같은 Material을 사용하는 것들만 합칠 준비
        CombineNearbyProps();
    }

    void CombineNearbyProps()
    {
        List<CombineInstance> combine = new List<CombineInstance>();
        Material sharedMaterial = null;

        // 예시: 특정 태그를 가진 오브젝트들을 찾아
        // 카메라 거리, Material 등을 고려하여 합칠 오브젝트 필터링
        GameObject[] props = GameObject.FindGameObjectsWithTag("CombinableProp");
        foreach (GameObject prop in props)
        {
            MeshFilter mf = prop.GetComponent<MeshFilter>();
            MeshRenderer mr = prop.GetComponent<MeshRenderer>();
            if (mf != null && mr != null)
            {
                // 첫 번째 prop의 Material을 기준으로 삼기
                if (sharedMaterial == null) sharedMaterial = mr.sharedMaterial;

                if (mr.sharedMaterial == sharedMaterial &&
                    Vector3.Distance(transform.position, prop.transform.position) < combineDistanceThreshold)
                {
                    CombineInstance ci = new CombineInstance();
                    ci.mesh = mf.sharedMesh;
                    ci.transform = mf.transform.localToWorldMatrix; // 월드 좌표계 기준으로 합치기
                    combine.Add(ci);

                    prop.SetActive(false); // 합쳐진 원본 오브젝트는 비활성화
                }
            }
        }

        // 새 Mesh 생성 및 할당
        Mesh newMesh = new Mesh();
        newMesh.CombineMeshes(combine.ToArray(), true, true); // true: 하나의 서브메쉬로 합침, true: 월드 좌표계 사용
        combinedMeshFilter.sharedMesh = newMesh;
        combinedMeshRenderer.sharedMaterial = sharedMaterial; // 합친 Material 할당
    }
}
```
**Opinion:** 레트로 개발자들은 하드웨어의 한계를 창의력과 극한의 효율성으로 극복했습니다. 우리는 그 정신을 배워야 합니다. 유니티가 제공하는 편리함 뒤에 숨겨진 "진정한 최적화"는 결국 개발자가 직접 데이터를 어떻게 구성하고, 어떻게 그릴 것인지에 대한 깊은 이해에서 나옵니다. "같은 것은 묶어서 처리한다"는 단순하지만 강력한 진리를 다시 한번 되새겨 봅시다.

### **문제 2: 비대한 메모리 사용량과 느린 로딩 시간**
**"우리 게임은 로딩이 왜 이렇게 오래 걸리고, 메모리는 왜 이렇게 많이 먹을까요?"**

최근 프로젝트에서 종종 듣는 질문입니다. 텍스처 압축, 에셋 번들 사용 등 기본적인 것들은 다 했는데도 말이죠. Profiler를 열어보면 분명 언로드되었어야 할 에셋들이 메모리에 둥둥 떠다니는 것을 발견하곤 합니다.

#### **레트로 콘솔의 지혜: 수동 메모리 관리와 에셋 스트리밍**
PS1 같은 CD-ROM 기반 콘솔은 매우 느린 I/O 속도와 2MB에 불과한 끔찍하게 적은 RAM을 가지고 있었습니다. (물론 CPU, GPU에도 메모리가 있었지만요). 이들에게는 게임 데이터 전체를 한 번에 로드한다는 것은 꿈도 못 꿀 이야기였죠.

1.  **명시적 메모리 할당/해제:** 개발자들이 메모리 블록을 직접 할당하고, 사용이 끝나면 해제하는 것이 일상다반사였습니다. `malloc`, `free`와 같은 함수들이 아니라, 특정 메모리 영역을 미리 예약해두고 거기서 텍스처나 모델 데이터를 로드하는 방식이 흔했습니다.
2.  **오버레이(Overlay) 및 스트리밍:** 레벨을 작게 쪼개거나, 배경음악이나 다음 레벨의 데이터를 현재 레벨이 진행되는 동안 미리 로드하는 '스트리밍' 기법을 적극 활용했습니다. 예를 들어, Final Fantasy 7 같은 게임은 배경 데이터가 바뀔 때마다 꽤 긴 로딩이 있었지만, 이는 당시 기술로는 최적의 방법이었습니다.

#### **유니티 적용: Addressables, Object Pooling, 그리고 메모리 프로파일링의 습관화**
유니티는 가비지 컬렉터(GC) 덕분에 개발자가 명시적으로 메모리를 관리할 필요가 줄었지만, 이는 동시에 메모리 누수나 비효율적인 자원 활용으로 이어지기 쉽습니다.

*   **Addressables 시스템:** 현대 유니티의 에셋 로딩 및 관리의 핵심입니다. 레트로 콘솔처럼 필요한 에셋만 정확히 로드하고, 사용하지 않을 때 명시적으로 언로드하는 체계를 제공합니다. 더 이상 `Resources.LoadAll` 같은 위험한 API에 의존하지 마세요!
*   **Object Pooling:** 특히 총알, 파티클, 적 AI 등 빈번하게 생성/소멸되는 오브젝트는 `Instantiate()`/`Destroy()` 비용이 매우 큽니다. 오브젝트 풀을 사용하여 오브젝트를 재활용하면 메모리 할당/해제 비용과 GC 발생 횟수를 크게 줄일 수 있습니다. 이는 레트로 개발자들이 오브젝트의 생명 주기를 직접 관리하던 방식과 일맥상통합니다.
*   **메모리 프로파일링 습관:** 유니티 프로파일러의 Memory 섹션은 당신의 든든한 친구입니다. 어느 에셋이 얼마나 많은 메모리를 차지하는지, 언제 GC가 발생하는지, 비활성화된 에셋이 여전히 메모리를 잡고 있지는 않은지 주기적으로 확인하는 습관을 들이세요.

```csharp
// 의사코드: 간단한 Object Pooling 구현
public class BulletPool : MonoBehaviour
{
    public GameObject bulletPrefab;
    public int initialPoolSize = 10;
    private Queue<GameObject> _pool = new Queue<GameObject>();

    void Awake()
    {
        for (int i = 0; i < initialPoolSize; i++)
        {
            GameObject bullet = Instantiate(bulletPrefab, transform);
            bullet.SetActive(false);
            _pool.Enqueue(bullet);
        }
    }

    public GameObject GetBullet()
    {
        if (_pool.Count == 0)
        {
            // 풀이 비었으면 새로 생성 (옵션: 풀 사이즈 확장)
            GameObject newBullet = Instantiate(bulletPrefab, transform);
            newBullet.SetActive(false); // 처음에는 비활성화 상태로
            return newBullet;
        }
        GameObject bulletToUse = _pool.Dequeue();
        bulletToUse.SetActive(true); // 활성화하여 사용 준비
        return bulletToUse;
    }

    public void ReturnBullet(GameObject bullet)
    {
        bullet.SetActive(false); // 비활성화
        // 필요에 따라 bullet의 상태(위치, 속도 등) 초기화
        _pool.Enqueue(bullet);
    }
}
```
**Opinion:** 메모리는 유한한 자원입니다. 레트로 개발자들이 가졌던 '메모리 프렌들리' 디자인 철학이 지금 우리에게 필요한 이유입니다. 그들은 바이트 하나, 텍스처 한 장의 해상도까지도 신중하게 결정했습니다. 우리도 그들처럼, "이 에셋이 정말 필요한가? 언제 로드하고 언제 언로드해야 할까?"라는 질문을 끊임없이 던져야 합니다.

### **문제 3: 과도한 연산량과 부하 분산의 실패**
**"게임이 특정 구간에서만 뚝뚝 끊겨요!"**

특정 복잡한 로직이나 물리 연산이 몰리는 곳에서 프레임이 급락하는 경험, 다들 있으시죠? 이는 CPU가 한 번에 너무 많은 작업을 처리하려 들 때 발생합니다.

#### **레트로 콘솔의 지혜: 고정 소수점 연산과 수동 최적화 루프**
NES의 6502 CPU는 부동 소수점(Floating Point) 연산을 지원하지 않았습니다. 모든 계산은 정수형으로 이루어졌고, 소수점은 개발자가 직접 관리하는 고정 소수점(Fixed Point) 방식으로 처리해야 했습니다. 이는 현대처럼 FPU(Floating Point Unit)가 없는 환경에서 연산 속도를 극대화하는 방법이었죠.

또한, 레트로 개발자들은 게임 루프를 극도로 최적화했습니다. 매 프레임 모든 것을 업데이트하는 대신, 오브젝트의 중요도나 화면상의 위치에 따라 업데이트 주기를 조절하거나, 특정 연산을 여러 프레임에 걸쳐 분산 처리하는 기법을 사용했습니다.

#### **유니티 적용: Jobs System, Burst Compiler, 그리고 Update 루프의 현명한 관리**
유니티는 우리에게 현대적인 멀티스레딩 및 최적화 도구를 제공합니다.

*   **Unity DOTS (Data-Oriented Tech Stack) - Jobs System & Burst Compiler:** 레트로 개발자들이 한정된 리소스로 최대한의 성능을 뽑아내기 위해 데이터를 촘촘하게 관리하고 최적화된 코드를 짰던 것처럼, Jobs System은 CPU 코어를 최대한 활용하여 연산을 병렬 처리하게 해주고, Burst Compiler는 C# 코드를 네이티브 코드로 변환하여 엄청난 속도 향상을 가져옵니다. 특히 데이터 중심 디자인(Data-Oriented Design)은 메모리 접근 패턴을 최적화하여 캐시 효율을 높이는 등, 레트로 개발자들이 데이터를 관리했던 방식의 현대적인 버전이라고 볼 수 있습니다.
*   **Update/FixedUpdate/LateUpdate의 현명한 사용:** 모든 로직을 `Update`에 몰아넣지 마세요. 물리 연산은 `FixedUpdate`에, UI 업데이트나 카메라 로직은 `LateUpdate`에 배치하는 등, 유니티의 라이프사이클을 이해하고 작업을 분산해야 합니다.
*   **커스텀 업데이트 스케줄러:** 모든 오브젝트가 매 프레임 업데이트될 필요는 없습니다. 먼 거리에 있는 적이나 배경 오브젝트는 5프레임에 한 번, 10프레임에 한 번만 업데이트되도록 커스텀 스케줄러를 구현할 수 있습니다. 이는 레트로 콘솔에서 리소스 제약 때문에 중요도에 따라 업데이트를 조절하던 방식과 동일합니다.

```csharp
// 의사코드: 간단한 커스텀 업데이트 스케줄러
public class CustomUpdater : MonoBehaviour
{
    public float updateInterval = 0.5f; // 0.5초마다 업데이트
    private float _timer;

    void Update()
    {
        _timer += Time.deltaTime;
        if (_timer >= updateInterval)
        {
            // 이 오브젝트의 핵심 로직을 여기에 넣기 (예: 멀리 있는 NPC AI)
            PerformExpensiveLogic();
            _timer -= updateInterval; // 타이머 리셋
        }
    }

    void PerformExpensiveLogic()
    {
        // ... 시간이 오래 걸리는 연산 (예: 경로 탐색, 복잡한 상태 업데이트)
        // 이 로직은 매 프레임 실행되지 않고, updateInterval 마다 한 번씩만 실행됨
    }
}
```
**Opinion:** 과거의 개발자들은 CPU 사이클 하나하나를 아끼기 위해 피나는 노력을 했습니다. 우리는 Jobs System과 Burst Compiler 같은 강력한 도구를 가졌지만, 그 본질은 여전히 '부하 분산'과 '효율적인 연산'에 있습니다. "모든 것을 매 프레임 처리할 필요는 없다"는 레트로 개발자들의 가르침을 명심해야 합니다.
