### **1. 문제 제기: 픽셀 시대의 한계와 핑 시대의 고통**

**[픽셀 시대의 문제: '그때 그 시절' 개발자들의 고뇌]**
1980년대 후반에서 90년대 중반, 닌텐도 개발자들은 지금과는 상상할 수 없을 정도로 열악한 환경에서 게임을 만들었습니다. CPU는 MHz 단위였고, RAM은 KB 단위였죠. 플로팅 포인트 유닛(FPU)은커녕 정수 연산조차 신중하게 다뤄야 했습니다. 하지만 그들은 놀랍도록 부드럽고, 재미있고, 결정론적인(Deterministic) 게임을 만들어냈습니다. 마리오가 절벽 위에서 점프하는 그 미묘한 타이밍, 링크의 칼날이 적에게 닿는 정교함은 모두 개발자들의 피와 땀으로 일궈낸 결과입니다.

**이 시대의 핵심 문제는 '제한된 자원 속에서 어떻게 안정적이고 일관된 60 FPS(또는 30 FPS) 게임 플레이를 구현할 것인가?'였습니다.**

**[핑 시대의 문제: 온라인 게임 개발자들의 숙명]**
그리고 시대는 흘러, 우리는 이제 수많은 플레이어가 동시에 접속하는 온라인 게임을 만듭니다. 하지만 여기서도 고통은 여전합니다. 바로 '네트워크 레이턴시'라는 녀우 때문이죠. 유저 간의 핑 차이는 심하면 수백 밀리초에 달하고, 이는 곧 '내가 총을 쐈는데 맞지 않았다'거나 '상대방이 순간이동한다' 같은 불쾌한 경험으로 이어집니다.

**이 시대의 핵심 문제는 '네트워크 지연 속에서 어떻게 모든 플레이어에게 공정하고 부드러운 게임 경험을 제공할 것인가?'입니다.**

두 시대의 문제는 다르지만, 그 해법을 찾아가는 과정에서 놀라운 공통점을 발견할 수 있습니다. 바로 '예측 가능한(Predictable)' 그리고 '효율적인(Efficient)' 시스템 구축이라는 목표입니다.

### **2. 레트로 닌텐도의 지혜: 자원 제약을 돌파하는 결정론적 설계**

레트로 닌텐도 개발자들이 자원 제약을 뚫고 안정적인 게임을 만든 핵심 비결 중 하나는 바로 **'결정론적 로직(Deterministic Logic)'**과 **'고정 소수점 연산(Fixed-Point Arithmetic)'**입니다.

**[문제: 느린 부동 소수점 연산과 비결정성]**
당시 CPU는 부동 소수점 연산을 처리하는 데 매우 취약했습니다. 속도 문제뿐 아니라, 서로 다른 컴파일러나 심지어 동일한 컴파일러의 최적화 설정에 따라 미묘하게 다른 부동 소수점 연산 결과가 나올 수도 있었습니다. 이는 게임의 물리나 위치 계산에 있어 비결정성을 야기할 수 있었죠.

**[해법: 고정 소수점 연산의 도입]**
개발자들은 이 문제를 해결하기 위해 고정 소수점 연산을 적극 활용했습니다. 실수(float) 대신 정수(integer)를 사용하되, 특정 비트(예: 하위 8비트)를 소수점 이하를 표현하는 데 할당하는 방식입니다. 예를 들어, `16.5`라는 숫자는 `16 * 256 + 0.5 * 256 = 4096 + 128 = 4224`와 같이 정수로 표현할 수 있습니다.

**핵심 로직 (의사 코드):**

```
// 고정 소수점 상수 (소수점 이하를 표현할 비트 수)
const int FRACTIONAL_BITS = 8; // 2^8 = 256, 즉 1은 256으로 표현
const int ONE_UNIT = 1 << FRACTIONAL_BITS; // 256

// 정수 -> 고정 소수점
fixed_t IntToFixed(int i) {
    return i * ONE_UNIT;
}

// 실수 -> 고정 소수점
fixed_t FloatToFixed(float f) {
    return (fixed_t)(f * ONE_UNIT);
}

// 고정 소수점 -> 정수
int FixedToInt(fixed_t f) {
    return f / ONE_UNIT;
}

// 고정 소수점 곱셈
// (A * B) / ONE_UNIT 와 동일하며, 비트 연산이 더 효율적
fixed_t FixedMul(fixed_t a, fixed_t b) {
    // 64비트 정수로 중간 결과를 계산하여 오버플로우 방지
    return (fixed_t)(((long long)a * b) >> FRACTIONAL_BITS);
}

// 고정 소수점 나눗셈
fixed_t FixedDiv(fixed_t a, fixed_t b) {
    // 64비트 정수로 중간 결과를 계산하여 정밀도 유지
    return (fixed_t)(((long long)a << FRACTIONAL_BITS) / b);
}

// 예시: 캐릭터 이동
fixed_t player_x_pos = FloatToFixed(10.5f);
fixed_t player_x_speed = FloatToFixed(0.1f);

// 매 프레임 업데이트
void UpdatePlayerPosition() {
    player_x_pos = player_x_pos + player_x_speed; // 덧셈은 일반 정수 덧셈과 동일
    // 화면에 그릴 때는 FixedToInt(player_x_pos) 사용
}
```
이러한 고정 소수점 연산은 **더 빠르고**, **항상 동일한 결과를 보장하는(결정론적인)** 물리 시뮬레이션을 가능하게 했습니다. 즉, 특정 시점에 동일한 입력을 주면 항상 동일한 다음 프레임 상태가 나오는 것이죠. 이는 레트로 게임이 프레임 단위로 완벽하게 예측 가능한 경험을 제공하는 데 필수적이었습니다.

### **3. 핑 시대의 해법: 레트로의 지혜로 롤백 넷코드 구현하기**

자, 이제 이 레트로의 지혜가 현대 네트워크 동기화, 특히 **롤백 넷코드(Rollback Netcode)**와 어떻게 연결되는지 살펴볼 시간입니다.

**[문제: 네트워크 레이턴시와 사용자 경험]**
기존의 딜레이 기반(Delay-based) 넷코드는 상대방의 입력이 도착할 때까지 기다렸다가 처리하므로 인풋 딜레이가 필연적으로 발생합니다. 이는 특히 격투 게임처럼 반응 속도가 중요한 장르에서 치명적이죠. 데드 레코닝(Dead Reckoning)은 예측으로 부드러움을 제공하지만, 예측이 틀렸을 때의 보정(Correction)이 눈에 띄게 되면 역시 불쾌감을 줍니다.

**[해법: 롤백 넷코드와 레트로 결정론의 만남]**
롤백 넷코드는 클라이언트가 자신의 입력을 즉시 처리하고 결과를 예측하여 보여줍니다. 이후 서버에서 상대방의 입력이 도착하면, 과거의 게임 상태로 '롤백'하여 모든 입력을 다시 시뮬레이션하고, 예측이 틀렸다면 그 결과를 반영하는 방식입니다.

**여기서 핵심은 바로 '결정론성(Determinism)'입니다!**
만약 게임 로직이 특정 입력에 대해 항상 동일한 다음 상태를 보장하지 못한다면, 롤백 후 재시뮬레이션 했을 때 클라이언트의 예측 결과와 서버의 검증 결과가 끊임없이 달라질 것입니다. 이는 엄청난 '개선점(Correction)'으로 이어져 게임 플레이가 심하게 끊기는 현상을 만듭니다.

**레트로 닌텐도 개발자들이 고정 소수점과 프레임 단위 로직으로 구현했던 바로 그 '결정론적인 게임 루프'가 롤백 넷코드의 핵심 기반이 됩니다.**

**롤백 넷코드의 핵심 로직 (아키텍처 설명):**

1.  **클라이언트 예측 (Client-Side Prediction):**
    *   플레이어 A는 자신의 입력을 받자마자 게임 상태를 업데이트하고 화면에 즉시 렌더링합니다. (인풋 딜레이 없음)
    *   동시에 이 입력을 서버로 보냅니다.
    *   플레이어 A는 다른 플레이어(B)의 입력이 아직 도착하지 않았으므로, B의 과거 상태를 기반으로 B의 다음 움직임을 예측합니다 (Dead Reckoning 기법 활용).

2.  **서버 검증 및 입력 동기화:**
    *   서버는 모든 플레이어의 입력을 받습니다.
    *   가장 오래된, 아직 처리되지 않은 모든 플레이어의 입력을 기준으로 게임 상태를 한 프레임씩 전진하며 시뮬레이션합니다. 이 결과가 '참(True)' 게임 상태가 됩니다.
    *   만약 모든 클라이언트의 입력이 동일한 프레임에 맞춰 도착하지 않는다면, 서버는 가장 느린 입력에 맞춰 기다리거나, 심지어 과거 입력을 요청하여 동기화 지점을 맞춥니다.

3.  **롤백 및 재시뮬레이션 (Rollback & Re-simulation):**
    *   서버로부터 다른 플레이어(B)의 '확정된' 과거 입력이 클라이언트 A에게 도착합니다.
    *   클라이언트 A는 자신이 과거에 예측했던 B의 상태가 실제 B의 입력과 달랐음을 인지합니다.
    *   클라이언트 A는 게임 상태를 해당 과거 프레임으로 '롤백'합니다 (과거 상태 저장).
    *   롤백된 시점부터 B의 '확정된' 입력을 포함하여 모든 플레이어의 입력을 다시 받아 게임 상태를 현재 프레임까지 '재시뮬레이션'합니다.
    *   재시뮬레이션 결과로 현재 프레임의 게임 상태가 업데이트되고, 이를 화면에 렌더링합니다. 만약 예측이 틀렸다면, 이 과정에서 순간적인 '고침(Correction)'이 발생하지만, 결정론적 로직 덕분에 그 폭이 최소화됩니다.

**의사 코드 (개념적 흐름):**

```
// 모든 게임 상태는 DeterministicState 객체에 저장
class DeterministicGameState {
    // ... 플레이어 위치, HP, 총알 위치 등 모든 게임 관련 변수
    // 이 변수들은 고정 소수점 등으로 처리되어야 함.

    // 해당 프레임의 입력을 적용하여 다음 상태로 전이 (결정론적!)
    void ApplyInputs(Input player1_input, Input player2_input) {
        // ... 물리 계산, 충돌 처리 등
        // 모든 계산은 DeterministicGameState 내부에서 이뤄지며 외부 요인에 의존하지 않음.
    }

    // 현재 상태를 저장 (롤백을 위해)
    DeterministicGameState SaveState() {
        // 깊은 복사
        return new DeterministicGameState(this);
    }

    // 저장된 상태를 복원
    void LoadState(DeterministicGameState saved_state) {
        // 깊은 복사
        // ...
    }
}

// 클라이언트 사이드
class ClientGameLoop {
    List<DeterministicGameState> game_state_history; // 과거 상태 저장
    InputQueue local_player_inputs; // 내 입력
    InputQueue remote_player_inputs_prediction; // 상대방 입력 (예측치)
    InputQueue remote_player_inputs_confirmed; // 상대방 입력 (서버 확정치)

    void Update() {
        // 1. 자신의 입력 처리 및 예측
        Input my_input = GetLocalInput();
        local_player_inputs.Add(current_frame, my_input);

        // 2. 현재 상태 저장 (롤백 대비)
        game_state_history[current_frame] = current_game_state.SaveState();

        // 3. 예측 기반으로 게임 상태 업데이트
        current_game_state.ApplyInputs(my_input, remote_player_inputs_prediction.Get(current_frame));
        Render(current_game_state);

        // 4. 서버로부터 확정된 입력 수신
        if (server_message.HasConfirmedInputs()) {
            int rollback_frame = server_message.LastConfirmedFrame;
            remote_player_inputs_confirmed.Merge(server_message.ConfirmedInputs); // 서버 확정 입력 반영

            // 5. 롤백!
            current_game_state.LoadState(game_state_history[rollback_frame]);

            // 6. 확정된 입력으로 재시뮬레이션
            for (int i = rollback_frame; i <= current_frame; i++) {
                Input player1_input = local_player_inputs.Get(i);
                Input player2_input = remote_player_inputs_confirmed.Get(i);
                current_game_state.ApplyInputs(player1_input, player2_input);
            }
            Render(current_game_state); // 최종 보정된 상태 렌더링
        }
    }
}
```

이 모든 과정이 부드럽게 작동하기 위해서는 `DeterministicGameState.ApplyInputs()` 메서드가 어떤 상황에서도 **항상 동일한 결과를 반환**해야 합니다. 레트로 닌텐도 개발자들이 고정 소수점 연산, 예측 가능한 물리 법칙, 그리고 엄격한 프레임 단위 로직으로 이를 달성했습니다. 그들의 기술은 수십 년이 지난 지금, 네트워크 지연을 극복하는 강력한 도구로 재탄생한 것이죠.

### **4. 데드 레코닝(Dead Reckoning)과 효율적인 상태 관리**

롤백 넷코드와 더불어, **데드 레코닝(Dead Reckoning)** 또한 레트로 게임의 효율적인 상태 관리 기법에서 영감을 얻을 수 있습니다. 데드 레코닝은 클라이언트가 서버로부터 받은 마지막 상태를 기반으로 엔티티의 다음 움직임을 예측하고, 서버는 주기적으로 실제 상태를 보내 예측 오류를 보정하는 방식입니다.

**[문제: 네트워크 대역폭 제한]**
모든 엔티티의 모든 상태를 매 프레임마다 보내는 것은 불가능에 가깝습니다. 특히 모바일 환경이나 대규모 멀티플레이어 게임에서는 네트워크 대역폭이 큰 제약이 됩니다.

**[해법: 최소한의 정보로 최대의 예측]**
레트로 게임은 제한된 메모리에서 캐릭터의 위치, 속도, 상태 등을 최소한의 비트로 표현하는 데 탁월했습니다. 예를 들어, `x, y, z` 좌표를 float 대신 16비트나 32비트 고정 소수점으로, 속도나 가속도 역시 마찬가지로 표현했습니다.

데드 레코닝에서도 이는 핵심입니다. 서버가 클라이언트에게 보내는 보정 데이터는 최소한의 정보여야 합니다. `(x, y, z)` 위치, `(vx, vy, vz)` 속도, `(ax, ay, az)` 가속도 등, 예측에 필요한 핵심 데이터만을 보내 예측 모델을 업데이트합니다. 레트로 게임에서 체득한 **'필요한 최소 정보로 상태를 완벽하게 재구성하는 능력'**이 데드 레코닝의 효율을 극대화하는 데 기여합니다.

**데드 레코닝 핵심 (클라이언트 측):**

```
class PredictedEntity {
    Vector3 last_known_pos;
    Vector3 last_known_vel;
    Vector3 last_known_accel; // 마지막으로 서버에서 받은 확정 상태

    float prediction_time_elapsed = 0.0f; // 예측 시작 이후 경과 시간

    void OnServerUpdate(ServerEntityState state) {
        // 서버에서 받은 확정 상태로 업데이트
        last_known_pos = state.pos;
        last_known_vel = state.vel;
        last_known_accel = state.accel;
        prediction_time_elapsed = 0.0f; // 예측 리셋
        // 예측 오류가 있다면 보간(interpolation)하여 부드럽게 맞춤
    }

    Vector3 GetPredictedPosition(float delta_time) {
        // Simple Euler Integration for prediction
        Vector3 current_vel = last_known_vel + last_known_accel * prediction_time_elapsed;
        Vector3 predicted_pos = last_known_pos + current_vel * prediction_time_elapsed + 0.5f * last_known_accel * prediction_time_elapsed * prediction_time_elapsed;
        return predicted_pos;
    }

    void Update(float delta_time) {
        prediction_time_elapsed += delta_time;
        // 클라이언트는 예측된 위치를 기반으로 렌더링
        current_display_pos = GetPredictedPosition(prediction_time_elapsed);
    }
}
```
여기에 중요한 것은 서버가 보내는 상태 정보가 바로 이 예측 모델을 업데이트하기에 최적화되어 있어야 한다는 점입니다. 레트로 게임처럼 한정된 비트로 최대한의 정보를 담아내려는 노력이 여기서 빛을 발합니다.

### **5. 개발자의 견해: '결정론'이라는 변치 않는 가치**

저는 이 두 시대의 기술적 접점을 보면서 게임 개발에서 '결정론(Determinism)'이라는 가치가 얼마나 중요한지 다시 한번 깨닫습니다. 레트로 닌텐도 개발자들은 하드웨어의 한계 때문에 어쩔 수 없이 결정론적 시스템을 설계했지만, 그 결과는 시대를 초월하는 견고함으로 이어졌습니다. 그리고 오늘날, 우리는 네트워크 레이턴시라는 또 다른 한계를 극복하기 위해 다시금 그 결정론이라는 원칙을 붙잡고 있습니다.

새로운 기술을 배우고 적용하는 것도 중요하지만, 때로는 옛 기술에서 그 근본 원리를 찾아보는 것이 훨씬 깊은 인사이트를 줄 때가 많습니다. '고정 소수점 연산'이나 '정수 기반 게임 루프' 같은 레트로 기법들은 단순히 오래된 방식이 아니라, 극한의 효율성과 예측 가능성을 추구하는 엔지니어링의 정수입니다.

여러분들이 롤백 넷코드나 데드 레코닝을 구현할 때 "내 게임 로직이 과연 결정론적인가?"라는 질문을 항상 던져보세요. 부동 소수점 연산의 미묘한 차이, 외부 라이브러리의 비결정적 동작, 랜덤 함수의 시드 관리 등, 사소해 보이는 부분들이 복잡한 네트워크 동기화 문제의 원인이 될 수 있습니다. 닌텐도 선배 개발자들의 지혜를 빌려, 굳건한 기초 위에서 예측 불가능한 네트워크 환경을 극복하는 멋진 게임을 만들어 나가시길 바랍니다.
