### 📜 문제 제기: AI의 뇌, 네트워크 지연 속에서 혼란에 빠지다

우리는 AI의 복잡한 의사결정 과정을 설계하기 위해 Behavior Trees(BTs)를 즐겨 사용합니다. 직관적인 시각화, 모듈화된 태스크, 그리고 계층적인 구조는 복잡한 AI 패턴을 구현하는 데 큰 도움을 주죠. 예를 들어, 적 AI가 '플레이어 추적 -> 엄폐 -> 총격 -> 재장전'과 같은 일련의 행동을 상황에 맞춰 수행하도록 만들 수 있습니다.

**문제는 멀티플레이어 환경에서 발생합니다.** 서버는 게임의 '진실'을 담당하고, 모든 클라이언트는 이 진실을 최대한 매끄럽게 보여줘야 합니다. 하지만 네트워크 지연(Latency)은 필연적이며, 이 지연 때문에 클라이언트의 게임 상태는 항상 서버보다 조금 뒤처지거나 예측에 의존하게 됩니다.

만약 AI의 Behavior Tree가 서버에서만 실행된다면, 그 결과물(AI의 이동, 공격)은 클라이언트로 전송될 때까지 지연됩니다. 클라이언트가 AI의 움직임을 그저 기다린다면 AI는 '순간이동'하거나 '뚝뚝 끊기는' 움직임을 보일 테고, 이는 플레이 경험을 심각하게 해칩니다.

그렇다고 클라이언트에서 AI BT를 완전히 실행하도록 하자니, 서버와의 상태 불일치(Desync) 문제가 불거집니다. AI가 클라이언트 A에서는 왼쪽으로 움직이기로 결정했는데, 클라이언트 B에서는 오른쪽으로 움직이기로 결정하는 '끔찍한 혼란'이 벌어질 수 있죠. 이는 공정성 문제뿐만 아니라 치명적인 버그의 온상이 됩니다.

여기서 우리가 사랑해 마지않는 **Rollback과 Dead Reckoning**이 등장하지만, 단순히 플레이어의 움직임에만 적용되는 게 아닙니다. **AI의 '뇌', 즉 Behavior Tree의 내부 상태까지도 동기화의 도마 위에 올려야 하는 상황**이 생기는 거죠.

### 💡 솔루션: AI Behavior Tree, 시공간을 초월하다 (Server-Authoritative AI with Rollback-Compatible BTs)

결론부터 말씀드리자면, 핵심은 **AI의 Behavior Tree 실행 자체를 서버에서 권위적으로(Authoritative) 관리하되, 이 AI의 내부 상태마저 Rollback 친화적으로 설계**하는 것입니다. 그리고 클라이언트는 이 서버 AI의 결과물을 **Dead Reckoning**을 이용해 예측하고 부드럽게 보간(Interpolate)하는 역할을 맡습니다.

#### 1. Deterministic AI Behavior Trees: 시간 여행을 위한 설계

Rollback 시스템은 과거의 특정 시점으로 게임 상태를 되돌린 후, 그 시점부터 다시 시뮬레이션을 진행합니다. 이때 AI의 Behavior Tree도 정확히 그 시점의 상태로 돌아가서 다시 '생각'하고 '행동'해야 합니다. 이를 위해서는 AI의 BT가 **완전한 결정론적(Deterministic)이어야 합니다.**

*   **비결정론적 요소 제거:** `rand()` 같은 함수는 절대 사용하면 안 됩니다. 만약 무작위성이 필요하다면, 게임 상태에 포함된 '시드(Seed)' 값을 이용한 결정론적 난수 생성기를 사용하고, 이 시드 또한 Rollback 대상에 포함시켜야 합니다.
*   **외부 입력:** AI의 의사결정에 영향을 미치는 모든 외부 입력(플레이어 위치, HP, 주변 오브젝트 등)은 반드시 서버의 권위적인 게임 상태에서 가져와야 합니다.
*   **내부 상태 관리:** BT가 현재 어떤 노드를 실행 중인지, 태스크 내의 타이머나 카운터 값, 그리고 AI의 '블랙보드(Blackboard)'에 저장된 모든 데이터는 Rollback을 위한 스냅샷 대상이 되어야 합니다.

**의사코드 (Pseudo-code) 예시:**

```cpp
// AI Behavior Tree의 상태를 저장하기 위한 구조체
struct AIBehaviorState
{
    // 현재 활성화된 BT 노드의 고유 ID 또는 경로
    NodeID activeNodeID; 
    // 블랙보드 (AI의 내부 변수들)
    map<string, Variant> blackboard;
    // 태스크 내부의 특정 상태 (예: Timer, Counter, Sub-state)
    map<string, int> taskInternalState; 
    // AI의 특정 '행동'이 시작된 타임스탬프 (Rollback 시 재조정 필요)
    GameTime currentActionStartTime;
    // 결정론적 난수 생성을 위한 시드
    uint32 randomSeed;
};

class MyGameAI
{
public:
    BehaviorTree bt;
    AIBehaviorState currentState;

    void Tick(GameWorldState& worldState, float deltaTime, uint32 currentTick)
    {
        // 서버의 현재 월드 상태와 이전 AI 상태를 기반으로 BT 실행
        bt.Execute(worldState, currentState, deltaTime, currentTick);
        
        // BT 실행 후 AI의 상태가 변경될 수 있음 (예: 새로운 노드 진입, 블랙보드 변경)
    }

    // Rollback을 위한 현재 AI 상태 저장
    AIBehaviorState SaveState()
    {
        return currentState; // 내부 상태 전체를 복사
    }

    // Rollback 후 특정 시점의 AI 상태로 복원
    void LoadState(const AIBehaviorState& stateToLoad)
    {
        currentState = stateToLoad; // 저장된 상태로 복원
        // 필요하다면, BT 내부의 '현재 실행 중인 노드' 등도 재설정
        bt.ResetToNode(currentState.activeNodeID);
    }
};

// 서버의 틱 루프에서 AI 처리 로직 (간략화)
void ServerGameLoop::ProcessTick(uint32 tickNumber)
{
    // ... 플레이어 입력 처리, 물리 업데이트 등 ...

    // Rollback 시스템의 핵심: 특정 틱의 상태를 저장
    GameWorldSnapshot snapshot = SaveGameWorldState(tickNumber);
    
    // 이 스냅샷에는 모든 AI의 AIBehaviorState도 포함되어야 함
    for (auto& ai : activeAIList)
    {
        snapshot.aiStates[ai.id] = ai.SaveState();
    }
    gameHistory.AddSnapshot(tickNumber, snapshot);

    // AI 업데이트
    for (auto& ai : activeAIList)
    {
        // AI는 서버의 권위적인 세계 상태를 기반으로 행동 결정
        ai.Tick(currentGameWorldState, FixedDeltaTime, tickNumber); 
        // AI의 결정에 따라 월드 상태 변경 (예: AI 이동, 공격)
    }

    // ... 네트워크 패킷 전송 등 ...
}

// Rollback 발생 시
void ServerGameLoop::PerformRollback(uint32 targetTick)
{
    // 과거 스냅샷 로드
    GameWorldSnapshot snapshot = gameHistory.GetSnapshot(targetTick);
    LoadGameWorldState(snapshot); // 월드 상태 복원

    for (auto& ai : activeAIList)
    {
        // 각 AI도 해당 틱의 상태로 복원
        ai.LoadState(snapshot.aiStates[ai.id]);
    }

    // targetTick부터 현재까지 다시 시뮬레이션
    for (uint32 t = targetTick; t < currentTick; ++t)
    {
        // SaveState, LoadState 없이 순수하게 Tick만 반복 (미래 예측)
        for (auto& ai : activeAIList)
        {
            ai.Tick(currentGameWorldState, FixedDeltaTime, t);
        }
        // 다른 게임 로직도 마찬가지로 재시뮬레이션
        // ...
    }
}
```

이러한 접근 방식은 AI가 과거로 돌아가더라도, 항상 동일한 입력에 대해 동일한 결정을 내릴 수 있도록 보장합니다.

#### 2. Dead Reckoning: 클라이언트의 시각적 마법

AI의 뇌가 서버에서 완벽하게 Rollback-ready 상태로 돌아간다고 해도, 그 결과물이 클라이언트로 전송되는 데는 여전히 지연이 있습니다. 이때 Dead Reckoning이 클라이언트 측에서 AI의 움직임을 부드럽게 보이도록 돕습니다.

*   **서버 -> 클라이언트 전송:** 서버는 AI의 현재 위치, 속도, 그리고 '의도하는 목적지'나 '현재 수행 중인 행동' 등 AI의 핵심적인 동기화 데이터를 클라이언트에 주기적으로 전송합니다.
*   **클라이언트 예측:** 클라이언트는 이 데이터를 받으면, AI가 다음 서버 업데이트가 올 때까지 어떻게 움직일지 예측합니다. 예를 들어, 서버가 'AI가 플레이어 X를 향해 시속 5km로 이동 중'이라고 알려주면, 클라이언트는 이 정보를 바탕으로 AI 모델을 부드럽게 움직이는 거죠.
*   **보정(Correction):** 다음 서버 업데이트가 도착하면, 클라이언트는 자신이 예측했던 AI의 위치와 서버가 알려준 실제 위치를 비교합니다. 만약 차이가 있다면, AI를 약간 보정하여 부드럽게 원래 위치로 '끌어당기거나' '밀어내는' 방식으로 위화감 없이 위치를 조정합니다.

Dead Reckoning은 AI의 '결정' 자체를 예측하는 것이 아니라, 서버가 결정한 '결과'인 움직임을 예측하고 보간하는 데 사용됩니다. AI의 BT가 아무리 복잡하게 움직임을 계산했더라도, 클라이언트는 그 최종적인 목적지와 속도만 받아서 부드럽게 보여주는 겁니다.

### 🎮 시니어 개발자의 썰: "AI는 절대 거짓말을 하지 않는다"

한번은 말이죠, 꽤나 복잡한 PvPvE 게임에서 AI 몬스터가 서버에서는 플레이어를 정확히 추격해서 공격하는데, 특정 클라이언트에서만 몬스터가 엉뚱한 벽에 박고 있거나 아예 움직이지 않는 기이한 버그를 잡느라 밤을 꼴딱 새운 적이 있습니다. 디버깅 도구를 붙여보니, 서버와 클라이언트 모두 AI의 `Tick` 함수는 잘 호출되는데, AI의 `Blackboard` 상태(특히 `TargetPlayerID`나 `LastKnownPlayerPosition`)가 서로 달랐던 거죠.

결국 원인은 비결정론적인 난수 생성과, AI의 특정 태스크 내부 타이머가 게임 틱(Tick)과 동기화되지 않고 클라이언트의 시스템 시간에 의존했던 부분이었습니다. 서버에서 Rollback을 수행할 때 AI의 상태도 같이 되돌려지는데, 이 타이머나 난수가 과거의 시점으로 정확히 돌아가지 않았던 거죠. AI가 과거로 '시간 여행'을 했는데, 뇌 속의 몇몇 기억만 그대로 남아버린 격이었습니다.

이 경험을 통해 저는 **"AI는 네트워크 게임에서 절대 거짓말을 하지 않아야 한다"**는 철학을 갖게 되었습니다. AI의 모든 의사결정 과정, 내부 상태는 서버의 권위적인 시공간 안에서만 존재해야 하며, Rollback이 발생했을 때도 완벽하게 과거를 재현하고 미래를 예측할 수 있어야 합니다. 클라이언트는 그저 AI의 '그림자'를 부드럽게 보여주는 역할에 충실할 뿐입니다.

### 🚀 아키텍처 다이어그램 (개념적)

```mermaid
graph TD
    subgraph Server (Authoritative)
        S_GT[Game Tick Loop] --> S_SH[State Snapshot & History]
        S_SH --> S_RB[Rollback / Re-simulation]
        S_RB -- Re-evaluates --> S_AI[AI Behavior Trees]
        S_AI -- AI State & Actions --> S_GS[Game World State]
        S_GS -- Game State Update --> S_NET[Network Sender]
    end

    subgraph Client (Predictive)
        C_NET[Network Receiver] --> C_DR[Dead Reckoning & Prediction]
        C_DR -- Smooths & Predicts --> C_AI_VIS[AI Visuals & Animation]
        C_DR -- Corrects --> C_GS[Client Game State]
    end

    S_NET -- AI State (Pos, Vel, Target, Action) --> C_NET
    S_AI -- AI Behavior State (for Rollback) --> S_SH
    S_SH -- Loads Previous State --> S_RB
    C_DR -- Occasionally Corrected By --> S_NET
```

**설명:**
*   **서버:** `Game Tick Loop`는 매 틱마다 `Game World State`를 업데이트하고, 이 상태와 모든 `AI Behavior Trees`의 내부 상태를 포함한 `State Snapshot`을 `History`에 저장합니다. `Rollback`이 필요하면, `History`에서 과거 스냅샷을 로드하여 `AI Behavior Trees`를 포함한 모든 게임 로직을 `Re-simulation`합니다. `Network Sender`는 AI의 현재 위치, 속도, 목표 등 핵심 동기화 정보를 클라이언트에 전송합니다.
*   **클라이언트:** `Network Receiver`를 통해 서버의 AI 동기화 데이터를 받습니다. `Dead Reckoning` 모듈은 이 데이터를 기반으로 AI의 `Visuals & Animation`을 예측하여 부드럽게 표현합니다. 서버로부터 새로운 데이터가 오면, 예측된 클라이언트 `Game State`를 서버 `Game State`에 맞춰 `Correction`합니다.

### 📝 결론: 보이지 않는 곳에서 빛나는 AI의 지능

AI Behavior Trees와 서버-클라이언트 동기화, 특히 Rollback과 Dead Reckoning의 결합은 단순히 기술적인 조합을 넘어, 멀티플레이어 게임에서 AI가 '살아 숨 쉬는 존재'로 느껴지게 하는 핵심적인 비결입니다. 서버는 AI의 '뇌'를 권위적으로 관리하고, 필요에 따라 과거로 시간 여행을 시켜 재사고(Re-thinking)하도록 만듭니다. 클라이언트는 이 서버 AI의 '신념'에 따라 움직이는 AI의 '그림자'를 부드럽게 보여주는 역할을 하는 거죠.

물론 이 과정은 쉽지 않습니다. Deterministic AI를 만드는 것은 생각보다 훨씬 많은 제약을 요구하며, Rollback 시스템에 AI의 복잡한 상태를 통합하는 것은 수많은 버그와의 싸움을 의미합니다. 하지만 그 모든 고난을 이겨냈을 때, 우리는 플레이어들이 "와, 저 AI 정말 똑똑하다!"라고 감탄하는 동시에, 그 어떤 네트워크 지연 속에서도 AI의 움직임이 '자연스럽다'고 느끼는 경이로운 경험을 선사할 수 있게 될 겁니다.

여러분들의 다음 프로젝트에서, 이 두 가지 키워드의 결합이 AI에게 진정한 생명력을 불어넣는 데 영감을 주었으면 좋겠습니다. 질문이나 여러분의 경험담이 있다면 언제든지 댓글로 남겨주세요!
