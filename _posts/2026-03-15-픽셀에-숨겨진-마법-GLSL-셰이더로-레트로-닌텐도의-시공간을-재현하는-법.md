### ✨ 사례 1: SNES 모드 7 (Mode 7) - 2D에서 3D 감각을!

**[문제]**: 1990년대 초반, SNES는 2D 콘솔이었습니다. 하지만 'F-Zero'나 'Super Mario Kart' 같은 게임들은 마치 3D 공간을 달리는 듯한 엄청난 속도감과 입체감을 선사했습니다. 어떻게 2D 화면에서 배경을 확대, 축소, 회전시켜 원근감을 만들어낼 수 있었을까요?

**[레트로 해결책]**: SNES의 PPU(Picture Processing Unit)에는 'Mode 7'이라는 특별한 배경 모드가 있었습니다. 이 모드는 하나의 배경 레이어에 대해 하드웨어적으로 아핀 변환(Affine Transformation)을 적용할 수 있었죠. 스케일, 회전, 이동을 위한 파라미터를 CPU에서 PPU 레지스터에 기록하면, PPU가 매 프레임마다 배경 타일을 화면에 그릴 때 이 변환 행렬을 사용하여 타일의 픽셀을 변형시켰습니다. 셰이더는 없었지만, 그 핵심에는 행렬 기반의 수학적 변환이 있었습니다.

**[GLSL로 재현하기]**: 우리는 이 Mode 7의 마법을 GLSL 프래그먼트 셰이더에서 정확히 구현할 수 있습니다. 셰이더의 목표는 화면의 각 픽셀(프래그먼트)에 대해 '이 픽셀은 배경 텍스처의 어느 위치에서 가져와야 하는가?'를 계산하는 것입니다.

```glsl
// Vertex Shader (간단하게 UV 전달)
in vec2 a_position;
in vec2 a_texCoord;
out vec2 v_texCoord;

void main() {
    gl_Position = vec4(a_position, 0.0, 1.0);
    v_texCoord = a_texCoord;
}

// Fragment Shader: SNES Mode 7 에뮬레이션
in vec2 v_texCoord;
uniform sampler2D u_backgroundTexture;
uniform vec2 u_resolution; // 화면 해상도
uniform vec2 u_scrollPos;  // 배경 스크롤 위치 (x, y)
uniform float u_rotation;  // 회전 각도 (라디안)
uniform float u_scale;     // 확대/축소 비율 (1.0 = 원본)
uniform vec2 u_perspectiveOffset; // 원근감을 위한 Y축 오프셋 보정

void main() {
    // 1. 프래그먼트 좌표를 0~1 범위로 정규화 (화면 중심 기준)
    vec2 screenCoord = v_texCoord * u_resolution;
    vec2 centeredCoord = (screenCoord - u_resolution * 0.5) / u_scale;

    // 2. 회전 적용
    float cosRot = cos(u_rotation);
    float sinRot = sin(u_rotation);
    vec2 rotatedCoord = vec2(
        centeredCoord.x * cosRot - centeredCoord.y * sinRot,
        centeredCoord.x * sinRot + centeredCoord.y * cosRot
    );

    // 3. (선택적) 원근감 효과: 화면 Y축 하단으로 갈수록 더 멀리 있는 것처럼 스케일/오프셋 조절
    // 이는 실제 Mode 7의 기능이라기보다는, 2D에서 3D 느낌을 더 강화하기 위한 추가적인 트릭입니다.
    // F-Zero나 마리오 카트의 지면이 멀어지는 느낌을 줄 수 있습니다.
    float yNormalized = screenCoord.y / u_resolution.y; // 0 (상단) ~ 1 (하단)
    float perspectiveFactor = mix(1.0, 0.5, yNormalized * u_perspectiveOffset.y); // Y가 커질수록 스케일 작아짐
    rotatedCoord *= perspectiveFactor;

    // 4. 스크롤 및 최종 텍스처 좌표 계산
    vec2 finalTexCoord = (rotatedCoord + u_scrollPos) / textureSize(u_backgroundTexture, 0);

    // 5. 텍스처 샘플링 및 출력
    // GL_REPEAT 텍스처 래핑 모드를 사용하면 배경이 자연스럽게 반복됩니다.
    gl_FragColor = texture(u_backgroundTexture, finalTexCoord);
}
```
**[제 경험담]**: 처음 Mode 7을 이해했을 때의 충격은 아직도 생생합니다. '와, 고작 저 해상도에 이런 수학적 변환만으로 저런 효과를 냈다니!' 싶었죠. 당시 개발자들은 리소스의 한계를 뛰어넘기 위해 각고의 노력을 했을 겁니다. 그리고 우리가 지금 GLSL로 이를 재현할 때, 단순히 코드를 베끼는 것을 넘어 그 '제약 속의 창의성'을 배우는 것이죠. 저 코드를 보면, 고작 몇 줄의 행렬 연산과 변수 조작만으로 평면 이미지가 3D 공간을 유영하는 듯한 환상을 만들 수 있다는 게 얼마나 경이로운가요!

### ⚙️ 사례 2: N64의 독특한 '폴리곤 룩'과 색상 양자화

**[문제]**: 닌텐도 64는 본격적인 3D 게임 시대를 열었지만, 당시의 하드웨어는 오늘날의 그래픽 카드와는 비교할 수 없을 정도로 제한적이었습니다. 특히 '지글거리는' 텍스처, 각진 폴리곤, 그리고 특정 색상의 '뭉개짐' 같은 독특한 시각적 특징이 있었습니다. 이를 현대 GLSL 셰이더로 의도적으로 재현하여 그 시절의 향수를 불러일으키고 싶다면 어떻게 해야 할까요?

**[레트로 해결책]**: N64는 부동 소수점 연산보다는 정수 연산을 선호했습니다. 이 때문에 정점 좌표나 텍스처 좌표가 정수 그리드에 '스냅'되는 듯한 효과를 보이기도 했죠. 또한, 색상 처리에서도 제한된 비트 깊이(예: 16비트 컬러)로 인해 색상이 양자화되거나 디더링되는 현상이 있었습니다. 깊이 버퍼의 정밀도도 낮아 원거리 오브젝트에서 'Z-fighting' 현상이 자주 발생하기도 했습니다. 이 모든 것이 의도된 것은 아니었지만, N64만의 독특한 그래픽 스타일을 만들어냈습니다.

**[GLSL로 재현하기]**: N64의 '폴리곤 룩'과 색상 양자화는 주로 버텍스 셰이더와 프래그먼트 셰이더에서 좌표와 색상을 '양자화'하는 방식으로 재현할 수 있습니다.

```glsl
// Vertex Shader: N64 스타일 정점 스냅핑
uniform mat4 u_mvpMatrix; // Model-View-Projection 행렬
uniform float u_quantizeFactor; // 정점 스냅핑 강도 (예: 100.0)
in vec3 a_position;
in vec3 a_normal;
in vec2 a_texCoord;

out vec3 v_normal;
out vec2 v_texCoord;
out vec3 v_worldPos; // 광원 계산을 위해 월드 포지션 전달

void main() {
    // 1. 정점 위치 양자화 (N64의 '그리드 스냅핑' 느낌)
    vec3 quantizedPos = floor(a_position * u_quantizeFactor) / u_quantizeFactor;

    gl_Position = u_mvpMatrix * vec4(quantizedPos, 1.0);
    v_normal = a_normal;
    v_texCoord = a_texCoord;
    v_worldPos = (u_modelMatrix * vec4(quantizedPos, 1.0)).xyz; // 월드 포지션도 양자화된 값으로
}


// Fragment Shader: N64 스타일 색상 양자화 및 간단한 조명
uniform sampler2D u_texture;
uniform vec3 u_lightDir; // 광원 방향 (정규화됨)
uniform vec3 u_lightColor;
uniform vec3 u_ambientColor;
uniform float u_numColorShades; // 색상 양자화 단계 수 (예: 8.0)

in vec3 v_normal;
in vec2 v_texCoord;
in vec3 v_worldPos;

void main() {
    // 1. 텍스처 샘플링
    vec4 texColor = texture(u_texture, v_texCoord);

    // 2. 간단한 Diffuse 조명 (N64는 주로 Gouraud 셰이딩을 썼지만, 여기서는 Per-pixel로 간단히)
    vec3 normal = normalize(v_normal);
    float diffuse = max(dot(normal, u_lightDir), 0.0);
    vec3 lighting = u_ambientColor + u_lightColor * diffuse;

    vec4 finalColor = vec4(texColor.rgb * lighting, texColor.a);

    // 3. 색상 양자화 (N64의 제한된 컬러 팔레트 느낌)
    finalColor.rgb = floor(finalColor.rgb * u_numColorShades) / u_numColorShades;

    gl_FragColor = finalColor;
}
```
**[제 경험담]**: N64는 텍스처 필터링(Bilinear)을 지원했지만, 해상도와 정점 정밀도의 한계 때문에 멀리 있는 텍스처는 종종 '뭉개지거나' '지글거리는' 느낌을 주었습니다. 저는 개인적으로 이런 불완전함이 N64만의 매력을 만든다고 생각합니다. 최신 그래픽에서 완벽한 리마스터도 좋지만, 때로는 의도적인 '결점'을 통해 그 시절의 감성을 자극하는 것이 더 강력할 때가 있죠. 위 셰이더는 버텍스 좌표를 양자화하여 각진 폴리곤 느낌을 강화하고, 최종 색상을 양자화하여 제한된 색상 팔레트의 느낌을 주는 가장 기본적인 방법입니다. N64의 독특한 안개(Fog) 효과나 'jittering'도 재현해볼 수 있지만, 이는 또 다른 깊은 이야기가 될 겁니다.

### 💡 왜 과거의 기술에서 현대의 영감을 찾아야 하는가?

어쩌면 여러분은 '왜 과거의 한계 기술을 GLSL로 굳이 재현해야 하는가?'라는 질문을 던질 수도 있습니다. 저의 대답은 명확합니다.

1.  **근본 원리 이해**: 레트로 기술은 최소한의 리소스로 최대한의 효과를 내려는 처절한 노력이 담겨 있습니다. 이는 곧 렌더링 파이프라인의 근본적인 수학적, 알고리즘적 원리를 이해하는 데 큰 도움이 됩니다. 이 깊은 이해는 우리가 새로운, 더 복잡한 효과를 만들 때도 견고한 기반이 됩니다.
2.  **최적화와 효율성**: 현대 GPU는 강력하지만, 여전히 성능은 중요합니다. 레트로 개발자들의 자원 절약 정신은 우리에게 효율적인 알고리즘 디자인과 최적화된 셰이더 작성법에 대한 귀중한 교훈을 줍니다.
3.  **창의적 제약**: 무한한 가능성은 때로 우리를 혼란스럽게 합니다. 과거의 제약을 의도적으로 도입하는 것은 새로운 시각과 창의적인 문제 해결 능력을 키우는 데 도움이 됩니다. '어떻게 하면 이 제약 안에서 멋진 것을 만들 수 있을까?'라는 질문은 강력한 동기가 됩니다.
4.  **역사적 통찰**: 게임 개발의 역사를 이해하는 것은 단순히 지식을 쌓는 것을 넘어, 우리가 현재 서 있는 위치를 깨닫고 미래의 방향을 가늠하는 데 중요한 통찰력을 제공합니다.
