### 문제: '풍요 속의 빈곤' – 현대 게임 개발의 역설

현대 게임 개발은 그야말로 '풍요'의 시대입니다. 기가바이트급 메모리, 테라플롭스 연산력을 가진 GPU, 수십 개의 코어를 가진 CPU… 상상할 수 있는 모든 것을 구현할 수 있는 듯 보이죠. 하지만 역설적이게도 우리는 여전히 성능 문제에 시달립니다. 프레임 드랍, 긴 로딩 시간, 버벅이는 UI, 모바일 기기에서의 발열 문제… 결국 이 모든 것은 '리소스 관리'의 실패로 귀결됩니다. 너무 많은 것을 한꺼번에, 비효율적으로 사용하려는 시도죠.

이 문제를 해결하기 위해, 저는 때때로 시간 여행을 떠납니다. 1980년대와 90년대, 극심한 제약 속에서 게임을 만들던 개발자들의 세상을요. 그들은 지금의 우리보다 훨씬 더 혹독한 '풍요 속의 빈곤'에 시달렸습니다. 그리고 그들의 기상천외한 해결책 속에서, 현대 유니티 최적화의 실마리를 발견하곤 합니다.

### 레트로 콘솔의 지혜: '제약은 창조의 어머니'

NES, SNES, 그리고 PS1은 각기 다른 시대의 아이콘이자, 각기 다른 종류의 '제약'을 대표합니다. 그리고 이 제약들은 개발자들에게 놀라운 창의력을 요구했죠.

#### 1. NES/SNES: 픽셀과 타일, 그리고 극도의 효율성

*   **문제:** VRAM (Video RAM)은 고작 수 킬로바이트, 스프라이트(Sprite) 동시 출력 개수 제한, 팔레트(Palette) 색상 제한. 이 극악무도한 환경에서 어떻게 풍성한 배경과 수많은 캐릭터를 표현할 수 있었을까요?
*   **해결책의 정수:**
    *   **타일 기반 렌더링 (Tile-based Rendering):** NES의 배경은 '타일'이라는 8x8 픽셀의 작은 조각들로 이루어져 있었습니다. 같은 타일은 메모리에 한 번만 저장되고, 화면에 여러 번 재활용되었죠. 이는 '데이터 중복 제거'와 '재활용'의 극한을 보여줍니다.
    *   **스프라이트 아틀라스 (Sprite Atlas)의 원조:** 모든 캐릭터 애니메이션 조각, UI 아이콘들은 하나의 큰 '스프라이트 시트'에 모여 있었고, 필요한 부분만 잘라내어 사용했습니다.
    *   **PPU (Picture Processing Unit)와 CPU의 협업:** PPU는 오직 화면을 그리는 일에만 집중하고, CPU는 게임 로직과 PPU에 데이터를 넘겨주는 역할만 수행했습니다. 각각의 역할 분담이 명확했죠.

#### 2. PlayStation 1: 3D의 태동, 그리고 '속임수'의 미학

*   **문제:** 어설픈 3D 하드웨어였습니다. 텍스처는 좌우로 찌그러지고(affine texture mapping issues), Z-버퍼가 없어 원근감에 따른 오브젝트 가림 처리가 힘들었고, 고정 소수점 연산은 정밀도 문제를 야기했습니다. 낮은 폴리곤 수 제한은 덤이었죠.
*   **해결책의 정수:**
    *   **적극적인 폴리곤 로드 조절 (LOD의 원조):** 멀리 있는 오브젝트는 최소한의 폴리곤으로, 가까이 있는 오브젝트는 상대적으로 많은 폴리곤으로 그렸습니다. 눈에 띄지 않는 곳은 아예 그리지 않았죠.
    *   **블러(Blur), 포그(Fog) 이펙트로 시각적 결함 감추기:** 텍스처 왜곡이나 지터링을 안개나 블러 효과로 자연스럽게 가렸습니다. 일종의 '시각적 최적화'였죠.
    *   **프리-렌더링된 배경 사용:** 3D 모델로 만들면 리소스가 터지는 배경은 아예 2D 이미지로 미리 렌더링해두고, 그 위에 3D 캐릭터를 올리는 방식으로 자원을 아꼈습니다.

### 유니티 최적화: 레트로의 지혜를 현대적으로 재해석하다

이런 레트로 콘솔 개발자들의 철학을 현대로 가져와 유니티 최적화에 적용하면 어떤 일이 벌어질까요? 저는 다음과 같은 '영감'을 얻었고, 이를 통해 실제 프로젝트에서 큰 성능 향상을 경험했습니다.

#### 1. 데이터 중심 설계 (Data-Oriented Design)의 강화

*   **레트로 교훈:** 타일과 스프라이트는 '데이터' 그 자체였습니다. PPU는 이 데이터를 효율적으로 읽고 그리는 데 최적화되어 있었죠.
*   **유니티 적용:** Unity DOTS (Data-Oriented Technology Stack)는 이 철학의 정점입니다. ECS (Entity Component System)는 데이터를 메모리에 효율적으로 배치하고, CPU 캐시 활용도를 높여줍니다. 기존 MonoBehaviour 중심의 객체 지향 설계가 가진 단점(메모리 파편화, 불필요한 데이터 포함)을 극복하죠.

    ```csharp
    // 레트로적 사고방식: 스프라이트 데이터를 메모리에 연속적으로 배치
    // NES Character Data Example (Conceptual)
    struct SpriteData {
        byte x_pos;
        byte y_pos;
        byte tile_index;
        byte attributes; // palette, priority, flip
    }
    // SpriteData[] sprites = new SpriteData[64]; // Limited, contiguous memory

    // Unity DOTS (ECS) Analogy
    public struct PositionComponent : IComponentData { public float2 Value; }
    public struct RotationComponent : IComponentData { public float Angle; }
    public struct SpeedComponent : IComponentData { public float Value; }

    // System이 특정 컴포넌트를 가진 엔티티만 순회하며 작업
    // 이는 PPU가 특정 스프라이트 데이터만 효율적으로 처리하던 방식과 유사
    public partial class MoveSystem : SystemBase
    {
        protected override void OnUpdate()
        {
            float deltaTime = SystemAPI.Time.DeltaTime;
            Entities.ForEach((ref PositionComponent pos, in SpeedComponent speed, in RotationComponent rot) =>
            {
                // 연속적인 데이터 처리
                pos.Value += new float2(math.cos(rot.Angle), math.sin(rot.Angle)) * speed.Value * deltaTime;
            }).ScheduleParallel();
        }
    }
    ```
    물론 DOTS 학습 곡선이 높지만, 극한의 성능이 필요할 때 '데이터를 어떻게 구조화할 것인가'에 대한 레트로의 답변은 여전히 유효합니다.

#### 2. 드로우 콜 최적화: '하나의 큰 덩어리' 전략

*   **레트로 교훈:** 스프라이트 시트, 타일셋은 드로우 콜 (Draw Call)의 원조입니다. 한 번에 많은 것을 그리기 위해 데이터를 묶는 거죠.
*   **유니티 적용:**
    *   **스프라이트 아틀라스 (Sprite Atlas):** 이건 말할 것도 없이 레트로의 핵심 개념을 그대로 가져온 것입니다. UI, 2D 게임에서 드로우 콜을 획기적으로 줄여줍니다.
    *   **메시 결합 (Mesh Combination):** PS1 개발자들이 적은 폴리곤을 효율적으로 관리했듯이, 비슷한 머티리얼을 사용하는 정적 오브젝트들은 하나의 메시로 합쳐버려 드로우 콜을 줄일 수 있습니다. 유니티의 Static Batching이 내부적으로 이런 역할을 하지만, 동적으로 생성되는 오브젝트나 특정 조건에서는 직접 메시를 결합하는 코드를 작성할 수도 있습니다.

    ```csharp
    // 문제: 너무 많은 작은 메쉬가 각각의 드로우 콜을 유발
    // 해결: 레트로적 사고방식 - 여러 기하학 데이터를 하나의 큰 덩어리로 묶기
    public static Mesh CombineMeshes(GameObject[] objectsToCombine)
    {
        List<CombineInstance> combineInstances = new List<CombineInstance>();
        List<Material> materials = new List<Material>();

        for (int i = 0; i < objectsToCombine.Length; i++)
        {
            MeshFilter meshFilter = objectsToCombine[i].GetComponent<MeshFilter>();
            MeshRenderer meshRenderer = objectsToCombine[i].GetComponent<MeshRenderer>();

            if (meshFilter == null || meshRenderer == null) continue;

            // 각 메쉬를 CombineInstance로 변환 (월드 좌표계로 변환)
            CombineInstance ci = new CombineInstance
            {
                mesh = meshFilter.sharedMesh,
                transform = meshFilter.transform.localToWorldMatrix
            };
            combineInstances.Add(ci);

            // 사용된 모든 머티리얼 수집 (실제로는 머티리얼 아틀라스가 필요)
            materials.AddRange(meshRenderer.sharedMaterials);

            // 원본 렌더러 비활성화
            meshRenderer.enabled = false;
        }

        Mesh newMesh = new Mesh();
        newMesh.indexFormat = UnityEngine.Rendering.IndexFormat.UInt32; // 큰 메쉬를 위해
        newMesh.CombineMeshes(combineInstances.ToArray(), true, true); // 서브메쉬 병합, 변환 매트릭스 사용
        newMesh.Optimize(); // 정점/인덱스 버퍼 최적화

        // 새로 생성된 메쉬를 사용할 GameObject에 할당
        // (머티리얼 처리 로직은 더 복잡해질 수 있음 - 머티리얼 아틀라스 필요)
        return newMesh;
    }
    ```
    이 코드는 여러 개의 작은 메시를 하나의 큰 메시로 합쳐서 드로우 콜을 하나로 줄이는 개념을 보여줍니다. PS1 개발자들이 한정된 폴리곤과 드로우 콜 안에서 최선을 다했듯이, 우리는 현대 엔진에서 유사한 방식으로 최적화를 추구하는 겁니다.

#### 3. 리소스 로드/언로드 및 오브젝트 풀링

*   **레트로 교훈:** NES는 VRAM이 부족해서 다음 화면에 필요한 타일/스프라이트 데이터만 그때그때 불러왔고, PS1도 메모리 한계로 인해 필요한 에셋만 로드하고 사용하지 않는 것은 바로 언로드했습니다.
*   **유니티 적용:**
    *   **어드레서블 에셋 시스템 (Addressables):** 필요한 에셋만 메모리에 로드하고, 사용이 끝나면 언로드하여 메모리 발자국(Memory Footprint)을 줄입니다.
    *   **오브젝트 풀링 (Object Pooling):** 총알, 파티클 이펙트처럼 자주 생성되고 파괴되는 오브젝트들은 미리 생성해두고 재활용합니다. 이는 GC (Garbage Collection) 발생을 줄여 프레임 드랍을 막아주는, 레트로 시대부터 이어져 온 거의 '교과서적인' 최적화 기법입니다.

#### 4. 시각적 '속임수'와 LOD (Level Of Detail)의 적극 활용

*   **레트로 교훈:** PS1의 안개 효과나 2D 배경 사용은 '제약'을 '특색'으로 바꾸는 영리한 시도였습니다.
*   **유니티 적용:**
    *   **LOD Group:** 유니티에 내장된 기능으로, 거리에 따라 모델의 디테일을 자동으로 조절합니다. PS1 개발자들이 수동으로 했던 일을 이제는 엔진이 자동으로 해줍니다.
    *   **Occlusion Culling:** 카메라에 보이지 않는 오브젝트는 아예 렌더링하지 않습니다. 레트로 게임에서 화면 밖의 스프라이트를 그리지 않던 것과 같은 맥락입니다.
    *   **Shader Optimization:** 복잡한 셰이더는 GPU에 큰 부담을 줍니다. PS1이 고정 소수점 연산으로 제한적이었던 것처럼, 현대 셰이더도 최대한 간소화하고, 텍스처 페치(Texture Fetch) 횟수를 줄이는 등 섬세한 관리가 필요합니다.

### 나의 견해: '왜'를 알아야 '어떻게'가 보인다

저는 가끔 후배 개발자들이 맹목적으로 최적화 기법을 적용하는 것을 봅니다. "ECS가 좋다니까 무조건 써야지!", "Draw Call이 나쁘다니까 무조건 줄여야지!" 하지만 '왜' 그것이 좋은지, '왜' 그 문제가 발생하는지 근본 원리를 이해하지 못하면, 특정 상황에서 오히려 독이 되거나 비효율적인 결과를 낳을 수 있습니다.

레트로 콘솔 아키텍처를 들여다보는 것은, 바로 이 '왜'를 이해하는 데 큰 도움을 줍니다. 제한된 자원 속에서 어떻게든 최고의 결과물을 뽑아내려던 개발자들의 고뇌와 해결책은, 현대의 우리가 마주하는 수많은 최적화 문제의 원형을 보여줍니다. 비록 하드웨어 스펙은 천지 차이지만, '자원을 효율적으로 사용한다'는 핵심 원칙은 변하지 않습니다.

이런 옛 방식의 접근을 통해 얻은 통찰력은 단순히 퍼포먼스 수치를 올리는 것을 넘어, 개발자로서 '문제를 다루는 능력' 자체를 향상시켜 줍니다. 마치 고전 문학이 현대인의 삶에도 깊은 울림을 주듯이, 과거의 기술적 제약은 오늘날의 우리에게도 '본질'을 꿰뚫어 볼 수 있는 지혜를 제공하는 셈이죠.

### 결론: 제약 속에서 피어난 창의성, 그리고 성능의 미래

현대 유니티 개발 환경은 레트로 콘솔 시절에 비하면 무한에 가까운 자유를 제공합니다. 하지만 이 자유 속에서도 우리는 여전히 '최적화'라는 영원한 숙제와 마주하고 있습니다.

이번 포스팅을 통해, 여러분이 NES, SNES, PS1 같은 레트로 콘솔 아키텍처에서 얻을 수 있는 깊이 있고 실무적인 영감들을 조금이나마 느끼셨기를 바랍니다. 과거의 개발자들이 제약 속에서 피워낸 창의성과 효율의 미학은, 오늘날 우리 유니티 개발자들에게도 여전히 유효한, 아니 어쩌면 더욱 중요해진 성능 최적화의 열쇠가 될 수 있습니다.
