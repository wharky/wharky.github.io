여기서 바로 우리의 해결사, **DOTS**가 등장합니다.

**[솔루션 1] DOTS: CPU를 위한 데이터 혁명**

DOTS는 '데이터 지향'이라는 철학을 기반으로 CPU 캐시 효율성을 극대화하고, 멀티코어 환경에서 병렬 처리를 손쉽게 구현할 수 있도록 설계된 강력한 도구 모음입니다. Entity Component System (ECS), C# Job System, Burst Compiler 이 세 가지 핵심 축이 맞물려 돌아가며, CPU 부하를 획기적으로 줄여줍니다.

ECS는 게임 오브젝트의 데이터를 '엔티티', '컴포넌트', '시스템'으로 분리하여 관리합니다. 특정 로직(시스템)이 필요한 데이터(컴포넌트)만 묶어서 처리하기 때문에 데이터 접근 패턴이 예측 가능해지고, CPU 캐시 히트율이 비약적으로 상승합니다. 여기에 C# Job System은 이 시스템 로직을 멀티스레드 환경에서 안전하고 효율적으로 실행할 수 있도록 돕고, Burst Compiler는 C# 코드를 네이티브 수준의 최적화된 기계어로 변환하여 극한의 성능을 끌어냅니다.

자, 이제 이 벌레 떼 10만 마리의 움직임을 DOTS로 처리해봅시다.

**[DOTS 핵심 로직 (Pseudo-code)]**

```csharp
// 1. 벌레의 상태를 정의하는 컴포넌트들 (IComponentData)
//    데이터는 최대한 작고, 인접하게 배치되도록 설계하는 것이 핵심!
public struct InsectMovementData : IComponentData {
    public float Speed;
    public float CurrentAngle;
    public float TurnSpeed;
    public float PhaseOffset; // 개체별 움직임 위상 차이
}

// 2. 벌레의 움직임을 처리하는 시스템 (ISystem or IJobEntity)
//    Batch 처리로 캐시 효율 극대화, Burst 컴파일러로 Native 속도
[BurstCompile]
public partial struct InsectMovementSystem : ISystem {
    public void OnUpdate(ref SystemState state) {
        float deltaTime = SystemAPI.Time.DeltaTime;
        float elapsedTime = SystemAPI.Time.ElapsedTime; // 전역 시간

        // IJobEntity를 활용하여 수십만 개의 벌레를 병렬 처리
        new MovementJob {
            DeltaTime = deltaTime,
            ElapsedTime = elapsedTime
        }.ScheduleParallel(); // 멀티스레드 병렬 실행
    }
}

[BurstCompile]
public partial struct MovementJob : IJobEntity {
    public float DeltaTime;
    public float ElapsedTime;

    // Execute 메서드는 각 엔티티에 대해 호출 (데이터만 전달!)
    void Execute(ref LocalTransform transform, ref InsectMovementData movementData) {
        // 현재 이동 방향을 기준으로 앞으로 전진
        float3 currentForward = transform.Forward();
        transform.Position += currentForward * movementData.Speed * DeltaTime;

        // 개체별 고유한 '위상 오프셋'을 이용한 procedural 회전
        // DOTS에서 복잡한 수치 계산은 Burst Compiler의 마법으로 매우 빠르게!
        movementData.CurrentAngle += movementData.TurnSpeed * DeltaTime *
                                     math.sin(elapsedTime * 0.5f + movementData.PhaseOffset);
        transform.Rotation = quaternion.Euler(0, movementData.CurrentAngle, 0);

        // 월드 경계를 벗어나면 다시 안으로 들어오도록 (간단한 예시)
        if (transform.Position.x > 100 || transform.Position.x < -100 ||
            transform.Position.z > 100 || transform.Position.z < -100) {
            transform.Position = float3.zero;
        }
    }
}
```
이제 10만 마리 벌레의 위치와 회전값, 심지어 개체별로 다른 미세한 움직임까지도 CPU에서 놀라울 정도로 빠르게 계산해낼 수 있습니다. 하지만 이 데이터들을 '어떻게 화면에 그릴 것인가'라는 두 번째, 더 큰 질문이 남습니다. 여기서 **Shader Math & GLSL**이 빛을 발할 차례입니다.

**[솔루션 2] Shader Math & GLSL: GPU를 위한 비주얼 연금술**

DOTS가 수많은 개체의 '상태'를 효율적으로 관리하고 계산했다면, 이제 그 결과물을 '아름답고 빠르고 대규모로' 시각화하는 것은 GPU의 몫입니다. 그리고 이 GPU의 언어가 바로 셰이더(Shader)이며, 그 핵심은 벡터, 행렬, 삼각 함수 등으로 이루어진 '셰이더 수학'입니다.

수십만 마리의 벌레를 하나하나 `MeshRenderer` 컴포넌트로 그리는 것은 드로우콜 오버헤드 때문에 불가능합니다. 여기서 우리는 GPU 인스턴싱(GPU Instancing)과 Compute Shader를 활용하여 DOTS가 준비한 데이터를 GPU에 효율적으로 넘기고, 셰이더 내에서 그리는 작업을 전담시킵니다.

**[DOTS와 Shader Math의 결합 아키텍처 (개념 설명)]**

1.  **DOTS 시스템**: 모든 벌레의 `LocalTransform` (위치, 회전, 스케일) 및 `MovementData`를 계산하고 업데이트합니다.
2.  **렌더링 데이터 준비 시스템 (DOTS)**: 특정 DOTS 시스템에서 수십만 개의 벌레 중 화면에 보여질 벌레들의 최종 `LocalTransform` 데이터를 추출하여, `ComputeBuffer` 또는 `StructuredBuffer`와 같은 GPU 버퍼로 복사합니다. 이 과정에서 DOTS가 제공하는 `NativeArray` 등의 고성능 컬렉션과 `memcpy` 같은 블록 복사를 활용하면 매우 빠릅니다.
3.  **GPU 인스턴싱 렌더링**: CPU에서는 단 한 번의 드로우콜(`Graphics.DrawMeshInstancedIndirect` 등)로, 준비된 `ComputeBuffer`를 참조하여 수십만 개의 벌레 메시를 GPU가 스스로 그려내도록 지시합니다.
4.  **Vertex Shader**: 각 벌레 인스턴스에 대해 실행됩니다. GPU 버퍼에서 해당 인스턴스의 `LocalTransform` 데이터를 읽어와 메시의 정점들을 월드 공간으로 변환합니다. 여기서 셰이더 수학(행렬 곱셈, 벡터 연산)이 핵심적으로 사용됩니다.
    *   **심화:** 단순히 위치만 읽어오는 것이 아니라, 벌레의 고유 `PhaseOffset` 값과 `_Time` 값을 이용해 셰이더 내에서 벌레의 날개짓 애니메이션이나 몸통의 미세한 흔들림을 프로시저럴하게 계산할 수 있습니다. `sin(_Time * Frequency + PhaseOffset)` 같은 함수가 이런 용도로 완벽하죠.
5.  **Fragment Shader**: 변환된 정점들을 기반으로 각 픽셀의 색상을 계산합니다. 빛의 방향, 노말 벡터, 재질 속성 등을 셰이더 수학으로 계산하여 사실적인 그림자와 하이라이트를 만들어냅니다. 여기서도 `dot` 연산, `normalize` 함수 등 벡터 수학이 필수적입니다.

**[Shader Math & GLSL 핵심 로직 (Pseudo-code)]**

```glsl
#version 450 core // GLSL 버전 명시 (Unity URP/HDRP의 HLSL도 유사)

// Vertex Shader: 각 정점(aPos)을 인스턴스 데이터에 맞춰 월드 공간으로 변환
layout(location = 0) in vec3 aPos;           // 메시의 로컬 정점 위치
layout(location = 1) in vec3 aNormal;         // 메시의 로컬 정점 노말
layout(location = 2) in vec2 aTexCoords;      // 메시의 UV 좌표

// ComputeBuffer에서 넘어온 인스턴스별 데이터 (DOTS가 채워줌)
// Unity의 경우 StructuredBuffer<T> 형태로 바인딩
struct InstanceData {
    mat4 modelMatrix; // 각 벌레의 월드 변환 행렬 (Position, Rotation, Scale)
    float perInstanceAnimationOffset; // DOTS에서 계산했거나, 셰이더 내에서 사용될 고유 오프셋
};
layout(std430, binding = 0) readonly buffer InstanceBuffer {
    InstanceData instances[];
} instanceBuffer;

uniform mat4 u_ViewProjectionMatrix; // 카메라의 View * Projection 행렬
uniform float u_GlobalTime; // 전역 시간 (CPU에서 넘겨줌)

out vec3 v_Normal;
out vec2 v_TexCoords;
out float v_InstanceAnimValue; // 프래그먼트 셰이더로 전달될 애니메이션 값

void main() {
    // gl_InstanceID: 현재 처리 중인 인스턴스의 고유 ID (GPU 인스턴싱에서 제공)
    InstanceData currentInstance = instanceBuffer.instances[gl_InstanceID];

    // 모델 행렬을 이용한 정점 변환: 로컬 -> 월드 공간
    mat4 model = currentInstance.modelMatrix;

    // 셰이더 내에서 프로시저럴 애니메이션 추가 (예: 벌레 몸통의 미세한 진동)
    // 수학 함수(sin)와 고유 오프셋을 활용하여 각 개체마다 다른 애니메이션 부여
    float wiggleAmplitude = 0.1;
    float wiggleFrequency = 5.0;
    float wiggle = sin(u_GlobalTime * wiggleFrequency + currentInstance.perInstanceAnimationOffset) * wiggleAmplitude;
    model[1][3] += wiggle; // Y축(위아래)으로 미세하게 움직임

    gl_Position = u_ViewProjectionMatrix * model * vec4(aPos, 1.0);

    // 노말 벡터 변환 (조명 계산용)
    v_Normal = mat3(transpose(inverse(model))) * aNormal;
    v_TexCoords = aTexCoords;
    v_InstanceAnimValue = wiggle; // 프래그먼트 셰이더에서도 활용 가능
}

// Fragment Shader: 각 픽셀의 색상 계산
uniform sampler2D u_MainTexture; // 벌레의 텍스처
uniform vec3 u_LightDirection;   // 광원 방향
uniform vec3 u_AmbientColor;     // 주변광 색상

in vec3 v_Normal;
in vec2 v_TexCoords;
in float v_InstanceAnimValue;

layout(location = 0) out vec4 FragColor;

void main() {
    // 텍스처 색상 샘플링
    vec4 baseColor = texture(u_MainTexture, v_TexCoords);

    // 조명 계산 (Lambertian diffuse model)
    vec3 normal = normalize(v_Normal);
    float diffuseFactor = max(dot(normal, -normalize(u_LightDirection)), 0.0);
    vec3 lighting = u_AmbientColor + diffuseFactor * vec3(1.0, 1.0, 1.0); // 백색 확산광

    // 최종 색상 = (조명 * 기본 색상) + (애니메이션 값에 따른 미세한 색상 변화)
    FragColor = baseColor * vec4(lighting, 1.0);
    // 예시: wiggle 값에 따라 미세하게 색상 변경
    FragColor.rgb += vec3(abs(v_InstanceAnimValue) * 0.5);
}
```

이 코드를 보면, DOTS가 넘겨준 `modelMatrix`와 `perInstanceAnimationOffset` 같은 간결한 데이터를 셰이더가 받아서, 복잡한 셰이더 수학 연산으로 10만 마리 벌레 각각의 위치 변환, 조명 계산, 심지어는 미세한 몸통 흔들림 애니메이션까지 처리하는 것을 알 수 있습니다. CPU는 데이터 계산에 집중하고, GPU는 그 데이터로 시각화에 집중하는 완벽한 분업이죠. 이로써 CPU 드로우콜은 하나로 줄고, GPU는 수십만 개의 정점을 효율적으로 병렬 처리하여 압도적인 성능을 달성합니다.

**[나의 주관적인 견해: DOTS는 뇌를, 셰이더 수학은 눈을 깨우다]**

제가 DOTS와 셰이더 수학을 깊게 파고들면서 느낀 것은, 이 둘은 마치 게임의 '뇌'와 '눈' 같다는 겁니다. DOTS는 게임 월드의 복잡한 논리와 수많은 개체의 상호작용을 차갑고 정확하게 계산하는 '뇌'의 역할을 합니다. 데이터의 흐름을 최적화하고, 불필요한 연산을 제거하며, 멀티스레드를 마치 하나의 거대한 논리 회로처럼 작동시키죠.

반면 셰이더 수학은 이 계산된 데이터들을 바탕으로 플레이어에게 보이는 '눈'의 역할을 담당합니다. 복잡한 벡터 연산과 행렬 변환으로 3D 공간을 구축하고, 빛과 색을 조합하여 생생한 시각적 경험을 창조합니다. 그 과정에서 `sin()`, `cos()`, `dot()`, `cross()` 같은 기본적인 수학 함수들이 마치 화가의 물감처럼, 게임 세상에 생명을 불어넣는 도구가 됩니다.

가장 중요한 건, 이 둘이 '데이터'라는 공통 분모를 통해 밀접하게 연결된다는 점입니다. DOTS는 렌더링에 필요한 최소한의 데이터를 효율적인 형태로 GPU에 전달하고, 셰이더는 이 데이터를 받아서 비주얼을 만들어냅니다. 이 '데이터 파이프라인'을 얼마나 매끄럽고 효율적으로 설계하느냐가 대규모 고성능 게임 개발의 성패를 가릅니다.
